Combat Extensions

Extension du systeme de combat a 3 tiers : arcs, fleches speciales, boucliers, set bonus armure, dagues, marteaux de guerre et pommes custom. Contexte PvP 1.8 — aucun cooldown d'attaque.

A faire Valterite — Tier 2 Aetherite — Tier 3 Erinium — Tier 4 19 nouveaux items 4 entites fleches

01 Contexte & Objectifs

Extension du systeme de combat

Le systeme de base Combat & Equipement definit trois tiers d'equipement (Valterite, Aetherite, Erinium) avec epees, outils et armures standards. Ce cahier des charges etend ce systeme avec de nouveaux types d'armes et d'objets consommables, en conservant la meme hierarchie de progression par tier.

L'objectif est d'enrichir le gameplay PvP en proposant des archetypes de combat varies — l'arc pour le combat a distance, la dague pour les joueurs furtifs, le marteau pour les batailles de groupe, le bouclier pour les tanks — sans pour autant creer une surpuissance qui rendrait certains builds obsoletes.

Contexte PvP 1.8

⚠
Regle absolue PvP 1.8 — Le serveur utilise un mod supprimant le cooldown d'attaque vanilla de Minecraft 1.9+. Cela signifie que les joueurs peuvent attaquer aussi vite que leur souris le permet (jitter click). En consequence : ne JAMAIS coder de bonus ou malus bases sur ATTACK_SPEED. Toutes les mecaniques de differentiation entre armes sont basees sur les degats, la portee, les effets speciaux et les zones d'effet.
  • Arcs — Differencies par les degats bruts, la vitesse de tirage et la velocite des fleches.
  • Dagues — Moins de degats bruts, mais bonus de backstab important. Reclame du positionnement.
  • Marteaux — Plus de degats et AoE frontale. Favorise les combats en groupe/donjon.
  • Boucliers — Reduction de degats via blocage actif. Tier Erinium repousse l'attaquant.
  • Set Bonus — Recompense passive pour les joueurs ayant un set complet d'un tier.
  • Pommes Custom — Consommables PvP couteux, alternatif a la Pomme d'Or Enchantee vanilla.

Conventions de developpement

  • Tous les items sont crees via EriItem.create() (EriAPI Content Builder). Pas de boilerplate Forge brut.
  • Tous les textes sont dans les fichiers de traduction fr_FR.lang et en_US.lang via les cles item.eriniumfaction.<id>.name.
  • Les event handlers utilisent EriEvents.on() ou @Mod.EventBusSubscriber.
  • Les entites fleches sont enregistrees via EntityRegistry.registerModEntity() dans le CommonProxy.
  • Java 8 uniquement — pas de var, pas de features Java 9+.

03 Arcs Custom

Trois arcs progressifs etendant la classe vanilla ItemBow. La progression se fait sur les degats, la velocite des fleches et la vitesse de tirage. L'arc d'Erinium est clairement superieur en toutes circonstances et represente un investissement de mid-game.

Propriete Arc de Valterite Arc d'Aetherite Arc d'Erinium
ID valterite_bow aetherite_bow erinium_bow
Degats vs arc vanilla +20% +35% +50%
Vitesse de tirage Identique au vanilla 20% plus rapide 30% plus rapide
Velocite de fleche +10% +20% +30%
Durabilite 500 750 1 200
Rarete UNCOMMON RARE EPIC + glow
Textures tirage 3 stades + idle 3 stades + idle 3 stades + idle

Implementation technique

  • Classe de base — ItemTieredBow extends ItemBow. Le constructeur prend un multiplicateur de degats, un multiplicateur de velocite et un facteur de duree de tirage.
  • Duree de tirage — Override de getMaxItemUseDuration(ItemStack). Retourner (int)(72000 / drawSpeedFactor) ou plus simplement reduire la duree en ticks : vanilla = 72 000, Aetherite = 60 000, Erinium = 50 400.
  • Degats et velocite — Override de onPlayerStoppedUsing(ItemStack, World, EntityLivingBase, int). Recalculer la charge en appliquant les multiplicateurs avant d'appeler le code de lancer de fleche.
  • Rendu de tirage — Enregistrer les textures via ModelBakery.registerItemVariants() et implémenter IItemPropertyGetter pour la propriete pull. Le modele JSON reference les textures aux valeurs 0.0, 0.65 et 0.9.
// ItemTieredBow — constructeur public ItemTieredBow(String tierName, float damageMultiplier, float velocityMultiplier, float drawSpeedFactor, int durability) { this.tierName = tierName; this.damageMultiplier = damageMultiplier; this.velocityMultiplier = velocityMultiplier; this.drawSpeedFactor = drawSpeedFactor; setMaxDamage(durability); } @Override public int getMaxItemUseDuration(ItemStack stack) { // Plus drawSpeedFactor est eleve, plus le tirage est rapide return (int) (72000 / drawSpeedFactor); } @Override public void onPlayerStoppedUsing(ItemStack stack, World world, EntityLivingBase entity, int timeLeft) { if (!world.isRemote && entity instanceof EntityPlayer) { EntityPlayer player = (EntityPlayer) entity; int charge = this.getMaxItemUseDuration(stack) - timeLeft; float chargeRatio = BowCharge.getArrowVelocity(charge); if (chargeRatio >= 0.1f) { EntityArrow arrow = new EntityTippedArrow(world, player); // Appliquer les multiplicateurs du tier arrow.shoot(player, player.rotationPitch, player.rotationYaw, 0.0f, chargeRatio * 3.0f * velocityMultiplier, 1.0f); arrow.setDamage(arrow.getDamage() * damageMultiplier); world.spawnEntity(arrow); } } }
⚠
Compatibilite enchantements Power — L'enchantement Power de l'arc vanilla multiplie les degats de base de l'arc. Puisque nos arcs custom overrident le calcul de degats, s'assurer que l'override prend en compte l'enchantement Power en lisant EnchantmentHelper.getEnchantmentLevel(Enchantments.POWER, stack) et en applicant le bonus additionnel apres le multiplicateur de tier.

Modeles JSON et textures de tirage

Chaque arc custom necessite 4 textures : idle (au repos) et 3 stades de tirage (pull0, pull1, pull2). La propriete pull dans le modele JSON reference ces textures.

// models/item/valterite_bow.json (extrait) { "parent": "item/bow", "textures": { "layer0": "eriniumfaction:items/valterite_bow" }, "overrides": [ { "predicate": { "pulling": 1, "pull": 0.00 }, "model": "eriniumfaction:item/valterite_bow_pulling_0" }, { "predicate": { "pulling": 1, "pull": 0.65 }, "model": "eriniumfaction:item/valterite_bow_pulling_1" }, { "predicate": { "pulling": 1, "pull": 0.90 }, "model": "eriniumfaction:item/valterite_bow_pulling_2" } ] }

04 Fleches Speciales

Quatre types de fleches custom offrant des effets speciaux a l'impact. Chaque type necessite une classe d'entite distincte heritant de EntityArrow.

Type ID Item Entite Effet a l'impact Stack
Fleche Valterite valterite_arrow EntityValteriteArrow Lenteur I pendant 2 secondes sur la cible 16
Fleche Aetherite aetherite_arrow EntityAetheriteArrow Ignore 30% de l'armure de la cible 16
Fleche Erinium erinium_arrow EntityEriniumArrow Degats doubles, non recuperable (consommee meme si ratee) 16
Fleche Explosive explosive_arrow EntityExplosiveArrow Petite explosion a l'impact (3 coeurs AoE en 3 blocs, pas de destruction de blocs) 8

Implementation : EntityValteriteArrow

  • Override de onHit(RayTraceResult) — si le hit est une entite, appliquer PotionEffect(MobEffects.SLOWNESS, 40, 0) (40 ticks = 2 secondes, amplificateur 0 = niveau I).
  • Appeler super.onHit(result) pour le comportement standard (degats physiques).

Implementation : EntityAetheriteArrow

  • La fleche Aetherite n'a pas d'effet de potion. Elle modifie le calcul des degats via un flag interne.
  • Override de attackEntityFrom(DamageSource, float) sur la cible — intercepter dans LivingHurtEvent : si la source de degats est une fleche Aetherite, reduire temporairement l'absorption d'armure de 30%.
  • Implementation recommandee : dans onHit(RayTraceResult), calculer les degats normaux, puis ajouter les degats bonus baseArmorValue * 0.3f comme degat magique non bloquable par l'armure (DamageSource.causeIndirectMagicDamage()).

Implementation : EntityEriniumArrow

  • Override de getDamage() pour retourner super.getDamage() * 2.0.
  • La fleche est consommee inconditionnellement — override de onHit() pour appeler setDead() et ne jamais dropper la fleche, qu'elle touche ou non.
  • Ne pas appeler la logique de recuperation vanilla dans onHitEntity().
// EntityExplosiveArrow — explosion a l'impact sans destruction de blocs @Override protected void onImpact(RayTraceResult result) { if (!world.isRemote) { Entity shooter = getShooter(); // Pas de destruction de blocs : mobGriefing = false, blockBreaking = false world.createExplosion(shooter, posX, posY, posZ, 1.5f, // force (1.5 = ~3 coeurs AoE) false); // causesFire = false // Overrider GameRules pour bloquer la destruction de blocs // ou utiliser World.newExplosion() avec Explosion.Mode.NONE setDead(); } }

Enregistrement des entites fleches

Les quatre entites sont enregistrees dans CommonProxy.preInit() via EntityRegistry.registerModEntity(). Les items fleches overrident ItemArrow.createArrow() pour retourner l'entite appropriee.

// Dans CommonProxy.preInit() EntityRegistry.registerModEntity( new ResourceLocation(EriniumFaction.MODID, "valterite_arrow"), EntityValteriteArrow.class, "valterite_arrow", 60, EriniumFaction.INSTANCE, 64, 1, true ); // Repeter pour les 3 autres entites (IDs 61, 62, 63)

Comportement distributeur

Enregistrer le comportement dispenseur pour chaque item fleche via BlockDispenser.DISPENSE_BEHAVIOR_REGISTRY.putObject(item, behavior) dans CommonProxy.init(). Le comportement cree et lance l'entite fleche custom depuis le distributeur.

ℹ
Detection des fleches speciales par l'arc — Override de ItemBow.findAmmo(EntityPlayer) ou utiliser ArrowLooseEvent pour verifier si le joueur possede des fleches speciales dans son inventaire. Priorite : Fleche Erinium > Fleche Aetherite > Fleche Valterite > Fleche vanilla.

05 Boucliers

Trois boucliers par tier avec des pourcentages de blocage differents. Equippables dans l'offhand (main gauche). La mecanique de blocage est basee sur le maintien du clic droit.

ID valterite_shield aetherite_shield erinium_shield
Blocage des degats 60% 75% 90%
Knockback sur blocage Aucun Aucun Repousse l'attaquant de 2 blocs
Durabilite 400 600 1 000
Slot Offhand Offhand Offhand
Rarete UNCOMMON RARE EPIC + glow

Implementation technique

  • Classe de base — ItemTieredShield extends ItemShield. Le constructeur prend un pourcentage de blocage et un flag knockback.
  • Detection du blocage — Dans ShieldBlockHandler (listener LivingHurtEvent) : verifier si la cible est un EntityPlayer, si son item offhand est un ItemTieredShield, et si player.isActiveItemStackBlocking() retourne true.
  • Reduction des degats — Modifier event.setAmount(event.getAmount() * (1.0f - blockPercentage)). Consommer de la durabilite en consequence.
  • Knockback Erinium — Pour le bouclier Erinium, apres la reduction de degats, appliquer un vecteur de velocite a l'attaquant en direction opposee a la cible : attacker.addVelocity(dx * 0.5, 0.1, dz * 0.5).
// ShieldBlockHandler — extrait @SubscribeEvent public void onLivingHurt(LivingHurtEvent event) { if (!(event.getEntityLiving() instanceof EntityPlayer)) return; EntityPlayer target = (EntityPlayer) event.getEntityLiving(); ItemStack offhand = target.getHeldItemOffhand(); if (offhand.isEmpty() || !(offhand.getItem() instanceof ItemTieredShield)) return; if (!target.isActiveItemStackBlocking()) return; ItemTieredShield shield = (ItemTieredShield) offhand.getItem(); float reduced = event.getAmount() * (1.0f - shield.getBlockPercentage()); event.setAmount(reduced); offhand.damageItem(1, target); if (shield.hasKnockback() && event.getSource().getImmediateSource() instanceof EntityLivingBase) { EntityLivingBase attacker = (EntityLivingBase) event.getSource().getImmediateSource(); Vec3d dir = attacker.getPositionVector().subtract(target.getPositionVector()).normalize(); attacker.addVelocity(dir.x * 0.6, 0.1, dir.z * 0.6); attacker.velocityChanged = true; } }
⚠
PvP 1.8 — Pas de timing — Le blocage du bouclier vanilla 1.9+ inclut une mecanique de timing (le bouclier doit etre "charge" avant de bloquer). En PvP 1.8, cette mecanique est inexistante. Notre implementation bloque instantanement des que isActiveItemStackBlocking() est vrai (clic droit maintenu).

06 Set Bonus Armure

Bonus passifs accords aux joueurs portant les 4 pieces d'armure du meme tier. Aucun nouvel item n'est cree. Le SetBonusHandler detecte la composition de l'armure et applique/supprime les bonus dynamiquement.

Tier Bonus de set complet Implementation Visuel
Valterite Regeneration passive hors-combat : 0.5 coeur toutes les 5 secondes. "Hors-combat" = aucun degat recu ou inflige dans les 5 dernieres secondes. player.heal(1.0f) toutes les 100 ticks si timer hors-combat >= 100 Particules bleues occasionnelles
Aetherite +10% de vitesse de deplacement (AttributeModifier persistant) SharedMonsterAttributes.MOVEMENT_SPEED modifier UUID fixe, applique/retire selon le set Particules violettes occasionnelles
Erinium -15% de tous les degats recus (reduction plate avant le calcul d'armure) Dans LivingHurtEvent : si set Erinium complet, event.setAmount(amount * 0.85f) Particules or/violet occasionnelles

SetBonusHandler — Logique de detection

  • Verifier les 4 slots d'armure (EntityEquipmentSlot.HEAD, CHEST, LEGS, FEET) a chaque PlayerTickEvent (Phase END, toutes les 20 ticks pour la performance).
  • Pour chaque slot, recuperer le registry name de l'item et le comparer aux prefixes "eriniumfaction:valterite_", "eriniumfaction:aetherite_", "eriniumfaction:erinium_".
  • Si les 4 pieces correspondent au meme tier, activer le bonus correspondant. Sinon, desactiver tous les bonus actifs.
  • Stocker le set actif par UUID : Map<UUID, String> activeSetBonus dans SetBonusHandler.
// SetBonusHandler — detection du set complet private String detectActiveSet(EntityPlayer player) { String[] tiers = { "valterite", "aetherite", "erinium" }; for (String tier : tiers) { boolean full = true; for (EntityEquipmentSlot slot : new EntityEquipmentSlot[]{ HEAD, CHEST, LEGS, FEET}) { ItemStack piece = player.getItemStackFromSlot(slot); if (piece.isEmpty()) { full = false; break; } String name = piece.getItem().getRegistryName().toString(); if (!name.startsWith("eriniumfaction:" + tier + "_")) { full = false; break; } } if (full) return tier; } return null; }

Bonus Aetherite — Gestion de l'AttributeModifier

L'UUID du modifier doit etre fixe et constant pour pouvoir le retirer proprement. Utiliser un UUID derive du modid.

private static final UUID SPEED_MODIFIER_UUID = UUID.fromString("a7b3c4d5-e6f7-8901-abcd-ef0123456789"); private static final String SPEED_MODIFIER_NAME = "EriniumFaction.AetheriteSetSpeed"; private void applyAetheriteSpeed(EntityPlayer player) { IAttributeInstance attr = player.getEntityAttribute( SharedMonsterAttributes.MOVEMENT_SPEED); if (attr.getModifier(SPEED_MODIFIER_UUID) == null) { attr.applyModifier(new AttributeModifier( SPEED_MODIFIER_UUID, SPEED_MODIFIER_NAME, 0.10, 1)); // Operation 1 = multiplicatif (baseValue * (1 + 0.10)) } } private void removeAetheriteSpeed(EntityPlayer player) { IAttributeInstance attr = player.getEntityAttribute( SharedMonsterAttributes.MOVEMENT_SPEED); AttributeModifier mod = attr.getModifier(SPEED_MODIFIER_UUID); if (mod != null) attr.removeModifier(mod); }

Particules client-side

Les particules decoratives sont spawned cote client uniquement dans PlayerTickEvent (Phase END, world.isRemote == true). Une chance de 5% par tick est suffisante pour un effet visual occasionnel.

  • Valterite — EnumParticleTypes.WATER_SPLASH avec teinte bleue.
  • Aetherite — EnumParticleTypes.PORTAL autour du joueur.
  • Erinium — EnumParticleTypes.ENCHANTMENT_TABLE avec teinte doree.

Detection du backstab — Mathematiques vectorielles

Le backstab est detecte en comparant le vecteur de regard de la cible avec le vecteur allant de l'attaquant vers la cible. Si l'attaquant se trouve dans le dos de la cible (arc de 120 degres derriere), le backstab s'applique.

// BackstabHandler — detection de la position dans le dos private boolean isBackstab(EntityLivingBase attacker, EntityLivingBase target) { // Vecteur du regard de la cible (direction ou elle regarde) Vec3d targetForward = target.getLookVec(); // Vecteur de la cible vers l'attaquant Vec3d targetToAttacker = attacker.getPositionVector() .subtract(target.getPositionVector()) .normalize(); // Produit scalaire : si negatif, l'attaquant est derriere la cible // dotProduct < -0.5 signifie angle > 120 degres (cos(120°) = -0.5) double dot = targetForward.dotProduct(targetToAttacker); return dot < -0.5; } @SubscribeEvent public void onLivingHurt(LivingHurtEvent event) { Entity src = event.getSource().getImmediateSource(); if (!(src instanceof EntityPlayer)) return; EntityPlayer attacker = (EntityPlayer) src; ItemStack held = attacker.getHeldItemMainhand(); if (!(held.getItem() instanceof ItemDagger)) return; if (isBackstab(attacker, event.getEntityLiving())) { event.setAmount(event.getAmount() * 1.5f); // Spawn de particules rouges cote client via packet spawnBackstabParticles(event.getEntityLiving()); } }

Creation via EriAPI

Les dagues sont creees comme des epees avec des degats inferieurs a ceux des epees de meme tier. La logique de backstab est entierement dans l'event handler, transparente pour EriItem.

// Dans EriniumItems.register() EriItem.create(EriniumFaction.MODID, "valterite_dagger") .rarity(EnumRarity.UNCOMMON) .creativeTab(EriniumCreativeTab.TAB) .tool() .type(ItemDefinition.ToolType.SWORD) .damage(2.0f) // 2.0 + 4 (base sword) = 6, mais en PvP 1.8 le base est different .attackSpeed(-2.4f) // ATTACK_SPEED ignoree en PvP 1.8, valeur formelle .durability(300) .enchantability(12) .harvestLevel(3) .done() .register();
✓
Visuel du backstab — Lors d'un backstab, spawner des particules CRIT_MAGIC sur la cible et jouer le son SoundEvents.ENTITY_PLAYER_ATTACK_CRIT au client de l'attaquant via un packet custom.

08 Marteaux de Guerre

Les marteaux font plus de degats que les epees et infligent une partie de ces degats aux ennemis adjacents (cone frontal). En contrepartie, chaque frappe impose un cooldown avant de pouvoir frapper a nouveau (2s / 2.5s / 3s selon le tier). C'est une arme de timing, pas de spam. Particulierement efficace dans les combats de groupe (sieges, events) et contre les mobs groupes.

Detection du cone frontal AoE

Le cone est calcule depuis la position de l'attaquant dans la direction de son regard. Seules les entites dans un angle de 90 degres de part et d'autre du vecteur de regard sont touchees.

// HammerAoeHandler — frappe AoE en cone frontal @SubscribeEvent public void onLivingHurt(LivingHurtEvent event) { Entity src = event.getSource().getImmediateSource(); if (!(src instanceof EntityPlayer)) return; EntityPlayer attacker = (EntityPlayer) src; ItemStack held = attacker.getHeldItemMainhand(); if (!(held.getItem() instanceof ItemWarHammer)) return; ItemWarHammer hammer = (ItemWarHammer) held.getItem(); float radius = hammer.getAoeRadius(); float aoeDmg = event.getAmount() * 0.60f; EntityLivingBase primaryTarget = event.getEntityLiving(); // AABB elargi autour de l'attaquant AxisAlignedBB bb = attacker.getEntityBoundingBox().grow(radius); List<EntityLivingBase> nearby = attacker.world.getEntitiesWithinAABB( EntityLivingBase.class, bb); Vec3d attackerLook = attacker.getLookVec(); for (EntityLivingBase entity : nearby) { // Exclure la cible principale et l'attaquant lui-meme if (entity == primaryTarget || entity == attacker) continue; // Exclure les allies (meme faction) if (isSameFaction(attacker, entity)) continue; // Verifier que l'entite est dans le cone frontal (angle < 90 degres) Vec3d toEntity = entity.getPositionVector() .subtract(attacker.getPositionVector()).normalize(); double dot = attackerLook.dotProduct(toEntity); if (dot < 0.0) continue; // dot < 0 = angle > 90 deg = derriere entity.attackEntityFrom( DamageSource.causePlayerDamage(attacker), aoeDmg); } // Particules de balayage (vanilla sweep) attacker.world.spawnParticle(EnumParticleTypes.SWEEP_ATTACK, primaryTarget.posX, primaryTarget.posY + primaryTarget.height / 2, primaryTarget.posZ, 0, 0, 0); }
ℹ
Protection contre les chaines de degats — Les degats AoE appliques via attackEntityFrom() declencheront a nouveau LivingHurtEvent. Pour eviter une boucle infinie, ajouter un flag NBT temporaire sur l'entite attaquee (ou un Set<Integer> d'entityId en cours de traitement) et le retirer apres la frappe.

Implementation technique

Les pommes custom sont creees comme des items de nourriture avec EriItem.create().food(). Pour appliquer plusieurs effets et gerer le cooldown, override de onItemUseFinish() est preferable a l'API .effect() chainee.

// ItemTieredApple — onItemUseFinish avec cooldown partage private static final UUID COOLDOWN_APPLE = UUID.randomUUID(); // UUID fixe @Override public ItemStack onItemUseFinish(ItemStack stack, World world, EntityLivingBase entity) { if (!world.isRemote && entity instanceof EntityPlayer) { EntityPlayer player = (EntityPlayer) entity; // Appliquer les effets du tier for (PotionEffect effect : this.effects) { player.addPotionEffect(new PotionEffect(effect)); } // Cooldown partage sur tous les items du groupe "custom_apples" player.getCooldownTracker().setCooldown(this, 600); // 600 ticks = 30 sec // Appliquer aussi le cooldown sur les autres pommes custom player.getCooldownTracker().setCooldown(EriniumItems.VALTERITE_APPLE, 600); player.getCooldownTracker().setCooldown(EriniumItems.AETHERITE_APPLE, 600); player.getCooldownTracker().setCooldown(EriniumItems.ERINIUM_APPLE, 600); stack.shrink(1); } return stack; }
⚠
Pomme Erinium — ingredient Enchanted Golden Apple — La Pomme d'Or Enchantee (Items.GOLDEN_APPLE avec metadata 1) est l'ingredient le plus rare du jeu. Ce choix delibere rend la Pomme d'Erinium exceptionnellement couteuse, justifiant ses effets combines (Regen III + Resistance II + Absorption IV) qui donnent un avantage decisif en PvP.

// Pomme Aetherite — Absorption IV pendant 30 secondes EriItem.create(EriniumFaction.MODID, "aetherite_apple") .rarity(EnumRarity.RARE) .creativeTab(EriniumCreativeTab.TAB) .maxStackSize(1) .food(0, 0.0f) // Pas de faim recuperee (c'est une pomme de combat) .alwaysEdible() // Mangeable meme a satiete pleine .register(); // Les effets sont geres dans la classe ItemTieredApple.onItemUseFinish()

11 Architecture Technique

Structure des packages

fr.eriniumgroup.eriniumfaction.content/ ├── EriniumItems.java — MODIFIER : ajouter les 19 nouveaux items ├── EriniumRecipes.java — MODIFIER : ajouter les nouvelles recettes ├── bow/ │ ├── ItemTieredBow.java — Classe de base des arcs custom (extends ItemBow) │ └── BowRenderHandler.java — Rendu cote client (pull animation, properties) ├── arrow/ │ ├── ItemTieredArrow.java — Item fleche custom (base) │ ├── EntityValteriteArrow.java — Entite fleche Valterite (slowness on hit) │ ├── EntityAetheriteArrow.java — Entite fleche Aetherite (armor pierce) │ ├── EntityEriniumArrow.java — Entite fleche Erinium (double damage, no recover) │ └── EntityExplosiveArrow.java — Entite fleche explosive (AoE, no block break) ├── shield/ │ └── ItemTieredShield.java — Bouclier custom (extends ItemShield) ├── weapon/ │ ├── ItemDagger.java — Dague (marker class, logique dans event handler) │ └── ItemWarHammer.java — Marteau de guerre (marker class + getAoeRadius()) ├── food/ │ └── ItemTieredApple.java — Pomme custom (effets multiples + cooldown partage) └── event/ ├── SetBonusHandler.java — Detection set complet + application/retrait des bonus ├── BackstabHandler.java — Detection backstab (vectoriel) + multiplicateur x1.5 ├── HammerAoeHandler.java — Frappe AoE en cone frontal (60% des degats) └── ShieldBlockHandler.java — Blocage personnalise + knockback Erinium

Enregistrement des event handlers

Tous les event handlers sont enregistres dans CommonProxy.init() via MinecraftForge.EVENT_BUS.register(). Les handlers qui ont besoin d'etre cotes client uniquement (particules) sont enregistres dans ClientProxy.init().

// Dans CommonProxy.init() MinecraftForge.EVENT_BUS.register(new SetBonusHandler()); MinecraftForge.EVENT_BUS.register(new BackstabHandler()); MinecraftForge.EVENT_BUS.register(new HammerAoeHandler()); MinecraftForge.EVENT_BUS.register(new ShieldBlockHandler());

Enregistrement des entites dans CommonProxy

// Dans CommonProxy.preInit() int entityId = 60; for (Class<? extends Entity> cls : new Class[] { EntityValteriteArrow.class, EntityAetheriteArrow.class, EntityEriniumArrow.class, EntityExplosiveArrow.class }) { String name = cls.getSimpleName().toLowerCase() .replace("entity", ""); EntityRegistry.registerModEntity( new ResourceLocation(EriniumFaction.MODID, name), cls, name, entityId++, EriniumFaction.INSTANCE, 64, 1, true); }

12 Integration avec les systemes existants

EriniumItems.java

Ajouter les 19 nouveaux items a la fin de la methode register(), en respectant les sections commentees existantes. Creer une section par categorie : // ======================== BOWS ========================, etc.

EriniumRecipes.java

Ajouter les nouvelles recettes dans register(). Utiliser les helpers registerToolRecipe() et registerArmorRecipe() existants pour les items simples. Creer des methodes helper specifiques pour les arcs, boucliers et pommes.

Integration avec le systeme d'enchantements

  • Arcs — Compatibles avec tous les enchantements d'arc existants (Power, Punch, Flame, Infinity). S'assurer que les calculs de degats custom dans ItemTieredBow prennent en compte EnchantmentHelper.
  • Dagues et Marteaux — Compatibles avec Sharpness, Fire Aspect, Knockback. Les enchantements speciaux d'Erinium (Backstab Amplifier, Aftershock) seront a definir dans le spec enchantements.
  • Boucliers — Compatibles avec Unbreaking et Mending. Un futur enchantement "Thorns de bouclier" pourrait remplacer le knockback manuel du bouclier Erinium.

Integration avec le systeme de Gemmes (Sockets)

  • Les arcs, dagues et marteaux supportent le systeme de sockets de gemmes defini dans le spec Accessoires & Gemmes.
  • Nombre de sockets par tier : Valterite = 1, Aetherite = 2, Erinium = 3.
  • Les gemmes compatibles : Rubis (degats), Saphir (vitesse/fleche), Topaze (knockback), Onyx (lifesteal).
  • Pour les boucliers : gemmes compatibles = Emeraude (reduction supplementaire), Onyx (lifesteal sur blocage).

Integration avec le systeme RPG

  • Stat Force — S'applique aux degats des dagues et marteaux exactement comme pour les epees.
  • Stat Agilite — Pourrait modifier la portee de detection du backstab ou le bonus de velocite des arcs.
  • Stat Resistance — S'applique apres le blocage du bouclier (deux couches de reduction).
  • Le set bonus Aetherite (+10% vitesse) se stack additivement avec le bonus de vitesse de la stat Agilite RPG.

Integration Faction

  • La detection des allies dans HammerAoeHandler et ShieldBlockHandler necessite d'interroger le FactionManager pour verifier si deux joueurs sont dans la meme faction ou des factions alliees.
  • Le backstab ne s'applique pas aux membres de la meme faction (friendly fire off par defaut).

16 Plan de Tests — Equilibrage PvP 1.8

Tests fonctionnels — Arcs

  • Verifier que le tirage plus rapide de l'Aetherite est perceptible visuellement (animation pull).
  • Confirmer que les modificateurs de degats et velocite sont corrects avec la valeur de charge maximale.
  • Tester la compatibilite avec Power I a V, Punch, Flame et Infinity.
  • Verifier que l'Infinity fonctionne avec les fleches speciales (elle ne devrait pas — les fleches speciales sont toujours consommees).

Tests fonctionnels — Fleches

  • Fleche Valterite : confirmer Lenteur I, 2 secondes exactes, sur joueur et mob.
  • Fleche Aetherite : mesurer les degats avec et sans armure, confirmer la reduction effective de 30%.
  • Fleche Erinium : confirmer qu'elle n'est jamais recuperee, meme sur miss ou bloc.
  • Fleche Explosive : confirmer qu'aucun bloc n'est detruit, rayon correct, degats AoE mesures.
  • Tester le comportement dans les distributeurs (dispenser).

Tests fonctionnels — Boucliers

  • Mesurer la reduction effective : 60%/75%/90% sur differentes valeurs de degats entrants.
  • Confirmer que le blocage ne consomme pas plus de durabilite que prevu.
  • Tester le knockback du bouclier Erinium dans differentes configurations (joueur contre mur, vide).
  • Verifier que le bouclier bloque les fleches, les degats de magie et les degats directs.

Tests equilibrage — Dagues vs Epees

  • Scenario normal : Dague Erinium (6.0) face a face vs Epee Valterite (6+3=9 en vanilla). La dague devrait perdre clairement.
  • Scenario backstab : Dague Erinium backstab = 9.0. Comparable a une epee standard, justifiant le risque de positionnement.
  • Tester le backstab contre des joueurs qui tournent rapidement (PvP 1.8 = rotation rapide). Le cone de 120 degres doit etre assez grand pour etre utilisable mais pas exploit.
  • Verifier que le backstab ne se declenche pas si l'attaquant est devant la cible a quelques degres pres.

Tests equilibrage — Marteaux en combat de groupe

  • Scenario 1v1 : Marteau Valterite (7.0) vs Epee Valterite. Le marteau est superieur en degats bruts mais ne justifie pas son cout eleve en 1v1 pur.
  • Scenario 1v3 : Marteau Erinium vs 3 joueurs groups. L'AoE de 3 blocs devrait toucher les 3 si bien positiones.
  • Confirmer qu'aucune boucle infinie ne se produit avec les degats AoE triggering LivingHurtEvent recursivement.
  • Mesurer le DPS total (cible principale + AoE) : ne devrait pas depasser 3x le DPS d'une epee.

Tests equilibrage — Boucliers en PvP

  • Un joueur avec Bouclier Erinium (90%) et set Erinium (-15%) = reduction totale de 91.5%. Confirmer que ce n'est pas annulant (verifier qu'un minimum de 5% des degats passe toujours).
  • Tester le cooldown de durabilite du bouclier : a combien de frappes peut-il tenir avant de casser ?

Tests equilibrage — Pommes Custom

  • Pomme Erinium (Regen III + Resistance II + Absorption IV) : en PvP, combien de temps survit un joueur sous ses effets ? Ne doit pas rendre le joueur unkillable.
  • Verifier que le cooldown partage de 30 secondes s'applique correctement entre les 3 types de pommes.
  • Tester que alwaysEdible() fonctionne meme avec la faim pleine.

Tests de performance

  • SetBonusHandler tourne toutes les 20 ticks par joueur : mesurer l'impact sur un serveur avec 50+ joueurs connects.
  • HammerAoeHandler avec AABB.grow() sur un serveur charge : s'assurer que la requete n'est pas trop couteuse.
  • Confirmer qu'aucun event handler ne provoque de lag spike visible lors des combats de masse (sieges faction).
