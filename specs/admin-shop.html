Admin Shop

Boutique administree par le serveur avec prix dynamiques bases sur l'offre et la demande. Les joueurs achetent et vendent des items a des prix qui fluctuent automatiquement en fonction de l'activite economique.

01 Contexte & Objectifs

Vision du systeme

L'Admin Shop est la source et le puits d'argent principal de l'economie du serveur. Contrairement au HDV ou les joueurs echangent entre eux, l'Admin Shop est une entite economique illimitee : il peut acheter et vendre une quantite infinie d'items. Cela garantit que chaque item a toujours un prix de reference et que les joueurs peuvent toujours vendre leurs surplus.

Pour eviter l'abus (acheter massivement un item bon marche, le revendre quand le prix monte), le shop utilise un systeme de fluctuation automatique des prix base sur l'offre et la demande :

  • Les joueurs achetent beaucoup : le prix d'achat (buy) augmente. La demande est forte, le shop augmente ses prix.
  • Les joueurs vendent beaucoup : le prix de rachat (sell) diminue. L'offre est forte, le shop baisse ses prix de rachat.
  • Retour a l'equilibre : sans transaction, les prix reviennent lentement vers leur valeur de base (decay naturel).

Objectifs de design

  • Reference de prix : donner une valeur de reference a chaque item du jeu. Les joueurs au HDV s'alignent naturellement sur les prix du shop.
  • Money sink : le spread buy/sell (le prix d'achat est toujours superieur au prix de vente) retire de l'argent de l'economie. Essentiel pour eviter l'inflation.
  • Anti-exploitation : la fluctuation empeche les joueurs de farmer un item en masse et de vider le shop d'argent. Plus ils vendent, moins le prix de rachat est interessant.
  • Simplicite : GUI clair avec des categories. Un joueur debutant doit pouvoir acheter son premier set d'outils en 30 secondes.
  • Configurabilite : tout est en JSON. Les admins ajoutent/retirent des items, ajustent les prix, sans recompiler le mod.

Ce que le systeme ne fait pas

  • Pas de stock limite : le shop a un stock infini en V1. Pas de "Rupture de stock".
  • Pas de monnaie Eris : l'Admin Shop fonctionne uniquement en Dollars ($). Les Eris sont reserves aux echanges joueur-a-joueur (HDV, trade).
  • Pas de craft : le shop ne fabrique rien. Il vend des items bruts, pas des equipements enchantes.

Stockage

  • Config statique : config/eriniumfaction/shop-config.json — defini par les admins, lu au demarrage, reloadable a chaud via commande.
  • Donnees dynamiques : ShopPriceData (extends WorldSavedData) — stocke les influences de prix. Sauvegarde automatique par Forge dans les donnees du monde.
ⓘ
Pourquoi WorldSavedData : Les influences de prix sont liees au monde et doivent survivre aux redemarrages. WorldSavedData est le mecanisme Forge natif pour cela. Pas de JSON custom necessaire — Forge gere la persistence automatiquement.

04 Fluctuation des Prix

Formule de prix dynamique

Les prix actuels sont calcules a partir du prix de base et de l'influence accumulee :

// Prix d'achat actuel (joueur achete au shop) currentBuyPrice = baseBuyPrice * (1.0 + buyInfluence * buyFactor) // Prix de vente actuel (joueur vend au shop) currentSellPrice = baseSellPrice * (1.0 + sellInfluence * sellFactor) // Contraintes currentBuyPrice = clamp(currentBuyPrice, baseBuyPrice * 0.5, baseBuyPrice * 3.0) currentSellPrice = clamp(currentSellPrice, baseSellPrice * 0.2, baseSellPrice * 2.0) currentSellPrice = min(currentSellPrice, currentBuyPrice * 0.9) // Toujours un spread minimum de 10%

Modification de l'influence

// Quand un joueur ACHETE N unites buyInfluence += N * influencePerUnit // La demande augmente sellInfluence -= N * influencePerUnit * 0.3 // Leger effet inverse sur la vente // Quand un joueur VEND N unites sellInfluence -= N * influencePerUnit // L'offre augmente, prix de rachat baisse buyInfluence -= N * influencePerUnit * 0.3 // Leger effet inverse sur l'achat

Decay (retour a l'equilibre)

Toutes les 6 heures en temps reel (configurable), un decay est applique a toutes les influences :

// Decay multiplicatif — l'influence se rapproche de 0 buyInfluence *= decayFactor // defaut: 0.85 (15% de reduction par cycle) sellInfluence *= decayFactor // Si |influence| < 0.01, ramener a 0 (eviter les flottants infinitesimaux)

Parametres configurables

ParametreDefautDescription
buyFactor0.15Sensibilite de l'influence sur le prix d'achat. 0.15 = +15% par point d'influence.
sellFactor0.20Sensibilite de l'influence sur le prix de vente. Plus sensible que l'achat.
influencePerUnit0.01Influence gagnee par unite achetee/vendue.
decayFactor0.85Multiplicateur de decay (0-1). Plus proche de 1 = retour lent.
decayIntervalMinutes360Intervalle du decay en minutes (defaut: 6h).
minBuyMultiplier0.5Prix d'achat minimum = baseBuyPrice * 0.5.
maxBuyMultiplier3.0Prix d'achat maximum = baseBuyPrice * 3.0.
minSellMultiplier0.2Prix de vente minimum = baseSellPrice * 0.2.
maxSellMultiplier2.0Prix de vente maximum = baseSellPrice * 2.0.
minSpreadPercent10Spread minimum entre buy et sell (en %). Garantit que le shop est toujours un money sink.
ⓘ
Exemple concret : Le diamant a un baseBuyPrice de $150 et baseSellPrice de $100. Apres que 500 joueurs aient achete 1000 diamants au total, la buyInfluence est a +10. Prix actuel : $150 * (1 + 10 * 0.15) = $150 * 2.5 = $375. Le prix a plus que double. Les joueurs arreteront d'acheter et le decay ramenera le prix a la normale en quelques jours.
⚠
Performance : Le calcul du prix est une simple multiplication. Pas de boucle, pas d'iteration. Le calcul est fait a chaque affichage dans le GUI et a chaque transaction. Cout negligeable meme avec 1000 joueurs.

Popup d'achat

  • Item : icone + nom de l'item.
  • Prix unitaire actuel : affiche en grand, avec indication de la tendance (fleche haut/bas par rapport au prix de base).
  • Prix de base : affiche en petit, barre si different du prix actuel.
  • Champ quantite : NumericField avec boutons rapides x1, x16, x64, xSTACK. Max = nombre qu'il peut acheter avec son solde.
  • Prix total : calcule en temps reel.
  • Solde joueur : affiche en bas.
  • Bouton Confirmer : "Acheter pour $X". Grise si solde insuffisant ou inventaire plein.

Popup de vente

  • Meme layout que l'achat, mais inverse.
  • Quantite : max = nombre de cet item dans l'inventaire du joueur.
  • Prix unitaire : prix de rachat actuel.
  • Bouton : "Vendre pour $X".

Indicateurs visuels

  • Fleche verte : prix actuel inferieur au prix de base (bonne affaire pour l'achat).
  • Fleche rouge : prix actuel superieur au prix de base (cher).
  • Pourcentage : affiche "+12%" ou "-8%" a cote du prix pour montrer l'ecart par rapport a la base.
  • Couleur du prix : vert si favorable au joueur, rouge si defavorable, blanc si ~base.

09 Regles Metier

Achat au shop

  • Verification : le serveur verifie que l'item existe dans la config, que le joueur a assez d'argent, et que son inventaire a de la place.
  • Inventaire plein : si le joueur ne peut pas recevoir TOUS les items demandes, le serveur calcule combien il peut recevoir et ajuste la quantite. L'achat est partiel (le joueur achete ce qu'il peut porter).
  • Debit : l'argent est debite du compte du joueur.
  • Items donnes : les items sont crees a partir du template (itemId + meta + nbt optionnel) et ajoutes a l'inventaire.
  • Influence mise a jour : buyInfluence augmente proportionnellement a la quantite achetee.

Vente au shop

  • Verification : le serveur verifie que l'item est rachetable (baseSellPrice != -1), que le joueur possede la quantite demandee.
  • Matching exact : l'item dans l'inventaire du joueur doit correspondre exactement a l'item du shop (meme ID + meta). Les items avec NBT custom ne sont PAS acceptes sauf si le shop les definit specifiquement.
  • Items retires : les items sont retires de l'inventaire cote serveur.
  • Credit : l'argent est credite sur le compte du joueur.
  • Influence mise a jour : sellInfluence modifiee.

Reload a chaud

  • Commande : /shopadmin reload relit le fichier JSON.
  • Influences preservees : les influences de prix actuelles (WorldSavedData) ne sont PAS remises a zero.
  • Nouveaux items : les items ajoutes dans la config sont immediatement disponibles avec une influence de 0.
  • Items supprimes : les items retires de la config disparaissent du shop. Leurs influences restent en WorldSavedData (inactives) et seront nettoyees au prochain decay.

10 Cas Limites & Securite

Anti-exploitation economique

  • Arbitrage : le spread minimum de 10% entre buy et sell rend l'arbitrage non profitable. Un joueur ne peut pas acheter et revendre pour gagner de l'argent.
  • Manipulation de prix : un joueur qui achete massivement pour faire monter les prix se penalise lui-meme (il paie de plus en plus cher). Le decay ramene les prix a la normale.
  • Vente en masse : un joueur qui farm un item et le vend en masse au shop voit le prix de rachat chuter rapidement. Le systeme s'auto-regule.
  • Items non-vendables : les items trop facilement farmables (cobble, dirt, etc.) ont baseSellPrice = -1. Le shop ne les rachete pas.

Overflow numerique

  • Prix : utiliser des long pour les prix. Clamp les influences pour eviter les valeurs extremes.
  • Quantite : limiter l'achat/vente a 2304 unites par transaction (36 stacks de 64).
  • Balance joueur : verifier que le credit ne depasse pas Long.MAX_VALUE.

Packet forgery

  • Item validation : le serveur verifie que l'item demande existe dans la config du shop. Pas de packet avec un itemId arbitraire.
  • Quantite validation : quantite > 0 et <= limite max.
  • Rate limiting : max 20 transactions par joueur par minute.

Concurrence

  • Tick thread unique : en Minecraft Forge, le server tick est single-threaded. Les transactions shop sont traitees dans le tick, donc pas de race condition. Les packets sont traites sequentiellement.
  • Prix coherent : le prix est calcule au moment exact de la transaction. Deux joueurs achetant dans le meme tick paient le meme prix (l'influence n'est mise a jour qu'apres chaque transaction).

11 Plan de Tests

Tests fonctionnels

  • Achat basique : acheter 16 diamants. Verifier items recus, argent debite, influence augmentee.
  • Vente basique : vendre 32 fer. Verifier items retires, argent credite, influence modifiee.
  • Item non vendable : tenter de vendre un item avec baseSellPrice = -1. Verifier le refus.
  • Solde insuffisant : tenter d'acheter sans assez d'argent. Verifier le refus.
  • Inventaire plein : remplir l'inventaire, acheter 64 items. Verifier l'achat partiel.
  • Categories : naviguer dans les categories et sous-categories. Verifier l'affichage correct.
  • Recherche : chercher "diamant". Verifier que le diamant apparait dans les resultats.

Tests de fluctuation des prix

  • Hausse d'achat : acheter 1000 diamants. Verifier que le prix d'achat a augmente significativement.
  • Baisse de vente : vendre 1000 fer. Verifier que le prix de rachat a diminue.
  • Clamp min/max : acheter massivement jusqu'au plafond. Verifier que le prix ne depasse pas baseBuyPrice * maxBuyMultiplier.
  • Spread minimum : manipuler les influences pour que sell > buy * 0.9. Verifier que le clamp de spread s'applique.
  • Decay : modifier l'influence, attendre le decay. Verifier que l'influence diminue de decayFactor.
  • Retour a zero : apres plusieurs cycles de decay, verifier que l'influence est bien ramenee a 0.

Tests admin

  • Reload : modifier le JSON, /shopadmin reload. Verifier que les changements sont pris en compte sans redemarrage.
  • Reset prix : /shopadmin resetprices. Verifier que toutes les influences sont a 0.
  • Info item : /shopadmin info minecraft:diamond. Verifier l'affichage des infos de prix.
  • Set influence : /shopadmin setinfluence minecraft:diamond 0 5.0 -2.0. Verifier les nouvelles valeurs.

Tests de securite

  • Item invalide : forger un packet shop_buy avec un itemId inexistant. Verifier pas de crash.
  • Quantite negative : forger un packet avec quantite = -1. Verifier le refus.
  • Config JSON invalide : mettre un JSON malformed, /shopadmin reload. Verifier log d'erreur et config precedente conservee.
  • Rate limit : envoyer 30 packets shop_buy en 1 seconde. Verifier que seuls 20 sont traites.

Tests de performance

  • 500 joueurs : simuler 500 joueurs ouvrant le shop simultanement. Verifier l'impact TPS (< 1ms).
  • Config large : charger un shop-config.json avec 500 items. Verifier le temps de chargement et la fluidite du GUI.
  • Decay 10,000 items : avoir 10,000 influences en memoire. Mesurer le temps du decay.
