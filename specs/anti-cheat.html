Securite — Detection de Triche

Anti Cheat

Systeme de detection automatique de clients modifies (hacked clients) et d'avantages injustes dans un environnement PvP 1.8. Detection server-side uniquement, violation progressive, punitions automatiques configurables.

Forge 1.12.2 PvP 1.8 (no cooldown) Server-side only Auto-punition 500-1000 joueurs EriAPI + Mixin

01Contexte & Objectifs

Contexte PvP 1.8

Le serveur utilise un mod PvP 1.8 (sans cooldown d'attaque). Cela change radicalement le paysage des triche par rapport a un serveur vanilla 1.12 :

  • CPS illimite — Sans cooldown, cliquer plus vite donne directement plus de DPS. Les auto-clickers sont un avantage massif.
  • KillAura — Peut attaquer plusieurs cibles par tick a 360 degres, devastateur sans cooldown.
  • Reach — Avec les "s-clicks" et le no-knockback, la portee d'attaque est critique.
  • Velocity/Anti-Knockback — Annuler le knockback permet d'encaisser des combos sans reculer, rompt completement l'equilibre PvP.
  • Speed — Se deplacer plus vite que les adversaires confere un avantage de poursuite/fuite decisif.
!
Note importante : L'attribut ATTACK_SPEED est inutile sur ce serveur (mod PvP 1.8). Ne jamais baser des checks anti-cheat sur ATTACK_SPEED. Tous les checks de combat se basent sur la vitesse de mouvement, les degats, le knockback et la distance.

Philosophie : detection server-side

L'anti-cheat est entierement cote serveur. Aucune verification client, aucun agent injecte. Cette approche est plus robuste (un client modifie ne peut pas mentir au serveur si le serveur valide lui-meme) mais requiert une calibration fine pour eviter les faux positifs.

Principes de design

  • Tolerance avant punition — Un joueur doit accumuler plusieurs violations avant toute action. Un seul faux positif isole ne punit pas.
  • Decay des violations — Les points de violation diminuent naturellement avec le temps pour ne pas penaliser un joueur longtemps apres un evenement de lag.
  • Configurabilite totale — Chaque check peut etre desactive independamment. Chaque seuil est ajustable sans recompilation.
  • Transparence staff — Le staff voit les violations en temps reel avec assez de contexte pour decider d'une action manuelle.
  • Zero impact gameplay — Les checks ne doivent jamais modifier le comportement d'un joueur legitime, meme sous lag.

02CheckMovement

Speed Hack

La vitesse maximale d'un joueur est calculee a chaque tick selon ses effets actifs, son etat (sprint, marche, aquatique, echelle, cheval) et les attributs modifies par le mod.

public class CheckMovement { // Calculer la vitesse max attendue en blocs/tick public double getMaxExpectedSpeed(EntityPlayerMP player) { double base = 0.22; // Vitesse marche vanilla (~4.3 blocs/s) if (player.isSprinting()) base *= 1.30; if (player.isOnLadder()) base = 0.12; if (player.isInWater()) base *= 0.80; if (player.isRiding()) base = getHorseSpeed(player); // Effets de potion PotionEffect speed = player.getActivePotionEffect(MobEffects.SPEED); if (speed != null) base *= (1 + (0.20 * (speed.getAmplifier() + 1))); PotionEffect slow = player.getActivePotionEffect(MobEffects.SLOWNESS); if (slow != null) base *= (1 - (0.15 * (slow.getAmplifier() + 1))); // Tolerance lag : +20% de marge return base * AntiCheatConfig.SPEED_TOLERANCE; } public void checkSpeed(EntityPlayerMP player, double distanceTraveled) { if (player.capabilities.isFlying) return; // Admin en vol double maxSpeed = getMaxExpectedSpeed(player); if (distanceTraveled > maxSpeed) { double ratio = distanceTraveled / maxSpeed; int weight = ratio > 2.0 ? 5 : ratio > 1.5 ? 3 : 1; ViolationTracker.getInstance().addViolation(player, "SPEED", weight); } } }

Fly Hack

  • Detection : joueur en l'air depuis plus de 4 ticks sans elytra, sans cheval volant, sans mode creatif, sans effet Levitation.
  • Tolerance : 4 ticks de vol correspond a un double saut de mod — la fenetre est suffisamment large pour eviter les faux positifs sur des sauts hauts.
  • Verification : position Y du joueur vs position Y attendue selon la gravite (chute libre).

NoFall

Le serveur tracked independamment la hauteur de chute de chaque joueur depuis le dernier sol touche. Quand un paquet d'atterrissage arrive, les degats attendus sont calcules et compares aux degats reellement infliges. Si l'ecart est > 2 hearts, violation.

Phase / NoClip

A chaque mise a jour de position, le serveur verifie si la position reportee par le joueur est dans un bloc solide. Si oui, le joueur est teleporte en arriere a la derniere position valide et une violation est enregistree.

Step Hack

Un joueur ne peut pas monter plus de 0.6 blocs par tick sans escalier ni dalle. Si la position Y augmente de plus de 0.65 blocs en un tick alors que le joueur est au sol (pas en saut), c'est un step hack.

Blink / Timer

  • Blink — Le timer check detecte un nombre anormal de paquets de mouvement. Vanilla envoie exactement 1 CPacketPlayer par tick. Si un joueur envoie plus de 25 paquets en 20 ticks (25% d'exces) : violation.
  • Timer — Si un joueur envoie moins de 5 paquets en 20 ticks (50% du normal) puis burst de 35+ paquets : violation blink.

TeleportHack

Si la distance entre deux positions consecutives depasse 10 blocs sans qu'un teleport serveur ait ete initie (via /tp, warp, faction home), violation severe de poids 8.

03CheckCombat

KillAura

La kill aura attaque automatiquement les cibles proches, souvent derriere le joueur ou sans line-of-sight. Detection multi-criteres :

  • Angle d'attaque — Si la cible est a plus de 120 degres derriere la direction de regard du joueur au moment de l'attaque : violation. (Vanilla 1.8 permet d'attaquer dans un cone de ~60 degres devant soi.)
  • Line-of-sight — Verifier qu'il n'y a pas de bloc solide entre le joueur et la cible au moment de l'attaque.
  • Multi-target rapid switch — Si un joueur attaque 3 entites differentes en moins de 5 ticks : violation killaura (pattern auto-swing).
  • Pas d'animation de swing — Voir NoSwing check dans CheckWorld.

Reach Hack

// Sur EntityAttackEvent : calculer la distance reelle de l'attaque @SubscribeEvent public void onAttack(AttackEntityEvent event) { if (!(event.getEntity() instanceof EntityPlayerMP)) return; EntityPlayerMP attacker = (EntityPlayerMP) event.getEntity(); Entity target = event.getTarget(); double distance = attacker.getPositionVector().distanceTo(target.getPositionVector()); double maxReach = AntiCheatConfig.MAX_REACH_DISTANCE; // 4.5 blocs par defaut if (distance > maxReach) { int weight = distance > maxReach * 1.5 ? 5 : 2; ViolationTracker.getInstance().addViolation(attacker, "REACH", weight); event.setCanceled(true); // Annuler l'attaque hors portee } }

La portee maximale par defaut est 4.5 blocs (3.0 vanilla PvP 1.8 + 1.5 de tolerance lag reseau). Configurable.

AutoClicker

Sur un serveur PvP 1.8, le jitter clicking humain peut atteindre 14-16 CPS soutenu, avec des pics a 18-20 CPS tres brefs. Un AutoClicker maintient 20+ CPS indefiniment. Detection :

  • Mesure du CPS sur une fenetre glissante de 10 secondes.
  • Si le CPS moyen depasse 18 sur plus de 5 secondes consecutives : violation.
  • Si le CPS depasse 25 meme brievement : violation immediate.
  • Analyse de la variance : un autoclicker a une variance CPS tres faible. Detecter une variance inter-click < 5ms sur 20 clicks consecutifs.

Criticals Illegaux

Un coup critique en PvP 1.8 requiert que le joueur soit en train de descendre dans les airs (position Y decroissante). Le serveur verrifie :

  • Si le joueur n'est pas en saut (Y velocity == 0 ou positif) et que le coup est critique : violation.
  • Si le joueur est clairement au sol (onGround = true, confirme server-side) : les critiques sont impossible, violation si tente.

Velocity / Anti-Knockback

Apres chaque coup recu, le serveur calcule le knockback attendu selon la force de frappe, l'enchantement Knockback, et l'effet Resistance. Il compare avec le deplacement reel du joueur dans les 3 ticks suivants :

  • Si le joueur se deplace de moins de 30% du knockback attendu apres un coup fort : violation velocity.
  • Tolerance : les joueurs avec potion de resistance ou dans l'eau ont des comportements differents — ces cas sont correctement pris en compte.

04CheckWorld

FastBreak

La vitesse de minage depend du type de bloc, de l'outil, des enchantements (Efficiency), de l'effet Haste, et de si le joueur est dans l'eau ou en l'air (malus). Le serveur calcule la duree minimale de minage pour chaque bloc :

public int getMinMiningTicks(EntityPlayerMP player, IBlockState state) { float hardness = state.getBlockHardness(player.world, player.getPosition()); float speed = player.getDigSpeed(state); if (player.canHarvestBlock(state)) { return (int) Math.ceil(hardness / speed * 1.5); // *1.5 = ticks } else { return (int) Math.ceil(hardness / speed * 5.0); // Outil inadapte = 5x plus long } }

Si le bloc est casse en moins de getMinMiningTicks() * 0.85 : violation FastBreak (15% de tolerance pour le lag).

FastPlace

Un joueur peut placer au maximum 1 bloc par tick (meme avec un autoclicker au CPS max). Si un joueur place 2+ blocs dans le meme tick : violation. Le systeme aussi surveille le taux de placement sur 20 ticks : max 20 blocs/seconde (1 par tick est le maximum physique).

Scaffold

Le scaffold automatique place des blocs sous les pieds du joueur pendant qu'il se deplace a vitesse sprint. Pattern detecte :

  • Joueur en sprint (> 0.20 blocs/tick de deplacement horizontal).
  • Placements de blocs consecutifs dans les chunks sous la position du joueur.
  • Si le pattern (sprint + placement bloc sous soi) se repete plus de 8 fois en 3 secondes : violation scaffold.

NoSwing

Tout coup porte ou tout bloc mine doit etre precede ou accompagne d'un paquet CPacketAnimation (swing de bras). Si une attaque ou un minage arrive sans swing dans les 2 ticks precedents : violation. Poids 2 par occurence.

05CheckPacket

Paquets invalides

Les paquets malformes ou hors-sequence sont interceptes par le Mixin sur NetHandlerPlayServer. Tout paquet avec des valeurs hors limites (positions NaN, infinies, ou hors monde) genere une violation immediate de poids 10 (quasi-certainement un client modifie).

Flood de paquets

  • Un joueur vanilla envoie environ 20-25 paquets par seconde (CPacketPlayer + interactions).
  • Si un joueur envoie plus de 500 paquets par seconde : violation severe + rate limiting force.
  • Si un joueur envoie plus de 1000 paquets par seconde : deconnexion immediate pour protection du serveur.

Timer manipulation (analyse temporelle)

Le timer cheat accelere toutes les actions du client (pas seulement les clics). Il est detecte par l'analyse de la frequence des paquets CPacketPlayer :

public class CheckPacket { // Ring buffer des timestamps des derniers paquets CPacketPlayer private final long[] movePacketTimes = new long[40]; // 40 derniers private int ringHead = 0; public void onMovePacket(EntityPlayerMP player) { long now = System.currentTimeMillis(); movePacketTimes[ringHead++ % 40] = now; // Calculer la frequence sur les 40 derniers paquets if (ringHead < 40) return; long oldest = movePacketTimes[ringHead % 40]; double packetsPerSec = 40000.0 / (now - oldest); // Normal: ~20 paquets/sec. Timer 1.2: ~24/sec. Timer 2.0: ~40/sec if (packetsPerSec > 26) { // 30% au-dessus de la normale int weight = packetsPerSec > 35 ? 5 : 2; ViolationTracker.getInstance().addViolation(player, "TIMER", weight); } } }

06CheckMining — Analyse X-Ray Statistique

Principe de detection

Le X-Ray ne peut pas etre detecte directement (le client voit les minerais, le serveur non). Cependant, le comportement de minage d'un joueur utilisant le X-Ray est statistiquement tres different d'un joueur legitime :

  • Un joueur normal mine des couloirs d'exploration, des branch mines, rencontre les minerais par hasard.
  • Un X-Rayer va directement aux veines de minerais sans exploration intermediaire.

Metriques collectees par joueur

Chest ESP

Si un joueur interagit avec un coffre cache (enterré, dans une cave non visitee) sans avoir explore la zone environment dans les 30 secondes precedentes : flag statistique. Ce n'est pas une violation directe mais un facteur de risque additionnel dans l'analyse globale.

07Systeme de Violations

ViolationTracker — Architecture

Chaque joueur possede un compte de violations independant par type de check. Les violations s'accumulent avec des poids differents selon la severite et la confiance du check.

Decay progressif

Toutes les 600 ticks (30 secondes), chaque joueur perd 1 point de violation. Cela assure que les joueurs qui ne triche plus retrouvent un niveau de violation propre apres quelques minutes. Un joueur avec 10 violations retombe a 0 en 5 minutes de jeu normal.

Tempban automatique

Si un joueur est kicke 3 fois pour triche dans la meme session de jeu (entre deux connexions), un ban temporaire configurable est automatiquement applique. Duree par defaut : 1 heure. Le staff peut annuler via /ac unban <player>.

09Architecture Technique

Package principal

Tout le code anti-cheat est dans fr.eriniumgroup.eriniumfaction.security.anticheat.

AntiCheatManager
Singleton central. Enregistre tous les checks, coordonne les events EriAPI, dispatche les violations.
CheckMovement
Speed, Fly, NoFall, Phase, Step, Blink, Timer, TeleportHack. Run sur PlayerMoveEvent.
CheckCombat
KillAura, Reach, AutoClicker, Criticals, Velocity. Run sur EntityAttackEvent.
CheckWorld
FastBreak, FastPlace, Scaffold, NoSwing. Run sur BlockBreak/Place events.
CheckPacket
Paquets invalides, flood, timer. Mixin sur NetHandlerPlayServer.
CheckMining
Analyse X-Ray statistique asynchrone. Ring buffer de records de minage par joueur.
ViolationTracker
Points par joueur en tableaux primitifs. Decay periodique. Persistance EriData.
PunishmentManager
Evaluate violations vs seuils. Kick, tempban, alertes staff, notifications.
AntiCheatConfig
Tous les seuils, poids, durees. Feature flags par check. Rechargeable via /ac reload.

11Systeme d'Exemptions

Exemptions permanentes par permission

Certains checks sont automatiquement bypasses pour les joueurs ayant des permissions specifiques :

PermissionChecks bypasses
eriniumfaction.bypass.flyFly, Speed (partiel), NoFall
eriniumfaction.bypass.speedSpeed uniquement
eriniumfaction.bypass.allTous les checks (pour les admins)
eriniumfaction.bypass.miningCheckMining / X-Ray uniquement

Exemptions contextuelles automatiques

  • Elytra — Check Fly desactive quand le joueur porte une elytra equipee et est en l'air.
  • Cheval — Checks Speed et Step ajustes selon la vitesse du cheval.
  • Effets de potion — Tous les calculs de vitesse/portee prennent en compte les effets actifs.
  • Lenteur de lag — Si le TPS serveur est < 15, la tolerance des checks de mouvement est automatiquement augmentee de 50% pour eviter les faux positifs dus au lag.
  • Teleportation recente — 2 secondes apres un /tp ou spawn : checks de mouvement desactives (position instable post-TP).

Snapshots de position (Evidence Replay)

Pour les checks de mouvement, le systeme enregistre les 20 derniers paquets CPacketPlayer de chaque joueur dans un ring buffer. Quand une violation est loggue, ces 20 positions sont sauvegardees dans le log avec timestamps pour permettre au staff de "rejouer" le mouvement suspect.

Alertes Discord (optionnel)

Si un webhook Discord est configure, les violations de niveau 3+ (kick) sont envoyes automatiquement avec :

  • Nom du joueur + UUID.
  • Type de triche detectee.
  • Points de violation au moment du kick.
  • Coordonnees.
  • ID d'incident pour correlation avec les logs.

Persistance des violations

Les points de violation et l'historique de kicks sont persistes via EriAPI DataManager. Ils survivent aux redemarrages serveur. Un joueur ne peut pas effacer son historique en se deconnectant.

13Performance

✓
Objectif : L'overhead total de l'anti-cheat ne doit pas depasser 0.5ms par joueur par tick. Avec 1000 joueurs et des checks exécutes seulement sur événement (pas toutes les ticks), l'impact réel est bien inférieur.

Strategies de performance

  • Run sur event, pas sur tick — CheckMovement s'execute sur PlayerMoveEvent (1 fois par paquet mouvement), CheckCombat sur AttackEvent (seulement pendant les combats). Pas de boucle sur tous les joueurs a chaque tick.
  • Tableaux primitifs — ViolationTracker utilise int[] et long[] indexes par slot joueur. Zero allocation d'objet, zero GC pressure.
  • Ring buffers de taille fixe — L'historique de positions (20 entrees) et le tracker CPS (100 entrees) utilisent des tableaux de taille fixe avec indice circulaire. Pas de listes qui grandissent.
  • Analyse X-Ray asynchrone — L'analyse statistique couteuse est faite off-thread toutes les 30 secondes, avec les resultats appliques sur le thread principal via EriScheduler.
  • Early exit — Chaque check commence par verifier les exemptions et les conditions rapides (is admin, is creative, has permission) avant tout calcul.
  • Calculs de collision en cache — Les AABB de blocs sont calcules une seule fois par check de phase et caches pour ce tick.

Tests de detection

[TEST-01] Speed hack : Utiliser un client de test pour envoyer des paquets CPacketPlayer a 2x la vitesse normale. Verifier detection en moins de 5 violations accumulees et kick a 10 points.
[TEST-02] Fly hack : Envoyer des paquets de position en vol libre (Y constant sans elytra). Verifier detection apres 4 ticks et accumulation de violations.
[TEST-03] KillAura angle : Simuler une attaque a 150 degres derriere la direction de regard. Verifier violation et annulation de l'attaque.
[TEST-04] Reach hack : Simuler des attaques a 6 blocs de distance. Verifier que l'attaque est annulee et violation enregistree.
[TEST-05] Velocity : Recevoir un coup avec knockback et rester sur place. Verifier detection et violation.
[TEST-06] AutoClicker : Envoyer 25 paquets d'attaque par seconde pendant 10 secondes. Verifier detection apres 5 secondes de CPS eleve.
[TEST-07] FastBreak : Casser un bloc de pierre avec la main en 2 ticks (normal : ~30 ticks). Verifier violation.
[TEST-08] Paquet invalide : Envoyer un CPacketPlayer avec des coordonnees NaN. Verifier kick immediat.
[TEST-09] Timer : Envoyer 40 paquets CPacketPlayer en 20 ticks. Verifier detection de l'acceleration.
[TEST-10] X-Ray simulation : Miner 300 blocs en allant directement aux minerais (ore rate 60%). Verifier detection en analyse async apres 30s.
[TEST-11] Faux positifs — Joueur normal : 1 heure de jeu normal avec sprint, combats legaux, minage legitime. Verifier zero violation.
[TEST-12] Faux positifs — Lag : Simuler un spike de lag (TPS 8) avec un joueur legitime. Verifier que la tolerance auto-lag empeche les violations Speed/Movement.
[TEST-13] Exemptions : Donner la permission bypass.fly a un joueur, verifier que le check Fly ne se declenche pas en mode vol.
[TEST-14] Commandes admin : Tester toutes les commandes /ac avec des joueurs fictifs. Verifier auto-completion et messages d'erreur propres.
[TEST-15] Performance 1000 joueurs : Stress test avec 1000 bots en combat simultane. Verifier que le tick time n'augmente pas de plus de 0.5ms/joueur.

Criteres de validation finale

  • Les 10 types de triche principaux sont detectes avec un taux de detection > 95% dans les conditions de test.
  • Taux de faux positifs < 0.1% sur 1 heure de jeu normal par 10 joueurs de test.
  • Overhead performance < 0.5ms/joueur/tick mesure avec 100 joueurs actifs.
  • Commande /ac info affiche des informations correctes et utiles.
  • Tous les kicks sont loggues avec l'ID d'incident dans logs/anticheat.log.
  • Le systeme de decay fait revenir un joueur a 0 violations en 5 minutes apres la derniere violation.
!
Calibration indispensable : Les seuils par defaut sont des estimations. Apres depoiement en production, une periode de 2 semaines de monitoring en mode "log only" (auto-kick desactive) est obligatoire pour ajuster les seuils selon le comportement reel des joueurs du serveur.
