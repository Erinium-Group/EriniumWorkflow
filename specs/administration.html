Systeme d'Administration

Outils de moderation en jeu pour gerir les joueurs : avertissements, mutes, bans temporaires et casier judiciaire persistant. Concu pour un serveur de 500-1000 joueurs simultanes.

Serveur-side Moderation EriData persistant EriChat pagine Planifie

01Contexte & Objectifs

Pourquoi ce systeme

Sur un serveur de 500-1000 joueurs, les modérateurs doivent disposer d'outils rapides, fiables et traçables. Les punitions doivent être persistantes entre les redémarrages, consultables par n'importe quel modérateur, et cohérentes avec les règles du serveur.

Le système remplace tout recours aux commandes vanilla (ban, kick) par des outils intégrés qui alimentent un casier judiciaire central par joueur, avec historique complet et escalade automatique configurable.

Objectifs principaux

  • Casier judiciaire persistant — chaque sanction est enregistrée et ne disparaît jamais sauf action admin explicite
  • Escalade automatique — N warns déclenchent automatiquement un mute ou un ban sans intervention humaine
  • Commandes complètes — warn, mute, unmute, tempban, unban, lookup, clearrecord
  • Performance — opérations en mémoire avec persistence NBT, aucune I/O sur le hot path des ticks
  • Intégration chat — le mute bloque les messages via le ChatManager existant, sans mixin fragile
ℹ
EriAPI requis : Ce système utilise EriChat pour les messages de modération riches, ChatPaginator pour /lookup, et le système de permissions du RankManager pour les checks de grade. Le casier judiciaire est stocké en fichier JSON dans config/EriniumFaction/judicial/ (hors du monde) pour survivre aux resets de map/saisons.

02Casier Judiciaire

Structure de donnees

Le casier est stocke dans un fichier JSON par joueur dans config/EriniumFaction/judicial/{uuid}.json. Ce dossier est hors du monde — il survit aux resets de map, changements de saison, et suppressions de monde. Il ne s'efface que manuellement via /clearrecord.

ⓘ
Pourquoi pas @EriData ? Les données @EriData sont stockées dans le dossier du monde (world/playerdata/). Lors d'un reset de map ou d'une nouvelle saison, le dossier monde est supprimé et le casier serait perdu. En stockant dans config/, les sanctions sont permanentes et indépendantes du monde.
// Fichier : config/EriniumFaction/judicial/{uuid}.json // Chargé en mémoire au login, sauvegardé à chaque modification public class JudicialRecord { // UUID du joueur (clé primaire) public String uuid; public String lastKnownName; // Historique brut — format: "TYPE:raison:staff:timestamp:durée" public List<String> entries = new ArrayList<>(); // Compteurs rapides (évite de rescanner entries à chaque check) public int warnCount = 0; public int muteCount = 0; public int kickCount = 0; public int banCount = 0; // Mute actif (0 = pas de mute, -1 = permanent, >0 = timestamp expiry) public long muteExpiry = 0L; public String muteReason = ""; // Sérialisation JSON via Gson public static JudicialRecord load(UUID uuid) { /* lit config/EriniumFaction/judicial/uuid.json */ } public void save() { /* écrit le fichier JSON */ } }

Format d'une entree

Chaque entrée dans entries est un String encodé avec le séparateur :. Le champ raison est URL-encodé pour gérer les espaces et les deux-points.

PositionChampExemple
[0]TypeWARN / MUTE / KICK / TEMPBAN
[1]Raison (URL-encoded)Langage+inapproprié
[2]Staff (username)AdminPseudo
[3]Timestamp UNIX (ms)1742000000000
[4]Durée en secondes (0 = permanent / N/A)1800

Methodes utilitaires sur JudicialRecord

  • addEntry(type, reason, staff, durationSeconds) — ajoute une entrée, incrémente le compteur correspondant
  • isMuted() — retourne true si muteExpiry == -1 ou muteExpiry > System.currentTimeMillis()
  • getMuteRemainingSeconds() — secondes restantes, 0 si expiré, Long.MAX_VALUE si permanent
  • getEntriesByType(String type) — filtre les entrées par type pour l'affichage
⚠
Limite de la liste : Chaque fichier JSON peut croître indéfiniment. Pour éviter des fichiers trop lourds, le système peut optionnellement archiver les entrées de plus de 90 jours dans un fichier log séparé (config/EriniumFaction/judicial/archive/{uuid}.log, configurable).

03Commande /warn

Syntaxe

/warn <joueur> <raison...>
/warn <joueur> <raison>
eriniumfaction.mod.warn
Ajoute un avertissement au casier judiciaire du joueur.
  • Le joueur reçoit un message rouge : "[AVERTISSEMENT] Vous avez reçu un avertissement. Raison : {raison}. Avertissements totaux : {N}"
  • L'entrée est ajoutée au casier avec type WARN
  • Le modérateur reçoit confirmation : "[Admin] Avertissement envoyé à {joueur} (warn #{N})"
  • Si le joueur est hors ligne : le warn est quand même enregistré (sur sa prochaine connexion, il reçoit le message)
  • Après l'enregistrement, vérification de l'escalade automatique
  • Auto-complétion sur l'argument joueur via PlayerArg.of("name")
  • Raison multi-mots supportée (tout ce qui suit le pseudo est la raison)

Messages EriChat

// Message au joueur puni EriChat.create() .text("[AVERTISSEMENT] ").color(TextFormatting.RED).bold() .text("Vous avez reçu un avertissement.").color(TextFormatting.WHITE) .text("\nRaison : ").color(TextFormatting.GRAY) .text(reason).color(TextFormatting.YELLOW) .text("\nAvertissements totaux : ").color(TextFormatting.GRAY) .text(String.valueOf(record.warnCount)).color(TextFormatting.RED).bold() .to(target); // Message au staff EriChat.create() .text("[Admin] ").color(TextFormatting.DARK_PURPLE).bold() .text("Avertissement envoyé à ").color(TextFormatting.GRAY) .text(targetName).color(TextFormatting.WHITE) .text(" (warn #" + record.warnCount + ")").color(TextFormatting.GOLD) .to(sender);

04Commandes /mute & /unmute

Syntaxe

/mute <joueur> <durée> <raison...> /unmute <joueur>
/mute <joueur> <durée> <raison>
eriniumfaction.mod.mute
Rend muet un joueur pour une durée donnée.
  • Formats de durée : 30s, 5m, 1h, 6h, 1d, 7d, 30d, permanent
  • Le champ muteExpiry est mis à jour dans JudicialRecord : timestamp d'expiration en ms, ou -1 si permanent
  • Le champ muteReason est enregistré pour l'affichage quand le joueur essaie de parler
  • L'entrée MUTE est ajoutée au casier (type, raison, staff, timestamp, durée en secondes)
  • Le joueur reçoit un message : "[MUTE] Vous avez été réduit au silence. Raison : {raison}. Expiration : {date ou Permanent}"
  • Si le joueur est hors ligne : les champs sont mis à jour, actifs à la prochaine connexion
  • Auto-complétion joueur + suggestions de durée (30m, 1h, 6h, 1d, 7d, permanent)
/unmute <joueur>
eriniumfaction.mod.mute
Retire le mute actif d'un joueur.
  • Remet muteExpiry = 0 et muteReason = ""
  • Confirmation au staff : "[Admin] Mute retiré pour {joueur}"
  • Notification au joueur s'il est en ligne : "[INFO] Votre mute a été levé."
  • Fonctionne sur joueurs en ligne et hors ligne

Integration avec le ChatManager

Dans le handler d'envoi de message chat côté serveur, avant de propager le message, vérifier le statut de mute :

// Dans ChatManager.handleChatMessage() ou via @SubscribeEvent sur ServerChatEvent if (AdminManager.getInstance().isMuted(player)) { long remaining = AdminManager.getInstance().getMuteRemainingSeconds(player); String expiry = remaining == Long.MAX_VALUE ? "Permanent" : formatDuration(remaining); EriChat.create() .text("[MUTE] ").color(TextFormatting.RED).bold() .text("Vous etes reduit au silence. Expiration : ").color(TextFormatting.GRAY) .text(expiry).color(TextFormatting.YELLOW) .to(player); event.setCanceled(true); }

Parsing de duree

// AdminManager.parseDuration(String input) → secondes, -1 si permanent, 0 si invalide "30s" → 30 "5m" → 300 "1h" → 3600 "1d" → 86400 "30d" → 2592000 "permanent" → -1

05Commandes /tempban & /unban

Syntaxe

/tempban <joueur> <durée> <raison...> /unban <joueur>
/tempban <joueur> <durée> <raison>
eriniumfaction.mod.tempban
Ban temporaire d'un joueur via la liste de bans vanilla de Minecraft.
  • Formats de durée : 30m, 1h, 1d, 7d, 30d, 365d
  • Utilise server.getPlayerList().getBannedPlayers() pour ajouter une entrée UserListBansEntry avec la date d'expiration calculée
  • Le joueur connecté est kické immédiatement avec un message formaté : "Vous avez été banni. Raison : {raison}\nExpiration : {date}"
  • L'entrée TEMPBAN est ajoutée au casier judiciaire
  • Confirmation au staff avec la durée et la raison
  • La vérification au login est assurée par le mécanisme vanilla de UserListBans
/unban <joueur>
eriniumfaction.mod.tempban
Retire le ban d'un joueur (en ligne ou hors ligne).
  • Appelle server.getPlayerList().getBannedPlayers().removeEntry(profile)
  • Supporte les joueurs hors ligne (recherche par nom dans GameProfile)
  • Confirmation au staff

Message de deconnexion

// Message de kick au joueur banni (ITextComponent) "§cVous avez été banni de ce serveur.\n" "§7Raison : §f{raison}\n" "§7Expiration : §e{date formatée}\n" "§8Contestation : discord.gg/eriniumgroup"

Date d'expiration — formatage

// Format : "22 mars 2026 à 18:30" (Locale française) new SimpleDateFormat("d MMMM yyyy 'à' HH:mm", Locale.FRENCH) .format(new Date(System.currentTimeMillis() + durationMs));

06Commande /lookup

Syntaxe

/lookup <joueur> [page]
/lookup <joueur> [page]
eriniumfaction.mod.lookup
Affiche le casier judiciaire complet d'un joueur via EriChat paginé.
  • Supporte les joueurs en ligne et hors ligne (recherche par nom dans les données sauvegardées)
  • En-tête : pseudo, UUID, statut actuel (en ligne/hors ligne), grade
  • Résumé des sanctions : N warns · N mutes · N kicks · N bans
  • Statut actif : si muté → "MUET — expire dans {durée}", si banni → "BANNI — expire le {date}"
  • Liste paginée des entrées (8 par page), de la plus récente à la plus ancienne
  • Chaque entrée : [TYPE] date — raison — par staff
  • Actions cliquables sur chaque entrée hors ligne (voir /lookup <joueur> <page>)
  • Auto-complétion sur le nom du joueur (joueurs en ligne + cache des joueurs connus)

Format d'affichage avec ChatPaginator

// En-tête et résumé EriChat.create() .text("══════ Casier de ").color(TextFormatting.DARK_PURPLE) .text(playerName).color(TextFormatting.WHITE).bold() .text(" ══════").color(TextFormatting.DARK_PURPLE) .to(sender); EriChat.create() .text(record.warnCount + " avertissements ").color(TextFormatting.GOLD) .text(record.muteCount + " mutes ").color(TextFormatting.AQUA) .text(record.banCount + " bans").color(TextFormatting.RED) .to(sender); // Liste paginée ChatPaginator paginator = ChatPaginator.create("/lookup " + playerName) .title("Historique des sanctions") .linesPerPage(8); for (String entry : record.entries) { paginator.add(formatEntry(entry)); // TYPE | date | raison (tronquée) | par staff } paginator.send(sender, page);

07Commande /clearrecord

Syntaxe

/clearrecord <joueur>
/clearrecord <joueur>
eriniumfaction.admin.clearrecord
Efface intégralement le casier judiciaire d'un joueur. Action irréversible.
  • Remet entries à une liste vide
  • Remet tous les compteurs à 0
  • Remet muteExpiry = 0 (retire le mute si actif)
  • Le ban éventuel via UserListBans n'est PAS retiré automatiquement (utiliser /unban séparément)
  • Un message de confirmation est demandé : "Confirmer ? Tapez /clearrecord {joueur} CONFIRMER"
  • Log serveur : "[ADMIN] {executeur} a effacé le casier de {joueur}"
  • Supporte joueurs hors ligne
!
Action irréversible : Le clearrecord supprime définitivement l'historique. Il est recommandé d'implémenter un système de sauvegarde ou d'archive vers fichier avant suppression (optionnel, configurable).

08Escalade Automatique

Principe

Après chaque warn enregistré, le système vérifie le nombre total de warns du joueur et déclenche automatiquement une sanction si un seuil configurable est atteint.

Seuil (warns)Action automatiqueConfig
3 warnsMute automatique — durée configurable (défaut 30min)autoMuteAfterWarns, autoMuteDuration
5 warnsTempban automatique — durée configurable (défaut 1h)autoBanAfterWarns, autoBanDuration

Implementation

// Dans AdminManager.addWarn() — appelé après toute commande /warn private void checkEscalation(EntityPlayerMP target, JudicialRecord record) { AdminConfig cfg = AdminConfig.getInstance(); if (record.warnCount == cfg.autoBanAfterWarns) { applyTempBan(target, cfg.autoBanDuration, "[AUTO] Seuil de warns atteint", "SYSTEME"); broadcastToStaff("[AUTO] " + target.getName() + " banni automatiquement (5 warns)"); } else if (record.warnCount == cfg.autoMuteAfterWarns) { applyMute(target, cfg.autoMuteDuration, "[AUTO] Seuil de warns atteint", "SYSTEME"); broadcastToStaff("[AUTO] " + target.getName() + " muté automatiquement (3 warns)"); } }

Broadcast staff

Les escalades automatiques sont annoncées à tous les joueurs en ligne ayant la permission eriniumfaction.mod.warn ou supérieure. Le message précise que c'est une action automatique pour distinguer des actions manuelles.

09Verification au Login

Mute au reconnect

Lors de l'événement PlayerLoggedInEvent, l'AdminManager vérifie si le joueur qui se connecte a un mute actif. Si oui, il envoie le message de rappel après un court délai (via EriScheduler) pour s'assurer que le joueur a bien reçu les messages de bienvenue.

// Dans AdminManager — appelé à chaque PlayerLoggedInEvent public void onPlayerLogin(EntityPlayerMP player) { JudicialRecord record = DataManager.get(player, JudicialRecord.class); // Notifier si muet if (record.isMuted()) { EriScheduler.delay(60, () -> { // 3 secondes après login EriChat.create() .text("[MUTE] ").color(TextFormatting.RED).bold() .text("Vous etes toujours reduit au silence. Expiration : ").color(TextFormatting.GRAY) .text(formatExpiry(record.muteExpiry)).color(TextFormatting.YELLOW) .to(player); }); } // Notifier les warns reçus hors ligne // (les warns offline sont marqués d'un flag "notifié" pour éviter le re-notif) }

Purge des mutes expires

Un check périodique toutes les 5 minutes (via EriScheduler.repeat) parcourt la liste des joueurs en ligne et nettoie les mutes expirés (met muteExpiry = 0 si le timestamp est passé). Cette opération est légère : O(joueurs en ligne), pas O(tous les joueurs).

10Architecture

AdminManager
Singleton principal. Gère toutes les opérations de sanction, la logique d'escalade, et les vérifications de mute. Point d'entrée unique pour le reste du mod.
JudicialRecord
@EriData PLAYER — casier judiciaire persistant. Stocke entries (List<String>), compteurs, muteExpiry, muteReason.
AdminCommand
EriCommand racine avec sous-commandes warn, mute, unmute, tempban, unban, lookup, clearrecord. Auto-complétion sur tous les arguments.
AdminConfig
Configuration centralisée : seuils d'escalade, durées auto, durées max de sanction.
DurationParser
Utilitaire statique. Convertit "30m", "1h", "7d", "permanent" en secondes (long). Valide le format et lève une exception descriptive si invalide.
AdminEventHandler
@Mod.EventBusSubscriber. Écoute PlayerLoggedInEvent, ServerChatEvent. Injecte les vérifications de mute dans le flux de chat.

13Performance

Contraintes serveur (500-1000 joueurs)

  • Aucune I/O sur le hot path — le check de mute dans ServerChatEvent est O(1) : lecture du champ muteExpiry en mémoire via DataManager.get()
  • Cache mémoire des mutes actifs — AdminManager maintient un Set<UUID> mutedPlayers en mémoire pour un check ultra-rapide sans accéder au DataManager à chaque message
  • Pas d'itération globale — aucune boucle sur tous les joueurs connectés dans les ticks. La purge des mutes expirés cible uniquement les joueurs en ligne
  • Persistence asynchrone optionnelle — les DataManager.save() peuvent être faits sur le thread d'I/O de Forge (déjà géré par EriAPI)
  • Lookup hors ligne — recherche par nom dans les fichiers de sauvegarde de l'overworld. Opération rare (modération), acceptable en synchrone
// Cache mémoire AdminManager — mis à jour lors des mutes/unmutes/login/logout private final Set<UUID> mutedPlayers = new HashSet<>(); public boolean isMuted(EntityPlayer player) { // Check rapide en O(1) sans accès NBT return mutedPlayers.contains(player.getUniqueID()); }
