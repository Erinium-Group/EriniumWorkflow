Refonte HUD — Barres de Vie & Status

Remplacement complet de toutes les barres de statut vanilla au-dessus de la barre d'action par un OverlayMod EriAPI minimaliste et elegant : vie, nourriture, armure, oxygene, monture et XP en barres sleek style RPG moderne.

Planifie Mixin GuiIngame OverlayMod EriAPI Client-Side Only Liquid Glass

01 Contexte & Objectifs

Le HUD vanilla de Minecraft 1.12.2 affiche la vie sous forme de coeurs pixelises, la nourriture en jambons, l'armure en icones metalliques et l'oxygene en bulles — un visuel fonctionnel pour un jeu arcade, mais totalement inadequat pour un serveur faction/PvP serieux avec une identite visuelle elaboree.

Sur un serveur PvP 1.8 (sans cooldown d'attaque), les informations de sante sont consultees en permanence. Chaque information manquante ou difficile a lire est un desavantage tactique. Le HUD doit etre instantanement lisible, ne pas polluer visuellement le reste de l'ecran, et refleter l'esthetique glass morphism du mod.

Problemes du HUD vanilla

  • Coeurs pixelises — Difficiles a lire en un coup d'oeil lors d'un combat rapide (PvP 1.8 jitter click)
  • Pas de valeurs numeriques — Impossible de savoir si on est a 5.5 ou 6 PV sans compter les demi-coeurs
  • Absorption non differenciee — Les coeurs dores se fondent dans le reste
  • Style totalement deconnecte — Icones 16x16 retro vs panneaux glass morphism violet/cyan du mod
  • Layout non optimal — Les barres vanilla sont dispersees et non centrees
  • XP bar inconsistante — Style graphique different du reste du HUD

Objectifs de la refonte

  • Lisibilite instantanee — Valeur numerique sur chaque barre, visible au premier coup d'oeil
  • Non distrayant — Barres fines, translucides, pas de surcharge visuelle
  • Coherence visuelle — Style Liquid Glass, palette violet/cyan, coins arrondis
  • Conditionnel intelligent — Armure cachee si 0, oxygene visible seulement sous l'eau, monture seulement si montee
  • Animations fluides — Lerp des valeurs pour eviter les sauts brusques
  • Performance — Construction unique de l'arbre de composants, mise a jour des valeurs seulement
⚠
Regle CLAUDE.md — Ne JAMAIS coder d'overlay brut (RenderGameOverlayEvent + OpenGL direct). TOUJOURS utiliser le systeme OverlayMod d'EriAPI (extends OverlayMod, register via OverlayModManager). L'implementation doit etre un StatusBarsOverlay extends OverlayMod.
⚠
PvP 1.8 — Le serveur utilise un mod PvP 1.8 sans cooldown d'attaque. L'attribut ATTACK_SPEED est inutile. Les animations et effets visuels du HUD ne doivent JAMAIS etre bases sur l'attack speed. La barre de vie est la reference principale de survie en combat.

02 Architecture Technique

L'implementation repose sur deux parties : un ensemble de Mixins qui annulent le rendu vanilla des barres de statut, et un StatusBarsOverlay qui s'appuie entierement sur EriAPI pour le rendu personnalise.

Classes Mixin cibles

Classe cible Methode annulee Couvre
net.minecraft.client.gui.GuiIngame renderHealth(ScaledResolution) Coeurs de vie + coeurs absorption
net.minecraft.client.gui.GuiIngame renderFood(ScaledResolution) Jambons de nourriture
net.minecraft.client.gui.GuiIngame renderArmor(ScaledResolution) Icones d'armure
net.minecraft.client.gui.GuiIngame renderAir(ScaledResolution) Bulles d'oxygene
net.minecraft.client.gui.GuiIngame renderMountHealth(ScaledResolution) Coeurs de monture
net.minecraft.client.gui.GuiIngame renderExperience(ScaledResolution) Barre d'XP vanilla

Toutes ces methodes sont injectees avec @Inject at HEAD, cancellable = true. La condition d'annulation est simple : si StatusBarsOverlay est actif (visible), appeler ci.cancel(). Sinon, laisser le vanilla s'afficher (fallback si EriAPI n'est pas charge).

@Mixin(GuiIngame.class) public class MixinGuiIngameStatusBars { @Inject(method = "renderHealth", at = @At("HEAD"), cancellable = true) private void cancelRenderHealth(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } @Inject(method = "renderFood", at = @At("HEAD"), cancellable = true) private void cancelRenderFood(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } @Inject(method = "renderArmor", at = @At("HEAD"), cancellable = true) private void cancelRenderArmor(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } @Inject(method = "renderAir", at = @At("HEAD"), cancellable = true) private void cancelRenderAir(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } @Inject(method = "renderMountHealth", at = @At("HEAD"), cancellable = true) private void cancelRenderMountHealth(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } @Inject(method = "renderExperience", at = @At("HEAD"), cancellable = true) private void cancelRenderExperience(ScaledResolution resolution, CallbackInfo ci) { if (StatusBarsOverlay.isActive()) ci.cancel(); } }

Registration de l'OverlayMod

// Dans ClientProxy.init() OverlayModManager.register(new StatusBarsOverlay()); // Structure de base de l'overlay public class StatusBarsOverlay extends OverlayMod { private static final int W = 640; // largeur design (HP 280 + gap 80 + food 280) private static final int H = 60; // hauteur design (armure + barres + xp + padding) public StatusBarsOverlay() { super("status_bars", "Status Bars HUD", W, H); setCategory("HUD"); // Centre en bas : x = 0.5 - (640/1920)/2 = 0.5 - 0.1667 = 0.3333 setDefaultPosition(0.333f, 0.930f); setLayer(OverlayLayer.POST_HOTBAR); } public static boolean isActive() { StatusBarsOverlay inst = (StatusBarsOverlay) OverlayModManager.get("status_bars"); return inst != null && inst.isVisible(); } }

04 Barre de Vie

La barre de vie est l'element le plus critique — elle doit etre lisible d'un coup d'oeil lors des combats PvP 1.8 a haute frequence de clic.

Sources de donnees

EntityPlayer player = Minecraft.getMinecraft().player; float hp = player.getHealth(); // HP actuels (0 a maxHP) float maxHp = player.getMaxHealth(); // HP max (vanilla = 20.0f) float absorption = player.getAbsorptionAmount(); // PV dores (0 par defaut) // HP max peut etre modifie par la stat Vigueur du systeme RPG // RpgManager applique le bonus en modifiant l'attribut MAX_HEALTH vanilla // donc getMaxHealth() retourne TOUJOURS la valeur correcte (avec bonus Vigueur)

Regles d'affichage

  • Texte — Format : "(int)hp / (int)maxHp" ex : "18/20" ou "18/30" si Vigueur augmente le max. Centrer dans la barre.
  • Absorption — Si absorption > 0, la section doree s'ajoute a la droite du fill HP. La largeur totale des deux fills ne peut pas depasser 280 px. Formule : fillHpW = 280 * (hp / (maxHp + absorption)), fillAbsW = 280 * (absorption / (maxHp + absorption)).
  • Texte avec absorption — Format : "18/20 +4" quand absorption > 0.
  • Flash de degat — Quand HP diminue, faire un shake(6) sur la barre de vie.
  • Pulsation danger — Quand HP < 20% du maxHP, activer une animation pingPong sur l'opacite de la barre (0.40f to 1.0f, 25 ticks, EASE_IN_OUT). Stopper quand HP remonte au-dessus de 25%.

Formules lerp

// Valeurs affichees (interpolees vers les valeurs reelles) private float displayHp = 20f; private float displayMaxHp = 20f; private float displayAbsorption = 0f; // Constante d'interpolation (par frame) private static final float LERP_SPEED = 0.15f; // Dans onFrame() — appele chaque frame private void updateDisplayValues() { float realHp = Minecraft.getMinecraft().player.getHealth(); float realMax = Minecraft.getMinecraft().player.getMaxHealth(); float realAbs = Minecraft.getMinecraft().player.getAbsorptionAmount(); // Lerp lineaire : valeur += (cible - valeur) * vitesse displayHp += (realHp - displayHp) * LERP_SPEED; displayMaxHp += (realMax - displayMaxHp) * LERP_SPEED; displayAbsorption += (realAbs - displayAbsorption) * LERP_SPEED; // Detecter perte de HP pour l'effet shake if (realHp < lastRealHp - 0.05f && hpBarBg != null) { hpBarBg.shake(6); } lastRealHp = realHp; }
✓
Lerp dans onFrame() vs onTick() — Le lerp doit se faire dans onFrame() (appele chaque frame rendue) et non onTick() (20/sec). Cela garantit une animation fluide quel que soit le FPS. La mise a jour de la largeur des fills GradientRectangle doit egalement se faire dans onFrame().

05 Barre de Nourriture

La barre de nourriture est positionnee a droite de la barre de vie. Elle est cachee quand le joueur monte une entite (remplacee par la barre de vie de monture).

Sources de donnees

FoodStats food = player.getFoodStats(); int foodLevel = food.getFoodLevel(); // 0 a 20 float saturation = food.getSaturationLevel(); // 0.0 a foodLevel (max 20) // saturation diminue en premier avant le food level

Regles d'affichage

  • Texte — Format : "19/20" centrer dans la barre. Toujours sur 20 (c'est le max vanilla).
  • Indicateur de saturation — Petit point brillant (ou micro-barre secondaire de 2px de hauteur en bas de la barre principale) de couleur jaune clair (#F1C40F) dont la largeur represente la saturation. Visible seulement si saturation > 0.
  • Alerte faim — Quand foodLevel <= 6 (30%), activer la pulsation pingPong sur la barre (identique au danger HP : 0.40f to 1.0f, 30 ticks, EASE_IN_OUT).
  • Masquage monture — Quand player.isRiding() est vrai, masquer entierement la barre food et afficher la barre de vie monture a sa place (voir section 08).
  • Lerp — Appliquer le lerp avec LERP_SPEED = 0.12f (un peu plus lent que HP pour un effet plus doux).
ℹ
Saturation non affichee comme valeur numerique — La saturation est un mecanisme technique que le joueur vanilla n'est pas cense voir explicitement. L'indicateur est visuel (micro-barre ou point) et subtil, pas un chiffre.

06 Indicateur d'Armure

L'armure est affichee sous forme d'un indicateur compact : une icone de bouclier et la valeur numerique, positionnes au-dessus de la barre de vie a gauche.

Source de donnees

// Valeur d'armure totale (0 a 20 points de base) int armorValue = TotalArmorValueHelper.getTotalArmorValue(player); // Ou via l'attribut direct : int armorValue = (int) player.getTotalArmorValue(); // methode vanilla GuiIngame

Regles d'affichage

  • Visibilite — L'indicateur est invisible si armorValue == 0. Il apparait avec un fadeIn(8) quand l'armure passe de 0 a > 0.
  • Position — Au-dessus du debut de la barre HP : x=0, y=0, dans la zone de design de l'overlay.
  • Icone — Symbole unicode ⛤ (bouclier) ou utiliser une texture de l'icone armure vanilla. Couleur : #5B9BD5.
  • Valeur — Affichee a droite de l'icone, format : simple entier "14". Couleur : #5B9BD5. Scale 0.65f.
  • Pas de lerp — L'armure change rarement et en valeur entiere, pas besoin d'interpolation.
  • Pas de saturation — Ne pas afficher la reduction de degats par enchant Protecion ou toughness ici, juste la valeur d'armure brute.
ℹ
Armure maximale — La valeur peut depasser 20 si la stat Resistance RPG ajoute des points d'armure virtuels. Afficher la valeur reelle retournee par getTotalArmorValue() sans la plafonner.

07 Barre d'Oxygene

La barre d'oxygene est un element conditionnel — elle n'existe visuellement que lorsque le joueur est sous l'eau et que son niveau d'air diminue. Elle apparait en fondu et disparait de meme.

Source de donnees

int air = player.getAir(); // 0 a 300 (300 = plein, en ticks) int maxAir = 300; // Constante vanilla // Detection : sous l'eau quand air < 300 et player est dans l'eau boolean underwater = player.isInsideOfMaterial(Material.WATER) || player.isInsideOfMaterial(Material.ICE); boolean showAir = underwater && air < maxAir;

Regles d'affichage

  • Apparition — Quand showAir passe de false a true : airGroup.fadeIn(12).
  • Disparition — Quand showAir passe de true a false (joueur sort de l'eau) : airGroup.fadeOut(20), puis masquer apres la fin du fadeOut.
  • Position — Centree entre les barres HP et food (x=240, y=24, w=160, h=6). Visible seulement dans cette zone — ne pas empieter sur les barres principales.
  • Format texte — A droite de la barre d'air : "<icone vague> (air/maxAir en secondes)" ex : "~ 8s" (air/20 arrondi). Scale 0.55f.
  • Urgence — Quand air < 60 (3 secondes), pulsation rapide : 0.3f to 1.0f, 15 ticks pingPong. Couleur de fill devient plus claire (#48CAE4).
  • Lerp — Appliquer lerp LERP_SPEED = 0.10f (plus lent, l'oxygene monte/descend lentement).
⚠
Gestion du groupe conditionnel — Creer un ContainerComponent airGroup qui contient le fond, le fill et le texte. Appeler airGroup.visible(false) a la construction. La visibilite est geree dans onTick() avec detection du changement d'etat.

08 Vie de Monture

Quand le joueur chevauche une entite (cheval, cochon, araignee, etc.), la barre de nourriture est remplacee par la barre de vie de la monture. La nourriture du joueur n'est pas pertinente en combat monte.

Source de donnees

boolean isRiding = player.isRiding(); if (isRiding) { Entity mount = player.getRidingEntity(); if (mount instanceof EntityLivingBase) { EntityLivingBase livingMount = (EntityLivingBase) mount; float mountHp = livingMount.getHealth(); float mountMaxHp = livingMount.getMaxHealth(); // Afficher la barre de monture en remplacement de la barre food } } // Cas specifique : cheval (EntityHorse) // getMaxHealth() sur un EntityHorse retourne la bonne valeur (15-30 HP selon le cheval)

Regles d'affichage

  • Transition — Quand le joueur monte : foodGroup.fadeOut(8) puis mountGroup.fadeIn(8). Transition inverse a la descente.
  • Format texte — "(int)mountHp / (int)mountMaxHp" ex : "15/20". Prefixe optionnel avec icone de cheval.
  • Gradient — Vert (#2ECC71 → #1A8449) pour distinguer immediatement de la vie du joueur.
  • Lerp — Appliquer lerp LERP_SPEED = 0.15f (identique a HP joueur).
  • Shake — Quand la vie de la monture diminue, appliquer mountBarBg.shake(4) (moins intense que pour le joueur).
  • Monture non-vivante — Si getRidingEntity() n'est pas une EntityLivingBase (ex: bateau, wagonnet), afficher la barre food normalement.

09 Barre d'XP

La barre d'XP est un element tres fin (4px de hauteur design) s'etendant sur toute la largeur du panneau, positionnee juste en dessous des barres de vie et nourriture. Elle affiche le niveau vanilla du joueur et le pourcentage de progression vers le prochain niveau.

ℹ
XP vanilla vs XP RPG — Cette barre affiche les points d'experience vanilla Minecraft (ceux obtenus en tuant des mobs, minant, etc. et visibles en bas de l'ecran). La barre XP RPG du systeme RPG est geree par RpgHudOverlay et affiche l'XP RPG separement. Les deux coexistent.

Sources de donnees

int xpLevel = player.experienceLevel; // Niveau vanilla (0, 1, 2, ...) float xpPct = player.experience; // Progression 0.0f a 1.0f vers le prochain niveau // Calcul de l'XP total necessaire pour le niveau actuel (formule vanilla) : // Niveau 0-15 : 2*lvl + 7 XP // Niveau 16-30 : 5*lvl - 38 XP // Niveau 31+ : 9*lvl - 158 XP

Regles d'affichage

  • Largeur fill — fillW = 640 * xpPct.
  • Label niveau — Positionne a gauche de la barre XP (x=0, y=36), format : "Nv.15". Couleur : #9B4FCF (violet-light). Scale 0.65f.
  • Gradient — Violet #6B2FA0 → Cyan #00E5FF (couleurs du mod EriniumFaction).
  • Level up flash — Quand le niveau augmente, faire un xpBar.pulse(10) et un bref fadeOut/fadeIn du label (rebirth visuel).
  • Lerp — LERP_SPEED = 0.08f (tres lent — la barre XP ne doit pas sauter brusquement lors d'un gain d'XP important).
  • Integration RpgHudOverlay — Cette barre XP vanilla n'interfere pas avec la barre XP RPG de RpgHudOverlay. Elles sont positionnees a des hauteurs differentes dans le repere de design 1920x1080.

10 Animations & Transitions

Les animations sont le coeur de l'experience visuelle du HUD. Elles doivent etre fluides, non intrusives et fournir un retour immediat sur les changements d'etat.

Algorithme de lerp (interpolation lineaire)

// Formule generale, appliquee dans onFrame() pour chaque valeur interpolee // displayValue converge vers targetValue a la vitesse LERP_SPEED chaque frame displayValue += (targetValue - displayValue) * LERP_SPEED; // Si la difference est negligeable, clamper pour eviter la derive infinie if (Math.abs(targetValue - displayValue) < 0.01f) { displayValue = targetValue; } // Conversion display → largeur de barre : int fillWidth = (int) (BAR_W * (displayValue / maxValue)); fillWidth = Math.max(0, Math.min(BAR_W, fillWidth));

Gestion des animations persistantes (danger)

// Champs de l'overlay pour tracker l'etat des animations de danger private Animation hpDangerAnim = null; private boolean hpInDanger = false; // Dans onTick() — verifier le changement d'etat de danger boolean nowInDanger = (player.getHealth() / player.getMaxHealth()) < 0.20f; if (nowInDanger && !hpInDanger) { hpInDanger = true; hpDangerAnim = Animation.create(0.40f, 1.0f, 25) .easing(Easing.EASE_IN_OUT) .pingPong() .scope("overlay") .onUpdate(v -> { if (hpBarFill != null) hpBarFill.setOpacity(v); }); AnimationManager.getInstance().play(hpDangerAnim); } else if (!nowInDanger && hpInDanger) { hpInDanger = false; if (hpDangerAnim != null) { hpDangerAnim.stop(); hpDangerAnim = null; } if (hpBarFill != null) hpBarFill.setOpacity(1.0f); }

11 Integrations

Integration RPG — stat Vigueur

La stat Vigueur du systeme RPG (RpgManager) modifie player.getMaxHealth() en ajoutant des PV supplementaires via l'attribut Forge MAX_HEALTH. La barre de vie StatusBarsOverlay lit player.getMaxHealth() directement — elle affiche automatiquement le bon maximum sans aucune integration supplementaire.

  • Pas de couplage direct avec RpgPlayerData
  • La valeur numerique affichee ("18/30") reflete correctement les 30 PV d'un personnage haut niveau
  • La barre s'adapte automatiquement a la largeur (toujours 280 px pour 100%)

Coexistence avec RpgHudOverlay

RpgHudOverlay affiche la barre XP RPG, le niveau RPG et le prestige en bas de l'ecran (position y=0.925f). StatusBarsOverlay est positionne a y=0.930f (juste en-dessous), ce qui cree une zone d'interface coherente centree en bas de l'ecran.

  • Aucun conflit de couche — les deux utilisent OverlayLayer.POST_HOTBAR
  • Positions par defaut calculees pour ne pas se superposer
  • L'utilisateur peut repositionner chaque overlay independamment via l'interface de gestion des overlays EriAPI

Coexistence avec MagicHudOverlay

MagicHudOverlay (barre de mana + slots de sorts) est positionne a y=0.875f. Aucune interaction avec les barres de statut. La mana est une ressource RPG independante des HP/food vanilla.

Barre d'XP vanilla vs XP RPG

Ce StatusBarsOverlay annule l'affichage de la barre XP vanilla via le Mixin renderExperience et la remplace par sa propre barre XP en bas du panneau. La barre XP RPG de RpgHudOverlay reste inchangee et continue d'afficher l'XP du systeme de progression RPG.

✓
Regle de bonne pratique — Ne jamais lire RpgPlayerData directement depuis StatusBarsOverlay. Les PV max doivent toujours passer par player.getMaxHealth() (attribut Minecraft). Si le systeme RPG modifie cet attribut, la barre de vie l'affichera automatiquement sans couplage.

12 Plan de Tests

Verification exhaustive de tous les etats et transitions possibles, en jeu en mode developpeur (runClient).

Tests de la barre de vie

  • HP plein (20/20) — barre entierement remplie, texte exact
  • HP partiel (9/20) — remplissage correct a 45%
  • HP critique (1/20, < 20%) — pulsation active, shake au degat
  • HP 0 — barre vide, pas d'erreur NaN/division par zero
  • HP max augmente (Vigueur RPG, ex: 30 PV max) — barre et texte s'adaptent
  • Absorption presente — section doree apres section rouge, texte "+4"
  • Absorption = 0 — section doree absente, pas de rendu parasite
  • Soin apres degat — lerp visible, pas de saut instantane

Tests de la barre de nourriture

  • Nourriture pleine (20/20) — barre entierement remplie
  • Nourriture faible (<= 6) — pulsation active
  • Saturation visible — micro-indicateur proportionnel a la saturation
  • Nourriture 0 — barre vide, pas d'erreur
  • Consommation d'aliment — lerp visible

Tests de l'indicateur d'armure

  • Armure complete (20 points) — valeur "20" affichee correctement

  • Armure > 20 (bonus RPG Resistance) — valeur superieure a 20 affichee sans troncature

Tests de la barre d'oxygene
