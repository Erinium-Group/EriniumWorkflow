Systeme de Chat

Refonte complete du chat Minecraft : canaux multiples, onglets filtrants, preview couleur en temps reel, mentions @, emojis, messages prives et GUI custom via Mixin + EriAPI.

Planifie Mixin GuiChat + GuiNewChat EriAPI Chat + Notification 5 Canaux 20+ Emojis

01 Contexte & Objectifs

Le chat vanilla de Minecraft 1.12.2 est extremement basique : un seul flux de messages sans filtrage, aucune notion de canal, aucune mise en forme riche, et une interface figee. Sur un serveur faction/PvP actif, le chat global devient vite illisible quand le chat de faction, les messages prives et le chat de groupe s'y melangent.

L'objectif de cette refonte est de remplacer completement l'experience chat par un systeme structure, visuellement riche et ergonomique, tout en preservant la compatibilite avec les systemes existants (RankChatHandler, FactionChatHandler).

Problemes du chat vanilla

  • Absence de canaux — Le chat faction, ally et global se chevauchent dans un seul flux
  • Pas de filtrage — Impossible de n'afficher que les messages de sa faction
  • Pas de mise en forme — Seuls les operateurs peuvent utiliser les couleurs (&)
  • Pas de notifications de mentions — Aucun signal quand quelqu'un vous cite
  • Interface rigide — GuiChat ne peut pas etre etendu sans Mixin
  • Pas d'emojis — Le chat reste purement textuel

Objectifs de la refonte

  • Canaux visibles et filtrants — 5 onglets, chacun filtre son canal
  • Preview en temps reel — Le joueur voit le rendu de son message avant d'envoyer
  • Mentions @ — Son de ping + highlight + notification pour la cible
  • Emojis — Shortcodes universels (:sword:, :heart:, etc.)
  • Messages prives integres — Onglet dedie, /msg, /r
  • Systeme d'ignore — Masquer les messages d'un joueur toxique
  • Performance — Traitement serveur efficace, stockage client-side propre
✓
Integration non destructive — Le nouveau systeme enveloppe les handlers existants (RankChatHandler, FactionChatHandler) au lieu de les remplacer. La logique de routage faction/ally deja codee est conservee et etendue.

02 Canaux de Chat

Le systeme comprend cinq canaux distincts, chacun avec sa portee, sa couleur et ses regles d'acces.

[G]
Global
Tous les joueurs. Aucune permission requise. Canal par defaut.
[F]
Faction
Membres de la faction uniquement. Necessite d'etre dans une faction.
[A]
Ally
Faction + factions alliees. Necessite une faction avec des allies.
[S]
Staff
Joueurs avec permission chat.staff. Communication interne equipe.
[PM]
Prives
Agrege tous les messages prives (envoyes et recus). Sans restriction.

Regles de routage

L'envoi d'un message suit la hierarchie suivante :

  1. Prefixe explicite — Si le message commence par ! (faction), @@ (ally) ou # (staff), il est route vers ce canal independamment de l'onglet actif.
  2. Canal actif — Sinon, le message est envoye dans le canal de l'onglet actuellement selectionne.
  3. Commande — /chat f, /msg <joueur>, etc. forcent le canal.
Canal Prefixe raccourci Commande Permission Portee
Global (aucun) /chat g Aucune Tous les joueurs
Faction ! au debut du message /chat f Etre dans une faction Membres de la faction
Ally @@ au debut du message /chat a Faction avec allies Faction + factions alliees
Staff # au debut du message /chat s chat.staff Joueurs avec chat.staff
Prive /msg <joueur> /msg, /pm, /r Aucune Destinataire unique
⚠
Priorite des prefixes — Le prefixe ! etant aussi utilise par certaines commandes, le systeme verifie en premier si le texte apres ! n'est pas une commande existante. Si c'est le cas, il laisse le comportement normal de Minecraft. Sinon, il route vers le canal faction.

Persistance du canal actif

Le canal actif d'un joueur est stocke dans son FactionPlayerData (champ chatMode deja existant). Ce champ est etendu pour prendre en charge les valeurs GLOBAL, FACTION, ALLY, STAFF. La persistance entre sessions est assurée par le DataManager EriAPI.

03 Format des Messages

Chaque message affiche dans le chat suit un format standardise qui inclut le canal, le grade, le pseudo et l'etiquette de faction. La construction finale utilise le pipeline RankChatHandler -> FactionChatHandler -> ChatManager.

Format general

[Canal] [Prefixe Grade] Pseudo [Tag Faction] » Message

Exemples par canal

// Global [G] [Admin] Steve [Vikings] » Bonjour tout le monde ! // Faction [F] [Membre] Alex [Vikings] » Raid ce soir a 20h // Ally [A] [Chef] Steve [Vikings -> Knights] » Reunion d'alliance // Staff [S] [Admin] Steve » Redemarrage dans 5 min // Prive (vu par l'expediteur) [PM] Steve -> Alex » Salut, tu veux trader ? // Prive (vu par le destinataire) [PM] Steve -> Vous » Salut, tu veux trader ?

Regles de construction

  • Tag du canal — Toujours en premiere position, couleur codee par canal (voir variables CSS)
  • Prefixe de grade — Fourni par RankManager via RankChatHandler (translateColors appele)
  • Pseudo — Colore avec la couleur du grade (rank.getColor())
  • Tag faction — Entre crochets, colore avec la couleur ARGB de la faction (argbToTextFormatting)
  • Separateur — Un fleche blanche §7» entre l'identite et le message
  • Message — Traite : codes & traduits si permission chat.color, emojis remplaces, mentions soulignees
Canal Couleur du tag Tag affiche Faction incluse
Global §f (blanc) [G] OUI
Faction §a (vert) [F] OUI
Ally §b (cyan) [A] OUI (+ cible)
Staff §e (jaune) [S] NON
Prive §d (mauve) [PM] NON

04 Systeme d'Onglets

L'interface chat dispose d'une barre d'onglets horizontale positionnee au-dessus de la saisie (rendue via le Mixin GuiChat). Chaque onglet correspond a un canal et filtre les messages affiches dans la fenetre de chat.

Comportement des onglets

  • Cliquer sur un onglet filtre l'affichage des messages ET bascule le canal d'envoi sur ce canal
  • L'onglet "Tout" affiche tous les messages de tous les canaux, codes par couleur
  • Un indicateur d'activite (point colore anime) apparait sur l'onglet quand il contient des messages non lus
  • L'onglet actif est mis en surbrillance (fond semi-transparent colore avec la couleur du canal)
  • Les onglets inaccessibles (ex : Faction sans faction, Staff sans permission) sont affiches en gris et desactives

Onglet "Tout" (All)

L'onglet special "Tout" n'est pas un canal d'envoi — il affiche uniquement. En mode "Tout", cliquer pour envoyer utilise le dernier canal actif avant de basculer sur "Tout". Un indicateur textuel discret indique le canal d'envoi courant : Envoi : Global.

Indicateurs non-lus

  • Chaque onglet maintient un compteur de messages non lus dans le ChatHistoryManager
  • Le compteur se remet a zero quand l'onglet est selectionne
  • Un petit point colore anime (pulse) apparait en haut a droite de l'onglet si compteur > 0
  • Pour les messages prives, le point est colore en mauve (--ch-private)
ℹ
Rendu EriAPI — La barre d'onglets est implementee avec des composants Button EriAPI en style outlined avec cornerRadius. Les indicateurs de non-lu utilisent un Circle avec l'animation component.breathe(40) pour le pulse. L'ensemble est rendu dans le Mixin GuiChat.

05 Preview Couleur en Temps Reel

La barre de saisie affiche au-dessus d'elle une ligne de preview qui montre en temps reel le rendu formate du message en cours de frappe. Cette fonctionnalite est particulierement utile pour les joueurs qui utilisent les codes couleur &.

Fonctionnement

  • A chaque frappe de touche, le texte saisi est passe dans ChatFormatter.previewFormat()
  • La methode applique la traduction des codes &0-9a-fk-or vers les codes Minecraft §
  • Elle remplace aussi les raccourcis emoji (:sword: → ⚔) pour la preview
  • Le resultat est affiche dans une Label EriAPI positionnee juste au-dessus de l'input, avec un fond semi-transparent
  • Si le message ne contient aucun code de formatage, la preview est masquee pour ne pas encombrer

Permission-gating

  • La preview couleur est visible par tous les joueurs, meme sans permission
  • Mais la traduction des codes & lors de l'envoi serveur n'est effectuee que si le joueur possede chat.color
  • Sans permission : les codes &x sont strippes du message (la methode stripColors du RankChatHandler est reutilisee)
  • La preview client-side affiche toujours les couleurs. Une mention discrete (couleurs non autorisees) est ajoutee si le joueur n'a pas la permission

Codes supportes

Code Effet Permission requise
&0 a &9Couleurs (noir, rouge, vert, jaune, bleu, etc.)chat.color
&a a &fCouleurs supplementaires (vert clair, cyan, rouge, mauve, blanc)chat.color
&lGraschat.color
&oItaliquechat.color
&nSoulignechat.color
&mBarrechat.color
&kObfusque (caracteres aleatoires)chat.color

⚠
&k obfusque — L'effet obfusque est autorise mais un cooldown de 30 secondes est applique apres l'envoi d'un message contenant &k pour eviter le spam visuel abusif. Ce cooldown est gere par ChatManager cote serveur.

06 Mentions (@)

Les mentions permettent d'alerter un joueur specifique en tapant @Pseudo dans le message. La cible recoit un signal sonore, une mise en evidence visuelle et une notification.

Detection des mentions

  • Le MentionHandler scanne le message apres traitement des couleurs/emojis
  • Toute sequence @NomJoueur est consideree comme une mention si NomJoueur correspond a un joueur en ligne (case-insensitive)
  • La detection s'arrete au premier espace ou caractere non-alphanumerique apres @
  • Les fausses mentions (joueur hors ligne ou inexistant) sont laissees telles quelles, non surlignees

Effets sur la cible mentionnee

  • Son de ping — SoundManager EriAPI joue un son "pling" (XP pickup vanilla, minecraft:entity.experience_orb.pickup)
  • Highlight visuel — Le @Pseudo est rendu avec un fond cyan/jaune dans le message affiche
  • Notification toast — NotificationManager EriAPI affiche un toast INFO de 3 secondes : "Steve vous a mentionne dans [canal]"
  • Indicateur de non-lu — L'onglet "Tout" montre un point d'alerte rouge si le joueur n'est pas sur l'onglet du canal ou la mention a ete faite

Mentions speciales

Mention Effet Permission
@everyone Mentionne tous les joueurs en ligne (ping + notification pour chacun) chat.mention.everyone
@faction Mentionne tous les membres en ligne de votre faction Etre dans une faction
@Pseudo Mentionne un joueur specifique Aucune

Autocomplete des mentions

  • Taper @ dans la barre de saisie puis appuyer sur TAB declenche l'autocomplete
  • L'autocomplete filtre les joueurs en ligne dont le pseudo commence par les lettres tapees apres @
  • Implemente via le Mixin GuiChat en interceptant la touche TAB
  • Les suggestions s'affichent dans une liste deroulante au-dessus de la saisie (meme style que l'autocomplete vanilla)
ℹ
Rendu du highlight — Le @Pseudo mentionne est rendu avec un background colore en utilisant le rendu manuel OpenGL dans le Mixin GuiNewChat (injection dans la methode de rendu de ligne). Techniquement un rectangle semi-transparent est dessine derriere la portion de texte correspondante.

Pipeline de remplacement

Le remplacement est effectue dans la methode ChatFormatter.replaceEmojis(String message) :

// Dans ChatFormatter.java public static String replaceEmojis(String message) { for (Map.Entry<String, String> entry : EmojiRegistry.getAll().entrySet()) { message = message.replace(entry.getKey(), entry.getValue()); } return message; } // Cas speciaux (texte colore) // :rage: → §cU+26A1§r (eclair rouge) // :gg: → §6§lGG§r (or + gras) // :rip: → §7§oRIP§r (gris + italique)

Autocomplete des emojis

  • Taper : dans la barre de saisie active le mode d'autocomplete emoji
  • Chaque caractere suivant filtre la liste des emojis dont le shortcode contient la chaine saisie
  • Appuyer sur TAB complete avec le premier emoji de la liste filtree
  • Appuyer sur TAB a nouveau cycle sur les emojis suivants
  • Les suggestions affichent le caractere emoji, son shortcode et une courte description
✓
Ajout d'emojis — Le registre EmojiRegistry est un simple LinkedHashMap<String, String> statique. Ajouter un emoji ne requiert qu'une seule ligne. Les shortcodes speciaux (texte colore) utilisent une map separee SPECIAL_EMOJIS avec la chaine de couleur complete comme valeur.

08 Messages Prives

Le systeme de messages prives ameliore le /tell vanilla avec un historique dedie, une commande de reponse rapide et des notifications sonores.

Commandes

Commande Alias Description
/msg <joueur> <message> /tell, /pm, /whisper Envoie un message prive
/r <message> /reply Repond au dernier MP recu

Format d'affichage

// Vu par l'expediteur : [PM] Vous -> Alex » Salut, tu veux trader ? // Vu par le destinataire : [PM] Steve -> Vous » Salut, tu veux trader ?

Notifications

  • Son — SoundManager.playNotification() est joue pour le destinataire
  • Toast EriAPI — Un toast INFO de type "Message de Steve" apparait en jeu via NotificationManager
  • Onglet Prives — Le compteur de non-lus de l'onglet "Prives" est incremente

Persistance du destinataire courant

  • Le dernier joueur ayant envoye un MP est stocke en ChatHistoryManager.lastPmSender (UUID)
  • Le dernier joueur a qui j'ai envoye un MP est stocke en lastPmRecipient
  • /r repond vers lastPmSender (le dernier qui m'a ecrit)
  • Si aucun MP n'a ete echange, /r affiche : "Aucune conversation recente."
  • Si le joueur se deconnecte, le destinataire est efface. Tenter /r affiche : "<Joueur> n'est plus connecte."
⚠
Joueur hors ligne — Si le destinataire d'un /msg est hors ligne, le message n'est pas delivre ni mis en attente. Un message d'erreur est renvoye a l'expediteur : "§c<Joueur> n'est pas connecte."

09 Selecteur d'Emojis (Popup GUI)

Un petit bouton emoji est affiche a droite de la barre de saisie. Un clic ouvre une fenetre popup flottante presentant une grille de tous les emojis disponibles.

UI du selecteur

  • Bouton d'ouverture — Un petit bouton carre ☺ (16x16 design px) a droite de l'input, style TexturedButton EriAPI ou Button outlined violet
  • Popup — Fenetre ContainerComponent EriAPI de 320x240 design px, style Liquid Glass (fond semi-transparent, backdrop blur, border subtile)
  • Grille d'emojis — GridLayout EriAPI de 5 colonnes avec espacement de 4px. Chaque cellule affiche le caractere emoji et son shortcode au hover (Tooltip)
  • Cliquer un emoji insere son shortcode dans la barre de saisie a la position du curseur
  • Fermeture — Clic en dehors du popup, touche Echap, ou re-clic sur le bouton emoji

Animation

  • Ouverture : popup.scaleIn(8) depuis le bouton emoji (origine en bas a droite)
  • Fermeture : popup.scaleOut(6)
  • Les cellules de la grille utilisent cell.pulse(5) au clic pour feedback visuel
ℹ
Classe EmojiPickerPopup — Cette classe etend ContainerComponent EriAPI. Elle est instanciee une seule fois et sa visibilite est toggle. Le popup est positionne en haut a droite de la barre de saisie en coordonnees 1920x1080.

10 Historique & Filtrage Client-Side

Chaque message est stocke client-side avec ses metadonnees de canal. Le filtrage par onglet est instantane et local — aucune requete serveur n'est necessaire.

Structure d'un message stocke

public class ChatMessage { public final ChatChannel channel; // GLOBAL, FACTION, ALLY, STAFF, PRIVATE public final ITextComponent rendered; // Composant texte Minecraft rendu public final long timestamp; // System.currentTimeMillis() public final UUID senderUuid; // null pour messages systeme public final boolean isMention; // true si ce joueur y est mentionne }

ChatHistoryManager

  • Singleton client-side (@SideOnly(Side.CLIENT))
  • Stocke jusqu'a 500 messages par canal (FIFO, le plus ancien est supprime quand la limite est atteinte)
  • L'onglet "Tout" utilise une liste unifiee de 1000 messages maximum
  • Chaque canal a son propre offset de scroll independant
  • Les compteurs de non-lus sont stores par canal dans une Map<ChatChannel, Integer>

Injection des messages

Le Mixin MixinGuiNewChat intercepte la methode printChatMessageWithOptionalDeletion de Minecraft. A chaque nouveau message, avant le rendu vanilla :

  1. Extraction du canal depuis le tag de prefixe du composant texte (ex : [G], [F], etc.)
  2. Detection si le message contient une mention du joueur local
  3. Creation d'un ChatMessage POJO et ajout dans le ChatHistoryManager
  4. Increment du compteur de non-lus si l'onglet actif n'est pas celui du message
  5. Si l'onglet actif est "Tout" ou le canal du message : laisser le rendu vanilla s'executer normalement
  6. Sinon : annuler le rendu vanilla via ci.cancel() (le message est stocke mais non affiche)

Scroll independant par onglet

  • La position de scroll de chaque onglet est sauvegardee en memoire
  • Changer d'onglet restaure la position de scroll de ce canal
  • Recevoir un nouveau message sur un onglet non actif ne modifie pas sa position de scroll

11 Systeme d'Ignore

Un joueur peut masquer tous les messages d'un autre joueur via /ignore. Cette preference est persistee cote serveur et appliquee a la reception du message (le message n'est pas envoye au joueur ignorant).

Persistance

  • La liste d'ignores est stockee dans un nouveau champ List<String> ignoredPlayers dans FactionPlayerData (UUIDs sous forme de String)
  • Le champ n'est PAS marque @Sync — il reste serveur-side uniquement
  • La persistence est assuree par le DataManager EriAPI (NBT, persistOnDeath = false)

Application du filtre

  • Dans ChatEventHandler.onServerChat(), avant d'envoyer le message a chaque destinataire, l'UUID de l'expediteur est verifie contre la liste d'ignores du destinataire
  • Si l'expediteur est ignore, le message n'est pas transmis a ce destinataire
  • L'ignore s'applique a tous les canaux (Global, Faction, Ally) mais pas aux messages systeme ni aux messages Staff

Limites

  • Maximum 50 joueurs ignores par joueur (protege contre l'abus de memoire)
  • Ignorer un membre de sa propre faction affiche un avertissement mais est autorise
  • Les admins avec chat.staff ne peuvent pas etre ignores (leurs messages Staff passent toujours)
ℹ
Pas de notification — Quand un joueur est ignore, l'expediteur ne recoit aucune notification et croit que son message a bien ete envoye. Cela evite les conflits en jeu.

12 GUI Chat Custom (Mixin + EriAPI)

Le remplacement du chat vanilla requiert deux Mixins distincts : un sur l'ecran de saisie (GuiChat) et un sur le rendu de l'historique (GuiNewChat).

MixinGuiChat — Ecran de saisie

Ce Mixin injecte du code dans GuiChat.initGui(), GuiChat.drawScreen() et GuiChat.keyTyped() pour ajouter :

  • La barre d'onglets (composants Button EriAPI) au-dessus de l'input
  • La ligne de preview (composant Label EriAPI) entre l'historique et l'input
  • Le bouton du selecteur d'emojis a droite de l'input
  • L'interception de TAB pour l'autocomplete (@mentions et :emojis)
  • La gestion du clic sur les onglets (mise a jour de l'onglet actif dans ChatHistoryManager)
@Mixin(GuiChat.class) public class MixinGuiChat { @Inject(method = "initGui", at = @At("RETURN")) private void onInitGui(CallbackInfo ci) { // Initialiser les composants EriAPI (tabs, preview, emoji btn) GuiChatCustom.INSTANCE.initComponents(this); } @Inject(method = "drawScreen", at = @At("RETURN")) private void onDrawScreen(int mouseX, int mouseY, float pt, CallbackInfo ci) { // Rendre les composants EriAPI par-dessus le chat vanilla GuiChatCustom.INSTANCE.render(mouseX, mouseY, pt); } @Inject(method = "keyTyped", at = @At("HEAD"), cancellable = true) private void onKeyTyped(char typedChar, int keyCode, CallbackInfo ci) { if (GuiChatCustom.INSTANCE.handleKey(typedChar, keyCode)) { ci.cancel(); // Intercepte TAB pour autocomplete } } }

MixinGuiNewChat — Historique et rendu

Ce Mixin intercepte GuiNewChat.printChatMessageWithOptionalDeletion() pour :

  • Extraire le canal et creer le ChatMessage dans le ChatHistoryManager
  • Annuler le rendu si le message n'appartient pas a l'onglet actif
  • Rendre les highlights de mentions (@Pseudo souligne/colore) sur les messages visibles

Composants EriAPI utilises

Composant Utilisation Position design (1920x1080)
Button (x5) Onglets Global, Faction, Ally, Staff, Prives + Tout Barre en bas, au-dessus de l'input
Label Ligne de preview couleur Au-dessus de l'input, sous les onglets
Button Bouton selecteur d'emojis A droite de la barre d'input vanilla
ContainerComponent EmojiPickerPopup Popup flottant au-dessus du bouton emoji
Circle Indicateurs de non-lus En haut a droite de chaque onglet
⚠
Coordonnees Mixin vs EriAPI — Les composants EriAPI sont en coordonnees 1920x1080 (ScaleManager auto). Mais les positions calculees dans le contexte d'un Mixin GuiChat (drawScreen) doivent utiliser ScaleManager.toScreen(x) pour convertir vers les coordonnees ecran reelles lors du positionnement relatif aux elements vanilla.

13 Traitement Serveur (Pipeline)

Toute la logique de traitement des messages est centralisee dans ChatEventHandler, qui intercepte le ServerChatEvent de Forge avec une priorite haute pour prendre la main avant les handlers existants.

Ordre de traitement

ServerChatEvent (EventPriority.HIGH) | v 1. Detection du canal (prefixe !, @@, # ou chatMode du joueur) | v 2. Traitement des codes couleur & (RankChatHandler.translateColors si chat.color) | v 3. Remplacement des emojis (ChatFormatter.replaceEmojis) | v 4. Detection des mentions @ (MentionHandler.detectMentions) | v 5. Construction du message final (ChatManager.buildMessage : tag canal + rank + faction) | v 6. Determination des destinataires (selon canal : tous, membres faction, allies, staff) | v 7. Filtrage ignore (retrait des joueurs qui ont ignore l'expediteur) | v 8. Envoi aux destinataires (player.sendMessage pour chacun) | v 9. Notifications de mentions (son + toast EriAPI via EriScheduler.delay(1, ...) pour eviter NPE) | v 10. Annulation du ServerChatEvent (event.setCanceled(true) — on gere l'envoi nous-memes)

Interaction avec les handlers existants

Le nouveau ChatEventHandler utilise la priorite EventPriority.HIGH. Les handlers existants (RankChatHandler et FactionChatHandler) utilisent la priorite par defaut (NORMAL). Comme le nouvel handler annule l'event a la fin, les anciens handlers ne s'executent plus pour les messages normaux.

La logique des anciens handlers est appelee directement depuis le nouveau pipeline plutot que via les events Forge — ceci garantit l'ordre d'execution et evite les conflits.

⚠
Ne pas supprimer les anciens handlers — RankChatHandler et FactionChatHandler restent dans le codebase et peuvent etre utiles pour des cas edge. Cependant, leurs methodes translateColors(), stripColors() et argbToTextFormatting() sont desormais utilisees comme utilitaires statiques appeles par ChatFormatter et ChatManager.

14 Architecture Technique

Structure des packages

fr.eriniumgroup.eriniumfaction.chat/ ├── ChatChannel.java // Enum : GLOBAL, FACTION, ALLY, STAFF, PRIVATE ├── ChatManager.java // Singleton serveur : pipeline, routage, construction ├── ChatMessage.java // POJO : channel, rendered, timestamp, senderUuid, isMention ├── ChatHistoryManager.java // @SideOnly(CLIENT) : stockage, filtrage, scroll, non-lus ├── ChatFormatter.java // translateColors, stripColors, replaceEmojis, previewFormat ├── EmojiRegistry.java // Map statique shortcode → unicode/couleur ├── MentionHandler.java // detectMentions, notifyMentioned, highlightInComponent ├── IgnoreManager.java // isIgnored, addIgnore, removeIgnore (delègue au DataManager) ├── gui/ │ ├── GuiChatCustom.java // Composants EriAPI : tabs, preview, emoji btn (singleton) │ └── EmojiPickerPopup.java // ContainerComponent EriAPI : grille emoji cliquable ├── command/ │ ├── ChatCommand.java // /chat [g|f|a|s] — changer de canal actif │ ├── MsgCommand.java // /msg, /pm, /r — messages prives │ ├── IgnoreCommand.java // /ignore, /unignore │ └── EmojiCommand.java // /emoji list ├── event/ │ └── ChatEventHandler.java // @SubscribeEvent ServerChatEvent (HIGH), pipeline complet ├── mixin/ │ ├── MixinGuiChat.java // Injection initGui, drawScreen, keyTyped │ └── MixinGuiNewChat.java // Injection printChatMessage, rendu historique └── config/ └── ChatConfig.java // Tous les toggles et valeurs numeriques

Responsabilites des classes cles

Classe Side Responsabilite
ChatManager Serveur Orchestrateur principal : recoit un message brut, renvoie le composant formate a envoyer a chaque destinataire
ChatFormatter Les deux Transformations pures : couleurs, emojis, preview. Pas d'etat, methodes statiques
ChatHistoryManager Client Memoire client : stocke, filtre, compte les non-lus, gere le scroll par canal
MentionHandler Serveur Scanne le texte, trouve les UUIDs des joueurs mentionnes, schedule les notifications
IgnoreManager Serveur Facade CRUD simple sur le champ ignoredPlayers du FactionPlayerData
EmojiRegistry Les deux Source de verite des emojis. Initialise au demarrage. Map immuable apres init
GuiChatCustom Client Singleton qui detient les composants EriAPI injectes dans GuiChat via Mixin

Enregistrement Mixin

Les deux nouvelles classes Mixin sont ajoutees dans mixins.eriniumfaction.json section "client" :

"client": [ "chat.MixinGuiChat", "chat.MixinGuiNewChat" ]
✓
Convention de nommage — Les mixins du sous-package chat sont references avec leur sous-package dans le JSON. Le refmap est genere automatiquement par l'annotation processor Mixin pour la production obfusquee.

15 Commandes

/chat

Sous-commande Alias Action Permission
/chat global /chat g Bascule le canal actif sur Global Aucune
/chat faction /chat f Bascule sur Faction (erreur si sans faction) Etre dans une faction
/chat ally /chat a Bascule sur Ally (erreur si sans allies) Faction avec allies
/chat staff /chat s Bascule sur Staff (erreur si sans permission) chat.staff

Auto-completion : La sous-commande suggere global, g, faction, f, ally, a, staff, s selon les canaux accessibles au joueur (filtre dynamique).

/msg, /pm, /r

Commande Syntax Auto-completion
/msg /msg <joueur> <message...> Noms des joueurs en ligne pour <joueur>
/pm Alias de /msg Idem
/tell Alias de /msg Idem
/r /r <message...> Aucune (message libre)

/ignore, /unignore

Commande Description Auto-completion
/ignore <joueur> Ajoute le joueur a la liste d'ignores Joueurs en ligne
/unignore <joueur> Retire le joueur de la liste d'ignores Joueurs actuellement ignores
/ignore list Affiche la liste de ses ignores Aucune

/emoji

Commande Description
/emoji list Affiche la liste paginee de tous les emojis disponibles (avec pagination EriAPI ChatPaginator, 10 par page)

Cle Type Valeur par defaut Description

chatEmojiEnabled boolean true Active le remplacement des shortcodes emoji

17 Integration avec les Systemes Existants

RankChatHandler

Ce handler conserve son annotation @Mod.EventBusSubscriber mais son @SubscribeEvent onChat n'est plus execute dans les flux normaux car le nouveau ChatEventHandler annule l'event en priorite HAUTE. Ses methodes statiques sont directement appelees :

  • RankChatHandler.translateColors(input) — appele dans ChatFormatter
  • RankChatHandler.stripColors(input) (rendu accessible ou recopie dans ChatFormatter)
  • La logique de formatage du grade (prefix, color, suffix) est appelee dans ChatManager.buildMessage()

FactionChatHandler

De meme, ce handler ne s'execute plus via events. Sa logique est integree :

  • La detection des membres en ligne (getOnlineMembers) est reutilisee dans ChatManager.getRecipients(FACTION)
  • La logique de routing ally (parcours des factions alliees) est reutilisee dans ChatManager.getRecipients(ALLY)
  • argbToTextFormatting est deplacee dans ChatFormatter comme methode statique publique

ChatMode enum

L'enum ChatMode existant (DEFAULT, FACTION, ALLY) est etendu avec GLOBAL et STAFF pour correspondre aux nouvelles valeurs :

public enum ChatMode { GLOBAL("eriniumfaction.chatmode.global"), // Remplace DEFAULT FACTION("eriniumfaction.chatmode.faction"), ALLY("eriniumfaction.chatmode.ally"), STAFF("eriniumfaction.chatmode.staff"); // Nouveau // DEFAULT est conserve comme alias de GLOBAL pour retrocompat }

FactionPlayerData

Le champ chatMode existant (String) est utilise pour stocker le canal actif. Le nouveau champ ignoredPlayers (List<String>) est ajoute dans la meme classe :

@EriData(modId = "eriniumfaction", scope = DataScope.PLAYER, persistOnDeath = false) public class FactionPlayerData { // ... champs existants ... public String chatMode = "GLOBAL"; // Etendu : + STAFF public List<String> ignoredPlayers = new ArrayList<>(); // Nouveau public UUID lastPmSender = null; // Nouveau (non-persistent) }

EriAPI

  • EriChat — utilise pour construire les messages riches (mentions hover, boutons de clic)
  • NotificationManager / OverlayToast — pour les toasts de mention et MP
  • EriScheduler — pour les notifications asynchrones (delay 1 tick pour eviter le contexte d'event)
  • Composants GUI Button, Label, ContainerComponent, GridLayout, Circle — pour la barre d'onglets et le popup emoji

18 Permissions

Toutes les permissions suivantes doivent etre ajoutees dans docs/permissions.md apres implementation.

Permission Description Grade par defaut
chat.color Autorise l'utilisation des codes couleur & dans les messages VIP et superieur
chat.staff Acces au canal Staff (/chat s, prefixe #) Modérateur et superieur
chat.mention.everyone Autorise @everyone (mention de tous les joueurs) Admin uniquement
chat.mention.faction Autorise @faction (mention de tous les membres de faction) Chef de faction et superieur
chat.bypass.ignore Les messages de ce joueur ne peuvent pas etre ignores Admin uniquement
chat.obfuscate Autorise l'utilisation du code &k (obfusque) VIP et superieur (avec chat.color)
ℹ
Verification — Toutes les permissions sont verifiees via RankManager.getInstance().hasPermission(player, "chat.xxx"), cohrent avec le systeme de grades existant.

19 Assets

Icones d'onglets (optionnelles)

Les onglets peuvent afficher soit du texte seul (Global, Faction, etc.) soit une icone 16x16 si les assets sont disponibles. Dans la version initiale, le texte seul est utilise pour eviter le blocage sur les assets.

  • tab_global.png — Icone bulle de dialogue (16x16)
  • tab_faction.png — Icone bouclier (16x16)
  • tab_ally.png — Icone lien / chaine (16x16)
  • tab_staff.png — Icone etoile / badge (16x16)
  • tab_private.png — Icone enveloppe (16x16)

Ces assets sont declares dans src/main/resources/assets/eriniumfaction/textures/gui/chat/ selon le workflow defini dans docs/ASSETS_WORKFLOW.md.

Bouton emoji

Si un asset texturé est desire : emoji_btn.png de 16x16 montrant un smiley stylise en accord avec la palette violet/cyan du mod. En l'absence d'asset, le composant Button affiche le caractere ☺ en UTF-8.

Sons

Le systeme n'introduit pas de nouveaux sons custom — il reutilise les sons vanilla :

  • Mention ping : minecraft:entity.experience_orb.pickup (volume 0.5, pitch 1.5)
  • MP recu : minecraft:entity.experience_orb.pickup (volume 0.7, pitch 1.0)

20 Plan de Tests

Tests fonctionnels — Canaux

  • Envoyer un message global → verifier reception par tous les joueurs
  • Envoyer un message faction → verifier reception uniquement par les membres, non-reception par les autres
  • Envoyer un message ally → verifier reception par faction + factions alliees, non-reception par ennemis/neutres
  • Envoyer un message staff sans permission → verifier echec avec message d'erreur
  • Utiliser les prefixes raccourcis (!, @@, #) → verifier routage correct
  • Basculer de canal avec /chat f alors que sans faction → verifier message d'erreur

Tests — Formatage

  • Message avec &6Hello sans permission chat.color → verifier stripping des codes
  • Message avec &6Hello avec permission chat.color → verifier couleur or dans le chat
  • Preview en temps reel : taper &6Bonjour → verifier ligne preview coloree
  • Message avec :sword: → verifier remplacement par ⚔ dans le chat
  • Message avec :gg: → verifier texte GG en or+gras
  • Message avec :shortcode_inexistant: → verifier qu'il reste tel quel

Tests — Mentions

  • Taper @Steve (joueur en ligne) → verifier son de ping pour Steve, toast de notification, highlight dans le message
  • Taper @JoueurInexistant → verifier absence de notification, texte non-souligne
  • Taper @everyone sans permission → verifier erreur de permission
  • Taper @everyone avec permission → verifier ping pour tous les joueurs en ligne
  • Autocomplete @ + TAB → verifier liste de joueurs en ligne

Tests — Messages Prives

  • /msg Steve Bonjour → verifier reception par Steve avec format [PM]
  • /r Reponse (apres avoir recu un MP) → verifier envoi au bon destinataire
  • /r sans MP precedent → verifier message "Aucune conversation recente"
  • /msg SteveHorsLigne message → verifier erreur "Joueur non connecte"
  • Reception MP → verifier son + toast + indicateur onglet Prives

Tests — Ignore

  • /ignore Steve → verifier que les messages de Steve n'arrivent plus
  • /ignore Steve dans faction partagee → verifier que messages faction de Steve ne arrivent plus
  • /unignore Steve → verifier retablissement de reception
  • /ignore list → verifier affichage correct de la liste
  • Steve ignore, Steve envoie un message → verifier que Steve ne sait pas qu'il est ignore
  • Admin avec chat.bypass.ignore → verifier que ses messages passent quand meme

Tests — GUI Onglets

  • Ouvrir le chat (T) → verifier presence de la barre d'onglets
  • Cliquer sur l'onglet Faction → verifier filtrage des messages (seuls les messages [F] visibles)
  • Recevoir un message sur un autre onglet → verifier apparition du point non-lu
  • Cliquer sur l'onglet ayant le point non-lu → verifier disparition du point
  • Cliquer sur le bouton emoji → verifier ouverture du popup
  • Cliquer sur un emoji dans le popup → verifier insertion du shortcode dans l'input
  • Appuyer sur Echap → verifier fermeture du popup sans fermer le chat

Tests — Performance

  • Spam de 100 messages/seconde → verifier absence de lag (pipeline asynchrone si necessaire)
  • 500 messages dans l'historique d'un canal → verifier que le 501e supprime le plus ancien
  • 10 joueurs avec @everyone simultanes → verifier pas de NPE, ordre de traitement correct
  • Scrollback de 500 messages avec onglet switch → verifier restauration de position fluide
ℹ
Environnement de test — Tous les tests sont effectues en mode dev (./gradlew runClient) avec au moins 2 clients connectes au serveur integre ou a un serveur local separe. Les tests de performance necessitent un profiler (VisualVM ou JProfiler avec JDK 8).
