Hotel des Ventes (HDV)

Systeme economique central permettant aux joueurs de vendre et acheter des items contre Dollars et Eris, avec vente directe, systeme de Lucky Blocks, encheres, et statistiques de marche en temps reel.

Planifie Dual Currency ($+Eris) Lucky Blocks Encheres Stats Marche Integration Mail Anti-Duplication

01 Contexte & Objectifs

Vision du systeme

Le HDV est la place de marche centralisee du serveur. Il remplace les shops manuels joueur-a-joueur par un systeme automatise, equitable et transparent. Les joueurs deposent des items a vendre, fixent un prix, et les acheteurs parcourent le catalogue. L'argent et les items sont echanges automatiquement.

Le HDV propose quatre modes de commerce distincts :

  • Vente directe (Marche) : deposer des items a un prix fixe par unite. L'acheteur choisit la quantite et paie immediatement.
  • Lucky Block : le vendeur remplit une grille 3x9 avec des items, fixe un prix par bloc. L'acheteur achete un bloc mystere contenant un item aleatoire de la grille. Mecanique de gacha/loterie integree.
  • Encheres : le vendeur met un item aux encheres avec un prix de depart et une duree. Les joueurs surencherissent. Le plus offrant gagne a la fin du timer.
  • Statistiques : le joueur place un item pour voir un graphe des ventes des 7 derniers jours (quantite vendue x prix unitaire moyen).

Objectifs de design

  • Economie saine : le HDV est le pilier de l'economie. Toutes les transactions sont tracees et les prix visibles favorisent l'equilibre offre/demande.
  • Double monnaie : chaque vente accepte Dollars OU Eris (pas les deux en meme temps). Le vendeur choisit la devise. Cela donne de la valeur aux deux monnaies.
  • Fun via Lucky Blocks : le systeme de Lucky Block ajoute une dimension ludique/gacha. Les joueurs peuvent tenter leur chance pour obtenir des items rares a prix reduit.
  • Transparence : les stats de marche permettent aux joueurs de fixer des prix justes bases sur l'historique reel des ventes.
  • Anti-scam : tout passe par le serveur. Pas de triche possible cote client.

Ce que le systeme ne fait pas

  • Pas de vente en personne : le HDV n'a pas de "shop physique" dans le monde. C'est un GUI accessible partout.
  • Pas de negociation : les prix sont fixes (sauf encheres). Pas de systeme d'offre/contre-offre.
  • Pas de credit : le joueur doit avoir l'argent disponible au moment de l'achat. Pas de paiement differe.
  • Pas de taxe : en V1, pas de commission prelevee par le systeme. A ajouter en V2 si necessaire pour le sink d'argent.

Stockage — WorldSavedData

// Acces au stockage AuctionWorldData data = (AuctionWorldData) world.getMapStorage() .getOrLoadData(AuctionWorldData.class, "erinium_hdv"); if (data == null) { data = new AuctionWorldData("erinium_hdv"); world.getMapStorage().setData("erinium_hdv", data); } // Structure NBT interne // AuctionWorldData contient des sous-maps : // - "listings" : NBTTagList de HDVListing serialises // - "luckyblocks": NBTTagList de HDVLuckyBlock serialises // - "auctions" : NBTTagList de HDVAuction serialises // - "stats" : NBTTagCompound avec sous-cles par jour (7 jours max) // Les ItemStacks sont serialises via ItemStack.writeToNBT()/new ItemStack(nbt)

Le HDV utilise un AuctionWorldData extends WorldSavedData unique stocke dans le world save. Toutes les donnees (listings, lucky blocks, encheres, historique des prix) sont dans un seul NBT compound. Le chargement complet en memoire au demarrage est acceptable car le volume est borne (limites par joueur).

Raison du choix WorldSavedData :

  • Reset saisonnier : quand le monde est reset pour une nouvelle saison, toutes les donnees HDV sont automatiquement wipees avec le monde. Aucun nettoyage manuel necessaire. Les joueurs ne peuvent PAS conserver des listings ou des items HDV entre saisons.
  • Coherence economique : le HDV fait partie de l'economie du monde. Un reset de monde = un reset complet du marche.
  • Serialisation NBT native : les ItemStacks sont sauvegardes avec ItemStack.writeToNBT() / new ItemStack(nbt), le format le plus fiable pour Minecraft.
⚠
Sauvegarde : Appeler auctionWorldData.markDirty() apres chaque modification. WorldSavedData est sauvegarde automatiquement par Minecraft lors des sauvegardes monde periodiques et a l'arret du serveur. Pour les operations critiques (achat, enchere), appeler world.getMapStorage().saveAllData() immediatement pour forcer la persistence.

Popup d'achat

Quand le joueur clique [Acheter], une popup apparait :

  • Affichage : icone de l'item, nom, prix unitaire, vendeur.
  • Champ quantite : NumericField (EriAPI) avec min=1, max=quantite disponible. Boutons rapides x1, x16, x64, MAX.
  • Prix total : calcule en temps reel (quantite x prix unitaire). Affiche en gros.
  • Solde joueur : affiche le solde actuel dans la devise concernee.
  • Bouton Confirmer : grise si solde insuffisant. Texte "Acheter pour $X" ou "Acheter pour XE".

Popup de mise en vente

Quand le joueur clique [+ VENDRE UN ITEM] :

  • Selection item : le joueur clique sur un item de son inventaire (rendu dans le GUI, pas un Container Forge). L'item est mis en surbrillance.
  • Quantite : NumericField, max = quantite possedee de cet item exact (meme ID + meme NBT).
  • Prix unitaire : NumericField (min=1). Le joueur fixe le prix PAR UNITE.
  • Devise : RadioGroup (Dollars / Eris).
  • Prix total affiche : quantite x prix unitaire (informatif seulement).
  • Bouton Confirmer : retire les items de l'inventaire du joueur et cree la listing.
ⓘ
Retrait des items : Les items sont retires de l'inventaire du joueur cote SERVEUR au moment de la creation de la listing. Le serveur verifie que le joueur possede bien les items en quantite suffisante. Cela previent la duplication.

Recherche et filtres

  • Recherche : TextField avec recherche par nom d'item (localized). Filtrage cote serveur pour eviter d'envoyer toutes les listings au client.
  • Filtre devise : Dollars, Eris, ou Tous.
  • Tri : Prix croissant, Prix decroissant, Plus recent, Plus ancien.
  • Categorie : Armes, Armures, Outils, Blocs, Materiaux, Nourriture, Potions, Divers. Basees sur les classes d'items Forge (ItemSword, ItemArmor, etc.).
  • Pagination : 10 listings par page. Navigation Prev/Next.

05 Page Lucky Block

Concept

Le Lucky Block est une mecanique de gacha integree au HDV. Le vendeur cree un "Lucky Block" en remplissant une grille de 3 lignes x 9 colonnes (27 slots) avec des items de son inventaire. Il fixe un prix par "bloc". Les acheteurs paient le prix d'un bloc et recoivent un item aleatoire parmi ceux de la grille.

Creation d'un Lucky Block (vendeur)

+====================================================================+ | CREER UN LUCKY BLOCK | |====================================================================| | | | Remplissez la grille avec les items a mettre en jeu : | | | | [Dia][Dia][Fer][Fer][Fer][Or ][Or ][ ][ ] Ligne 1 | | [Fer][Fer][Fer][Fer][Or ][Or ][Or ][ ][ ] Ligne 2 | | [Or ][Or ][Or ][Or ][Or ][Boi][Boi][ ][ ] Ligne 3 | | | | Slots remplis : 18/27 | | Prix par bloc : [$______] Devise : [$ / Eris] | | | | Revenu max estimee : $X * 18 blocs = $Y | | | | [ CONFIRMER ] | +====================================================================+
  • Grille : le joueur clique sur un slot vide de la grille, puis clique sur un item de son inventaire pour le placer. Clic droit sur un slot rempli pour le retirer.
  • Minimum : au moins 3 slots doivent etre remplis pour creer un Lucky Block.
  • Items retires : les items sont retires de l'inventaire au moment de la confirmation (serveur).
  • Prix par bloc : le vendeur fixe le prix que chaque acheteur paiera pour un tirage.
  • Nombre de blocs : egal au nombre de slots remplis. Chaque bloc correspond a exactement un slot.

Achat d'un Lucky Block (acheteur)

  • Vue liste : les Lucky Blocks actifs sont listes avec : nom du vendeur, prix par bloc, nombre de blocs restants, et un apercu des items possibles.
  • Apercu : l'apercu montre les items qui ETAIENT dans la grille a la creation (previewSlots). Ce preview est fige et ne change JAMAIS apres la creation, meme quand des blocs sont achetes. Les acheteurs ne savent donc pas quels items ont deja ete gagnes.
  • Achat : le joueur clique "Acheter un bloc", paie le prix, et recoit un item aleatoire parmi les slots restants.
  • Livraison : l'item gagne est ajoute directement a l'inventaire du joueur. Si l'inventaire est plein, l'item est envoye par mail.
  • Animation : une animation de "roulette" montre les items defiler avant de s'arreter sur l'item gagne (purement cosmetique cote client, le resultat est deja decide cote serveur).

Regles du tirage

  • Selection aleatoire : le serveur choisit un slot aleatoire parmi les slots NON ENCORE GAGNES. Chaque slot a une probabilite egale (1/N ou N = slots restants).
  • Pas de remplacement : une fois un slot gagne, il est retire du pool. Le dernier acheteur recoit le dernier item restant avec certitude.
  • Preview inchange : le preview public montre toujours les 27 slots originaux. L'acheteur voit "Diamant, Fer, Or..." mais ne sait pas si le Diamant a deja ete gagne.
⚠
Anti-triche Lucky Block : Le tirage aleatoire DOIT utiliser SecureRandom (pas Random) pour eviter la prediction. Le resultat est decide cote serveur AVANT d'envoyer l'animation au client. Le client ne recoit que le resultat final, pas le seed.

06 Page Encheres

Interface

+====================================================================+ | ENCHERES ACTIVES | |====================================================================| | | | [Epee Dia SV] | Actuelle: $1,200 | PvPKing | Fin: 2h34m | [Encherir]| | [Armure P4] | Actuelle: 85E | - | Fin: 5h12m | [Encherir]| | [Totem] | Actuelle: $3,500 | Rich01 | Fin: 23m | [Encherir]| | | | [+ METTRE AUX ENCHERES] | +====================================================================+

Creation d'une enchere

  • Item : le joueur selectionne un item de son inventaire (un seul stack).
  • Prix de depart : NumericField, minimum 1.
  • Devise : RadioGroup (Dollars / Eris).
  • Duree : choix parmi des presets : 1h, 6h, 12h, 24h, 48h, 72h.
  • Confirmation : l'item est retire de l'inventaire. L'enchere est creee.
  • Non annulable : une enchere ne peut PAS etre annulee par le vendeur une fois creee. Cela protege les encherisseurs.

Surenchere

  • Montant minimum : la surenchere doit etre au moins currentBid + minBidIncrement. L'increment minimum est configurable (defaut: 10% du prix actuel, minimum 1).
  • Debit immediat : le montant de l'enchere est debite immediatement du compte du joueur. Cela garantit que l'encherisseur a les fonds.
  • Remboursement automatique : quand un joueur est surencheri, son montant est rembourse immediatement (si en ligne) ou par mail (si hors ligne).
  • Extension de duree : si une enchere est placee dans les 5 dernieres minutes, la duree est etendue de 5 minutes (anti-snipe). Maximum 3 extensions par enchere.

Fin de l'enchere

  • Gagnant : l'item est envoye au gagnant par mail (type AUCTION, message "Enchere gagnee — [Item]").
  • Vendeur : l'argent (montant de l'enchere finale) est envoye au vendeur par mail (type AUCTION, message "Votre enchere pour [Item] s'est terminee — +$X").
  • Aucun encherisseur : si personne n'a encheri, l'item est renvoye au vendeur par mail (type AUCTION, message "Votre enchere pour [Item] n'a recu aucune offre").
  • Verification : la tache HDVAuctionTask verifie chaque seconde (20 ticks) si des encheres ont expire et les traite.
ⓘ
Anti-snipe : L'extension de 5 minutes empeche les joueurs de placer une enchere a la derniere seconde pour gagner sans laisser le temps aux autres de reagir. Avec un maximum de 3 extensions, une enchere peut durer au plus 15 minutes de plus que prevu.
🔧
Implementation — livraison hors ligne : utiliser MailManager.getInstance().sendMail(UUID, ...) pour envoyer les items et l'argent aux joueurs hors ligne. Le MailManager fonctionne par UUID et ne necessite pas que le joueur soit connecte. Les commandes admin (/mailadmin, /rank, /money) supportent egalement les UUID pour cibler des joueurs hors ligne.

12 Cas Limites & Securite

Anti-duplication

  • Retrait serveur : les items sont retires de l'inventaire du joueur cote SERVEUR quand il cree une listing/lucky/enchere. Le client n'a aucun controle.
  • Verification de possession : le serveur verifie le slot d'inventaire specifie et compare l'item avec ce que le client pretend vendre. Si ca ne correspond pas, refus.
  • Race condition achat : les achats sont synchronises par un lock sur la listing. Deux joueurs achetant le meme item en meme temps : le premier est traite, le second recoit "Quantite insuffisante".
  • NBT matching : pour les ventes directes, l'item est stocke en template. L'acheteur recoit des copies exactes du template (meme NBT). Pas de substitution possible.

Anti-abus economiques

  • Self-buy : un joueur ne peut PAS acheter ses propres items (vente directe + lucky blocks). Exception: il peut encherir sur ses propres encheres (pour augmenter le prix) mais c'est a ses risques.
  • Prix aberrants : prix minimum = 1, prix maximum = configurable (defaut: 1,000,000,000). Empeche les transferts deguises a 0$ et les overflow.
  • Items blacklisted : certains items ne peuvent pas etre vendus (bedrock, command blocks, items admin, etc.). Liste configurable.
  • Stacks invalides : verification que l'item est un ItemStack valide (pas null, pas vide, pas count 0).

Deconnexion pendant operation

  • Pendant la creation d'un Lucky Block : si le joueur se deconnecte apres avoir "place" des items dans la grille (cote client) mais avant de confirmer, rien ne se passe — les items n'ont pas ete retires.
  • Pendant un achat : le serveur traite les packets de maniere atomique. Soit l'achat est completement traite avant la deconnexion, soit il ne l'est pas du tout.
  • Enchere en cours : la deconnexion n'affecte pas les encheres. L'enchere continue. Le remboursement est envoye par mail si necessaire.

Expiration

  • Ventes directes : apres expiration, les items restants sont renvoyes au vendeur par mail.
  • Lucky Blocks : apres expiration, les items des slots NON GAGNES sont renvoyes au vendeur par mail.
  • Encheres : gerees par HDVAuctionTask (voir section Encheres).
⚠
Rate limiting : Limiter les operations HDV a 10 actions par joueur par minute (achats + mises en vente + encheres). Cela previent les bots et les scripts. Chaque joueur a un compteur decrement toutes les 6 secondes.

13 Plan de Tests

Tests Marche

  • Vente + Achat : joueur A vend 64 diamants a $150/u. Joueur B achete 32. Verifier : B recoit 32 diamants, B debite $4,800, listing mise a jour (32 restants), A recoit $4,800 par mail.
  • Achat quantite exacte : acheter les 64 restants. Listing supprimee. Items livres.
  • Achat quantite superieure : tenter d'acheter 100 quand il en reste 64. Verifier le refus.
  • Solde insuffisant : tenter d'acheter sans assez d'argent. Verifier le refus.
  • Annulation : annuler une vente avec 32/64 restants. Verifier 32 items renvoyes.
  • Self-buy : tenter d'acheter son propre item. Verifier le refus.
  • Limite listings : creer 21 listings (limite 20). Verifier le refus de la 21e.

Tests Lucky Block

  • Creation : remplir 18 slots, confirmer. Verifier items retires de l'inventaire, lucky block cree avec 18 blocs.
  • Achat : acheter un bloc. Verifier item aleatoire recu, blocs restants = 17.
  • Apercu inchange : acheter 5 blocs. Verifier que le preview montre toujours les 18 items originaux.
  • Dernier bloc : acheter le dernier bloc. Verifier que le joueur recoit l'item restant avec certitude.
  • Minimum slots : tenter de creer un lucky block avec 2 slots. Verifier le refus.
  • Annulation : annuler avec 12/18 restants. Verifier 12 items renvoyes.

Tests Encheres

  • Creation + Bid : creer une enchere a $100. Joueur B encheri $150. Verifier : B debite $150, enchere mise a jour.
  • Surenchere : joueur C encheri $200. Verifier : B rembourse $150 (mail ou direct), C debite $200.
  • Increment minimum : tenter d'encherir $155 sur $150 (increment < 10%). Verifier le refus.
  • Fin enchere : attendre la fin. Verifier : gagnant recoit l'item par mail, vendeur recoit l'argent par mail.
  • Aucun encherisseur : attendre la fin sans enchere. Verifier : item renvoye au vendeur.
  • Anti-snipe : encherir dans les 5 dernieres minutes. Verifier l'extension de 5 minutes.
  • Non annulable : tenter d'annuler une enchere. Verifier le refus.
  • Self-bid : le vendeur encheri sur sa propre enchere. Verifier le debit correct.

Tests Stats

  • Affichage : effectuer 10 ventes de diamants sur 3 jours. Verifier que le graphe affiche correctement les 3 points de donnees.
  • Pas de donnees : chercher un item jamais vendu. Verifier message "Aucune donnee".
  • Retention : verifier que les stats de plus de 7 jours sont supprimees.

Tests de securite

  • Packet forge : envoyer un packet d'achat avec un slot d'inventaire inexistant. Verifier pas de crash.
  • Race condition : deux joueurs achetent le dernier item en meme temps. Verifier qu'un seul recoit l'item.
  • Duplication : vendre un item puis tenter de le re-vendre (le slot est vide). Verifier le refus.
  • Blacklist : tenter de vendre un item blackliste. Verifier le refus.
