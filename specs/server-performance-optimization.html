Infrastructure — Audit TPS & Correctifs

Optimisation Performance Serveur

Audit de scalabilite revele des bottlenecks critiques qui rameneraient le TPS a 8-10 a pleine capacite (1000 joueurs). Ce cahier documente chaque systeme defaillant, sa cause racine, et les correctifs precis a appliquer pour descendre de 14-20ms/tick a 3-4ms/tick.

CRITIQUE — 14-20ms/tick Cible — 3-4ms/tick 500-1000 Joueurs Forge 1.12.2 Serveur Unique 8 Systemes

01Contexte & Resultats de l'Audit

Infrastructure concernee

  • Serveur unique — Pas de BungeeCord, pas de split horizontal. Tout le monde dans la meme JVM.
  • 500 a 1000 joueurs simultanees — Heure de pointe en semaine/weekend. C'est la charge de reference pour toutes les estimations de cette spec.
  • ~20 mods actifs — Dont Dimensional Threading pour le multithreading des dimensions.
  • PvP 1.8 — Pas de cooldown d'attaque, combats frequents, beaucoup d'effets de potion actifs en permanence.
  • Monde unique — Tous les joueurs coexistent dans le meme overworld avec des biomes custom, des claims de faction, des territoires.

Constat de l'audit

!
TPS projete a 1000 joueurs : 8-10 TPS — sans correctifs, le serveur sera injouable a pleine capacite. Les systemes identifies ci-dessous ajoutent collectivement 14 a 20ms par tick, alors que le budget total d'un tick a 20 TPS est de 50ms.

02RpgEventHandler — Optimisation des Perks Tick

!
Cout actuel : 2-3ms/tick a 1000 joueurs. Fichier : RpgEventHandler.java.

Probleme 1 — Iteration globale des joueurs

Les perks Bastion, Frappe et Forme Cristalline sont geres dans un ServerTickEvent (ou equivalent) qui appelle server.getPlayerList().getPlayers() et itere la liste entiere a chaque tick. A 1000 joueurs, c'est 1000 iterations de logique perk chaque 50ms, meme si aucun joueur n'a le perk actif.

AVANT — iteration globale (a eviter)// Dans un @SubscribeEvent(ServerTickEvent) ou similaire : for (EntityPlayerMP player : server.getPlayerList().getPlayers()) { // Execute pour CHAQUE joueur, CHAQUE tick, meme sans perk actif RpgPlayerData data = DataManager.get(player, RpgPlayerData.class); if (data.hasPerk("BASTION")) { ... } if (data.hasPerk("FRAPPE")) { ... } if (data.hasPerk("FORME_CRISTALLINE")) { ... } }
APRES — PlayerTickEvent (Forge batch)// Forge declenche PlayerTickEvent.Type.END pour chaque joueur, en batch. // Pas d'iteration manuelle : Forge gere la distribution. @SubscribeEvent public void onPlayerTick(TickEvent.PlayerTickEvent e) { if (e.phase != TickEvent.Phase.END || !(e.player instanceof EntityPlayerMP)) return; EntityPlayerMP player = (EntityPlayerMP) e.player; RpgPlayerData data = DataManager.get(player, RpgPlayerData.class); if (data == null) return; // Sortir immediatement si aucun perk tick-base n'est actif if (!data.bastionActive && !data.frappeActive && !data.formeCristallineActive) return; if (data.bastionActive) tickBastion(player, data); if (data.frappeActive) tickFrappe(player, data); if (data.formeCristallineActive) tickFormeCristalline(player, data); }

Probleme 2 — 7 HashMaps UUID non nettoyees

Le handler maintient 7 HashMap<UUID, *> pour tracker des etats de perk par joueur (cooldowns, compteurs, flags). Ces maps ne sont jamais nettoyees a la deconnexion. Avec 1000 joueurs uniques par jour, la memoire utilisee par ces maps croit indefiniment jusqu'au redemarrage.

// Exemple de maps typiquement presentes : private final Map<UUID, Integer> bastionCooldown = new HashMap<>(); private final Map<UUID, Long> frappeLastHit = new HashMap<>(); private final Map<UUID, Boolean> cristallineOn = new HashMap<>(); // ... 4 autres maps similaires // =================================================== // SOLUTION : deplacer ces champs dans RpgPlayerData // et ajouter un cleanup PlayerLoggedOutEvent // =================================================== // Dans RpgPlayerData (annote @EriData ou champ directement) : public int bastionCooldown = 0; public long frappeLastHitTime = 0L; public boolean formeCristallineOn = false; // ... etc. // Dans RpgEventHandler — cleanup obligatoire : @SubscribeEvent public void onPlayerLogout(PlayerEvent.PlayerLoggedOutEvent e) { // Si des maps residuelles subsistent, les vider ici UUID id = e.player.getUniqueID(); bastionCooldown.remove(id); frappeLastHit.remove(id); cristallineOn.remove(id); // ... toutes les autres maps }

Solution complete

  • Migrer vers PlayerTickEvent — remplacer le ServerTickEvent global par des handlers individuels. Forge dispatche ces events player-par-player, ce qui evite l'iteration manuelle.
  • Early return — si aucun perk tick-base n'est actif pour ce joueur, retourner immediatement. Cout marginal pour 95% des joueurs sans perk actif.
  • Champs dans RpgPlayerData — migrer les 7 HashMaps vers des champs primitifs directement dans la classe data. Acces O(1) sans hashage de UUID, et persistence automatique via EriAPI.
  • PlayerLoggedOutEvent — ajouter un handler de cleanup pour toute map residuelle.
2-3ms
Cout actuel / tick
<0.2ms

03CheckMovement — Anti-cheat Mouvement

!
Cout actuel : 3-5ms/tick a 1000 joueurs. Fichier : CheckMovement.java.

Probleme — Calculs couteux a chaque tick

Le systeme anti-cheat applique plusieurs verifications de mouvement (speed, fly, nofall, phase, step, blink) pour chaque joueur a chaque tick. Les operations les plus couteuses sont :

  • Math.sqrt() — utilise pour calculer la distance parcourue. Operation en virgule flottante couteuse, invoquee 1000 fois/tick a 1000 joueurs.
  • 2-3 appels world.getBlockState() — pour les verifications fly/nofall/phase. Chaque appel peut impliquer un lookup de chunk et de section. Multiplicatif a 1000 joueurs.
  • Frequence : chaque tick — aucune reduction de frequence meme quand un joueur est immobile.
AVANT — sqrt + getBlockState chaque tick// Exemple typique dans CheckMovement : double dx = player.posX - player.prevPosX; double dz = player.posZ - player.prevPosZ; double dist = Math.sqrt(dx*dx + dz*dz); // COUTEUX — sqrt inutile if (dist > MAX_SPEED) flag(player, "speed"); // Puis verifications blocs : IBlockState below = world.getBlockState(pos.down()); // appel 1 IBlockState feet = world.getBlockState(pos); // appel 2 IBlockState head = world.getBlockState(pos.up()); // appel 3
APRES — distanceSq + cache + every 2 ticks// Comparaison en distance au carre — pas besoin de sqrt : double dx = player.posX - player.prevPosX; double dz = player.posZ - player.prevPosZ; double distSq = dx*dx + dz*dz; // Pas de sqrt ! double maxSpeedSq = MAX_SPEED * MAX_SPEED; // Constante pre-calculee if (distSq > maxSpeedSq) flag(player, "speed"); // Reduire la frequence des verifications bloc : if (player.ticksExisted() % 2 != 0) return; // Verif tous les 2 ticks // Mettre en cache BlockPos pour eviter les allocations : BlockPos feetPos = player.getPosition(); IBlockState below = world.getBlockState(feetPos.down()); // Reutiliser below pour nofall ET phase — un seul getBlockState boolean onGround = below.getMaterial().isSolid(); if (!onGround && player.onGround) checkNoFall(player); // Reutilise onGround

Solution complete

  • Remplacer Math.sqrt() par distanceSq — comparer des distances au carre. MAX_SPEED devient MAX_SPEED_SQ, pre-calcule une fois comme constante statique. Fonctionnellement identique, 0 appel sqrt.
  • Cache et reutilisation de BlockPos — calculer getPosition() une seule fois par check, stocker le resultat. Reutiliser un IBlockState deja recupere pour plusieurs verifications (ex: le bloc du sol sert a la fois pour fly et nofall).
  • Frequence : tous les 2 ticks — les verifications de mouvement tolerent un delai de 100ms sans impact sur la detection. Divise le cout total par 2. Utiliser player.ticksExisted % 2 == 0 comme garde.
  • Combiner les verifications par type de bloc — grouper les appels getBlockState de facon a ne lire chaque position qu'une seule fois (below, feet, head) et brancher toutes les verifications depuis ces 3 resultats.
!
Ne pas desactiver le check speed/fly — reduire la frequence a 2 ticks est suffisant. Ne pas monter au-dela de 3-4 ticks pour rester efficace contre le speed hack.

04FactionMilestoneTicker — Perks de Territoire

!
Cout actuel : 5-10ms de spike toutes les 20 ticks (1 seconde). Fichier : FactionMilestoneTicker.java.

Probleme — getFactionAt() + addPotionEffect() sur tous les joueurs

Toutes les 20 ticks, le ticker itere tous les joueurs en ligne et, pour chaque joueur :

  • Appelle getFactionAt() (lookup de chunk + recherche de faction proprietaire).
  • Si en territoire de faction, applique jusqu'a 3 addPotionEffect() consecutifs.
  • N'a aucune logique de cache — addPotionEffect() est appele meme si l'effet est deja actif avec une duree suffisante.

A 1000 joueurs, cela represente jusqu'a 3000 appels addPotionEffect() et 1000 appels getFactionAt() en un seul tick. Les effets de potion Forge declenchent des events supplementaires et des recalculs d'attributs a chaque application.

AVANT — re-application systematique toutes les 20 ticks// Toutes les 20 ticks, pour chaque joueur : String factionAt = getFactionAt(player); // Lookup chunk a chaque fois if (factionAt != null) { FactionInfo faction = getFactionInfo(factionAt); if (faction.hasMilestone("FORTERESSE")) { player.addPotionEffect(new PotionEffect(MobEffects.RESISTANCE, 40, 0)); // Meme si deja actif ! player.addPotionEffect(new PotionEffect(MobEffects.STRENGTH, 40, 0)); player.addPotionEffect(new PotionEffect(MobEffects.SPEED, 40, 0)); } }
APRES — cache territoire + re-application seulement quand necessaire// Dans RpgPlayerData ou FactionCapData — champs de cache : public String cachedTerritoryFaction = null; // Faction du territoire actuel public int territoryPerkRefreshTick = 0; // Prochain tick de refresh // Dans FactionMilestoneTicker (toujours toutes les 20 ticks) : @SubscribeEvent public void onPlayerTick(TickEvent.PlayerTickEvent e) { if (e.phase != TickEvent.Phase.END) return; if (e.player.ticksExisted % 20 != 0) return; EntityPlayerMP player = (EntityPlayerMP) e.player; // Reutiliser le cache de territoire de FactionTerritoryHandler // (evite de recalculer getFactionAt() ici) FactionCapData cap = EriCap.get(player, FactionCapData.class); String territory = cap.cachedTerritoryFaction; if (territory == null) return; // Pas en territoire de faction FactionInfo faction = getFactionInfo(territory); if (faction == null || !faction.hasMilestone("FORTERESSE")) return; // Re-appliquer seulement si l'effet expire bientot (<40 ticks restants) PotionEffect resistance = player.getActivePotionEffect(MobEffects.RESISTANCE); if (resistance == null || resistance.getDuration() < 40) { player.addPotionEffect(new PotionEffect(MobEffects.RESISTANCE, 60, 0)); } // Idem pour STRENGTH et SPEED }

Solution complete

  • Reutiliser le cache territoire de FactionTerritoryHandler — ce handler (voir section 08) maintiendra deja un champ cachedTerritoryFaction dans la capability joueur. MilestoneTicker le lit directement, eliminant son propre appel getFactionAt().
  • Conditionner addPotionEffect() a la duree restante — verifier getActivePotionEffect(effet).getDuration() < 40 avant d'appliquer. Si l'effet a encore plus de 2 secondes, ne rien faire. Eliminera ~80-90% des appels addPotionEffect().
  • Duree 60 ticks au lieu de 40 — appliquer des effets de duree 60 ticks (3 secondes) avec un refresh a <40 ticks laisse une marge confortable sans sur-application.
  • PlayerTickEvent au lieu d'un timer global — evite l'iteration manuelle de la liste des joueurs.
5-10ms
Cout actuel / 20 ticks
<1-2ms
Cible / 20 ticks

05TabListNameHandler + NameplateSyncHandler — Reseau

!
Cout actuel : 3-5ms + ~2.5MB/min de bande passante toutes les 40 ticks. Fichiers : TabListNameHandler.java, NameplateSyncHandler.java.

Probleme — Sync complete toutes les 2 secondes

Les deux handlers partagent le meme anti-pattern : toutes les 40 ticks, ils iterent tous les joueurs et envoient a tous les clients une mise a jour du nom — meme si rien n'a change depuis le dernier envoi. A 1000 joueurs :

  • TabList — 1000 * 1000 = 1 million de paires (emetteur, destinataire). Meme si factorise en broadcast, chaque packet TabList contient les 1000 entrees et alloue un PacketBuffer neuf a chaque cycle.
  • Nameplate — Envoi d'un packet de nom par joueur visible. Si 50 joueurs sont dans le meme chunk, chacun recoit 49 packets de mise a jour toutes les 2 secondes.
  • Bande passante totale — Estimee a ~2.5MB/min uniquement pour ces deux systemes.
AVANT — sync complete a intervalle fixe// TabListNameHandler — toutes les 40 ticks : if (tickCount % 40 == 0) { for (EntityPlayerMP player : server.getPlayerList().getPlayers()) { String tabName = buildTabName(player); // Construit a chaque fois sendTabUpdate(player, tabName); // Envoye meme si inchange } }
APRES — delta updates + cache des noms formates// Cache des noms formates par UUID : private static final Map<UUID, String> cachedTabNames = new HashMap<>(); // Appele seulement lors d'evenements qui changent le nom : // - RankChangeEvent (montee de grade) // - FactionJoinEvent / FactionLeaveEvent // - PlayerLevelUpEvent (nouveau niveau RPG) // - PlayerLoggedInEvent (init initiale) public static void invalidateTabName(EntityPlayerMP player) { String newName = buildTabName(player); String old = cachedTabNames.put(player.getUniqueID(), newName); if (!newName.equals(old)) { sendTabUpdate(player, newName); // Envoye seulement si change } } // Cleanup a la deconnexion : @SubscribeEvent public void onLogout(PlayerEvent.PlayerLoggedOutEvent e) { cachedTabNames.remove(e.player.getUniqueID()); }

Solution pour NameplateSyncHandler

Meme logique de delta update : ne resyncer le nameplate d'un joueur que lorsque ses donnees affichees changent (grade, nom de faction, niveau). Ajouter un champ lastSyncedNameplateHash dans la capability joueur et comparer le hash avant envoi.

// Dans la capability joueur — hash du dernier nameplate envoye : public int lastNameplateHash = 0; // Dans NameplateSyncHandler : public static void syncNameplateIfChanged(EntityPlayerMP player) { String nameplate = buildNameplate(player); int hash = nameplate.hashCode(); FactionCapData cap = EriCap.get(player, FactionCapData.class); if (cap.lastNameplateHash == hash) return; // Rien de change cap.lastNameplateHash = hash; broadcastNameplateUpdate(player, nameplate); }

Solution complete

  • Supprimer les timers a intervalle fixe — aucun tick-based loop pour la TabList ou les nameplates.
  • Invalidation evenementielle — appeler invalidateTabName() et syncNameplateIfChanged() uniquement dans les evenements qui modifient les donnees affichees : RankChangeEvent, FactionJoinEvent, FactionLeaveEvent, PlayerLevelUpEvent, PlayerLoggedInEvent.
  • Cache des noms formates — eviter de reconstruire les strings de nom formatees a chaque tick. Les stocker dans une Map indexee par UUID.
  • Cleanup PlayerLoggedOutEvent — retirer les entrees des maps de cache a la deconnexion.

06SetBonusHandler — Detection de Set d'Armure

!
Cout actuel : ~0.6ms / 20 ticks. Fichier : SetBonusHandler.java.

Probleme 1 — Allocations String pour comparaison d'items

Toutes les 20 ticks, pour chaque joueur, le handler verifie si les 4 pieces d'armure forment un set complet en appelant getRegistryName().toString() sur chaque piece. Cette operation alloue une nouvelle String a chaque verification, soit 16 allocations de String par joueur par 20 ticks (4 pieces × 4 slots). A 1000 joueurs, c'est 16 000 Strings allouees toutes les secondes, pressurant le GC.

AVANT — comparaison par registry name (String)for (ItemStack stack : player.getArmorInventoryList()) { String name = stack.getItem().getRegistryName().toString(); // Nouvelle String a chaque fois if (name.equals("eriniumfaction:faction_chestplate")) { ... } }
APRES — comparaison par reference d'Item (zero allocation)// Les instances d'Item sont des singletons enregistres dans le registre Forge. // Comparer la reference directe est O(1) et n'alloue rien. private static final Item FACTION_HELMET = Item.getByNameOrId("eriniumfaction:faction_helmet"); private static final Item FACTION_CHESTPLATE = Item.getByNameOrId("eriniumfaction:faction_chestplate"); // ... etc. (initialises une seule fois au PostInit) for (ItemStack stack : player.getArmorInventoryList()) { Item item = stack.getItem(); // Pas de String, comparaison de reference if (item == FACTION_HELMET) { ... } if (item == FACTION_CHESTPLATE) { ... } }

Probleme 2 — 3 HashMaps non nettoyees

Le handler maintient 3 HashMap<UUID, *> pour tracker l'etat du bonus de set actif. Ces maps ne sont jamais nettoyees a la deconnexion du joueur.

07InactiveEntityCuller — Copie de Liste d'Entites

!
Cout actuel : 2-3ms de spike tous les 80 ticks. Fichier : InactiveEntityCuller.java.

Probleme — Copie complete de loadedEntityList

Toutes les 80 ticks, le culler execute new ArrayList<Entity>(world.loadedEntityList) pour creer une copie defensive de la liste des entites chargees avant d'iterer. A 20 000 entites chargees (estimation raisonnable pour un serveur PvP de 1000 joueurs), cette allocation copie 20 000 references Object et cree une ArrayList de grande taille, pressurant immediatement le GC.

AVANT — copie defensive de la liste entiere// Toutes les 80 ticks : List<Entity> entities = new ArrayList<Entity>(world.loadedEntityList); // Copie ~20k references for (Entity e : entities) { if (shouldCull(e)) world.removeEntity(e); }
APRES — iteration directe avec protection ConcurrentModification// Option A — reutiliser une ArrayList statique (zero allocation) : private static final ArrayList<Entity> toRemove = new ArrayList<>(); // Toutes les 80 ticks : toRemove.clear(); // Vider sans desallouer le tableau interne for (int i = 0; i < world.loadedEntityList.size(); i++) { Entity e = world.loadedEntityList.get(i); if (shouldCull(e)) toRemove.add(e); } for (Entity e : toRemove) world.removeEntity(e); // Note : toRemove est un champ statique, pas d'allocation. // Ne pas utiliser dans un contexte multi-thread sans synchronisation.

Solution complete

  • Option A (recommandee) — utiliser une ArrayList statique comme buffer de collecte des entites a supprimer. Appeler clear() entre les cycles (ne desalloue pas le tableau interne, juste remet la taille a 0). Iteration directe sur loadedEntityList avec index pour eviter l'iterator.
  • Option B — iterer loadedEntityList en arriere avec un index decroissant et appeler remove(i) directement. Plus simple mais fragile si d'autres threads modifient la liste (a verifier selon le contexte threading du serveur).
  • Contrainte thread safety — loadedEntityList est modifiee sur le thread principal Minecraft. Tant que le culler s'execute dans un ServerTickEvent (thread principal), l'option A est sans risque.

08FactionTerritoryHandler — Cle de Chunk par String

!
Cout actuel : 1-2ms/tick. Fichier : FactionTerritoryHandler.java.

Probleme — Concatenation de String comme cle de chunk

Pour tracker dans quel chunk chaque joueur se trouve, le handler construit une cle sous forme de String par concatenation : chunkX + ":" + chunkZ + ":" + dim. Cette operation :

  • Alloue une nouvelle String a chaque evaluation (chaque tick, pour chaque joueur).
  • Implique le boxing de 3 entiers en String.
  • Genere une pression GC continue sur le thread principal.

A 1000 joueurs par tick, cela represente 1000 allocations de String uniquement pour les cles de chunk du handler de territoire.

AVANT — cle String (1 allocation par joueur par tick)String key = chunkX + ":" + chunkZ + ":" + dim; // Nouvelle String a chaque appel String faction = territoryMap.get(key);
APRES — cle long packee (zero allocation)// Packer chunkX, chunkZ, dim en un seul long. // dim tient sur 8 bits (dimensions -128 a 127), chunkX et chunkZ sur 28 bits chacun. public static long chunkKey(int chunkX, int chunkZ, int dim) { return ((long)(dim & 0xFF) << 56) | ((long)(chunkX & 0x0FFFFFFF) << 28) | (long)(chunkZ & 0x0FFFFFFF); } // Utilisation (zero allocation) : long key = chunkKey(chunkX, chunkZ, dim); String faction = territoryMap.get(key); // Map<Long, String> au lieu de Map<String, String> // Optionnellement, utiliser Long2ObjectOpenHashMap de FastUtil pour eviter le boxing Long : // Long2ObjectOpenHashMap<String> territoryMap — entierement sans boxing

Bonus — Cache du territoire courant par joueur

En complement de la cle packee, stocker le resultat du lookup de territoire dans la capability du joueur (cachedTerritoryFaction). Ce champ sera lu par FactionMilestoneTicker (section 04) evitant un double lookup par tick.

// Dans FactionCapData (capability joueur) : public String cachedTerritoryFaction = null; // Null = pas en territoire public long lastChunkKey = 0L; // Pour detecter le changement de chunk // Dans FactionTerritoryHandler.onPlayerTick : long key = chunkKey(playerChunkX, playerChunkZ, dim); if (cap.lastChunkKey == key) return; // Pas change de chunk, pas de recalcul cap.lastChunkKey = key; cap.cachedTerritoryFaction = territoryMap.get(key); // Update le cache

09Memory Leaks — HashMaps non nettoyees

!
Impact : croissance indefinie des HashMaps jusqu'au redemarrage du serveur. Concerne RpgEventHandler.java (7 maps) et SetBonusHandler.java (3 maps).

Mecanisme du leak

Les HashMaps indexees par UUID sont alimentees a la connexion d'un joueur (ou a la premiere action qui les touche) mais ne sont jamais videes quand le joueur se deconnecte. Chaque entree consume de la memoire heap (UUID 16 bytes + cle boxing + valeur). Avec 1000 joueurs uniques par jour de serveur en heure de pointe :

  • RpgEventHandler — 7 maps, chacune avec potentiellement 1000+ entrees = des dizaines de milliers d'objets UUID/valeur orphelins.
  • SetBonusHandler — 3 maps, meme probleme.
  • Apres une semaine de uptime — les maps peuvent contenir des milliers d'entrees invalides de joueurs desormais deconnectes, occupant inutilement de la RAM et ralentissant les lookups (plus le tableau de hachage est grand, plus la resolution de collision est lente).

Checklist de nettoyage

RpgEventHandler — Ajouter un @SubscribeEvent PlayerEvent.PlayerLoggedOutEvent qui appelle map.remove(player.getUniqueID()) pour les 7 maps.
SetBonusHandler — Idem pour les 3 maps.
Envisager la migration vers les champs EriData/EriCapability — les modules EriAPI gerent automatiquement le cycle de vie des donnees par joueur et les serialisent en NBT. Elimine completement le besoin de HashMaps separees.
Audit complementaire — verifier les autres handlers du projet pour toute HashMap/Set indexe par UUID non nettoye. Grep : HashMap<UUID dans tout le projet.
Test de regression — apres le fix, verifier avec un profiler Java (VisualVM ou async-profiler) que les maps restent a taille stable apres 100 connexions/deconnexions en dev.

11Plan de Tests

Tests fonctionnels par systeme

RpgEventHandler — Verifier que les perks Bastion, Frappe et Forme Cristalline fonctionnent toujours apres migration vers PlayerTickEvent. Tester l'activation, la desactivation et le cooldown.
CheckMovement — Verifier que speed hack et fly hack sont toujours detectes avec le check a 2 ticks. Tester avec un joueur en mouvement normal (aucun faux positif).
FactionMilestoneTicker — Verifier que les effets de potion de territoire sont bien appliques en entrant dans un territoire de faction avec le milestone actif. Verifier qu'ils expirent en quittant le territoire.
TabList + Nameplate — Se connecter avec 2 joueurs, verifier l'affichage initial. Changer de grade : verifier que la TabList se met a jour immediatement. Changer de faction : idem.
SetBonusHandler — Equiper un set complet, verifier le bonus. Retirer une piece, verifier la suppression du bonus.
InactiveEntityCuller — Spawner 100 mobs inactifs, attendre 80 ticks, verifier qu'ils sont bien supposes etre culls selon les criteres du handler.
FactionTerritoryHandler — Verifier la detection du territoire (entree, sortie, changement de faction du territoire) avec la nouvelle cle long.

Tests de performance

Profiler avant/apres — Utiliser Spark (plugin/mod) ou async-profiler pour mesurer les ms/tick de chaque systeme avant et apres optimisation.
Simulation de charge — Utiliser un bot de stress-test (ex: Minecraft bot framework) pour simuler 100-500 joueurs en dev et verifier que les gains de perf sont proportionnels.
Memory leak test — Connecter/deconnecter 200 joueurs uniques en dev, verifier avec VisualVM que les HashMaps restent a taille stable (ne grossissent plus).
GC pressure — Apres les optimisations String, verifier que la frequence des GC mineur diminue via les logs JVM (-verbose:gc).
Validation TPS — Avec 500 bots en jeu apres toutes les optimisations, le TPS doit etre stable a 18-20. Mesurer via /tps ou Forge debug screen.
