Erigineering — Editeur d'Animation

Application desktop d'animation pour Minecraft Java 1.12.2 modde. La modelisation des entites se fait sur Blockbench, les textures sur Photoshop/Paint.net. Erigineering se concentre exclusivement sur l'animation avec 3 categories : block, item et entity.

En cours Electron + Three.js Migration EriAnim 3 categories animation

01 Contexte & Vision

Pourquoi Erigineering

Le projet EriniumFaction utilise actuellement Blockbench pour la modelisation 3D et EriAnim (outil interne Electron + Three.js) pour l'animation de blocs. Le probleme principal :

  • EriAnim limite aux blocs : EriAnim ne supporte que les animations de blocs. Les items et entites n'ont pas d'outil d'animation.
  • Pas de centralisation : Les modeles sont faits sur Blockbench, les textures sur Photoshop/Paint.net, et les animations sur EriAnim. L'animation merite un outil dedie et complet.

Vision

Erigineering est un editeur d'animation pour Minecraft modde. La modelisation se fait sur Blockbench, les textures sur Photoshop/Paint.net/GIMP. Erigineering se concentre exclusivement sur l'animation. Il reprend la stack technique d'EriAnim (Electron 41+ / Three.js 0.152+ / Node.js) et l'etend :

  • Modelisation (Blockbench) : La modelisation des entites, blocs et items se fait sur Blockbench. Le format JSON Blockbench supporte nativement les cubes float, le per-face UV et le multi-texture. Erigineering charge ces fichiers JSON.
  • Animation : Migration complete d'EriAnim (timeline, 7 types de keyframes, 13 easings, events) etendue a 3 categories : block, item, entity.
  • Textures (externes) : Les textures sont creees dans des outils externes (Photoshop, Paint.net, GIMP). Erigineering les charge pour l'affichage dans le viewport mais ne les edite pas.

Ce que Erigineering remplace

Outil actuel Usage Remplace par
EriAnim Animation de blocs (.erianim.json) Erigineering (migration + 3 categories : block, item, entity)

Ce que Erigineering ne fait pas

  • Pas de modelisation : La modelisation 3D (cubes, groupes, pivots) se fait dans Blockbench.
  • Pas d'edition de texture : Les textures sont creees dans des outils externes (Photoshop, Paint.net, GIMP). Erigineering charge les textures pour le viewport mais ne les edite pas.
  • Pas de rendu final identique : Le rendu Three.js est une approximation du rendu OpenGL 1.1 de Minecraft. De legeres differences visuelles sont attendues.
  • Pas de plugin Minecraft : Erigineering n'est pas un mod. C'est une application desktop independante qui exporte des fichiers exploitables par le mod.

Communication IPC

L'application suit le pattern Electron standard : le main process (Node.js) gere les operations I/O (lecture/ecriture de fichiers, ZIP, dialogues systeme) et le renderer process (Chromium) gere le viewport 3D, les editeurs, et l'interface. La communication passe par les channels IPC d'Electron :

Channel IPC Direction Usage
project:save Renderer → Main Sauvegarder le projet .erigineering (ZIP)
project:load Renderer → Main Charger un projet depuis le disque
file:export Renderer → Main Exporter JSON, .erianim.json
file:import Renderer → Main Importer Blockbench JSON, .erianimproject
dialog:open Renderer → Main Ouvrir un dialogue fichier natif
dialog:save Renderer → Main Ouvrir un dialogue sauvegarde natif
window:detach Renderer → Main Detacher un panel dans une fenetre separee
ⓘ
Securite Electron : Le preload script expose uniquement les channels IPC listes ci-dessus via contextBridge. Le renderer n'a PAS acces a require('fs') ou require('child_process'). Toutes les operations fichier passent par le main process.

03 Modelisation (Blockbench)

Delegation a Blockbench

La modelisation des entites se fait sur Blockbench en mode Java Block/Item. Le format JSON Blockbench supporte nativement les cubes float, le per-face UV, et le multi-texture. Erigineering charge ces fichiers JSON pour l'animation. EriAPI cote Minecraft lit le meme format JSON Blockbench pour le rendu des entites en jeu.

Feature Blockbench Erigineering
Cubes float OUI (JSON block) NON (modelisation retiree)
Per-face UV OUI NON
Multi-texture OUI NON
Animation NON (pas EriAnim) OUI (3 categories)
Seats NON Metadata dans animation
ⓘ
Workflow : Modeliser l'entite sur Blockbench (mode Java Block/Item) → exporter le JSON → ouvrir dans Erigineering → animer → exporter le .erianim.json.

05 Systeme d'Animation — Feature Principale

Feature principale d'Erigineering

L'animation est la raison d'etre d'Erigineering. Le code source d'EriAnim (tools/erianim-editor/) est migre integralement et etendu a 3 categories : block, item et entity. L'editeur charge des modeles Blockbench JSON (format block/item) et permet de les animer avec le systeme EriAnim existant. Le format de sortie reste .erianim.json, identique a l'existant.

Workflow : Modeliser sur Blockbench → ouvrir le JSON dans Erigineering → animer → exporter le .erianim.json.

Types de keyframes

Les 7 types de keyframes existants sont conserves. L'animation fonctionne pour les blocs ET les entites (meme format).

Type Propriete Valeur Description
rotation Rotation absolue [x, y, z] degres Definit la rotation absolue du groupe a un instant t.
translate Translation absolue [x, y, z] unites Definit le deplacement du groupe par rapport a sa position d'origine.
scale Echelle [x, y, z] facteur Definit l'echelle du groupe (1.0 = taille normale).
spin Rotation continue [x, y, z] deg/tick Rotation continue a vitesse constante. Ignore l'easing.
rotate Rotation relative [x, y, z] degres Ajoute une rotation relative a la rotation actuelle.
visible Visibilite true / false Montre ou cache le groupe a un instant t.
texture Texture swap texture_key Change la texture du groupe a un instant t (pour les animations de texture).

Events d'animation

Les events sont des actions declenchees a un instant precis de la timeline, sans interpolation :

Event Parametres Description
callback Nom du callback (string) Declenche un callback Java enregistre cote mod. Utilise pour la logique metier (degats, spawn, etc.).
sound ResourceLocation du son Joue un son a la position du bloc/entite.

hide_group Nom du groupe Cache un groupe (equivalent a visible=false).
show_group Nom du groupe Affiche un groupe (equivalent a visible=true).
cut Aucun Coupe l'animation (saut instantane, pas d'interpolation vers la keyframe suivante).

Nouveau : 3 categories d'animation

EriAnim supporte actuellement les animations de blocs. Erigineering etend le systeme a 3 categories :

Categorie Description Modele source Statut
Block Animation de blocs 3D (lootbox, machines, etc.) JSON Blockbench EXISTANT
Item Animation d'items 3D (pioche, epee, baguette, lance). Meme workflow que les blocs : modeliser l'item en 3D, l'animer avec des keyframes sur les groupes. JSON Blockbench NOUVEAU
Entity Animation d'entites custom (monstres, PNJ, turrets). Charge un JSON Blockbench (mode block/item) avec per-face UV et multi-texture. JSON Blockbench NOUVEAU

Les 3 categories utilisent le meme format d'animation (.erianim.json), les memes types de keyframes (rotation, translate, scale, spin, rotate, visible, texture), et la meme timeline. La seule difference est le type de modele charge et le renderer cote EriAPI.

  • Cas d'usage Item : Epee qui tourne en idle, pioche qui vibre en minant, baguette qui pulse, arc qui se plie en chargeant.
  • Cas d'usage Entity : Monstre qui marche (cycle jambes), PNJ qui parle (animation bouche), turret qui tourne et tire.
✓
Compatibilite : Les fichiers .erianim.json existants (blocs) restent 100% compatibles. Les categories item et entity ajoutent un champ "category" dans le JSON mais le format de base est identique.

06 Textures (outils externes)

Pas d'editeur de texture

Les textures sont creees dans des outils externes (Photoshop, Paint.net, GIMP). Erigineering ne contient pas d'editeur de texture.

Les textures PNG sont chargees par Erigineering uniquement pour l'affichage dans le viewport 3D lors de l'animation. Elles ne sont pas modifiees par l'application.

ⓘ
Workflow textures : Creer/modifier les textures dans Photoshop, Paint.net ou GIMP → les sauvegarder en PNG → Erigineering les charge automatiquement pour l'affichage viewport quand un JSON Blockbench les reference.

07 Viewport 3D

Rendu Three.js

Le viewport 3D est le coeur de l'application. Il utilise Three.js pour le rendu WebGL et offre une experience interactive fluide :

Camera et navigation

  • Orbit camera : Middle click pour orbiter autour du modele. Rotation libre sur les axes X et Y.
  • Pan : Shift + Middle click pour deplacer la camera lateralement.
  • Zoom : Molette de la souris. Zoom progressif avec acceleration.
  • Focus : Double-clic sur un cube pour centrer la camera dessus. Touche . (point) pour centrer sur la selection.
  • Vues predefines : Numpad 1 (face), 3 (cote), 7 (dessus), 5 (toggle ortho/perspective).

Elements visuels

  • Grille au sol : Grille 16x16 (taille d'un bloc Minecraft) avec lignes secondaires. Couleur configurable.
  • Axes XYZ : Axes colores (X=rouge, Y=vert, Z=bleu) affiches dans le coin du viewport et au centre du modele.
  • Gizmos : Outils de manipulation interactifs — translate (fleches), rotate (anneaux), scale (cubes). Switch avec les raccourcis G, R, S.
  • Selection highlight : Le cube/groupe selectionne est entoure d'un outline colore. Multi-selection avec Ctrl+click.

Modes de rendu

Mode Raccourci Description
Texture T Rendu avec les textures appliquees sur les faces. Mode par defaut.
Solid Z Couleur unie par cube (gris). Utile pour voir la forme sans distraction des textures.
Wireframe F Affiche uniquement les aretes des cubes. Utile pour voir la structure interne.
X-ray Alt+Z Semi-transparent. Permet de voir les cubes caches derriere d'autres cubes.

Eclairage

  • Ambient : Lumiere d'ambiance douce, pas d'ombre portee. Simule l'eclairage Minecraft.
  • Directionnelle : Lumiere directionnelle optionnelle pour mieux voir les reliefs.
  • Flat : Mode flat — pas d'ombrage du tout. Chaque face a sa couleur exacte. Ideal pour le pixel art.

Options du viewport

  • Background : Couleur unie (defaut: gris fonce), gradient deux couleurs, ou image custom.
  • Preview animation : Bouton Play dans le viewport pour previsualiser l'animation en temps reel.
  • Snap to grid : Options de snap configurable : 1 (bloc), 0.5 (demi-bloc), 0.25 (quart), 0.1 (precision), off (libre).
  • Duplication : Ctrl+D pour dupliquer les cubes/groupes selectionnes. Ctrl+Shift+D pour dupliquer et placer avec le gizmo.
  • Undo/Redo : Toutes les operations viewport sont enregistrees dans l'historique global.

Chargement JSON Blockbench

Erigineering charge les fichiers JSON Blockbench sans conversion :

  • Cubes : Les coordonnees from/to sont chargees telles quelles.
  • Groupes : La hierarchie de groupes/outliner est chargee avec les pivots pour l'animation.
  • UV per-face : Les UV per-face du format Blockbench sont utilises directement.
  • Textures : Les textures embeddees (base64) ou referencees (chemin) sont chargees pour l'affichage dans le viewport.
  • Animations : Les animations Blockbench (format molang/bedrock) ne sont PAS importees — elles utilisent un systeme incompatible. Seules les animations EriAnim sont supportees.

09 Format de Projet .erigineering

Structure interne

Le fichier .erigineering est un ZIP renomme. Il contient l'integralite du projet dans un seul fichier partageable :

monmodele.erigineering (ZIP renomme) ├── project.json Metadonnees, settings, version du format ├── model.json Reference au JSON Blockbench (groupes, cubes, UV, faces) └── animations/ Tous les fichiers .erianim.json ├── idle.erianim.json └── attack.erianim.json

project.json

{ "format_version": 1, "name": "Monstre des Abysses", "author": "EriniumGroup", "created": "2026-04-14T12:00:00Z", "modified": "2026-04-14T15:30:00Z", "type": "entity", // "block", "item", ou "entity" "mod_id": "eriniumfaction", "settings": { "grid_snap": 0.25, "grid_visible": true, "background_color": "#1A1A2E", "render_mode": "textured" } }

model.json

Le format interne du modele est une version etendue qui conserve toutes les informations d'edition (selection, visibilite editeur, metadata des calques). A l'export, seules les donnees necessaires au runtime Minecraft sont incluses.

Champ Type Description
textures object Map texture_key → chemin vers le fichier PNG externe (reference, pas embarque).
groups array Hierarchie de groupes (meme structure que le format export entite, avec metadata additionnelle).
display object Transforms d'affichage (optionnel, pour les animations d'items en main via EriAnim).
✓
Partage simplifie : Un fichier .erigineering contient le modele, les animations et les settings. Les textures restent des fichiers PNG externes references par le JSON Blockbench.

10 Mises a jour EriAPI necessaires

Vue d'ensemble

EriAPI doit pouvoir charger un JSON Blockbench (format block/item) comme modele d'entite. Le systeme AnimatedBlockModel / AnimatedBlockTESR existant peut etre reutilise car le format est le meme. Les entites, items et blocs partagent le meme format JSON Blockbench.

AnimatedEntityRenderer

Un nouveau renderer qui charge un JSON Blockbench standard et joue les animations .erianim.json pour les entites :

  • Classe : AnimatedEntityRenderer extends RenderLiving
  • Chargement : Au preInit, le renderer charge les fichiers JSON Blockbench depuis assets/modid/models/entity/ en reutilisant le parser existant (AnimatedBlockModel).
  • Rendu : Meme pipeline que les blocs animes — le format JSON est identique (elements, groups, faces, UV per-face).
  • Animation : Les fichiers .erianim.json sont charges et appliques aux groupes du modele, exactement comme pour les blocs.

AnimatedItemRenderer

Extension du systeme d'animation pour supporter les items tenus en main :

  • AnimatedItemRenderer : Un nouveau TESR/renderer d'item qui charge un modele 3D Blockbench et joue les animations .erianim.json — meme pipeline que les blocs animes.
  • Champ "category" : Le .erianim.json contient un champ category ("block", "item", ou "entity") pour que EriAPI sache quel renderer utiliser.

EriEntity builder

Nouveau builder fluent dans le Content system d'EriAPI pour enregistrer des entites custom. La methode .model() accepte un JSON Blockbench standard :

EriEntity.create("eriniumfaction", "monstre") .model("eriniumfaction:entity/monstre") .texture("body", "eriniumfaction:textures/entity/monstre/body") .texture("armor", "eriniumfaction:textures/entity/monstre/armor") .animation("idle", "eriniumfaction:animations/entity/monstre/idle") .animation("attack", "eriniumfaction:animations/entity/monstre/attack") .hitbox(0.8f, 1.8f) .register();

Le builder gere automatiquement :

  • Registration : Enregistrement de l'entite dans le registre Forge (EntityEntry).
  • Renderer : Association automatique du EriEntityRenderer avec les textures et le modele JSON.
  • Animations : Chargement et association des fichiers .erianim.json references.
  • Hitbox : Configuration de la bounding box (largeur, hauteur).
  • Spawn egg : Generation optionnelle d'un spawn egg (couleurs configurables).
  • Seats : Positions d'assise configurables (voir ci-dessous).

Seats — Points d'assise

Un systeme de seats permet de definir un ou plusieurs points d'assise sur une entite. Indispensable pour les montures, vehicules, et meubles interactifs. Les seats sont definis dans le builder Java, pas dans un JSON custom.

  • Multi-seats : Une entite peut avoir autant de seats que necessaire (ex: un vehicule 4 places, un dragon avec un pilote + un passager).
  • Position float : Chaque seat a une position [x, y, z] en coordonnees locales de l'entite (float).
  • Rotation : Rotation du joueur assis (yaw offset) — pour orienter le passager dans la bonne direction.
  • Attachement a un groupe : Un seat peut etre attache a un groupe du modele. Quand le groupe bouge (animation), le seat suit. Ex: le seat du pilote attache au groupe "cockpit" qui tourne.
  • Lock camera : Option pour verrouiller la camera du joueur assis (utile pour les vehicules — le joueur regarde devant).

Dans le builder EriAPI :

EriEntity.create("eriniumfaction", "vehicule") .model("eriniumfaction:entity/vehicule") .seat(0.0f, 1.2f, -0.3f) // seat 0 : pilote .seat(0.0f, 1.2f, 0.8f) // seat 1 : passager .seat(0.6f, 0.8f, 0.0f, "turret") // seat 2 : attache au groupe "turret" .register();
ⓘ
Seats dans le builder Java : Les seats sont definis dans le code Java via le builder EriEntity, pas dans un fichier JSON. Cela simplifie le format et evite un JSON custom pour les entites.
⚠
Impact EriAPI : Ces modifications sont consequentes. Elles doivent etre planifiees dans une version majeure d'EriAPI (ex: 2.0.0). La retrocompatibilite avec les systemes existants (AnimatedBlockTESR, EriItem, etc.) doit etre preservee.

13 Performance

Objectifs

Metrique Objectif Mesure
FPS viewport 60 FPS constant Avec un modele de 1000+ cubes, textures 128x128
Undo/Redo < 16ms (1 frame) Retour instantane, meme avec 500 operations en historique
Chargement projet < 2 secondes Projet .erigineering de 50 MB (textures haute-res)

Import Blockbench < 3 secondes Modele Blockbench complexe avec textures embeddees

Strategies d'optimisation

  • Diff-based Undo : L'historique ne stocke pas des copies completes du modele a chaque etape. Il stocke des diffs (deltas) — seuls les champs modifies sont enregistres. Chaque undo/redo applique le diff inverse.
  • Lazy texture loading : Les textures ne sont chargees en VRAM que lorsqu'elles sont visibles dans le viewport.
  • Instanced rendering : Pour les modeles avec beaucoup de cubes identiques (grid array), Three.js InstancedMesh est utilise pour rendre tous les cubes en un seul draw call.
  • Frustum culling : Les cubes hors du champ de vision de la camera ne sont pas rendus. Three.js le gere nativement mais les groupes entiers sont aussi culles.
  • Texture atlas : Pour le rendu viewport, les multiples textures d'une entite sont compilees en un atlas temporaire pour minimiser les texture binds. L'atlas est regenere uniquement quand une texture change.
  • Web Worker pour l'export : La serialisation JSON et la compression ZIP du format .erigineering sont effectuees dans un Web Worker pour ne pas bloquer l'UI.
✓
Benchmark cible : Un modele d'entite avec 200 cubes, 3 textures 64x64, et une animation de 60 keyframes doit pouvoir etre manipule (rotation, translation, preview animation) a 60 FPS sur un GPU integre (Intel UHD 630).

15 Plan de Developpement (Phases)

5 phases progressives

Le developpement suit un plan en 5 phases. Chaque phase produit un livrable testable. Les phases sont sequentielles — chaque phase depend de la precedente.

Phase Nom Contenu Livrable Statut
1 Scaffold + Migration EriAnim Creer le projet Electron, migrer tout le code d'EriAnim, verifier que l'editeur d'animation fonctionne dans la nouvelle structure. App Electron fonctionnelle avec l'editeur d'animation EriAnim integre. FAIT
2 Architecture + page d'accueil Architecture du projet, page d'accueil avec choix du type de projet (block, item, entity), navigation entre les pages. Application structuree avec page d'accueil fonctionnelle. FAIT
3 Support 3 categories animation Charger des JSON Blockbench (block, item, entity) dans l'editeur d'animation. Ajouter le champ "category" au projet. Preview dans le viewport. Animer des blocs, items et entites depuis des JSON Blockbench.
4 EriAPI : AnimatedEntityRenderer + EriEntity builder AnimatedEntityRenderer qui charge un JSON Blockbench comme entite, EriEntity builder, AnimatedItemRenderer. EriAPI capable de rendre les entites et items animes depuis les JSON Blockbench.
5 Polish, raccourcis, UX, performance Raccourcis configurables, optimisation performance, multi-fenetre, UX polish. Application complete, polie, prete pour la production.
⚠
Phase 4 critique : La phase 4 (EriAPI : AnimatedEntityRenderer) est la plus risquee car elle impacte le code Java du mod. Elle necessite une version majeure d'EriAPI et doit etre testee exhaustivement en jeu pour verifier que les entites et items animes s'affichent correctement dans Minecraft.

16 Limitations & Futur

Limitations actuelles

  • Pas de modelisation : Erigineering ne permet pas de creer ou modifier des modeles 3D. La modelisation se fait dans Blockbench.
  • Pas d'edition de texture : Erigineering ne contient pas d'editeur de texture. Les textures sont creees dans des outils externes (Photoshop, Paint.net, GIMP).
  • Rendu approximatif : Le rendu Three.js (WebGL 2.0) est une approximation du rendu OpenGL 1.1 de Minecraft. Les differences possibles : eclairage, blending, mipmap, rendu des faces transparentes.
  • Pas d'import d'animations Blockbench : Le format d'animation Blockbench (Bedrock/molang) est incompatible avec .erianim.json. Les animations doivent etre recreees manuellement.

Futur (post-v1.0)

  • Integration launcher : Preview in-game directe depuis Erigineering via le launcher EriniumFaction. Le modele est envoye au client de dev et affiche en temps reel dans Minecraft.
  • Plugin system : API JavaScript pour creer des plugins (outils custom, formats d'export additionnels, generateurs proceduraux).
  • Collaboration : Edition collaborative en temps reel (comme Figma). Plusieurs artistes travaillent sur le meme modele simultanement.
  • Marketplace de modeles : Partage de modeles .erigineering entre les createurs de la communaute EriniumFaction.
  • Support 1.16+ : Si EriniumFaction migre un jour vers une version plus recente, adapter les exports pour les nouveaux formats JSON Minecraft.

Metriques de succes

  • Workflow : Un artiste peut animer un modele complet sans quitter Erigineering apres la modelisation dans Blockbench.
  • Productivite : Le temps de creation d'un modele anime est reduit par rapport au workflow EriAnim.
  • Qualite : Les animations exportees fonctionnent sans modification dans le mod EriniumFaction.
  • Adoption : Toutes les nouvelles animations du projet sont creees dans Erigineering.
