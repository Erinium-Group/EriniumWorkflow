01Contexte & Vision

Perimetre de ce cahier

EriAPI est le framework Minecraft Forge 1.12.2 utilise par EriniumFaction comme couche de base. Ce cahier des charges couvre quatre grands domaines :

  • Nouveaux composants GUI — 9 composants s'integrant dans le systeme EriGuiScreen existant (Dropdown, ToggleSwitch, Modal, PlayerHead, ItemStackRenderer, Divider, Badge, Accordion, Carousel)
  • Systeme d'overlay modulaire — architecture complete inspiree de Lunar Client, remplacant le systeme EriOverlay actuel par un systeme que le joueur controle entierement
  • ProgressBar Enhanced — composant de barre de progression polyvalent avec 9 formes geometriques pour les overlays
  • Composants overlay — 6 composants legers optimises pour le rendu en jeu (FloatingText, IconBar, OverlayProgressBar, AnchoredGroup, BlinkingIndicator, OverlayToast)
⚠
Règle absolue — Rétro-compatibilité
Le système d'overlay existant (EriOverlay, OverlayManager, Anchor, OverlayLayer, OverlayEventHandler) est remplacé par le nouveau système modulaire (OverlayMod). Ne JAMAIS supprimer ces fichiers. Ils doivent être marqués @Deprecated avec un message renvoyant vers OverlayMod et OverlayModManager. Le code existant qui les utilise doit continuer à fonctionner.

Philosophie : EriAPI fournit les outils, pas le contenu

💡
Regle fondamentale
EriAPI ne fournit AUCUN mod d'overlay concret. Elle fournit uniquement l'infrastructure. C'est EriniumFaction (ou n'importe quel autre mod) qui cree les OverlayMod specifiques (barre de mana, HUD de faction, timer de sort, etc.).
⚠
Règle de style — Liquid Glass vs rendu en jeu
Liquid Glass s'applique UNIQUEMENT aux GUI EriAPI (GUI Manager, GUI Éditeur, GUI ModSettings) et à l'affichage des mods dans le menu de configuration position/taille (bordures et fond semi-transparent pour visualiser les limites du mod).

En jeu (rendu réel), EriAPI n'impose AUCUN style aux mods d'overlay. Le développeur choisit lui-même le fond, les bordures, le style de son mod — ou n'en met aucun. Un mod peut être totalement transparent, avoir un fond custom, ou n'importe quel style. C'est à lui de gérer le rendu visuel de son contenu.

Résumé : EriAPI = liquid glass pour ses propres écrans de config. OverlayMod en jeu = liberté totale du développeur.

Comparaison : ancien vs nouveau systeme d'overlay

Ancien systeme (EriOverlay)Nouveau systeme (OverlayMod)
Position fixee par le developpeurPosition controlee par le joueur (drag & drop)
Taille fixe par le developpeurScale ajustable par le joueur
Activation/desactivation : codeActivation/desactivation : GUI ingame
Aucune persistance des preferencesPersistance JSON automatique
Aucun GUI de gestionGUI Manager + GUI Editeur (liquid glass)
Pas de concept de "mod d'overlay"Concept clair : chaque feature = un OverlayMod

Compatibilite

Les classes EriOverlay, OverlayManager, Anchor, OverlayLayer et OverlayEventHandler sont conservees mais marquees @Deprecated avec un message renvoyant vers OverlayMod. Aucun code existant ne casse.

Dropdown / Select

Menu deroulant permettant de choisir une valeur parmi une liste. Supporte la recherche textuelle, la navigation clavier et une animation d'ouverture/fermeture fluide.

Constructeur

new Dropdown(int x, int y, int width, int height)

API

MethodeParametre(s)Description
addOption(label, value)String, StringAjoute une entree dans la liste
setSelected(value)StringSelectionne programmatiquement une valeur
getSelected()—Retourne la valeur actuellement selectionnee
onChange(consumer)Consumer<String>Callback appele quand la selection change
placeholder(text)StringTexte affiche quand rien n'est selectionne
maxVisibleItems(n)intNombre max d'items visibles avant scroll (defaut 6)
searchable(enabled)booleanActive le champ de recherche filtrante

Comportement

  • Click sur le composant ouvre la liste deroulante vers le bas (ou vers le haut si pas d'espace)
  • Click en dehors de la liste ferme la liste
  • Scroll dans la liste si plus de maxVisibleItems options
  • Highlight de l'option survolee au hover
  • Navigation clavier : fleches haut/bas, Entree pour valider, Echap pour fermer
  • Animation d'ouverture/fermeture : hauteur slide + fade opacity

Exemple

Dropdown dd = new Dropdown(660, 400, 300, 40) .placeholder("Choisir un rang...") .searchable(true) .maxVisibleItems(8) .onChange(val -> applyRank(val)); dd.addOption("Membre", "member"); dd.addOption("Officier", "officer"); dd.addOption("Chef", "leader"); root.add(dd);

Modal / Dialog

Popup modale s'affichant au-dessus du GUI courant. Bloque les interactions avec le fond tant qu'elle est ouverte. Ferme sur Echap ou click du bouton de fermeture.

Methodes statiques (factory)

MethodeParametre(s)Description
Modal.confirm(title, msg, onConfirm, onCancel)String, String, Runnable, RunnableDialog de confirmation avec boutons Confirmer / Annuler
Modal.alert(title, msg, onClose)String, String, RunnableDialog d'alerte avec un seul bouton OK
Modal.custom(title, width, height)String, int, intModal vide retournant un ContainerComponent pour y ajouter des composants

Comportement

  • Fond sombre semi-transparent (0x88000000) sur tout l'ecran
  • Panneau central en style liquid glass : fond rgba(14,14,22,0.90), bordure rgba(255,255,255,0.12), cornerRadius(16)
  • Animation fade in + scale in a l'ouverture
  • Echap appelle onCancel / onClose et ferme
  • Les interactions (click, scroll, touches) sont absorbees par la modal — le GUI sous-jacent ne recoit rien

Exemple

// Dialog de confirmation Modal.confirm( "Dissoudre la faction ?", "Cette action est irreversible. Tous les membres seront expulses.", () -> factionManager.dissolve(player), () -> {} // annulation : ne rien faire ); // Modal custom ContainerComponent content = Modal.custom("Parametres avances", 500, 400); content.add(new Label(20, 20, 460, 20, "Option X")); content.add(new ToggleSwitch(380, 15, 50, 26));

03Systeme d'Overlay Modulaire

🌠
Concept cle : inspire de Lunar Client
Le joueur devient maitre de son HUD. Il active/desactive chaque element, le deplace ou il veut, ajuste sa taille. Tout se fait via des GUI en jeu. Le developpeur cree seulement des OverlayMod — EriAPI gere le reste.

3.1 — OverlayMod (classe de base)

Classe abstraite que tout developpeur etend pour creer un element d'overlay configurable par le joueur. Chaque OverlayMod possede son propre ContainerComponent racine.

Champs internes

ChampTypeDescription
idStringIdentifiant unique (ex: "eriniumfaction:mana_bar")
displayNameStringNom affiche dans le GUI manager
descriptionStringDescription courte pour le GUI manager
categoryStringCategorie de regroupement (ex: "Combat", "Faction")
rootContainerComponentConteneur racine du mod en design pixels
designWidth / designHeightintTaille de base du mod en pixels design
enabledbooleanSi le mod est actif ou non
posXPercent / posYPercentfloatPosition en % de l'ecran (0.0 = haut/gauche, 1.0 = bas/droite)
scalefloatEchelle de rendu (0.5 a 2.0, defaut 1.0)
builtbooleanFlag lazy-build : buildOverlay() n'est appele qu'une fois

Methodes a surcharger

MethodeObligatoireDescription
protected abstract void buildOverlay()OuiConstruction des composants dans getRoot(). Appele une seule fois au premier rendu.
protected void onTick()NonLogique par tick : mise a jour des valeurs, textes, progress bars. Appele seulement si enabled.
protected void onEnabled()NonAppele quand le joueur active le mod.
protected void onDisabled()NonAppele quand le joueur desactive le mod.

API publique

MethodeParametre(s)Description
getRoot()—Retourne le ContainerComponent racine
setCategory(category)StringDefinit la categorie du mod
setDescription(desc)StringDefinit la description
setDefaultPosition(xPct, yPct)float, floatPosition par defaut (premier lancement, avant chargement JSON)
setMinScale(min)floatScale minimum autorisee (defaut 0.5)
setMaxScale(max)floatScale maximum autorisee (defaut 2.0)
rebuild()—Force la reconstruction des composants (reset built + appel buildOverlay)
getId()—Retourne l'id du mod
getDisplayName()—Retourne le nom affiche
isEnabled()—Retourne l'etat actif/inactif
getScale()—Retourne la scale actuelle
getPosXPercent() / getPosYPercent()—Position en % de l'ecran

Exemple complet — Barre de Mana

public class ManaBarMod extends OverlayMod { private OverlayProgressBar manaBar; private Label manaLabel; public ManaBarMod() { super("eriniumfaction:mana_bar", "Barre de Mana", 220, 36); setCategory("Magie"); setDescription("Affiche votre mana actuelle en bas de l'ecran."); setDefaultPosition(0.5f, 0.92f); // centre en bas setMinScale(0.5f); setMaxScale(2.0f); } @Override protected void buildOverlay() { manaBar = new OverlayProgressBar(0, 0, 220, 16) .fillColor(0xFF00A5FF) .backgroundColor(0x50000000) .cornerRadius(4); manaLabel = new Label(0, 20, 220, 14, "100 / 100") .color(0xCCFFFFFF) .scale(0.8f) .align(Label.Align.CENTER); getRoot().add(manaBar); getRoot().add(manaLabel); } @Override protected void onTick() { FactionCapData cap = EriCap.get(Minecraft.getMinecraft().player, FactionCapData.class); float cur = cap.mana, max = cap.maxMana; manaBar.value(cur / max); manaLabel.setText(String.format("%.0f / %.0f", cur, max)); } }

Enregistrement dans ClientProxy

// ClientProxy.init() OverlayModManager.getInstance().register(new ManaBarMod()); OverlayModManager.getInstance().register(new FactionHudMod()); OverlayModManager.getInstance().register(new SpellCooldownMod()); OverlayModManager.getInstance().loadConfig(); // apres tous les register

3.3 — OverlayModEventHandler (interne)

Classe interne a EriAPI, annotee @Mod.EventBusSubscriber avec modid = "eriapi" et value = Side.CLIENT. Le developpeur n'a pas a la toucher.

  • RenderGameOverlayEvent.Post (ElementType.ALL) → appelle OverlayModManager.renderAll(partialTicks)
  • ClientTickEvent (phase END) → appelle OverlayModManager.tickAll()
  • Verifie !Minecraft.getMinecraft().gameSettings.hideGUI (F1 masque tout)
  • Verifie que le GUI courant n'est pas GuiOverlayEditor (en mode edition, le manager gere le rendu directement)

Comportement technique

  • Fond : le monde Minecraft est rendu normalement, un quad noir rgba(0,0,0,0.30) est pose par-dessus
  • Rendu des mods : chaque mod activé est affiché avec un wrapper liquid glass (fond rgba(255,255,255,0.06), bordure rgba(255,255,255,0.12)) pour rendre ses limites visibles — ce wrapper n'existe PAS en jeu, uniquement dans cet écran de configuration
  • Hover : survol d'un mod → bordure blanche pointillee autour de sa zone
  • Selection : click → bordure cyan pleine, handles de resize (petits carres de 8px) aux 4 coins
  • Drag : maintenir click et deplacer → le mod suit le curseur. Position convertie en % ecran a la release
  • Resize : drag d'un handle de coin → ajuste le scale du mod. Clampe entre minScale et maxScale du mod
  • Snap to grid : si actif, snapping a une grille de 16px. Indicateur visuel : points de grille semi-transparents
  • Snap to center : guides visuels cyan (lignes fines) quand le centre du mod s'approche a moins de 8px du centre horizontal ou vertical de l'ecran
  • Clamping : les mods ne peuvent pas sortir de l'ecran. Position clampee a [0, 1] apres soustraction de la taille du mod
  • Click droit : menu contextuel avec "Reinitialiser position" et "Reinitialiser taille"
  • ESC : sauvegarde automatique + fermeture
  • Barre d'outils : en haut, fond liquid glass. Boutons : "Grille on/off", "Tout reinitialiser", "Sauvegarder & Fermer"

3.6 — Persistance JSON

Les preferences du joueur sont sauvegardees dans .minecraft/config/eriapi/overlay_mods.json via Gson.

Format JSON

{ "eriniumfaction:mana_bar": { "enabled": true, "posX": 0.5, "posY": 0.92, "scale": 1.0 }, "eriniumfaction:faction_hud": { "enabled": true, "posX": 0.02, "posY": 0.02, "scale": 0.9 }, "eriniumfaction:spell_cooldowns": { "enabled": false, "posX": 0.5, "posY": 0.85, "scale": 1.0 } }

Regles de chargement

  • Sauvegarde declenchee automatiquement quand GuiOverlayEditor se ferme et quand saveConfig() est appele manuellement
  • Chargement au FMLInitializationEvent cote client, apres que tous les mods ont appele register()
  • Si un mod enregistre n'a pas d'entree dans le JSON : les valeurs par defaut definies dans son constructeur sont utilisees
  • Si le fichier JSON n'existe pas : il est cree avec les valeurs par defaut de tous les mods
  • Les positions etant en %, elles survivent aux changements de resolution

3.7 — ModSettings (options configurables par mod)

Chaque OverlayMod peut declarer des options configurables via la methode buildSettings(ModSettings). EriAPI genere automatiquement un GUI de settings en style Liquid Glass a partir de ces declarations. Le developpeur ne code aucun GUI de settings — il declare ses options et EriAPI fait le reste.

⚙
Zero boilerplate
Declarer buildSettings() suffit. EriAPI cree automatiquement le bouton "Options" dans le GUI Manager, le GUI GuiModSettings avec tous les composants adequats, la persistance JSON et les callbacks en temps reel.

Packages

// Composants GUI fr.eri.eriapi.gui.components/ ├── Dropdown.java ├── ToggleSwitch.java ├── Modal.java ├── PlayerHead.java ├── ItemStackRenderer.java ├── Divider.java ├── Badge.java ├── Accordion.java ├── Carousel.java ├── ProgressBarEnhanced.java └── Shape.java (enum : HORIZONTAL, VERTICAL, CIRCULAR, PIE, DIAMOND, HEXAGON, OCTAGON, PENTAGON, SQUARE) // Overlay modulaire fr.eri.eriapi.overlay/ ├── OverlayMod.java ← Classe de base abstraite ├── OverlayModManager.java ← Singleton (remplace OverlayManager) ├── OverlayModEventHandler.java ← @Mod.EventBusSubscriber client-only ├── OverlayModConfig.java ← Persistance Gson → JSON ├── components/ │ ├── FloatingText.java │ ├── IconBar.java │ ├── OverlayProgressBar.java │ ├── AnchoredGroup.java │ ├── BlinkingIndicator.java │ └── OverlayToast.java ├── settings/ │ ├── ModSettings.java ← Builder de declaration d'options │ ├── SettingEntry.java ← Entree d'option (key, label, type, value, default) │ └── SettingType.java ← Enum (TOGGLE, SLIDER, SLIDER_INT, DROPDOWN, COLOR_PICKER, TEXT_FIELD, KEYBIND, SEPARATOR, INFO) └── editor/ ├── GuiOverlayManager.java ← GUI activation/desactivation ├── GuiOverlayEditor.java ← GUI drag & drop └── GuiModSettings.java ← GUI settings genere automatiquement (liquid glass) // Deprecated (retro-compatibilite) fr.eri.eriapi.overlay/ ├── @Deprecated EriOverlay.java ├── @Deprecated OverlayManager.java ├── @Deprecated Anchor.java ├── @Deprecated OverlayLayer.java └── @Deprecated OverlayEventHandler.java

Relations et heritage

  • OverlayMod contient un ContainerComponent par composition (pas d'heritage GUI)
  • Tous les composants GUI (Label, ProgressBarEnhanced, etc.) sont utilisables dans OverlayMod.getRoot().add(...)
  • GuiOverlayManager et GuiOverlayEditor etendent EriGuiScreen
  • OverlayModConfig utilise Gson (deja disponible dans le classpath Minecraft Forge) pour la serialisation JSON
  • OverlayModEventHandler ecoute RenderGameOverlayEvent.Post et ClientTickEvent — uniquement cote client
  • OverlayToast est enregistre automatiquement comme un OverlayMod special par EriAPI au chargement, permettant au joueur de positionner la zone de toasts via le GUI Editeur

Flux de rendu

// Chaque frame (RenderGameOverlayEvent.Post, ElementType.ALL) OverlayModEventHandler.onRender(event) { if (gameSettings.hideGUI) return; if (currentGui instanceof GuiOverlayEditor) return; OverlayModManager.getInstance().renderAll(event.getPartialTicks()); } // renderAll() pour chaque OverlayMod enabled : for (OverlayMod mod : enabledMods) { if (!mod.built) { mod.buildOverlay(); mod.built = true; } float screenX = mod.posXPercent * screenWidth; float screenY = mod.posYPercent * screenHeight; // Appliquer scale + translation + rendre le ContainerComponent ScaleManager.renderAt(mod.getRoot(), screenX, screenY, mod.scale, partialTicks); }
