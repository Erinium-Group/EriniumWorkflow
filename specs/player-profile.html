Systeme de Profil Joueur

Un GUI complet de profil joueur avec deux onglets : statistiques de jeu (kills, deaths, KDA, blocs mines/places) et gestion des cosmetiques (inventaire, equipement, preview 3D en temps reel).

01 Contexte & Objectifs

Vision du systeme

Le profil joueur est un hub personnel regroupant toutes les informations et personnalisations liees a un joueur. C'est l'ecran de reference pour consulter ses stats PvP, gerer ses cosmetiques equipes, et voir un apercu de son personnage en temps reel.

Le profil est accessible par le joueur lui-meme (keybind ou commande), et peut aussi etre consulte pour d'autres joueurs (commande avec argument). Quand on consulte le profil d'un autre joueur, les stats sont visibles mais les actions de gestion de cosmetiques sont desactivees.

Objectifs de design

  • Centralisation : Un seul endroit pour voir ses stats et gerer ses cosmetiques. Pas de menus disperses.
  • Feedback visuel : Le joueur voit immediatement l'effet d'un changement de cosmetique sur son modele 3D dans la preview.
  • Fluidite : Navigation par onglets, transitions animees, chargement instantane (donnees locales @Sync).
  • Social : Pouvoir consulter le profil d'un autre joueur encourage la competition (stats) et l'exhibition (cosmetiques).

Dependances

  • Systeme de cosmetiques : Le profil depend de CosmeticPlayerData et CosmeticRegistry (voir spec cosmetics.html).
  • EriAPI Data : Les stats joueur sont stockees via @EriData.
  • EriAPI GUI : Le GUI utilise EriGuiScreen, TabView, ScrollPanel, GridLayout, etc.

02 Structures de Donnees

ProfilePlayerData (@EriData)

Donnees de statistiques du joueur, persistees et synchronisees au client.

@EriData(modId = "eriniumfaction", scope = DataScope.PLAYER, persistOnDeath = true) public class ProfilePlayerData { // --- Stats PvP --- @Sync public int kills = 0; // Nombre total de kills @Sync public int deaths = 0; // Nombre total de morts @Sync public int assists = 0; // Nombre total d'assists // --- Stats monde --- @Sync public int blocksMined = 0; // Nombre total de blocs mines @Sync public int blocksPlaced = 0; // Nombre total de blocs places // --- Metadata --- public long firstJoinTime = 0; // Timestamp de la premiere connexion @Sync public long totalPlayTime = 0; // Temps de jeu total en secondes }

Champs calcules (non stockes)

Ces valeurs sont calculees a la volee depuis les champs stockes :

Stat Formule Affichage
KDA Ratio (kills + assists) / max(1, deaths) Arrondi a 2 decimales. Ex: "2.45"
Kill/Death Ratio kills / max(1, deaths) Arrondi a 2 decimales. Ex: "1.87"
Temps de jeu formate totalPlayTime converti en jours/heures/minutes Ex: "12j 5h 32m"

Tracking des stats

Les stats sont incrementees via des event listeners Forge :

Stat Event Forge Conditions
kills LivingDeathEvent La source de degats est un joueur (verifier event.getSource().getTrueSource() instanceof EntityPlayer). La victime est un autre joueur.
deaths LivingDeathEvent L'entite morte est un joueur.
assists LivingDeathEvent Un joueur qui a inflige des degats a la victime dans les 10 derniers secondes avant sa mort, mais qui n'est pas le tueur. Necessite un cache temporaire de derniers dommages (Map<UUID, List<DamageEntry>>).
blocksMined BlockEvent.BreakEvent Le joueur casse un bloc en mode survie. Les blocs creatifs ne comptent pas.
blocksPlaced BlockEvent.PlaceEvent Le joueur place un bloc en mode survie.
totalPlayTime TickEvent.PlayerTickEvent Incremente de 1 toutes les 20 ticks (= 1 seconde). Utiliser un compteur de ticks pour ne PAS verifier chaque tick (perf avec 500+ joueurs).
⚠
Performance du tracking : Le totalPlayTime ne doit PAS incrementer a chaque tick. Utiliser un compteur modulo 20 (if (player.ticksExisted % 20 == 0) totalPlayTime++). Pour les assists, le cache de derniers dommages est nettoye toutes les 200 ticks (10 sec) pour eviter la croissance memoire. Les entrees expirees sont supprimees paresseusement.

Systeme de tracking des assists

public class AssistTracker { // victimUUID -> liste de (attackerUUID, timestamp) private final Map<UUID, List<DamageEntry>> recentDamage = new HashMap<>(); // Appele dans LivingHurtEvent public void recordDamage(UUID victim, UUID attacker) { recentDamage .computeIfAbsent(victim, k -> new ArrayList<>()) .add(new DamageEntry(attacker, System.currentTimeMillis())); } // Appele dans LivingDeathEvent — retourne les UUIDs des assistants public Set<UUID> getAssists(UUID victim, UUID killer) { List<DamageEntry> entries = recentDamage.remove(victim); if (entries == null) return Collections.emptySet(); long cutoff = System.currentTimeMillis() - 10_000; // 10 secondes Set<UUID> assists = new HashSet<>(); for (DamageEntry e : entries) { if (e.timestamp >= cutoff && !e.attacker.equals(killer)) { assists.add(e.attacker); } } return assists; } } static class DamageEntry { final UUID attacker; final long timestamp; }

03 GUI Profil — Layout General

Acces

  • Keybind : Touche configurable (defaut : P). Enregistree via EriKeys.create("eriniumfaction.open_profile"), contexte KeyContext.IN_GAME. Ouvre le profil du joueur qui appuie.
  • Commande : /profile (sans arguments) ouvre le profil du joueur qui tape la commande. /profile <player> ouvre le profil d'un autre joueur (consultation uniquement).

Structure de l'ecran (1920x1080 design pixels)

+==================================================================+ | PROFIL DE SteveKiller42 Faction : Les Templiers | | Membre depuis le 15/03/2026 Temps de jeu : 12j 5h 32m | +===========================+======================================+ | | | | +-------------------+ | [ STATISTIQUES ] [ COSMETIQUES ] | | | | | | | | PREVIEW 3D | | Contenu de l'onglet actif... | | | DU JOUEUR | | (voir sections 04 et 05) | | | | | | | | (rotation au | | | | | drag souris) | | | | | | | | | +-------------------+ | | | | | +===========================+======================================+

Elements structurels

  • Header (haut) : Nom du joueur en grand (scale 2.5), nom de la faction (si applicable), date de premiere connexion, temps de jeu total. Fond violet gradie translucide.
  • Panneau gauche (400px largeur) : Preview 3D du joueur en pied avec ses cosmetiques equipes. Le joueur peut cliquer-glisser pour faire tourner le modele. Fond sombre avec un cercle lumineux sous les pieds du modele.
  • Panneau droit (reste de la largeur) : Systeme d'onglets (TabView EriAPI) avec deux onglets : "Statistiques" et "Cosmetiques".
  • Style : Liquid Glass — panneaux translucides, bordures subtiles, blur d'arriere-plan, animations de transition entre onglets (slideIn/slideOut).

Mode consultation (profil d'un autre joueur)

Quand le joueur consulte le profil d'un autre, le GUI est identique SAUF :

  • Le header indique "Profil de [NomAutreJoueur]" au lieu du nom du joueur local.
  • L'onglet Cosmetiques est en lecture seule : les boutons "Equiper" et "Desequiper" sont absents. Le joueur voit les cosmetiques equipes par l'autre joueur mais ne peut pas les modifier.
  • La preview 3D montre le skin et les cosmetiques de l'autre joueur.

04 Onglet Statistiques

Layout de l'onglet

+======================================================+ | [STATISTIQUES] [COSMETIQUES] | +------------------------------------------------------+ | | | --- COMBAT --- | | +----------+ +----------+ +----------+ +--------+| | | 245 | | 87 | | 62 | | 2.45 || | | Kills | | Deaths | | Assists | | KDA || | +----------+ +----------+ +----------+ +--------+| | | | --- MONDE --- | | +----------------+ +----------------+ | | | 12,456 | | 8,231 | | | | Blocs mines | | Blocs places | | | +----------------+ +----------------+ | | | +======================================================+

Cartes de statistiques

Chaque statistique est affichee dans une carte individuelle (Rectangle arrondi, fond translucide sombre). La valeur numerique est en grand (scale 3.0, couleur cyan pour les positives, rouge pour les negatives). Le label est en dessous en petite police (slate).

Carte Valeur Couleur Format
Kills ProfilePlayerData.kills Vert (#2ECC71) Entier avec separateur de milliers (ex: "1,245")
Deaths ProfilePlayerData.deaths Rouge (#E74C3C) Entier avec separateur de milliers
Assists ProfilePlayerData.assists Cyan (#00E5FF) Entier avec separateur de milliers
KDA Calcule : (kills + assists) / max(1, deaths) Or (#F1C40F) Decimal a 2 chiffres (ex: "2.45")
Blocs mines ProfilePlayerData.blocksMined Cyan (#00E5FF) Entier avec separateur de milliers
Blocs places ProfilePlayerData.blocksPlaced Cyan (#00E5FF) Entier avec separateur de milliers

Sections

  • "COMBAT" : Premiere rangee — Kills, Deaths, Assists, KDA. Separateur visuel (ligne horizontale avec label "COMBAT" en lettres capitales, couleur or).
  • "MONDE" : Deuxieme rangee — Blocs mines, Blocs places. Meme separateur avec label "MONDE".
ⓘ
Animation a l'ouverture : Les cartes de stats apparaissent avec un fadeIn(10) + slideIn(Direction.LEFT, 20, 8) en sequence (0.1 sec de decalage entre chaque carte), donnant un effet de cascade visuel.

05 Onglet Cosmetiques

Layout de l'onglet

+======================================================+ | [STATISTIQUES] [COSMETIQUES] | +------------------------------------------------------+ | | | Filtrer par slot : [Tous] [Casque] [Plastron] ... | | Filtrer par collection : [Toutes] [Samourai] ... | | | | +--------+ +--------+ +--------+ +--------+ | | | Casque | |Plastron| |Bottes | |Sac a | | | |Samourai| |Samourai| |Samourai| |dos | | | | (slot) | | (slot) | | (slot) | |Dragon | | | |[EQUIPE]| |[EQUIPER]| |[EQUIPER]| |[EQUIPER]| | | +--------+ +--------+ +--------+ +--------+ | | | | (ScrollPanel si beaucoup de cosmetiques) | | | +======================================================+

Filtres

Deux barres de filtres en haut de l'onglet :

  • Filtre par slot : Boutons radio horizontaux — "Tous", "Casque", "Plastron", "Jambieres", "Bottes", "Sac a dos". Defaut : "Tous". Filtre les cosmetiques affiches par leur slot.
  • Filtre par collection : Boutons radio horizontaux — "Toutes" + un bouton par collection qui contient au moins 1 cosmetique possede. Defaut : "Toutes".

Les deux filtres sont cumulatifs. Selectionner "Casque" et "Samourai" n'affiche que les casques de la collection Samourai que le joueur possede.

Grille de cosmetiques

Les cosmetiques sont affiches dans un ScrollPanel contenant un GridLayout (4 colonnes, espacement 16px). Chaque cellule est une carte de cosmetique :

  • Fond : Rectangle arrondi translucide (120x160 design pixels).
  • Nom : Nom du cosmetique (Label, center aligned, scale 1.0).
  • Collection : Nom de la collection en petit sous le nom (slate, scale 0.8).
  • Slot : Badge indiquant le slot (ex: "HELMET") en bas a gauche.
  • Bouton action :
    • Si le cosmetique est actuellement equipe dans son slot : bouton vert "EQUIPE" (cliquable pour desequiper).
    • Si le cosmetique n'est pas equipe : bouton bleu "EQUIPER" (cliquable pour equiper). Si un autre cosmetique est deja dans ce slot, il est automatiquement remplace.
  • Hover : Bordure lumineuse, scale légère (1.02), tooltip avec la description de la collection.

Actions d'equipement

  • Equiper : Cliquer "EQUIPER" envoie un PacketCosmeticEquip (action "equip") au serveur. Le serveur valide (le joueur possede bien ce cosmetique), met a jour equippedCosmetics, sauvegarde, sync au client, et broadcast aux joueurs proches. La preview 3D a gauche se met a jour immediatement.
  • Desequiper : Cliquer "EQUIPE" (sur un cosmetique deja equipe) envoie un PacketCosmeticEquip (action "unequip"). Le slot est vide, le rendu vanilla est restaure.
ⓘ
Seuls les cosmetiques possedes : L'onglet Cosmetiques n'affiche que les cosmetiques que le joueur possede (CosmeticPlayerData.ownedCosmetics). Les cosmetiques non possedes ne sont pas visibles ici — ils sont dans la boutique (voir spec cosmetics.html).

06 Preview 3D du Joueur

Principe

Le panneau gauche du GUI affiche un rendu 3D du modele du joueur (ou du joueur consulte) en pied, avec ses cosmetiques equipes. C'est un rendu OpenGL en temps reel dans un ScissorBox delimite au panneau gauche.

Fonctionnalites de la preview

  • Rotation : Le joueur peut cliquer-glisser (clic gauche maintenu) pour faire tourner le modele sur l'axe Y (horizontal). La rotation est fluide avec un easing sur le relachement (inertie).
  • Zoom : La molette de la souris (quand le curseur est sur le panneau) permet de zoomer/dezoomer. Limites : zoom x0.5 a x2.0. Easing sur le zoom.
  • Skin : Le rendu utilise le skin reel du joueur (texture de peau Mojang). Pour un autre joueur, son skin est charge via le NetworkPlayerInfo disponible cote client.
  • Cosmetiques : Les cosmetiques equipes sont rendus sur le modele, exactement comme ils apparaissent en jeu. Quand le joueur equipe/desequipe un cosmetique dans l'onglet droite, la preview se met a jour instantanement.
  • Animation idle : Le modele execute une animation idle legere (balancement lent du corps, respiration subtile) pour donner une impression de vie.
  • Eclairage : Eclairage fixe (une lumiere directionnelle a 45 degres depuis le haut-gauche + lumiere ambiante douce) pour mettre en valeur les cosmetiques 3D.
  • Socle : Un cercle lumineux (gradient radial, couleur violet/cyan) sous les pieds du modele, avec un reflet subtil du modele (miroir a faible opacite).

Implementation technique

// Dans le drawScreen() du GUI, apres le rendu des composants // Le rendu du joueur utilise GuiInventory.drawEntityOnScreen() comme base // modifie pour supporter la rotation libre et les cosmetiques GlStateManager.enableDepth(); GlStateManager.pushMatrix(); // Scissor pour ne pas deborder du panneau gauche ScissorHelper.enable(panelX, panelY, panelWidth, panelHeight); // Translation au centre du panneau float centerX = panelX + panelWidth / 2.0f; float centerY = panelY + panelHeight * 0.85f; // pieds en bas float scale = 80.0f * zoomFactor; // taille du modele // Rendu de l'entite avec rotation custom drawEntityOnScreen(centerX, centerY, scale, rotationYaw, -15.0f, entity); ScissorHelper.disable(); GlStateManager.popMatrix(); GlStateManager.disableDepth();

07 Packets Reseau

Vue d'ensemble

Packet Direction Declencheur Contenu
PacketProfileRequest Client → Serveur Le joueur ouvre le profil d'un autre joueur (/profile <name>) UUID du joueur cible
PacketProfileData Serveur → Client Reponse a PacketProfileRequest Toutes les stats du joueur cible (kills, deaths, assists, blocksMined, blocksPlaced, totalPlayTime, firstJoinTime) + cosmetiques equipes + nom de faction
ⓘ
Profil de soi-meme : Quand le joueur ouvre son propre profil, aucun packet n'est envoye. Les donnees sont deja disponibles localement grace au @Sync de ProfilePlayerData et CosmeticPlayerData. Le PacketProfileRequest n'est utilise que pour consulter le profil d'un autre joueur.

Securite des packets

  • Rate limiting : Un joueur ne peut envoyer qu'un PacketProfileRequest toutes les 2 secondes (cooldown serveur). Les requetes excessives sont ignorees.
  • Joueur hors ligne : Si le joueur cible est hors ligne, le serveur repond avec un packet d'erreur (message chat : "Ce joueur n'est pas en ligne"). Les donnees offline ne sont pas chargees (performance).
  • UUID forge : Le serveur valide que l'UUID correspond a un joueur connecte avant de repondre.

09 Cas Limites

Stats a zero

Un nouveau joueur a toutes ses stats a 0. Le KDA est affiche comme "0.00" (pas de division par zero grace au max(1, deaths)). Le temps de jeu est "0h 0m". Les cartes sont quand meme affichees — pas de message "Aucune stat disponible".

Joueur sans faction

Si le joueur n'est dans aucune faction, le champ "Faction" dans le header est remplace par "Sans faction" en gris italique.

Joueur sans cosmetiques

Si le joueur ne possede aucun cosmetique, l'onglet Cosmetiques affiche un message centre : "Vous ne possedez aucun cosmetique. Visitez la boutique pour en obtenir !" avec un bouton "Ouvrir la boutique" qui ouvre le GUI cosmetic shop.

Profil d'un joueur deconnecte pendant la consultation

Si le joueur consulte deconnecte pendant que le GUI est ouvert, les donnees deja chargees restent affichees (pas de fermeture automatique du GUI). Le header affiche un badge "HORS LIGNE" en rouge. Les cosmetiques equipes restent affiches tels qu'ils etaient au moment du chargement.

Overflow des compteurs

Les compteurs sont des int (max 2,147,483,647). Un joueur qui mine plus de 2 milliards de blocs est peu probable, mais si cela arrive, le compteur clippe a Integer.MAX_VALUE plutot que de wraparound. Verifier avant l'incrementation : if (data.blocksMined < Integer.MAX_VALUE) data.blocksMined++.

Assists : auto-assist

Un joueur ne peut PAS obtenir un assist sur sa propre mort (tomber dans le vide apres avoir ete frappe). Le tueur est explicitement exclus de la liste d'assists. Un joueur qui se suicide (mort sans source de degats joueur) ne genere aucun assist.

Cosmetique possede mais supprime du registre

Si un cosmetique possede n'existe plus dans le CosmeticRegistry, il est affiche dans la grille avec un style "indisponible" (grise, barre, tooltip "Ce cosmetique a ete retire"). Le bouton "Equiper" est desactive. Si le cosmetique etait equipe, il a ete desequipe lors du reload (voir spec cosmetics.html).

10 Plan de Tests

Tests fonctionnels — Stats

  • Kill d'un joueur incremente kills du tueur et deaths de la victime
  • Un joueur qui inflige des degats dans les 10 sec avant un kill obtient un assist
  • Le tueur ne recoit PAS d'assist sur son propre kill
  • Les degats infliges il y a plus de 10 secondes ne comptent pas comme assist
  • Casser un bloc en mode survie incremente blocksMined
  • Placer un bloc en mode survie incremente blocksPlaced
  • Les blocs casses/places en creatif ne comptent PAS
  • Le totalPlayTime s'incremente de 1 par seconde de jeu en ligne
  • Toutes les stats survivent a la mort du joueur (persistOnDeath)
  • Toutes les stats survivent au redemarrage du serveur

Tests fonctionnels — GUI

  • Le GUI s'ouvre via keybind (P) et commande (/profile)
  • /profile <name> ouvre le profil d'un autre joueur en lecture seule
  • La preview 3D affiche le skin correct et les cosmetiques equipes
  • La rotation par drag fonctionne fluidement
  • Le zoom par molette fonctionne avec limites
  • L'onglet Statistiques affiche toutes les stats avec les bonnes valeurs
  • Le KDA est calcule correctement (pas de division par zero)
  • L'onglet Cosmetiques n'affiche que les cosmetiques possedes
  • Les filtres par slot et par collection fonctionnent (cumulatifs)
  • Equiper un cosmetique met a jour la preview 3D immediatement
  • Desequiper un cosmetique restaure le rendu vanilla dans la preview
  • Le mode consultation (autre joueur) desactive les boutons Equiper

Tests de performance

  • Le tracking totalPlayTime utilise un modulo 20, pas de tick inutile
  • Le cache d'assists est nettoye regulierement (pas de fuite memoire)
  • Le PacketProfileRequest a un rate limit de 2 sec
  • Ouvrir le profil avec 200+ cosmetiques possedes : pas de lag
  • La preview 3D ne cause pas de chute de FPS significative

Tests de securite

  • PacketProfileRequest avec UUID inexistant : rejete avec message d'erreur
  • Spam de PacketProfileRequest : rate limited, pas de surcharge serveur
  • /profile set par un joueur non-admin : permission refusee
  • /profile reset par un joueur non-admin : permission refusee
