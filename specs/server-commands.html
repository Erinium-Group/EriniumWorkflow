Systeme complet de commandes de teleportation et d'utilitaires pour le serveur EriniumFaction : spawn, homes, warps, TPA, /back, RTP et commandes d'administration, avec warmup, cooldowns et integration au systeme de grades (permissions a valeur entiere).

01 Contexte & Objectifs

Contexte

EriniumFaction est un serveur PvP/Faction 1.8 (sans cooldown d'attaque). Les joueurs ont besoin d'un ensemble de commandes de mobilite pour naviguer dans le monde, rejoindre leurs allies, retourner a leur base et acceder aux points d'interet du serveur. Ces commandes doivent s'integrer parfaitement au systeme de grades existant (RankManager, PermissionResolver) et utiliser EriAPI pour la planification, le chat enrichi et la persistence des donnees.

Objectifs fonctionnels

  • Spawn : point d'ancrage central du serveur, accessible a tous avec warmup.
  • Homes : bases personnelles, nombre limite par grade (permission a valeur entiere homes.max=N).
  • Warps : points d'interet administres, filtres par permission individuelle par warp.
  • TPA : teleportation sociale entre joueurs avec systeme de demande/acceptation.
  • Back : retour rapide apres une mort ou une teleportation, ideal pour recuperer du loot.
  • RTP : decouverte du monde via teleportation aleatoire securisee.
  • Admin TP : commandes d'administration sans warmup ni cooldown.

Contraintes techniques

  • Java 8 obligatoire — pas de features Java 9+.
  • EriAPI pour toutes les commandes (EriCommand), la planification (EriScheduler), le chat (EriChat) et les donnees joueur (DataManager).
  • Le serveur utilise PvP 1.8 : ne jamais se baser sur ATTACK_SPEED.
  • Les warmups et cooldowns sont configures via CommandConfig.java (modifiable sans recompilation).
  • Les cooldowns sont en memoire uniquement (reset au redemarrage du serveur — intentionnel).
  • Toutes les commandes avec des arguments dynamiques doivent avoir de l'auto-completion.
  • Toute nouvelle permission doit etre documentee dans docs/permissions.md.
ⓘ
Priorite d'implementation : Spawn → Homes → Warps → TPA → Back → RTP → Admin TP. Le systeme de warmup partage doit etre cree en premier car il est utilise par toutes les commandes de teleportation.

02 Systeme Spawn — /setspawn & /spawn

Vue d'ensemble

Le systeme de spawn est le point d'ancrage central du serveur. L'administrateur definit un point de spawn unique via /setspawn et les joueurs peuvent s'y teleporter a tout moment via /spawn, soumis a un warmup et un cooldown configurables.

WarmupNon

StockageSpawnData (WorldSavedData)

Cette commande enregistre la position exacte du joueur (monde, X, Y, Z, yaw, pitch) dans SpawnData. Elle remplace le spawn precedemment defini. Un message de confirmation est envoye avec les coordonnees.

Commande /spawn

ParametreValeur
Syntaxe/spawn

WarmupOui — configurable (spawnWarmup, defaut 3s)
CooldownOui — configurable (spawnCooldown, defaut 30s)

⚠
Cas limite : si aucun spawn n'a ete defini via /setspawn, la commande affiche un message d'erreur : "Aucun spawn n'a ete configure. Contactez un administrateur." Elle ne teleporte pas au spawn vanilla de la map (0,0 world).

Comportement detaille

  • Le joueur execute /spawn.
  • Verification du cooldown — si en cooldown, affichage du temps restant en secondes.
  • Si le spawn n'est pas configure → message d'erreur, arret.
  • Lancement du warmup via TeleportWarmup avec affichage titre/actionbar.
  • Si le joueur bouge de plus de 0.5 bloc → annulation et notification.
  • A expiration du warmup → teleportation au spawn, enregistrement de la position precedente dans BackTracker.
  • Application du cooldown post-teleportation.

SpawnData — WorldSavedData

03 Systeme Homes — /sethome, /home, /delhome, /homes

Vue d'ensemble

Les homes sont des points de teleportation personnels enregistres par chaque joueur. Le nombre maximum de homes qu'un joueur peut posseder est determine par la permission a valeur entiere homes.max=N de son grade. Si aucun grade ne definit cette valeur, la valeur par defaut de CommandConfig.homeMaxDefault s'applique (defaut : 1).

Tableau des commandes

CommandeSyntaxePermissionWarmupCooldown
/sethome /sethome [nom] Aucune Non Non
/home /home [nom] Aucune homeWarmup homeCooldown
/delhome /delhome <nom> Aucune Non Non
/homes /homes Aucune Non Non

/sethome [nom]

  • Le parametre nom est optionnel. Si omis, le home s'appelle home.
  • Le nom est en minuscule, alphanumerique, max 16 caracteres. Caracteres speciaux refuses.
  • Verification du nombre de homes existants vs homes.max du grade.
  • Si la limite est atteinte → message d'erreur avec le nombre maximum et la commande /delhome.
  • Ecrasement silencieux si un home du meme nom existe deja (mise a jour de la position).
  • Stockage via HomePlayerData (EriData, persistant apres mort).

/home [nom]

  • Le parametre nom est optionnel. Si omis, utilise home.
  • Si le home demande n'existe pas → message d'erreur listant les homes disponibles.
  • Processus : verification cooldown → lancement warmup → teleportation → enregistrement position precedente dans BackTracker → application cooldown.
  • L'auto-completion liste uniquement les homes du joueur executant la commande.

/delhome <nom>

  • Argument obligatoire. L'auto-completion liste les homes du joueur.
  • Si le home n'existe pas → message d'erreur.
  • Demande de confirmation optionnelle (configurable via CommandConfig.homeDeleteConfirm).

/homes

Liste tous les homes du joueur avec leurs coordonnees et leur dimension via EriChat.

// Exemple de sortie chat de /homes --- Vos homes (2/3) --- home → Overworld X:-142 Y:64 Z:351 base2 → Overworld X:890 Y:72 Z:-220

Un helper statique LocationSerializer.serialize(world, x, y, z, yaw, pitch) et LocationSerializer.deserialize(String) sera cree dans le package command/.

04 Systeme Warps — /setwarp, /delwarp, /warp, /warps

Vue d'ensemble

Les warps sont des points de teleportation globaux administres par les operateurs du serveur. Chaque warp possede sa propre permission warp.<nom>. Un joueur sans cette permission ne peut pas voir le warp dans l'auto-completion et ne peut pas l'utiliser.

Tableau des commandes

CommandeSyntaxePermissionWarmupCooldown
/setwarp /setwarp <nom> command.setwarp Admin Non Non
/delwarp /delwarp <nom> command.delwarp Admin Non Non
/warp /warp [nom] warp.<nom> par warp warpWarmup warpCooldown
/warps /warps Aucune Non Non

/setwarp <nom>

  • Cree un warp au nom donne a la position exacte de l'administrateur.
  • Si un warp du meme nom existe deja → confirmation ou ecrasement selon config.
  • La permission warp.<nom> est generee automatiquement avec le warp et documentee dans la sortie de la commande.
  • Le nom du warp est en minuscule, alphanumerique + tirets, max 24 caracteres.
  • Stockage dans WarpWorldData (WorldSavedData).

/delwarp <nom>

  • Supprime le warp. L'auto-completion liste tous les warps existants (admin voit tout).
  • Confirmation obligatoire : le joueur doit re-executer la commande avec --confirm ou cliquer sur un bouton chat.

/warp [nom]

  • Si nom est omis → envoie la liste des warps accessibles (equivalent a /warps).
  • Verification de la permission warp.<nom> — si refusee : "Ce warp n'existe pas ou vous n'avez pas la permission d'y acceder." (message identique pour cacher l'existence du warp).
  • Processus standard : cooldown → warmup → teleportation → BackTracker → cooldown.
  • L'auto-completion ne liste que les warps pour lesquels le joueur a la permission.

/warps

Liste tous les warps auxquels le joueur a acces, avec coordonnees et dimension. Utilise EriChat avec pages si la liste est longue (ChatPaginator).

05 Systeme TPA — Demandes de Teleportation

Vue d'ensemble

Le systeme TPA permet aux joueurs de se teleporter les uns vers les autres apres accord mutuel. Deux types de demandes existent : /tpa (demande d'aller vers le joueur cible) et /tpahere (demande au joueur cible de venir chez vous). Les demandes expirent automatiquement apres un delai configurable (tpaTimeout, defaut 60s).

Tableau des commandes

CommandeSyntaxePermissionDescription
/tpa /tpa <joueur> command.tpa Demande a se teleporter vers le joueur cible
/tpahere /tpahere <joueur> command.tpahere Demande au joueur cible de venir chez vous
/tpaccept ou /tpyes /tpaccept Aucune Accepte la demande TPA en cours
/tpdeny ou /tpno /tpdeny Aucune Refuse la demande TPA en cours

Flux de demande TPA

Joueur A execute /tpa JoueurB │ ├─ [Echec] A n'a pas la permission command.tpa ├─ [Echec] B n'est pas connecte ├─ [Echec] A a deja une demande en attente (annuler avec /tpdeny) └─ [OK] │ ├─ TpaManager.addRequest(A → B, type=TPA) ├─ Planification expiration dans tpaTimeout ticks ├─ Message A : "Demande envoyee a JoueurB. Expire dans 60s." └─ Message B (EriChat clickable) : "[JoueurA veut se teleporter vers vous]" [ACCEPTER] [REFUSER] JoueurB clique [ACCEPTER] → execute /tpaccept │ └─ [OK] │ ├─ TpaManager retire la demande ├─ Lancement warmup pour le joueur qui se deplace (A dans /tpa, B dans /tpahere) └─ Apres warmup → teleportation → BackTracker → cooldown tpa

Message chat cliquable (EriChat)

").color(TextFormatting.GRAY) .text("[ACCEPTER]").color(TextFormatting.GREEN).bold() .command("/tpaccept") .hover("Cliquer pour accepter la demande") .text(" ") .text("[REFUSER]").color(TextFormatting.RED).bold() .command("/tpdeny") .hover("Cliquer pour refuser la demande") .to(targetPlayer);

TpaManager — Singleton

  • Stockage en memoire : Map<UUID, TpaRequest> — cle = UUID du joueur qui doit repondre.
  • Un joueur ne peut avoir qu'une seule demande en attente en tant que demandeur ET en tant que cible.
  • Nouvelle demande recue alors qu'une est deja en attente : annulation de l'ancienne, remplacement.
  • Expiration via EriScheduler.delay() — si la demande n'a pas ete traitee, elle est supprimee et les deux joueurs sont notifies.
  • Cooldown post-teleportation : tpaCooldown applique au demandeur.

TpaRequest — Structure de donnee

public class TpaRequest { public enum Type { TPA, TPAHERE } public final UUID requester; // Joueur qui a execute /tpa ou /tpahere public final UUID target; // Joueur qui doit repondre public final Type type; // TPA = requester se teleporte, TPAHERE = target se teleporte public final long createdAt; // System.currentTimeMillis() public ScheduledTask expiryTask; // Task EriScheduler pour l'expiration }
⚠
Cas limite : si le demandeur ou la cible se deconnecte avant l'acceptation, la demande est annulee immediatement via l'event PlayerEvent.PlayerLoggedOutEvent. Le TpaManager s'abonne a cet event via EriEvents.

06 Systeme Back — /back

Vue d'ensemble

La commande /back teleporte le joueur a sa derniere position notable : soit sa derniere position avant une teleportation, soit sa derniere position de mort. C'est une commande tres utile pour recuperer son loot apres une mort inattendue.

ParametreValeur
Syntaxe/back
Permissioncommand.back
WarmupNon (teleportation immediate)
CooldownbackCooldown (defaut 60s)
PersistanceMemoire uniquement (reset a la deconnexion)

Sources de la position de retour

  • Mort : position au moment de la mort, via l'event LivingDeathEvent (filtre sur EntityPlayerMP). Priorite haute.
  • Teleportation : position du joueur juste avant toute teleportation effectuee par ce systeme (spawn, home, warp, tpa, rtp, admin tp).
  • L'evenement le plus recent l'emporte : si le joueur meurt puis fait /spawn, /back le retourne au point de spawn (derniere teleportation), pas au lieu de mort.

BackTracker — Classe utilitaire

public class BackTracker { private static final Map<UUID, SavedLocation> lastPositions = new HashMap<>(); // Appelee par chaque commande de tp ET par l'event de mort public static void save(EntityPlayerMP player) { lastPositions.put(player.getUniqueID(), new SavedLocation(player.world, player.posX, player.posY, player.posZ, player.rotationYaw, player.rotationPitch)); } public static SavedLocation get(UUID uuid) { return lastPositions.get(uuid); // null si aucune position enregistree } // Nettoyer a la deconnexion du joueur public static void clear(UUID uuid) { lastPositions.remove(uuid); } }

Comportement

  • Si aucune position n'est enregistree (joueur n'a jamais teleporte ni est mort depuis sa connexion) → message d'erreur : "Aucune position de retour disponible."
  • Apres le /back → la position de retour est consommee (une seule utilisation par position).
  • La position de retour du /back lui-meme est sauvegardee dans BackTracker (permet un double /back pour faire des allers-retours).

07 Teleportation Aleatoire — /rtp

Vue d'ensemble

La commande /rtp teleporte le joueur a un emplacement aleatoire dans le monde principal, a une distance configurable du spawn. Elle garantit un atterrissage sur un sol solide (pas dans le vide, pas dans la lave, pas dans l'eau). Elle est desactivee dans la dimension monde minage.

ParametreValeur
Syntaxe/rtp
Permissioncommand.rtp
WarmupOui — utilise le warmup standard (3s)
CooldownrtpCooldown (defaut 300s / 5 min)
Dimension autoriseeOverworld uniquement (dimension 0)
Rayon minrtpMinRadius (defaut 1000 blocs du spawn)
Rayon maxrtpMaxRadius (defaut 5000 blocs du spawn)

Algorithme de recherche de position (RtpHelper)

RtpHelper.findSafeLocation(world, minRadius, maxRadius) Boucle (max 10 tentatives) │ ├─ Generer angle aleatoire (0-360°) ├─ Generer distance aleatoire [minRadius, maxRadius] ├─ Calculer X = spawnX + cos(angle) * distance ├─ Calculer Z = spawnZ + sin(angle) * distance ├─ Forcer le chargement du chunk (ChunkLoader) ├─ Trouver Y : descendre depuis 255 jusqu'au premier bloc solide │ ├─ [Echec] Y < 5 (vide) ├─ [Echec] Bloc sous les pieds = LAVA ├─ [Echec] Bloc au niveau des pieds ou de la tete != AIR └─ [OK] Retourner (X, Y+1, Z) Si 10 tentatives echouees → message d'erreur, cooldown non applique

Comportement

  • Si le joueur est dans la dimension monde minage → message d'erreur : "Le RTP n'est pas disponible dans la dimension minage."
  • Le calcul de la position est effectue de maniere synchrone (simple scan Y, pas de generation de chunks complexe).
  • Apres teleportation reussie → position precedente sauvegardee dans BackTracker + cooldown applique.
  • Si la recherche echoue → le cooldown N'est PAS applique (le joueur peut reessayer immediatement).
⚠
Performance : forcer le chargement d'un chunk distant peut provoquer une legere latence. Le calcul est fait en main thread, mais reste acceptable car la boucle est limitee a 10 iterations maximum et le scan Y est O(255).

08 Administration — /tp, /tphere, /tppos

Vue d'ensemble

Ces commandes sont reservees aux administrateurs et operateurs. Elles ne sont soumises ni au warmup ni au cooldown. Elles permettent une gestion rapide des joueurs sur le serveur. La position precedente du joueur deplace est sauvegardee dans BackTracker.

CommandeSyntaxePermissionDescription
/tp /tp <joueur> command.tp Admin Se teleporter vers un joueur instantanement
/tphere /tphere <joueur> command.tphere Admin Teleporter un joueur vers soi instantanement
/tppos /tppos <x> <y> <z> command.tppos Admin Se teleporter a des coordonnees precises

Details /tp <joueur>

  • L'auto-completion liste tous les joueurs en ligne.
  • La position de l'executant est sauvegardee dans BackTracker avant la teleportation.
  • Message de confirmation : "Teleportation vers [joueur] effectuee."

Details /tphere <joueur>

  • L'auto-completion liste tous les joueurs en ligne.
  • La position du joueur deplace est sauvegardee dans BackTracker (il pourra utiliser /back).
  • Notification au joueur deplace : "Vous avez ete teleporte vers [admin] par [admin]."

Details /tppos <x> <y> <z>

  • Les arguments X, Y, Z sont des entiers ou des decimaux.
  • La dimension cible est celle de l'executant (pas de changement de dimension via cette commande).
  • Verification Y : si Y < 0 ou Y > 255 → message d'erreur.
  • Position precedente sauvegardee dans BackTracker.

09 Systeme de Warmup — Delai avant Teleportation

Concept

Le warmup est un delai obligatoire entre le declenchement d'une teleportation et son execution. Durant ce delai, si le joueur se deplace de plus de 0.5 bloc, la teleportation est annulee. Un retour visuel clair (titre/actionbar) informe le joueur du compte a rebours.

Classe TeleportWarmup

public class TeleportWarmup { /** * Demarre un warmup pour le joueur. * @param player Le joueur concerne * @param seconds Duree du warmup (depuis CommandConfig) * @param onSuccess Callback execute a la fin du warmup * @param onCancel Callback execute si annulation (mouvement ou deconnexion) */ public static void start(EntityPlayerMP player, int seconds, Runnable onSuccess, Runnable onCancel) { BlockPos startPos = player.getPosition(); final int totalTicks = seconds * 20; final int[] ticksElapsed = {0}; // Affiche le titre initial sendWarmupTitle(player, seconds); // Memoriser la task pour pouvoir l'annuler final ScheduledTask[] taskHolder = {null}; taskHolder[0] = EriScheduler.repeat(20, () -> { if (!player.isEntityAlive()) { taskHolder[0].cancel(); onCancel.run(); return; } ticksElapsed[0] += 20; int secondsLeft = seconds - (ticksElapsed[0] / 20); // Verification mouvement if (CommandConfig.warmupCancelOnMove) { double dx = player.posX - startPos.getX(); double dz = player.posZ - startPos.getZ(); if (Math.sqrt(dx*dx + dz*dz) > 0.5) { taskHolder[0].cancel(); sendCancelTitle(player); onCancel.run(); return; } } if (ticksElapsed[0] >= totalTicks) { taskHolder[0].cancel(); onSuccess.run(); } else { sendWarmupTitle(player, secondsLeft); } }); // Stocker la task active pour l'annuler si le joueur redemarre /spawn, /home, etc. ActiveWarmups.put(player.getUniqueID(), taskHolder[0]); } }

Retour visuel

  • Titre : grande police centree affichant "Teleportation dans Xs..." avec sous-titre "Ne bougez pas". Couleur degradee de or vers vert au fil du compte a rebours.
  • Annulation : titre rouge "Teleportation annulee !" + son d'erreur.
  • Les titres sont envoyes via player.connection.sendPacket(new SPacketTitle(...)).

ActiveWarmups

Un Map<UUID, ScheduledTask> en memoire reference les warmups actifs. Si un joueur lance une nouvelle commande de teleportation alors qu'un warmup est en cours, l'ancien est annule silencieusement et le nouveau demarre. Cela evite les conflits de teleportation simultanee.

✓
Bonne pratique : le warmup n'est pas utilise pour les commandes admin (/tp, /tphere, /tppos) et pour /back. Ces commandes teleportent instantanement.

10 Systeme de Cooldown

Architecture

Le CooldownManager est un singleton en memoire gerant les cooldowns par joueur et par type de commande. Les cooldowns ne persistent pas au redemarrage du serveur — c'est intentionnel et simplifie l'implementation sans recourir a du stockage supplementaire.

CooldownManager

public class CooldownManager { private static final CooldownManager INSTANCE = new CooldownManager(); public static CooldownManager getInstance() { return INSTANCE; } // Structure : UUID joueur → (cle commande → timestamp expiration en ms) private final Map<UUID, Map<String, Long>> cooldowns = new HashMap<>(); /** Retourne true si le joueur est en cooldown pour cette commande */ public boolean isOnCooldown(UUID uuid, String key) { Map<String, Long> map = cooldowns.get(uuid); if (map == null) return false; Long expiry = map.get(key); return expiry != null && System.currentTimeMillis() < expiry; } /** Retourne le temps restant en secondes (0 si pas en cooldown) */ public int getRemaining(UUID uuid, String key) { Map<String, Long> map = cooldowns.get(uuid); if (map == null) return 0; Long expiry = map.get(key); if (expiry == null) return 0; return (int) Math.max(0, (expiry - System.currentTimeMillis()) / 1000); } /** Applique un cooldown de 'seconds' secondes */ public void set(UUID uuid, String key, int seconds) { cooldowns.computeIfAbsent(uuid, k -> new HashMap<>()) .put(key, System.currentTimeMillis() + (seconds * 1000L)); } /** Nettoyer a la deconnexion (optionnel, evite les fuites memoire) */ public void clear(UUID uuid) { cooldowns.remove(uuid); } }

// A ajouter dans RankManager.java /** * Extrait la valeur entiere d'une permission de la forme "key=N". * Scanne tous les grades du joueur (avec heritage) et retourne la valeur la plus haute. * Si aucune permission de ce type n'est trouvee, retourne defaultValue. * * @param player Le joueur * @param key La cle de permission (ex. "homes.max") * @param defaultValue Valeur par defaut si aucun grade ne definit la permission * @return La valeur entiere la plus haute trouvee parmi tous les grades */ public int getPermissionValue(EntityPlayerMP player, String key, int defaultValue) { if (player.canUseCommand(2, "")) return Integer.MAX_VALUE; // OP = illimite int max = defaultValue; String prefix = key + "="; // 1. Verifier les permissions custom du joueur List<String> playerPerms = playerPermissions.get(player.getUniqueID().toString()); if (playerPerms != null) { for (String p : playerPerms) { if (p.startsWith(prefix)) { try { max = Math.max(max, Integer.parseInt(p.substring(prefix.length()))); } catch (NumberFormatException ignored) {} } } } // 2. Verifier les permissions du grade (avec heritage) RankInfo rank = getPlayerRank(player.getUniqueID()); if (rank != null) { Set<String> effective = PermissionResolver.getEffectivePermissions(rank, rankLookup()); for (String p : effective) { if (p.startsWith(prefix)) { try { max = Math.max(max, Integer.parseInt(p.substring(prefix.length()))); } catch (NumberFormatException ignored) {} } } } return max; }

{ "name": "&6VIP", "priority": 10, "prefix": "&6[VIP] ", "inherits": ["default"], "permissions": [ "homes.max=5", // Ce grade permet 5 homes "command.tpa", "command.tpahere", "command.back", "command.rtp", "warp.marche", "warp.pvp" ] }
ⓘ
Heritage : si le grade VIP herite du grade PREMIUM (qui definit homes.max=10), la methode retournera 10 (la valeur la plus haute parmi tous les grades dans la chaine d'heritage). Cela permet une escalade fluide des limites selon la hierarchie des grades.

Schema de serialisation des positions

Toutes les positions sont serialisees sous la forme de chaine de caracteres comma-separated compatible avec le type Map<String, String> d'EriData et les champs String de WorldSavedData.

// Format : "dimensionName,x,y,z,yaw,pitch" // Exemple : "overworld,-142.5,64.0,351.0,-90.0,0.0" // Exemple nether : "the_nether,50.0,80.0,-30.0,45.0,5.0" public class LocationSerializer { public static String serialize(World world, double x, double y, double z, float yaw, float pitch) { String dimName = world.provider.getDimensionType().getName(); return dimName + "," + x + "," + y + "," + z + "," + yaw + "," + pitch; } public static SavedLocation deserialize(String s) { String[] parts = s.split(","); return new SavedLocation(parts[0], Double.parseDouble(parts[1]), Double.parseDouble(parts[2]), Double.parseDouble(parts[3]), Float.parseFloat(parts[4]), Float.parseFloat(parts[5])); } }

15 Auto-completion Contextuelle

Principe

Conformement aux conventions de code du projet, tous les arguments de commande doivent avoir de l'auto-completion. L'auto-completion est contextuelle : elle filtre les suggestions en fonction du joueur executant la commande (ses homes, les warps auxquels il a acces, etc.).

CommandeArgumentSource des suggestionsFiltre
/home nom Homes du joueur executant DataManager.get(player, HomePlayerData.class).homes.keySet()
/delhome nom Homes du joueur executant Identique a /home
/warp nom Warps auxquels le joueur a acces RankManager.hasPermission(player, "warp." + warpName)
/delwarp nom Tous les warps (admin) Aucun — admin voit tout
/tpa joueur Joueurs en ligne PlayerArg.of("joueur") — fourni par EriAPI
/tpahere joueur Joueurs en ligne Identique a /tpa
/tp admin joueur Joueurs en ligne PlayerArg.of("joueur")
/tphere joueur Joueurs en ligne PlayerArg.of("joueur")
/tppos x, y, z Position actuelle du joueur Suggestions dynamiques : coordonnees actuelles comme valeur par defaut

Implementation de l'auto-completion dynamique

// Supplier de noms de homes pour le joueur executant // Utilise dans HomeCommand et DelHomeCommand private static final Function<CommandContext, String[]> HOME_SUPPLIER = ctx -> { EntityPlayerMP player = ctx.getSenderAsPlayerMP(); if (player == null) return new String[0]; HomePlayerData data = DataManager.get(player, HomePlayerData.class); return data.homes.keySet().toArray(new String[0]); }; // Supplier de warps filtres par permission private static final Function<CommandContext, String[]> WARP_SUPPLIER = ctx -> { EntityPlayerMP player = ctx.getSenderAsPlayerMP(); WarpWorldData warps = WarpWorldData.get(/* overworld */); return warps.warps.keySet().stream() .filter(name -> player == null || RankManager.getInstance().hasPermission(player, "warp." + name)) .toArray(String[]::new); }; // Usage dans EriCommand (la methode exacte depend de l'API EriCommand) cmd.arg(StringArg.of("nom").suggestsDynamic(HOME_SUPPLIER));
⚠
Securite de l'auto-completion : la liste des warps dans l'auto-completion de /warp est filtree par permission. Un joueur ne voit que les warps auxquels il a acces. Cela empeche la decouverte de warps prives par enumeration Tab.
