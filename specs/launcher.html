Cahier des charges · EriniumFaction

Launcher ElectronJS

Le launcher est le point d'entree unique pour jouer sur EriniumFaction. Il gere l'authentification Discord OAuth, le telechargement et la verification des fichiers du jeu, la configuration, et le lancement de Minecraft 1.12.2 avec CleanRoom (fork de Forge compatible Java 21+) et le token d'authentification. Le telechargement vanilla est gere par minecraft-java-core.

Planifie ElectronJS Discord OAuth Auto-Update Liquid Glass UI Windows / Linux / Mac

01 Contexte & Objectifs

Pourquoi un launcher custom

Le serveur fonctionne en online-mode=false (crack). Sans launcher custom, n'importe qui peut se connecter avec n'importe quel pseudo. Le launcher est le verrou d'acces : il authentifie le joueur via Discord OAuth, collecte son HWID, et injecte un token JWT dans Minecraft. Sans ce token, le mod serveur refuse la connexion.

Objectifs

  • Authentification securisee — Discord OAuth comme unique methode de connexion, coherent avec le site web
  • Distribution des mods — telecharger, verifier et mettre a jour automatiquement tous les mods et configs requis
  • Anti-triche — empecher l'ajout de mods non autorises, verifier l'integrite des fichiers avant chaque lancement
  • Experience utilisateur premium — interface Liquid Glass fluide, transitions animees, chargement rapide
  • Multi-plateforme — Windows (prioritaire), Linux et macOS en secondaire
  • Auto-mise a jour — le launcher et les fichiers du jeu se mettent a jour automatiquement

Contraintes

  • Telechargement initial — le launcher inclut le runtime Electron (~150-200 Mo). Les fichiers du jeu (MC + CleanRoom + mods) sont telecharges separement apres la premiere connexion
  • RAM minimale — le launcher lui-meme ne doit pas depasser 200 Mo de RAM
  • Temps de demarrage — moins de 3 secondes entre le clic sur l'icone et l'affichage de l'ecran splash
  • Compatibilite Java — CleanRoom supporte Java 21+. Detecter et utiliser Java 21 ou superieur (Corretto/Adoptium). Si absent, proposer l'installation automatique ou laisser minecraft-java-core le telecharger

Architecture IPC (Inter-Process Communication)

Electron separe le main process (Node.js, acces systeme complet) et le renderer process (page web, sandbox). Toute communication passe par l'IPC bridge :

07 Authentification Discord OAuth

Flow detaille

Le launcher utilise le meme flow Discord OAuth que le site web. Le joueur n'a qu'UN seul compte, cree sur le site via Discord. Le launcher est un second client OAuth de la meme application Discord.

Etape par etape

  • 1. Clic "Se connecter avec Discord" — le renderer envoie auth:discord-login via IPC
  • 2. Main process cree un serveur HTTP local — port aleatoire dans la plage 49152-65535. Le serveur n'ecoute que sur 127.0.0.1 (pas accessible depuis l'exterieur). Un state aleatoire (32 octets hex) est genere pour la protection CSRF
  • 3. Ouverture navigateur systeme — shell.openExternal(discordOAuthUrl) avec les parametres : client_id, redirect_uri=http://localhost:PORT/callback, scope=identify email, state=RANDOM_STATE
  • 4. Joueur autorise sur Discord — Discord redirige vers http://localhost:PORT/callback?code=XXX&state=YYY
  • 5. Serveur local recoit le callback — verifie que state correspond. Affiche une page HTML "Connexion reussie, vous pouvez fermer cet onglet" dans le navigateur. Ferme le serveur HTTP local
  • 6. Echange du code — le main process envoie le code OAuth + le HWID au site web API : POST https://erinium.fr/api/auth/discord/callback avec { code, hwid, source: "launcher", redirect_uri }
  • 7. Site web API — echange le code contre un token Discord, recupere les infos joueur, verifie l'email, verifie l'anciennete du compte Discord, verifie que le pseudo MC est defini, genere un JWT + refresh token
  • 8. Reponse — le launcher recoit { jwt, refreshToken, expiresIn, user: { mcName, discordName, avatar, rank, faction, ... } }
  • 9. Stockage securise — JWT et refresh token stockes via safeStorage.encryptString(). Infos joueur gardees en memoire
  • 10. Transition — le renderer est notifie via auth:token-received, la fenetre redimensionne en mode principal (1200x700)

Refresh token

  • Duree de vie JWT — 1 heure. Le launcher verifie l'expiration avant chaque lancement de jeu
  • Duree de vie refresh token — 30 jours. Stocke dans safeStorage
  • Refresh automatique — quand le JWT est expire mais le refresh token est valide, appel POST /api/auth/refresh avec { refreshToken, hwid }. Recoit un nouveau JWT
  • Refresh echoue — si le refresh token est aussi expire ou invalide → ecran de login Discord

Deconnexion

  • Action — supprime JWT et refresh token du safeStorage, vide les infos joueur en memoire
  • Optionnel — appel POST /api/auth/logout pour invalider le refresh token cote serveur
  • Transition — retour a l'ecran login (fenetre redimensionne en 400x600)
!
Securite — le client_secret Discord ne doit JAMAIS etre dans le code du launcher. L'echange code → token se fait via l'API du site web (serveur Vercel) qui detient le secret.

08 Securite & Anti-Tamper

HWID Fingerprint

Le main process collecte un fingerprint hardware unique au demarrage. Ce HWID est envoye avec chaque requete d'authentification et compare cote serveur.

const os = require('os'); const { execSync } = require('child_process'); const crypto = require('crypto'); function collectHWID() { const cpu = os.cpus()[0].model; const interfaces = os.networkInterfaces(); const mac = Object.values(interfaces) .flat() .find(i => !i.internal && i.mac !== '00:00:00:00:00:00')?.mac; let disk = ''; if (process.platform === 'win32') { disk = execSync('wmic diskdrive get serialnumber') .toString().trim().split('\n')[1]?.trim() || ''; } else { disk = execSync('lsblk --nodeps -no serial /dev/sda') .toString().trim() || ''; } const raw = `${cpu}|${os.platform()}${os.arch()}|${mac}|${disk}`; return crypto.createHash('sha256').update(raw).digest('hex'); }

safeStorage (stockage chiffre)

  • Electron safeStorage — utilise le keychain OS (Windows DPAPI, macOS Keychain, Linux Secret Service) pour chiffrer les tokens
  • Stockage — les tokens chiffres sont ecrits dans un fichier local. Sans les credentials OS de l'utilisateur, ils sont illisibles
  • Fallback — si safeStorage n'est pas disponible (certains Linux), utiliser un chiffrement AES-256 avec une cle derivee du HWID

Protection du token au passage Launcher → Jeu

  • Argument JVM obfusque — le token est passe au process Java via --etk=XXX (nom court, non-evident). Pas --eriniumToken en clair dans les args visibles dans le Task Manager
  • Alternative : variable d'environnement — au lieu d'un argument JVM (visible dans tasklist /v), passer le token via une variable d'environnement ephemere (process.env.ETK = jwt) qui n'apparait pas dans la ligne de commande du processus. Le mod Java lit System.getenv("ETK")
  • Nettoyage apres lecture — le launcher supprime la variable d'environnement immediatement apres le spawn du process Java (le process fils a deja herite la valeur)
  • Pas de fichier temporaire — le token ne transite JAMAIS par un fichier sur le disque (pas de /tmp/token.txt)

Anti-tamper du launcher

  • Code signing — le binaire Electron est signe avec un certificat de signature de code. Windows affiche un avertissement si le binaire n'est pas signe
  • ASAR chiffre — le paquet ASAR (archive du code JS) est chiffre. Empeche la lecture directe du code source
  • Detection DevTools — si les DevTools Electron sont ouverts (raccourci Ctrl+Shift+I ou F12), afficher un avertissement et logger l'evenement. En production, desactiver completement les DevTools
  • Obfuscation JS — le code du renderer est minifie et obfusque via webpack + terser. Le code du main process est aussi obfusque (javascript-obfuscator)
  • Integrite du processus — le mod client peut verifier que le processus parent (launcher) a le bon hash. Si le launcher est modifie, le token sera invalide

Blocage des launchers tiers

  • Token obligatoire — le mod EriniumFaction ne demarre PAS sans un token valide. Un joueur qui met le mod dans CurseForge, MultiMC, Prism ou n'importe quel autre launcher → crash immediat au preInit
  • Pas de contournement — le token ne peut etre obtenu QUE via notre launcher (Discord OAuth → site API → JWT → safeStorage → passage au process Java). Aucun autre launcher ne connait ce flow
  • HWID lie au token — meme si un attaquant recupere un token (dump memoire), il est lie au HWID de la machine qui l'a demande. Utilise sur une autre machine → rejete par le serveur
!
Attention — la securite du launcher est une couche de dissuasion, pas une protection absolue. Un attaquant motive peut toujours reverse-engineer Electron. La vraie securite est cote serveur (validation JWT, HWID, anti-VPN). Le launcher rend l'attaque plus couteuse en temps et en effort.

09 Lancement du Jeu

Sequence de lancement

Quand le joueur clique sur "JOUER", le launcher execute cette sequence :

  • 1. Verifier le token JWT — si expire, refresh automatique. Si refresh echoue → ecran login
  • 2. Verifier Minecraft vanilla — via minecraft-java-core, verifier le cache MC 1.12.2. Si absent → telechargement (versions, librairies, assets, natives)
  • 3. Verifier CleanRoom — verifier que cleanroom-universal.jar est present et son hash est correct. Si absent ou mismatch → telecharger depuis CDN ou GitHub releases
  • 4. Verifier les fichiers custom — comparer le manifeste local avec le manifeste distant (voir spec game-download.html). Telecharger les fichiers changes/manquants
  • 5. Verifier l'integrite — re-hasher les fichiers critiques (mods) pour confirmer leur integrite
  • 6. Construire la commande Java — assembler le classpath (MC vanilla + CleanRoom + mods) et la ligne de commande complete avec le tweaker CleanRoom
  • 7. Lancer le processus — spawn Java avec les arguments (ou utiliser la methode de lancement de minecraft-java-core)
  • 8. Monitorer — suivre le processus (exit code, stdout/stderr)

10 Auto-Updater du Launcher

Mecanisme

Le launcher utilise electron-updater (integre a electron-builder) pour se mettre a jour automatiquement. Les releases sont hebergees sur GitHub Releases (repo prive) ou sur notre propre serveur.

Flow de mise a jour

  • 1. Verification — au demarrage (ecran splash), appel autoUpdater.checkForUpdates()
  • 2. Mise a jour disponible — l'ecran splash affiche "Mise a jour disponible (v1.1.0)..." + barre de progression
  • 3. Telechargement — telechargement automatique en arriere-plan. Progression affichee
  • 4. Installation — une fois telecharge, autoUpdater.quitAndInstall() relance le launcher avec la nouvelle version
  • 5. Signature — la mise a jour est signee. electron-updater verifie automatiquement la signature avant d'appliquer

Versionning

  • Format — Semantic Versioning (X.Y.Z). Majeure = breaking changes, Mineure = nouvelles features, Patch = bugfixes
  • Stockage version — dans package.json du projet Electron
  • Affichage — version visible sur l'ecran splash et dans le footer de l'ecran principal

14 Modele de Menace

Attaque identifiee : Creation automatisee de comptes crack + exploit OP + VPN rotatif

Scenario : Un attaquant scripte la creation de centaines de comptes, se connecte avec des VPN differents a chaque fois (rendant les IP bans inutiles), et exploite une faille pour obtenir les permissions OP. Avec online-mode=false, il n'y a pas de paywall Mojang — l'attaque est gratuite et illimitee.

Impact : Griefing massif, vol de donnees faction, destruction du monde, perte de joueurs.

Couches de defense

La securite repose sur un modele en couches. Chaque composant (site web, launcher, mod client, mod serveur) ajoute une barriere supplementaire. Un attaquant doit contourner TOUTES les couches pour reussir.

15 Anti-Bot Inscription WEBSITE

Mur anti-bot : L'inscription passe obligatoirement par Discord OAuth. Un compte Discord verifie (email confirme) est le prerequis minimal. Cela elimine les bots simples car Discord impose ses propres verifications (email, captcha, anciennete potentielle).

Mesures detaillees

  • Discord OAuth obligatoire — pas de formulaire email/mot de passe. Le joueur clique "Se connecter avec Discord", autorise l'application, et est redirige vers le site avec son profil Discord (ID, username, avatar, email verifie)
  • Compte Discord verifie requis — le scope OAuth inclut identify et email. Seuls les comptes Discord avec un email verifie sont acceptes. Rejeter les comptes sans email verifie
  • Pseudo Minecraft obligatoire — apres la premiere connexion Discord, le joueur doit choisir un pseudo MC unique (popup sur le site). Pas de pseudo = pas de token = pas de jeu
  • Anciennete Discord optionnelle — possibilite de rejeter les comptes Discord crees il y a moins de X jours (configurable, defaut : 7 jours). Anti compte jetable
  • A2F optionnelle (TOTP) — TOTP (Google Authenticator / Authy) propose mais pas impose. Pour ceux qui veulent securiser leur compte. Configure sur le dashboard du site
  • Verification nouvel appareil/IP — si le compte se connecte depuis un appareil (HWID) ou une IP non reconnus, envoyer un code de verification par email Discord (via DM bot ou notification site). L'appareil/IP est ensuite whiteliste pour ce compte
  • Rate limit inscription — max 3 comptes par IP/24h, max 1 compte par Discord ID (1 Discord = 1 compte serveur), captcha sur le formulaire si necessaire
  • Honeypot — champs invisibles dans les formulaires. Les bots les remplissent, les humains non → reject

16 Securite Mod Client MOD

Regle absolue : Le mod NE PEUT PAS fonctionner sans le launcher officiel. Aucun launcher tiers (CurseForge, MultiMC, Prism, TLauncher, etc.) ne doit pouvoir lancer le jeu avec nos mods.

Double verification — 2 niveaux independants

Niveau 1 — Au lancement du jeu (preInit, cote client) : Le mod verifie qu'un token existe. Si absent → crash immediat. Cela empeche de lancer le jeu sans le launcher officiel — meme en solo, meme sans serveur. Le jeu est totalement inutilisable sans le launcher.

Niveau 2 — A la connexion serveur (cote serveur) : Le serveur valide la signature du token, le HWID, l'anti-VPN et les bans. Cela empeche les faux tokens — meme si quelqu'un fabrique un token bidon pour lancer le jeu, le serveur le rejette.

Niveau 1 : Verification au lancement (preInit)

Ce check se fait au demarrage du jeu, dans FMLPreInitializationEvent, AVANT que le menu principal ne s'affiche. C'est le gardien qui empeche toute utilisation du mod hors launcher.

  • Lecture du token — lire l'argument JVM --etk=XXX (nom obfusque). Si absent → crash immediat avec message "Utilisez le launcher officiel EriniumFaction pour jouer. Telechargez-le sur eriniumfaction.fr"
  • Crash volontaire — le jeu ne doit PAS demarrer. Pas de menu principal, pas de solo, pas de LAN — crash net avec un ecran d'erreur clair. Utiliser throw new RuntimeException() dans le preInit pour forcer l'arret
  • Validation format — verifier que le token a le format JWT (3 parties base64 separees par des points). Si format invalide → crash avec message "Token invalide. Relancez le launcher."
  • Verification expiration — decoder le payload JWT (sans verifier la signature — la cle secrete ne doit JAMAIS etre dans le client) et verifier que exp n'est pas depasse. Si expire → crash avec message "Session expiree. Relancez le launcher."
  • Anti-debug — detecter si un debugger Java est attache (ManagementFactory.getRuntimeMXBean().getInputArguments() contient -agentlib:jdwp). Si oui → crash immediat

Niveau 2 : Validation serveur (a la connexion)

Ce check se fait quand le joueur rejoint le serveur. Le client envoie le token + HWID au serveur qui fait la vraie validation cryptographique. Detaille dans la section S17 (Securite Mod Serveur).

  • Envoi du token — a la connexion (FMLNetworkEvent.ClientConnectedToServerEvent), le client envoie un packet custom contenant : JWT + HWID
  • Collecte HWID cote Java — System.getProperty("os.arch"), Runtime.getRuntime(), NetworkInterface.getHardwareAddress(), disk serial via ProcessBuilder("wmic diskdrive get serialnumber"). Hasher le tout en SHA-256
  • Serveur valide — signature JWT (RS256), expiration, UUID, HWID match, anti-VPN, rate limit. Si invalide → kick immediat

Protection du token contre la recuperation

  • Token en memoire uniquement — le token est lu depuis les JVM args au preInit puis stocke dans un private static final dans une classe non-exportee. Les JVM args sont ensuite nettoyes de la memoire (remplacer l'entree dans ManagementFactory.getRuntimeMXBean().getInputArguments() est impossible, mais on peut obfusquer le nom de l'argument)
  • Pas de fichier — le token n'est JAMAIS ecrit sur le disque cote client (pas de log, pas de config, pas de crash report). Les crash reports Minecraft sont filtres pour supprimer les JVM args contenant "eriniumToken"
  • Pas dans les logs — le Logger du mod filtre toute chaine contenant "eriniumToken" ou ressemblant a un JWT (regex eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+)
  • Obfuscation du nom d'argument — au lieu de --eriniumToken=XXX, utiliser un nom non-evident comme --etk=XXX ou un argument encode. Un attaquant qui regarde les args JVM dans le Task Manager ne voit pas immediatement que c'est un token d'auth
  • Token a courte duree — le JWT expire en 1h. Meme si un attaquant recupere le token (dump memoire, etc.), il n'a qu'1h pour l'utiliser. Et le HWID est verifie cote serveur → le token est lie au hardware de la machine qui l'a demande
  • Anti-process scan — detecter si un outil de dump memoire est en cours d'execution (ex: ProcessBuilder("tasklist") et chercher des processus connus : Cheat Engine, x64dbg, Process Hacker). Si detecte → refuser la connexion au serveur

17 Securite Mod Serveur SERVEUR MOD

Validation et protection

  • Validation JWT complete — verifier signature (RS256 avec cle publique du site), expiration, UUID correspond au joueur qui se connecte
  • Verification HWID — comparer le HWID envoye avec celui enregistre sur le compte (via API du site ou cache local). Si mismatch → kick + flag le compte
  • Anti-VPN/Proxy — a la connexion, checker l'IP du joueur contre une base VPN (IPHub API, IP2Proxy, ou base locale). Si VPN detecte → kick avec message "Desactivez votre VPN"
  • Timeout auth — si le client n'envoie pas le token dans les 5 secondes apres connexion → kick
  • /ban = triple ban — quand un admin fait /ban joueur, ca ban simultanement : UUID + IP + HWID. Triple securite. Le joueur est banni peu importe s'il change de compte (UUID), de reseau (IP) ou les deux — son hardware est marque
  • /ipban et /hwidban separees — pour les cas ou on veut ban uniquement une IP (ex: VPN partage) ou uniquement un HWID (ex: cybercafe) sans affecter les autres dimensions
  • Ban check au login — a PlayerLoggedInEvent, verifier IP et HWID (envoye par le client dans le auth packet) contre la liste des bans. Kick immediat avec message explicatif si banni
  • Rate limit connexion — max 5 connexions par HWID/heure, max 10 par IP/heure

18 Permissions & Detection MOD SERVEUR

Permissions — zero exploit OP

Regle absolue : aucun joueur ne doit JAMAIS pouvoir devenir OP via le jeu.
  • Supprimer /op et /deop — intercepter via Mixin ou event. Ces commandes ne doivent fonctionner QUE depuis la console serveur physique
  • Systeme de permissions custom — ne PAS utiliser l'OP vanilla. Utiliser notre propre systeme (AdminManager existant) avec des roles granulaires
  • Admin auth separee — pour les actions admin in-game, exiger un second token (admin token) genere depuis le panel web admin avec A2F
  • Audit log — logger TOUTE commande admin avec timestamp, joueur, IP, HWID, commande exacte. Stocke en base de donnees, pas en fichier local
  • Command injection audit — auditer toutes les commandes du mod pour verifier qu'aucune ne permet d'injecter des commandes serveur. Ne jamais eval() ou passer du player input brut a server.commandManager.executeCommand()
  • Permission escalation prevention — aucune commande ne doit permettre de modifier les permissions d'un autre joueur sauf depuis la console ou avec admin token

Detection comportementale

  • Pattern bot — login → action → logout rapide en boucle. Si detecte > 3 fois en 10 min → temp ban 24h
  • Flood de comptes — si > 5 comptes avec le meme HWID → flag pour review manuelle + bloquer les nouveaux comptes sur ce HWID
  • Mouvement suspect — vitesse de deplacement anormale, teleportation, reach anormal. L'anticheat existant (CheckCombat, CheckMovement) doit etre renforce
  • Action apres connexion — si un joueur execute des commandes admin dans les 10 premieres secondes de connexion → auto-kick + flag. Un vrai admin prend son temps

HWID Fingerprint (cote mod Java)

String cpu = System.getenv("PROCESSOR_IDENTIFIER"); String os = System.getProperty("os.name") + System.getProperty("os.arch"); byte[] mac = NetworkInterface.getByInetAddress( InetAddress.getLocalHost()).getHardwareAddress(); String disk = exec("wmic diskdrive get serialnumber"); // Windows // Linux: /sys/block/sda/device/serial ou lsblk --nodeps -no serial String raw = cpu + "|" + os + "|" + hex(mac) + "|" + disk.trim(); String hwid = SHA256(raw);

Compatibilite CleanRoom

!
Toutes les classes de securite doivent rester compatibles Java 8. Pas de var, pas de HttpClient (Java 11+). Utiliser HttpURLConnection ou Netty pour les appels HTTP. Pas de ProcessHandle (Java 9+) — utiliser ProcessBuilder pour les commandes systeme. C'est CleanRoom qui gere la compatibilite Java 21+, le code source reste en Java 8.
