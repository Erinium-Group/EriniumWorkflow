Infrastructure — Optimisation Serveur

Server Performance

Systeme d'optimisation complet pour maintenir 20 TPS stables avec 500 a 1000 joueurs simultanees sur un seul serveur Forge 1.12.2, sans BungeeCord, avec un monde faction et une dimension de minage.

20 TPS Stable Forge 1.12.2 Mixin 500-1000 Joueurs Single Server Async I/O

01Contexte & Objectifs

Infrastructure cible

  • Architecture — Serveur unique, pas de BungeeCord ni de split par dimension sur des processus distincts.
  • Dimensions — Monde faction principal (overworld) + dimension de minage (dim custom).
  • Mods — 20+ mods actifs dont Dimensional Threading (gere le multithreading des dimensions).
  • Joueurs — 500 a 1000 joueurs simultanees en heure de pointe.
  • PvP — Serveur PvP 1.8 : combats frequents, effets de potion, mouvements rapides = beaucoup d'entites et de paquets.

Perimetre de cette spec

i
Exclusion : Le threading des dimensions est gere par le mod Dimensional Threading — ne pas dupliquer cette logique. Cette spec couvre tout le reste : entites, chunks, paquets, ticks, I/O, memoire, reseau.

Analyse du bottleneck Forge 1.12.2

Les principaux goulets d'etranglement sur un serveur Forge 1.12.2 avec 1000 joueurs sont, par ordre d'impact :

  • Tick entites (40-60% du tick) — Chaque entite (mob, item, fleche, animal) execute sa logique IA a chaque tick.
  • Generation de chunks (pic 100+ms) — La generation synchrone d'un chunk peut bloquer tout un tick.
  • Tick TileEntities (15-25%) — Hoppers, fours, coffres actifs tickent a chaque tick.
  • Sauvegarde monde (pic 200+ms toutes les 5 min) — L'auto-save bloque le thread principal sous Forge 1.12.2.
  • Paquets reseau (10-20%) — Serialisation et envoi des mises a jour d'entites a 1000 clients.

02Optimisation des Entites

EntityTickOptimizer — Throttling par distance

La logique IA d'un mob a 64 blocs de tout joueur consomme du CPU sans aucun impact visible. Le throttling reduit la frequence de tick des entites eloignees :

Distance du joueur le plus procheFrequence de tickImpact IA
0 – 24 blocsChaque tick (100%)IA complete
24 – 48 blocs1 tick sur 2 (50%)Mouvements legerement ralentis
48 – 96 blocs1 tick sur 4 (25%)Comportement ralenti, invisible au joueur
96+ blocs ou chunk inactiveSkip IA entierementL'entite existe mais ne pense pas
@Mixin(World.class) public abstract class MixinWorldEntityTick { @Inject(method = "updateEntityWithOptionalForce", at = @At("HEAD"), cancellable = true) private void throttleEntityTick(Entity entity, boolean forceUpdate, CallbackInfo ci) { if (entity instanceof EntityLivingBase && !(entity instanceof EntityPlayer)) { int throttle = EntityTickOptimizer.getThrottle(entity); if (throttle > 1) { long tick = entity.world.getTotalWorldTime(); if ((tick + entity.getEntityId()) % throttle != 0) { ci.cancel(); // Skip ce tick } } } } }

Entity Activation Range

Les entites au-dela de 32 blocs de tout joueur entrent en mode "inactif" : leurs ticks IA sont entierement sautes. Elles continuent d'exister dans le monde et de se deplacer selon l'inertie, mais ne calculent pas de nouvelles decisions d'IA.

  • Mobs hostiles — Desactivation IA a 32 blocs.
  • Animaux / mobs passifs — Desactivation IA a 24 blocs.
  • Items au sol — Desactivation complete a 16 blocs (pas de mouvement physique non plus).
  • Fleches / projectiles — Toujours actifs (courte duree de vie, impact joueur possible).
  • EntityPlayer — Jamais throttle.

Mob Cap par section de chunk

Le mob cap vanilla s'applique a l'echelle du monde entier (70 mobs hostiles par monde). Sur un serveur 1000 joueurs, cela est completement insuffisant. Le cap est remplace par un cap par section de chunk (16x16x16) :

  • Maximum 10 mobs hostiles par section de chunk.
  • Maximum 20 animaux par section de chunk.
  • Suppression automatique des mobs en surplus (les plus anciens en premier).

Culling des entites inactives

Une entite qui est restee dans un chunk non charge pendant plus de 5 minutes (configurable) est automatiquement supprimee du serveur. Cela evite l'accumulation d'entites dans des zones non visitees.

03Optimisation des Chunks

Probleme de la generation synchrone

Sous Forge 1.12.2, la generation d'un nouveau chunk bloque le thread principal. Avec 1000 joueurs se deplacant dans toutes les directions, plusieurs chunks peuvent etre generes en serie dans le meme tick, causant des pics de latence de 100-500ms.

ChunkLoadOptimizer — Chargement asynchrone

public class ChunkLoadOptimizer { // Queue de chunks a generer, traites 2 par tick max private final Queue<ChunkGenerationTask> generationQueue = new ConcurrentLinkedQueue<>(); public static final int MAX_CHUNKS_PER_TICK = 2; // configurable @SubscribeEvent public void onServerTick(TickEvent.ServerTickEvent event) { if (event.phase != TickEvent.Phase.START) return; int processed = 0; while (!generationQueue.isEmpty() && processed < MAX_CHUNKS_PER_TICK) { ChunkGenerationTask task = generationQueue.poll(); long before = System.currentTimeMillis(); task.generate(); PerformanceMonitor.recordChunkGenTime(System.currentTimeMillis() - before); processed++; } } }

Pre-generation de chunks

Le vecteur de deplacement de chaque joueur est calcule et les chunks dans la direction de deplacement sont pre-mis en queue de generation avec une priorite elevee avant que le joueur ne les atteigne.

Distance de vue dynamique (AutoViewDistance)

La distance de vue globale du serveur est ajustee automatiquement selon le nombre de joueurs connectes :

Le changement de view distance se fait progressivement (1 chunk par 30 secondes) pour eviter un pic de chargement/dechargement massif lors de transitions.

Dechargement agressif de chunks

Un chunk sans joueur dans un rayon de 2 chunks est decharge immediatement apres 30 ticks (1.5 secondes) sans joueur. Forge vanilla attend 600 ticks (30 secondes). Sur 1000 joueurs, cette reduction libere des centaines de chunks en memoire.

04Optimisation des Paquets Reseau

Probleme : surcharge paquets a 1000 joueurs

Le systeme d'entity tracker vanilla envoie des mises a jour de position a tous les joueurs dans le rayon de tracking pour chaque entite a chaque tick. Avec 1000 joueurs :

  • Si 1000 joueurs sont proches les uns des autres (combat faction) : jusqu'a 1000 * 1000 = 1,000,000 paquets de position par tick.
  • Le debit reseau peut depasser 100 MB/s uniquement pour les paquets d'entites.

PacketOptimizer — Batch updates

// Mixin sur EntityTrackerEntry pour regrouper les updates @Mixin(EntityTrackerEntry.class) public abstract class MixinEntityTrackerEntry { @Shadow private int updateFrequency; @Inject(method = "<init>", at = @At("RETURN")) private void adjustUpdateFrequency(Entity entity, int range, int freq, boolean sendVel, CallbackInfo ci) { if (entity instanceof EntityLivingBase && !(entity instanceof EntityPlayer)) { // Mobs : update toutes les 2 ticks au lieu de chaque tick updateFrequency = Math.max(freq, PerformanceConfig.ENTITY_UPDATE_FREQ_TICKS); } } }

Rate limiting par joueur

Chaque connexion joueur a une limite de paquets sortants par tick (configurable, defaut 200 paquets/tick). Si cette limite est depassee, les paquets de basse priorite (positions d'entites lointaines) sont droppes jusqu'au prochain tick.

Batch Tab List Updates

Les mises a jour de la tab list (tab player) sont regroupees et envoyees en batch toutes les 2 secondes (40 ticks) plutot qu'en temps reel. Cela reduit de 40x le nombre de paquets SPacketPlayerListItem.

Chat queuing anti-flood

Les messages chat sont limites a 1 par 500ms par joueur. Les messages en surplus sont soit droppes, soit mis en file avec un indicateur de delai. Empeche les tempetes de paquets chat lors d'evenements.

05Distribution des Ticks

Probleme : congestion en debut de tick

Par defaut, tous les TileEntities tickent au debut du tick serveur (phase WORLD_TICK). Avec 10,000 TileEntities dans le monde, ils consomment tous le CPU dans la meme fenetre, creant un pic de charge.

TickDistributor — Repartition sur N ticks

public class TickDistributor { private static final int DISTRIBUTION_WINDOW = 4; // Repartir sur 4 ticks // Partitionner les TileEntities en 4 groupes par hash de position public static boolean shouldTickThisTick(TileEntity te, long worldTick) { int group = (te.getPos().hashCode() & 0x7FFFFFFF) % DISTRIBUTION_WINDOW; return (worldTick % DISTRIBUTION_WINDOW) == group; } }

En pratique, chaque TileEntity ne ticke qu'1 fois tous les 4 ticks au lieu de chaque tick. Pour la plupart des TileEntities (coffres, balises, spawners inactifs), cela est imperceptible. Les hoppers ont leur propre systeme de throttling.

Hopper Throttling

Les hoppers sont les TileEntities les plus couteux de Minecraft. Le throttling conditionnel :

  • Hopper isole (< 10 hoppers dans le chunk) — tick normal (toutes les 8 ticks, comportement vanilla).
  • Chunk avec 10-30 hoppers — tick toutes les 16 ticks.
  • Chunk avec 30+ hoppers — tick toutes les 32 ticks + alerte admin (potentielle machine exploit).

Redstone throttling

La propagation redstone illimitee peut provoquer des boucles infinies ou des pics de lag. Un compteur de profondeur limite les mises a jour redstone a 512 par tick et par zone de 16x16x16.

Random tick speed optimization

Le random tick speed affecte la culture de blocs et la poussee de l'herbe. En dessous de 18 TPS, le random tick speed est reduit automatiquement de 3 a 1 pour alleager la charge, puis restaure quand le TPS revient au-dessus de 19.

06Data & I/O Asynchrone

Sauvegarde monde asynchrone

L'auto-save vanilla bloque le thread principal pendant la serialisation et l'ecriture des chunks et des donnees joueur. Sur un monde charge avec 1000 joueurs, cela peut prendre 200-500ms toutes les 5 minutes — un spike TPS majeur visible par tous.

// Mixin sur MinecraftServer.saveAllWorlds pour rendre la save async @Mixin(MinecraftServer.class) public abstract class MixinMinecraftServer { @Inject(method = "saveAllWorlds", at = @At("HEAD"), cancellable = true) private void asyncSaveAllWorlds(boolean suppressLog, CallbackInfo ci) { if (PerformanceConfig.ENABLE_ASYNC_SAVE) { AsyncWorldSaver.getInstance().scheduleSave(); ci.cancel(); // La save vanilla est remplacee par notre impl async } } }

La sauvegarde async fonctionne en deux phases :

  • Phase 1 (thread principal, < 5ms) — Serialisation des chunks en memoire (NBTTagCompound) sans I/O disque.
  • Phase 2 (thread I/O dedie) — Ecriture des NBTTagCompound serialises dans les fichiers .mca. Completement off-thread.

Batching de sauvegarde joueur

Les donnees NBT de 1000 joueurs sont sauvegardees en batch toutes les 5 minutes au lieu d'etre sauvegardees individuellement a chaque deconnexion. La sauvegarde batch est faite sur un thread I/O dedie.

Cache de fichiers region

Les fichiers .mca (region files) frequemment accedes sont gardes ouverts avec un cache LRU de 32 fichiers. Cela evite les overheads d'ouverture/fermeture de fichiers pour les zones frequemment visitees.

EriAPI DataManager — Backend optimise

Pour 1000 joueurs, stocker les donnees dans des fichiers NBT individuels (1 fichier par joueur) represente 1000 acces disque distincts lors de l'auto-save. Le DataManager est configure en mode batch write : toutes les donnees joueur sont serialisees en un seul fichier JSON ou dans une simple base SQLite en memoire.

07Optimisation Memoire

Object pooling

Minecraft cree et detruit des millions d'objets temporaires par tick (BlockPos, Vec3d, AxisAlignedBB). Chaque allocation declenche eventuellement le GC Java, causant des pauses. Le pooling reutilise ces objets :

// Pool de BlockPos mutables (thread-local pour la securite) public class BlockPosPool { private static final ThreadLocal<BlockPos.MutableBlockPos> POOL = ThreadLocal.withInitial(BlockPos.MutableBlockPos::new); public static BlockPos.MutableBlockPos get() { return POOL.get(); } }

Le pool est appele dans les hotspots identifies par profiling : boucles de recherche d'entites, calculs de path-finding, detection de collisions.

Reduction NBT des mobs

Les mobs accumulent des tags NBT optionnels (historique de degats, custom name vide, tags inutilises par les mods). Un pass de nettoyage NBT retire les tags vides ou redondants lors de la serialisation des entites.

Chunk palette optimization

Le systeme de palette de blocs de Minecraft 1.12.2 utilise un tableau de long pour stocker les blocs par chunk. Les chunks avec peu de types de blocs distincts utilisent des palettes plus petites (4 bits/bloc au lieu de 8). Cette optimisation est deja en partie presente dans Forge mais peut etre amelioree pour les chunks du monde de minage.

Limit des entites items au sol

Les items droppes au sol s'accumulent dans les zones PvP actives. Une limite de 200 items par chunk section est appliquee. Au-dela, les plus anciens items sont supprimes (avec log si l'item est rare).

08AutoViewDistance

Ajustement dynamique de la distance de vue

L'AutoViewDistance surveille le TPS et le nombre de joueurs en temps reel et ajuste la distance de vue du serveur progressivement :

public class AutoViewDistance { private int currentViewDist = 10; private int targetViewDist = 10; private long lastAdjustTick = 0; public void tick(long worldTick) { if (worldTick - lastAdjustTick < 600) return; // Check toutes les 30s int players = FMLCommonHandler.instance().getMinecraftServerInstance() .getCurrentPlayerCount(); double tps = PerformanceMonitor.getInstance().getAverageTPS(60); // moy 60s targetViewDist = calculateTarget(players, tps); if (targetViewDist != currentViewDist) { int step = targetViewDist > currentViewDist ? 1 : -1; currentViewDist += step; FMLCommonHandler.instance().getMinecraftServerInstance() .setPlayerListViewDistance(currentViewDist); } lastAdjustTick = worldTick; } }

Des qu'un admin modifie manuellement la view distance via commande, l'AutoViewDistance se met en pause pendant 10 minutes avant de reprendre les ajustements automatiques.

09Architecture Technique

Package principal

Tout le code d'optimisation est isole dans le package fr.eriniumgroup.eriniumfaction.performance.

Singleton central. Coordonne tous les sous-systemes d'optimisation. Point d'entree d'initialisation.

Throttling IA par distance + activation range. Culling entites inactives. Cap mob par section.

PerformanceMonitor
TPS tracker, entity counts, chunk stats, packet rates. Fournit les donnees a /servperf.

Tous les seuils et options configurables. Rechargeable en live. Valeurs par defaut conservatrices.

10Monitoring TPS & Metriques

TPS Tracker

Le TpsTracker calcule le TPS en temps reel avec trois moyennes glissantes :

  • TPS instantane — Mis a jour a chaque tick.
  • Moyenne 1 minute — Fenetre glissante sur les 60 derniers ticks.
  • Moyenne 5 minutes — Fenetre glissante sur les 6000 derniers ticks.
  • Moyenne 15 minutes — Fenetre glissante sur les 18000 derniers ticks.

Alertes automatiques

  • TPS < 15 — Alerte console + notification Discord (si webhook configure).
  • Tick time > 100ms — Log du composant coupable (entites, chunks, TEs) pour diagnostique.
  • Generation chunk > 50ms — Log avec coordonnees du chunk pour investigation.
  • Memoire heap > 85% — Alerte + reduction temporaire de la view distance de 2.

14Plan de Tests

Tests de performance

[TEST-01] Baseline : Mesurer le TPS sans aucune optimisation active sur un serveur de test avec 100 bots explorant la carte pendant 10 minutes.
[TEST-02] Entity throttling : Spawner 5000 mobs dans une zone. Activer l'optimisation. Mesurer la reduction de tick time. Verifier que les mobs proches du joueur ont toujours un comportement normal.
[TEST-03] Chunk generation : Faire explorer 100 bots dans des zones non generees. Comparer le tick time avec et sans la queue de generation. Verifier l'absence de monde corrompu.
[TEST-04] Auto-save async : Provoquer 5 cycles d'auto-save consécutifs. Mesurer le TPS pendant chaque save. Verifier que le TPS reste > 19 pendant la save async.
[TEST-05] Packet rate : Simuler un combat 50v50. Mesurer le volume de paquets generes. Verifier que le rate limiting ne cause pas de desync notable cote client.
[TEST-06] AutoViewDistance : Connecter progressivement 800 bots. Verifier que la view distance se reduit automatiquement selon les seuils configures.
[TEST-07] Hopper throttling : Placer 50 hoppers dans un chunk. Verifier le throttling et l'absence de duplication d'items dans les machines legales.
[TEST-08] Stress test 1000 joueurs : Simuler 1000 connexions avec des bots effectuant des actions variees (combat, mining, commerce). Mesurer TPS sur 30 minutes.
[TEST-09] TE Distribution : Remplir un monde avec 10,000 TileEntities actifs. Verifier reduction tick time de 15% minimum vs baseline.
[TEST-10] Compatibilite mods : Verifier que les optimisations ne cassent pas les mecaniques des 20 mods actifs (pas de duplication, pas de comportement IA casse, pas de corruption de redstone).

Criteres de validation

  • 500 joueurs : TPS moyen > 19.5 sur 30 minutes de test continu.
  • Aucun rapport de comportement de jeu casse ou anormal par les testeurs.
  • Impact auto-save < 100ms mesure sur un world plein.
  • Compatibilite confirmee avec tous les mods du pack.
  • Commande /servperf affiche des metriques coherentes et utiles.
!
Important : Les optimisations doivent etre activees/desactivables individuellement via PerformanceConfig pour permettre un diagnostique fin en cas de bug. Ne jamais activer toutes les optimisations d'un coup en production.
