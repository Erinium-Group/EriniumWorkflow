Monde Minage

Dimension dediee a la recolte de ressources : invincibilite totale, minerais exclusifs, reset automatique toutes les 7 jours et structure de spawn flottante. Le monde principal reste intact.

01 Contexte & Objectifs

Problematique

Dans un serveur faction/PvP classique, le minage intensif des ressources rares (Valterite, Aetherite, Erinium) dans le monde principal pose plusieurs problemes majeurs :

  • Devastation du monde principal — les joueurs creusent des milliers de tunnels, rendant le monde "gruyere" et visuellement degrade
  • Pression PvP constante — les mineurs sont des cibles faciles, rendant l'activite miniere frustrante
  • Epuisement des ressources — sans systeme de renouvellement, les minerais rares disparaissent definitivement
  • Inequite entre factions — les factions dominantes monopolisent les zones riches en minerais

Solution : Monde Minage dedié

Le Monde Minage est une dimension parallele accessible via la commande /minage. Il sert exclusivement a la recolte de ressources et offre un environnement sur et equitable pour tous les joueurs.

Regles fondamentales

  • Invincibilite absolue — aucun type de degat ne peut tuer un joueur dans cette dimension (void, chute, lave, mobs, PvP)
  • Pas de claims — la revendication territoriale est completement desactivee dans cette dimension
  • Pas de lacs de lave souterrains — la generation de lacs de lave underground est supprimee pour un minage securise
  • Minerais moddes exclusifs — Valterite, Aetherite et Erinium ne generent PAS dans l'Overworld, uniquement ici
  • Minerais vanilla normaux — charbon, fer, or, diamant, etc. generent normalement dans les deux mondes
ℹ
Strategie economique — En confinant les minerais moddes au Monde Minage, on cree un cycle naturel : les joueurs doivent regulierement visiter la dimension pour se ravitailler, et le reset periodique garantit que personne ne peut stockpiler indefiniment un avantage. Cela equilibre la progression des factions et maintient une economie vivante.

02 Dimension

Enregistrement de la dimension

La dimension est enregistree au demarrage du serveur via l'API Forge standard. L'ID de dimension est configurable pour eviter les conflits avec d'autres mods.

// Dans MiningWorldManager.init() — appele depuis CommonProxy.preInit() public static final int MINING_DIM_ID = MiningConfig.miningWorldDimId; // defaut: 2 DimensionManager.registerDimension(MINING_DIM_ID, DimensionType.register( "mining_world", "_mining", MINING_DIM_ID, MiningWorldProvider::new, false ));

Generation de terrain

MiningChunkGenerator enveloppe le generateur vanilla de l'Overworld (ChunkGeneratorOverworld) en interceptant la phase de population pour supprimer les lacs de lave souterrains.

Deux approches possibles, la seconde est preferee :

// Approche 1 : Intercepter ChunkGeneratorOverworld (Mixin) // @Inject dans populate() pour annuler LAVA_LAKE // Approche 2 (preferee) : EriEvents sur PopulateChunkEvent EriEvents.on(PopulateChunkEvent.Populate.class) .filter(e -> e.getWorld().provider.getDimension() == MiningWorldManager.MINING_DIM_ID) .filter(e -> e.getType() == PopulateChunkEvent.Populate.EventType.LAVA_LAKE) .handle(e -> e.setCanceled(true));
✓
Lacs de lave de surface — Seuls les lacs de lave souterrains sont supprimes (Y < 60). Les lacs de surface peuvent etre maintenus pour l'ambiance visuelle, mais les joueurs sont de toute facon invincibles. Il est recommande de supprimer egalement les lacs de surface pour rendre le monde plus praticable. Annuler l'event LAVA_LAKE supprime les deux.

Coordonnees et topographie

Le monde utilise la meme echelle de coordonnees que l'Overworld (1:1 — pas de ratio 8:1 comme le Nether). La topographie est similaire a l'Overworld vanilla : collines, caves, biomes normaux. La structure de spawn (ile flottante) est placee au centre (0, Y, 0) a chaque reset.

⚠
ID de dimension — L'ID par defaut est 2. Sur certains serveurs avec d'autres mods installes (ex : Twilight Forest = 7, Aether = 3, etc.), ce numero peut entrer en conflit. L'administrateur DOIT configurer miningWorldDimId en consequence. Une detection de conflit au demarrage (log WARN) est recommandee.

03 Generation de Minerais

Principe d'exclusivite

Les minerais moddes (Valterite, Aetherite, Erinium) ne generent UNIQUEMENT dans le Monde Minage. Cette exclusivite est enforcee dans MiningOreGenerator via un check de dimension, et dans EriniumWorldGenerator existant qui doit etre modifie pour ignorer ces ores dans l'Overworld.

// EriniumWorldGenerator.java — modification requise @Override public void generate(Random random, int chunkX, int chunkZ, World world, IChunkGenerator chunkGenerator, IChunkProvider chunkProvider) { int dimId = world.provider.getDimension(); // Minerais vanilla : generent partout (Overworld + Mining) if (dimId == 0 || dimId == MiningWorldManager.MINING_DIM_ID) { generateVanillaOres(random, chunkX, chunkZ, world); } // Minerais moddes : UNIQUEMENT dans le monde minage if (dimId == MiningWorldManager.MINING_DIM_ID) { generateMiningOres(random, chunkX, chunkZ, world); } }

04 Protection des Joueurs

Invincibilite totale

Dans le Monde Minage, les joueurs sont completement invincibles. Toutes les sources de degats sont annulees via trois events Forge distincts pour une couverture complete :

// MiningWorldProtection.java // Annule TOUS les degats dans la dimension minage EriEvents.on(LivingAttackEvent.class) .filter(e -> e.getEntityLiving() instanceof EntityPlayer) .filter(e -> isInMiningDim(e.getEntityLiving())) .handle(e -> e.setCanceled(true)); EriEvents.on(LivingHurtEvent.class) .filter(e -> e.getEntityLiving() instanceof EntityPlayer) .filter(e -> isInMiningDim(e.getEntityLiving())) .handle(e -> e.setCanceled(true)); EriEvents.on(LivingDamageEvent.class) .filter(e -> e.getEntityLiving() instanceof EntityPlayer) .filter(e -> isInMiningDim(e.getEntityLiving())) .handle(e -> e.setCanceled(true)); private static boolean isInMiningDim(EntityLivingBase entity) { return entity.world.provider.getDimension() == MiningWorldManager.MINING_DIM_ID; }

Protection contre le void

Le void ne genere pas de DamageSource standard avant un certain seuil. Une protection supplementaire teleporte le joueur vers le Y de spawn si sa position passe sous Y=0 :

// Verification sur PlayerTickEvent (cote serveur) EriEvents.on(TickEvent.PlayerTickEvent.class) .filter(e -> e.phase == TickEvent.Phase.END) .filter(e -> isInMiningDim(e.player)) .filter(e -> e.player.posY < -5.0) .handle(e -> { // Teleporter au centre (spawn de la structure) e.player.setPositionAndUpdate(0.5, MiningWorldManager.SPAWN_Y, 0.5); });
✓
Faim desactivee ? — Bien que les degats de faim soient bloques, la barre de faim continue de diminuer. Le joueur ne peut pas mourir de faim mais peut se retrouver avec une barre vide (qui empeche le sprint en vanilla). Il est recommande d'ajouter un FoodStatsSetFoodLevelEvent ou de maintenir la nourriture a 20/20 via un tick periodique si ce comportement est juge genant.

Comportement des mobs hostiles

Les mobs hostiles peuvent toujours spawner dans le Monde Minage (la nuit, dans les caves sombres), mais ne peuvent pas blesser les joueurs. Ils peuvent etre tues normalement pour l'XP et le loot. Ce comportement est intentionnel : les mobs apportent de la dynamique sans danger.

05 Restrictions Territoriales

Interdiction de claims

Les factions ne peuvent pas revendiquer de territoire dans le Monde Minage. Cette restriction est appliquee au niveau du handler de protection faction, avant toute logique de claim.

// Dans FactionProtectionHandler ou CommandFactionClaim // Verifier la dimension avant tout traitement if (player.world.provider.getDimension() == MiningWorldManager.MINING_DIM_ID) { EriChat.create() .text("[Monde Minage] ").color(TextFormatting.GOLD) .text("Les claims sont interdits dans cette dimension.").color(TextFormatting.RED) .to(player); return; }

Fonctionnalites faction desactivees

Overclaim (vol de claim) INTERDIT Pas de claims, donc pas d'overclaim
Protection de territoire N/A Aucun chunk n'est protege

PvP faction INTERDIT Invincibilite couvre le PvP

ℹ
Philosophie "zone neutre" — Le Monde Minage est une zone neutre par design. Aucune faction ne peut y etablir de domination. Tous les joueurs, quelle que soit leur faction ou leur absence de faction, ont un acces egal aux ressources. C'est le seul endroit du jeu ou les ennemis travaillent cote a cote sans se battre.

06 Commande /minage

Description generale

La commande /minage teleporte le joueur vers le Monde Minage (ou retour dans l'Overworld si deja dans le Monde Minage). Elle est la seule methode d'acces et de sortie de la dimension.

Parametre Valeur
Nom de la commande /minage
Arguments Aucun
Permission requise Aucune (tous les joueurs)
Permission admin minage.admin (bypass cooldown)
Cooldown 30 secondes (configurable)
Utilisable en combat Oui (pas de detection de combat)
Package fr.eriniumgroup.eriniumfaction.mining.command

Logique de teleportation

Joueur execute /minage | v Est dans Monde Minage ? | | OUI NON | | v v Teleporter vers Cooldown actif ? Overworld | | (position sauvee OUI NON ou spawn) | | v v Message Monde Minage cooldown desactive ? | | OUI NON | | v v Message Teleporter desactive au spawn de la dim

Sauvegarde de position

Quand un joueur entre dans le Monde Minage, sa position dans l'Overworld est sauvegardee. A son retour, il est teleporte a cette position. Si aucune position n'est sauvegardee (premiere visite), il est envoye au spawn de l'Overworld.

// Donnees sauvegardees via EriCap ou @EriData @EriData(modId = "eriniumfaction", scope = DataScope.PLAYER) public class MiningPlayerData { // Position de retour dans l'Overworld public double returnX = Double.NaN; public double returnY = Double.NaN; public double returnZ = Double.NaN; // Timestamp de derniere utilisation (pour cooldown) public long lastMinageUse = 0L; }

Messages affichés au joueur

A l'arrivee dans le Monde Minage, plusieurs informations sont envoyees au joueur via EriChat :

Les claims sont interdits.") .color(TextFormatting.GREEN) .to(player);

Gestion du cooldown

Le cooldown est verifie cote serveur via le timestamp de derniere utilisation. Les joueurs avec la permission minage.admin (administrateurs) contournent le cooldown.

long now = System.currentTimeMillis(); long cooldownMs = MiningConfig.miningWorldCooldown * 1000L; long elapsed = now - data.lastMinageUse; if (elapsed < cooldownMs && !RankManager.hasPermission(player, "minage.admin")) { long remaining = (cooldownMs - elapsed) / 1000L; EriChat.create() .text("Cooldown : encore ").color(TextFormatting.RED) .text(remaining + "s").color(TextFormatting.YELLOW) .to(player); return; }

07 Structure de Spawn

Concept

Le point de spawn du Monde Minage est une ile flottante positionnee au centre de la dimension (X=0, Z=0) a une hauteur elevee (Y=80 environ). Cette structure est un fichier .nbt place par le jeu a chaque reset de la dimension.

ℹ
Fichier de structure — Le fichier NBT est stocke a assets/eriniumfaction/structures/mining_spawn.nbt. Ce fichier est cree une fois avec l'outil Structure Block de Minecraft en mode SAVE. La structure doit etre simple, robuste et facilement lisible par le TemplateManager de Forge.

Specification de la structure

Caracteristique Detail
Nom du fichier mining_spawn.nbt
Emplacement assets/eriniumfaction/structures/
Position dans le monde X=0, Y=80, Z=0 (coin NW de la structure)
Forme Ile flottante circulaire
Blocs recommandes Stone, Grass, Dirt, Cobblestone, Mossy Cobblestone
Taille approximative 15x6x15 (LxHxP)
Point de spawn joueur Centre de la structure, bloc superieur (Y=86 env.)

Placement via TemplateManager

La structure est placee via l'API TemplateManager de Minecraft apres la generation du premier chunk au centre. Ce processus est declenche lors de chaque reset du monde.

public static void placeSpawnStructure(WorldServer miningWorld) { TemplateManager templateManager = miningWorld.getStructureTemplateManager(); ResourceLocation structureLoc = new ResourceLocation( EriniumFaction.MODID, "mining_spawn" ); Template template = templateManager.get(miningWorld.getMinecraftServer(), structureLoc); if (template == null) { EriniumFaction.LOGGER.error("Structure mining_spawn.nbt introuvable !"); return; } BlockPos origin = new BlockPos(0, SPAWN_Y, 0); PlacementSettings settings = new PlacementSettings() .setRotation(Rotation.NONE) .setMirror(Mirror.NONE) .setIgnoreEntities(false); template.addBlocksToWorld(miningWorld, origin, settings); SPAWN_Y_SURFACE = SPAWN_Y + template.getSize().getY() - 1; EriniumFaction.LOGGER.info("Structure de spawn placee a Y={}", SPAWN_Y_SURFACE); }

Chargement des chunks de spawn

Avant de placer la structure, les chunks au centre doivent etre charges. Ceci est fait en forcant le chargement des chunks adjacents (0,0) via world.getChunk() avant l'appel a placeSpawnStructure().

// Forcer le chargement des chunks autour du spawn for (int cx = -2; cx <= 2; cx++) { for (int cz = -2; cz <= 2; cz++) { miningWorld.getChunk(cx, cz); // Charge si pas deja en memoire } } placeSpawnStructure(miningWorld);

08 Systeme de Reset

Vue d'ensemble

Le reset automatique efface et regenere le Monde Minage toutes les 7 jours (configurable). Ce systeme garantit que les ressources sont perpetuellement renouvelees et qu'aucune faction ne peut monopoliser durablement les zones riches.

Persistance du timer

MiningWorldResetData etend WorldSavedData et est attache a l'Overworld (dimension 0). Il survive aux redemarrages du serveur car il est sauvegarde dans les donnees du monde principal.

public class MiningWorldResetData extends WorldSavedData { public static final String KEY = "MiningWorldResetData"; public long nextResetTimestamp = 0L; // Unix millis public boolean structurePlaced = false; // Structure deja posee ? public static MiningWorldResetData get(WorldServer overworld) { MapStorage storage = overworld.getMapStorage(); MiningWorldResetData data = (MiningWorldResetData) storage.getOrLoadData(MiningWorldResetData.class, KEY); if (data == null) { data = new MiningWorldResetData(); data.nextResetTimestamp = System.currentTimeMillis() + MiningConfig.miningWorldResetDays * 86_400_000L; storage.setData(KEY, data); } return data; } @Override public void readFromNBT(NBTTagCompound nbt) { nextResetTimestamp = nbt.getLong("nextResetTimestamp"); structurePlaced = nbt.getBoolean("structurePlaced"); } @Override public NBTTagCompound writeToNBT(NBTTagCompound nbt) { nbt.setLong("nextResetTimestamp", nextResetTimestamp); nbt.setBoolean("structurePlaced", structurePlaced); return nbt; } }

Procedure de reset

Quand le timestamp de reset est atteint, la procedure suivante est executee de maniere sequentielle :

PROCEDURE DE RESET │ ├─ 1. Broadcast final (message a tous les joueurs en ligne) │ ├─ 2. Teleporter les joueurs presents dans la dim minage │ → Vers leur position sauvegardee dans l'Overworld │ → Ou vers le spawn de l'Overworld si pas de position │ ├─ 3. Decharger la dimension │ → DimensionManager.unloadWorld(miningWorld) │ → Attendre que tous les chunks soient sauvegardes │ ├─ 4. Supprimer le dossier de dimension │ → Suppression recursive du dossier │ → world_name/DIM{id}/ (ex: world/DIM2/) │ ├─ 5. Mettre a jour le timer │ → nextResetTimestamp = now + resetDays * 86400000 │ → structurePlaced = false │ → Sauvegarder MiningWorldResetData │ └─ 6. La dimension regenere naturellement → Au prochain /minage, la dim est recree par Forge → placeSpawnStructure() est appele au premier acces

Implementation du reset

public class MiningWorldResetManager { private static MiningWorldResetManager INSTANCE; private MiningWorldResetData data; // Appele sur ServerTickEvent (toutes les 20 ticks = 1 sec) public void tick(WorldServer overworld) { if (data == null) data = MiningWorldResetData.get(overworld); long now = System.currentTimeMillis(); // Verifier si la structure doit etre placee (apres un reset) if (!data.structurePlaced) { WorldServer miningWorld = getMiningWorld(overworld); if (miningWorld != null) { placeSpawnStructure(miningWorld); data.structurePlaced = true; data.markDirty(); } } // Verifier si le reset est du if (now >= data.nextResetTimestamp) { executeReset(overworld); } // Verifier les seuils d'avertissement checkWarnings(now); } private void executeReset(WorldServer overworld) { WorldServer miningWorld = getMiningWorld(overworld); // 1. Broadcast EriChat.create() .text("[Monde Minage] ").color(TextFormatting.GOLD).bold() .text("Reset en cours ! Tous les joueurs sont teleportes.").color(TextFormatting.RED) .broadcast(); // 2. Teleporter joueurs if (miningWorld != null) { for (EntityPlayerMP player : new ArrayList<>(miningWorld.playerEntities)) { teleportToOverworld(player, overworld); } // 3. Decharger DimensionManager.unloadWorld(MiningWorldManager.MINING_DIM_ID); } // 4. Supprimer le dossier deleteDimensionFolder(overworld); // 5. Mettre a jour le timer data.nextResetTimestamp = System.currentTimeMillis() + MiningConfig.miningWorldResetDays * 86_400_000L; data.structurePlaced = false; data.markDirty(); } }

Des avertissements sont envoyes a tous les joueurs connectes avant le reset. Un EriScheduler verifie periodiquement les seuils.

// Verification des seuils (appele chaque tick serveur) private static final long[] WARNING_THRESHOLDS = { 30L * 60 * 1000, // 30 min 10L * 60 * 1000, // 10 min 5L * 60 * 1000, // 5 min 1L * 60 * 1000 // 1 min }; private void checkWarnings(long now) { long remaining = data.nextResetTimestamp - now; for (long threshold : WARNING_THRESHOLDS) { // Envoyer le message si on vient juste de passer le seuil // (dans une fenetre de 20 ticks = 1 seconde pour eviter les doublons) if (remaining > threshold && remaining <= threshold + 1000) { broadcastWarning(threshold); } } }

Suppression du dossier de dimension

private void deleteDimensionFolder(WorldServer overworld) { File worldDir = overworld.getSaveHandler().getWorldDirectory(); File dimDir = new File(worldDir, "DIM" + MiningWorldManager.MINING_DIM_ID); if (dimDir.exists()) { try { FileUtils.deleteDirectory(dimDir); // Apache Commons IO EriniumFaction.LOGGER.info("Dossier DIM{} supprime avec succes", MiningWorldManager.MINING_DIM_ID); } catch (IOException e) { EriniumFaction.LOGGER.error("Echec suppression DIM{}", MiningWorldManager.MINING_DIM_ID, e); } } }
⚠
Apache Commons IO — Pour la suppression recursive de dossier, utiliser FileUtils.deleteDirectory() d'Apache Commons IO qui est deja presente dans le classpath de Minecraft/Forge. Eviter d'implementer sa propre recursion de suppression de fichiers.

Teleportation inter-dimensions

La teleportation entre dimensions utilise l'API Forge ForgeHooks.onTravelToDimension() suivi de EntityPlayerMP.travelToDimension(). La position de destination est passee via les attributs du joueur avant le voyage.

public static void teleportToMiningWorld(EntityPlayerMP player) { // Sauvegarder position Overworld MiningPlayerData data = DataManager.get(player, MiningPlayerData.class); data.returnX = player.posX; data.returnY = player.posY; data.returnZ = player.posZ; data.lastMinageUse = System.currentTimeMillis(); DataManager.save(player, MiningPlayerData.class); // Teleporter vers le spawn de la dimension player.travelToDimension(MiningWorldManager.MINING_DIM_ID); player.setPositionAndUpdate( 0.5, MiningWorldManager.SPAWN_Y_SURFACE, 0.5 ); } public static void teleportToOverworld(EntityPlayerMP player, WorldServer overworld) { MiningPlayerData data = DataManager.get(player, MiningPlayerData.class); player.travelToDimension(0); // Overworld if (!Double.isNaN(data.returnX)) { player.setPositionAndUpdate(data.returnX, data.returnY, data.returnZ); } else { // Pas de position sauvegardee : spawn Overworld BlockPos spawn = overworld.getSpawnPoint(); player.setPositionAndUpdate(spawn.getX() + 0.5, spawn.getY(), spawn.getZ() + 0.5); } }

Cas limites importants

  • Serveur redemarre pendant un reset en cours — La procedure de reset est atomique : si interrompue, le timer est mis a jour en dernier (etape 5). Au redemarrage, si le dossier de dimension existe encore mais le timer est depasse, un nouveau reset doit etre declenche.
  • Joueur deconnecte dans la dim minage pendant un reset — Le joueur doit reapparaitre dans l'Overworld a sa reconnexion. Verifier la dimension du joueur dans PlayerLoggedInEvent et teleporter si necessaire.
  • Multiplicateur fractionnaire — Avec 1.5x pour 4 veines/chunk, on doit generer 6 veines. Utiliser une variable accumulatrice pour les fractions : accumulated += multiplier; while(accumulated >= 1.0) { generateVein(); accumulated -= 1.0; }
  • ID de dimension conflictuel — Logger un WARNING au demarrage si l'ID configuré est deja utilise par une autre dimension.
  • Structure NBT corrompue ou absente — Logger ERROR et teleporter le joueur au bloc le plus haut au centre (0, 0) comme fallback.
  • Overworld inaccessible pendant teleportation retour — Cas impossible en pratique, mais gerer l'exception potentielle de travelToDimension().
