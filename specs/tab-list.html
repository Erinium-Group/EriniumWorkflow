Refonte Tab List — Liste des Joueurs

Remplacement complet de la liste de joueurs vanilla (TAB) par un panneau glass morphism personnalise : 3 colonnes de 15 joueurs, tetes de skin, grades colores et indicateur de ping vanilla.

Planifie Mixin GuiPlayerTabOverlay Client-Side Only 3 Colonnes x 15 45 Joueurs Max

01 Contexte & Objectifs

La liste de joueurs vanilla de Minecraft 1.12.2 (affichee en maintenant TAB) est fonctionnelle mais visuellement generique et totalement deconnectee de l'identite visuelle d'EriniumFaction. Elle affiche les pseudos en texte blanc sur fond gris, sans notion de grade, sans couleur de faction, et avec un indicateur de ping minimaliste.

Sur un serveur faction/PvP actif, la tab list est un element d'interface consulte en permanence — pendant les combats pour evaluer les ennemis presents, pendant la gestion de faction pour verifier les membres en ligne. Elle doit refleter l'univers du mod.

Problemes du rendu vanilla

  • Aucun grade visible — Les prefixes de grade ne sont pas affichables sans modifier le packet S38PacketPlayerListItem, ce qui implique des complications serveur
  • Pas de tete de joueur — Le vanilla affiche juste le pseudo en texte
  • Fond gris generique — Aucune coherence avec le style Liquid Glass du mod
  • Couleurs de rang absentes — Tous les joueurs ont le meme style visuel
  • Ping basique — Les barres vanilla sont petites et non colorisees selon le seuil
  • Pas de mise en page adaptee — Le vanilla empile les joueurs en une ou deux colonnes sans controle de la presentation

Objectifs de la refonte

  • Immersion visuelle — Panneau Liquid Glass coherent avec les autres GUI du mod (violet/cyan, translucide, coins arrondis)
  • Information riche — Tete de skin, grade colore, ping colore par niveau
  • Lisibilite optimale — Layout 3 colonnes, lignes alternees, texte anti-aliase
  • Tri hierarchique — Grades les plus hauts en premier, puis ordre alphabetique
  • Performance — Rendu conditionnel (seulement si TAB enfonce), cache joueurs, pas de recalcul inutile
  • Client-side uniquement — Zero modification serveur, zero packet custom, toutes les donnees sont deja disponibles
✓
Approche non destructive — Le Mixin annule uniquement le rendu vanilla, sans toucher a la logique de gestion des joueurs de Minecraft. Toutes les donnees (profils, ping, skins) continuent d'etre gerees normalement par le client vanilla.

02 Architecture Technique

L'implementation est purement client-side. Le rendu vanilla est annule via un Mixin, et remplace par un arbre de composants EriAPI rendus dans le contexte overlay. Aucun packet, aucune donnee serveur supplementaire n'est necessaire.

Classe cible du Mixin

  • Classe : net.minecraft.client.gui.GuiPlayerTabOverlay
  • Methode : renderPlayerlist(int width, ScoreObjective objective, float partialTicks)
  • Injection : @Inject at HEAD, cancellable = true
  • Action : Appel de ci.cancel() apres le rendu custom pour empecher le rendu vanilla
@Mixin(GuiPlayerTabOverlay.class) public class MixinGuiPlayerTabOverlay { @Inject(method = "renderPlayerlist", at = @At("HEAD"), cancellable = true) private void onRenderPlayerlist(int width, ScoreObjective objective, float partialTicks, CallbackInfo ci) { // Recuperer l'etat du bouton TAB if (!Minecraft.getMinecraft().gameSettings.keyBindPlayerList.isKeyDown()) { return; // Tab non enfonce : ne rien faire (vanilla s'en charge aussi) } // Deleguer au renderer custom TabListRenderer.INSTANCE.render(width, partialTicks); ci.cancel(); // Annuler le rendu vanilla } }

Classe de rendu principale

Toute la logique de rendu est encapsulee dans un singleton TabListRenderer (@SideOnly(Side.CLIENT)) dans le package fr.eriniumgroup.eriniumfaction.tablist. Ce singleton construit et maintient un arbre de composants EriAPI en memoire (un ContainerComponent racine), le reconstruit quand la liste de joueurs change, puis le dessine a chaque frame via ScaleManager.

fr.eriniumgroup.eriniumfaction.tablist/ ├── TabListRenderer.java Singleton — construit l'arbre EriAPI et gere le rendu ├── TabListCache.java Cache des joueurs tries (MAJ toutes les 20 ticks) └── mixin/ └── MixinGuiPlayerTabOverlay.java Mixin — annule le rendu vanilla Enregistrement dans mixins.eriniumfaction.json : "client": ["MixinGuiPlayerTabOverlay"] Arbre de composants (reconstruit a chaque invalidation du cache) : ContainerComponent root (0, 0, 1920, 1080) ├── Rectangle bg panel (panelX, 20, 700, panelH) — fond glass ├── Rectangle header bg (panelX, 20, 700, 36) — fond header violet ├── Label header title (panelX, 28, 350, 20) — "ERINIUM FACTION" ├── Label player count (panelX+350, 28, 350, 20) — "32 / 45 joueurs" └── per-player (x15 x3 cols) ├── Rectangle row bg (colX, rowY, colW, 22) — alternance subtile ├── TexturedRect head (colX+4, rowY+3, 16, 16) — face skin └── Label name (colX+24, rowY+5, nameW, 14) — prefixe + pseudo (ping : rendu OpenGL helper isole — voir section 07)

04 Donnees Affichees par Entree

Chaque ligne joueur affiche quatre elements dans l'ordre gauche → droite :

1. Tete de joueur (skin face)

  • Texture : info.getLocationSkin() — ResourceLocation du skin charge par Minecraft
  • Rendu : region (8, 8, 8, 8) de la texture skin (couche de base du visage)
  • Superposition : region (40, 8, 8, 8) pour la couche chapeau/hat (avec GL11.glEnable(GL11.GL_BLEND))
  • Taille rendue : 8x8 pixels design (scalee automatiquement)
  • Si le skin n'est pas encore charge (premiers ticks apres connexion) : afficher le skin Steve par defaut (new ResourceLocation("textures/entity/steve.png"))

2. Prefixe de grade + pseudo

  • Source : info.getDisplayName() si non null et non vide — contient le prefixe grade formate par le serveur
  • Fallback : construire manuellement avec FactionPlayerData.rankPrefix + espace + info.getGameProfile().getName()
  • Rendu : composant Label EriAPI avec la chaine formatee — les codes §x du prefixe sont interprete par le fontRenderer sous-jacent. Le Label gere automatiquement la troncature avec "..." si le texte depasse sa largeur design.
  • Troncature : configurer le Label avec la largeur exacte de la zone texte — EriAPI applique clipToWidth automatiquement lors du rendu

3. Indicateur de ping

  • Source : info.getResponseTime() en millisecondes
  • Rendu : icone barres signal vanilla (texture Gui.ICONS, region variable selon le niveau)
  • Position : aligne a droite de la colonne, centre vertical sur la ligne
  • Details dans la section 05
⚠
getDisplayName() null — Le serveur n'envoie pas toujours un displayName. Toujours verifier info.getDisplayName() != null avant de l'utiliser. En cas de null, utiliser le nom brut du profil et construire le prefixe depuis FactionPlayerData.

private void renderPingIcon(int ping, int x, int y) { // Selectionner la region selon le ping int iconV; int color; if (ping < 0) { iconV = 216; color = 0xFF8892A4; // gris (pas encore mesure) } else if (ping < 150) { iconV = 176; color = 0xFF2ECC71; // vert — 5 barres } else if (ping < 300) { iconV = 184; color = 0xFFA8D850; // vert clair — 4 barres } else if (ping < 600) { iconV = 192; color = 0xFFF1C40F; // jaune — 3 barres } else if (ping < 1000) { iconV = 200; color = 0xFFE67E22; // orange — 2 barres } else { iconV = 208; color = 0xFFE74C3C; // rouge — 1 barre } // Appliquer la teinte coloree int r = (color >> 16) & 0xFF; int g = (color >> 8) & 0xFF; int b = color & 0xFF; GlStateManager.color(r / 255f, g / 255f, b / 255f, 1.0f); // Binder la texture des icones vanilla Minecraft.getMinecraft().getTextureManager().bindTexture(Gui.ICONS); // Fond gris (barres vides) drawTexturedModalRect(x, y, 0, 176 + 5 * 8, 10, 8); // Barres colorees par-dessus drawTexturedModalRect(x, y, 0, iconV, 10, 8); GlStateManager.color(1f, 1f, 1f, 1f); // Reset couleur }

06 Tri des Joueurs

La liste des joueurs est triee selon deux criteres successifs : d'abord par priorite de grade (descendant — grade le plus haut en premier), puis par ordre alphabetique du pseudo en cas d'egalite.

Algorithme de tri

// Dans TabListCache.java private List<NetworkPlayerInfo> buildSortedList() { Collection<NetworkPlayerInfo> raw = Minecraft.getMinecraft().getConnection().getPlayerInfoMap(); return raw.stream() .sorted(Comparator // 1. Par priorite de grade (decroissant) .comparingInt((NetworkPlayerInfo info) -> getRankPriority(info)) .reversed() // 2. Par nom alphabetique (croissant) .thenComparing(info -> info.getGameProfile().getName().toLowerCase())) .limit(45) // Maximum 45 joueurs affiches .collect(Collectors.toList()); } // Obtenir la priorite de grade depuis FactionPlayerData private int getRankPriority(NetworkPlayerInfo info) { UUID uuid = info.getGameProfile().getId(); EntityPlayer player = Minecraft.getMinecraft().world .getPlayerEntityByUUID(uuid); if (player == null) return 0; // Grade inconnu : priorite minimale FactionPlayerData data = DataManager.get(player, FactionPlayerData.class); if (data == null) return 0; // rankPriority est un champ @Sync int dans FactionPlayerData return data.rankPriority; }

Prerequis — champ rankPriority

Pour que le tri fonctionne, FactionPlayerData doit exposer un champ @Sync int rankPriority correspondant a la priorite du grade du joueur (valeur numerique stockee dans le RankInfo, plus haute = grade superieur). Ce champ doit etre mis a jour par le serveur lors d'un changement de grade et synchronise au client via le mecanisme @Sync du DataManager EriAPI.

ℹ
Joueurs non charges — Si un EntityPlayer n'est pas present dans world.getPlayerEntityByUUID() (joueur distant dont l'entite n'est pas chargee), la priorite retourne 0 (grade par defaut). Le tri reste correct, ces joueurs apparaissent en fin de liste.

07 Rendu via Composants EriAPI

Le rendu repose entierement sur l'arbre de composants EriAPI. Toutes les coordonnees sont en espace design 1920x1080 — chaque composant appelle ScaleManager.scaleX/Y() en interne pour convertir vers les pixels ecran reels. EriAPI gere son propre etat OpenGL (blend, couleurs, matrices de texte) par composant : il n'est pas necessaire de pousser/popper un etat GL global autour de l'appel a root.render(), sauf un GL11.glPushMatrix() / glPopMatrix() de securite autour de l'ensemble du rendu dans le Mixin.

✓
EriAPI pour les overlays — Les composants EriAPI (ContainerComponent, Rectangle, Label, TexturedRect) fonctionnent en dehors d'un EriGuiScreen. Chaque composant convertit ses coordonnees design en coordonnees ecran via ScaleManager.scaleX/Y() lors du rendu — aucune projection manuelle n'est necessaire. Il suffit d'appeler root.render(mouseX, mouseY, partialTicks) depuis le Mixin, encadre d'un simple GL11.glPushMatrix() / glPopMatrix(). Seul le rendu ping (barres signal vanilla) reste un helper OpenGL minimal isole.

08 Animations

Les animations sont minimales et pertinentes — elles ne doivent pas distraire pendant un combat. L'accent est mis sur une ouverture/fermeture fluide et un retour visuel discret.

Fade d'ouverture/fermeture

  • Quand TAB est enfonce, le panneau apparait avec un fade-in de l'opacite : 0.0 → 1.0 en 6 ticks (0.3s)
  • Quand TAB est relache, le panneau disparait avec un fade-out : 1.0 → 0.0 en 4 ticks (0.2s)
  • Implemente via un float tabAlpha interpolee dans TabListRenderer.tick() (appele depuis ClientTickEvent)
  • L'alpha est applique sur le ContainerComponent racine via root.setOpacity(tabAlpha) — EriAPI propage automatiquement l'opacite effective a tous les composants enfants via getEffectiveOpacity()
private float tabAlpha = 0f; private static final float FADE_IN_SPEED = 1f / 6f; // 6 ticks private static final float FADE_OUT_SPEED = 1f / 4f; // 4 ticks public void tick() { boolean tabDown = Minecraft.getMinecraft().gameSettings .keyBindPlayerList.isKeyDown(); if (tabDown) { tabAlpha = Math.min(1f, tabAlpha + FADE_IN_SPEED); } else { tabAlpha = Math.max(0f, tabAlpha - FADE_OUT_SPEED); } if (root != null) root.setOpacity(tabAlpha); // propague a tous les enfants }

Lignes alternees

  • Les lignes d'index impair (0-based) ont un fond rgba(255,255,255,0.025), les lignes paires sont transparentes
  • Pas d'animation sur les lignes — effet purement statique pour la lisibilite

Absence de scroll

  • Si plus de 45 joueurs sont connectes, seuls les 45 premiers selon le tri sont affiches
  • Pas de scroll animate — la tab list est un affichage rapide, pas un menu
  • Un message discret dans le footer indique le nombre total si > 45 (voir section 11)

09 Performance & Strategie de Cache

La tab list est rendue a chaque frame tant que TAB est enfonce. Sans cache, le tri et la resolution des donnees joueur seraient effectues 60+ fois par seconde. Un cache avec invalidation periodique est indispensable.

TabListCache

Element Strategie Frequence de mise a jour
Liste triee des joueurs Reconstruite apres chaque tri, stockee dans une List Toutes les 20 ticks (1 seconde)
Noms formates (displayName) Stockes dans une Map<UUID, String> Mis a jour uniquement si la liste change
Skins (ResourceLocation) Pas de cache necessaire — Minecraft gere le sien Appel direct a info.getLocationSkin()
Priorites de grade Lues depuis FactionPlayerData (@Sync, deja en memoire) Pas de cache supplementaire
// TabListCache — actualisation periodique private int ticksSinceLastUpdate = 0; private static final int UPDATE_INTERVAL = 20; // ticks public void onClientTick() { ticksSinceLastUpdate++; if (ticksSinceLastUpdate >= UPDATE_INTERVAL) { ticksSinceLastUpdate = 0; rebuildCache(); // Retri + reconstruction des noms } } // Forcer la mise a jour immediate (appele a la connexion/deconnexion d'un joueur) public void invalidate() { ticksSinceLastUpdate = UPDATE_INTERVAL; }

Ecouter les evenements de connexion

Pour invalider le cache immediatement quand un joueur rejoint ou quitte le serveur, ecouter :

  • Connexion : ClientConnectedToServerEvent ou FMLNetworkEvent.ClientConnectedToServerEvent
  • Deconnexion joueur : surveiller la taille de getPlayerInfoMap() — si elle change entre deux ticks, invalider

Cache de l'arbre de composants

  • L'arbre de composants (ContainerComponent root) est le cache de rendu : il est reconstruit une fois par rebuildCache(), puis reutilise tel quel a chaque frame
  • Seul root.setOpacity(tabAlpha) est appele a chaque tick pour le fade — aucune autre modification de l'arbre n'est faite par frame
  • Les textures skin sont gerees par Minecraft (pas de cache additionnel) ; TexturedRect appelle bindTexture une fois par composant par frame — c'est incontournable pour les skins distincts
  • EriAPI gere lui-meme l'etat OpenGL (blend, matrice, couleurs) — ne pas dupliquer ces appels autour de root.render()

10 Integrations

La tab list s'appuie sur trois systemes existants du projet sans les modifier.

RankManager

  • Utilise indirectement via FactionPlayerData.rankPrefix (champ @Sync) et FactionPlayerData.rankPriority (champ @Sync a ajouter)
  • Ne jamais appeler RankManager directement depuis le rendu client — il peut etre absent ou inaccessible selon le contexte
  • Le prefixe colore est disponible soit dans info.getDisplayName() (si le serveur le pousse), soit construit depuis FactionPlayerData.rankPrefix

FactionPlayerData

11 Cas Limites

0 joueurs connectes

  • Si getPlayerInfoMap() retourne une collection vide, afficher le panneau avec uniquement le header et un message centre : §7Aucun joueur en ligne
  • La hauteur minimale du panneau est fixee a HEADER_H + 30px pour eviter un panneau de 1px

Plus de 45 joueurs (serveur > 45 slots)

  • Les 45 premiers selon le tri sont affiches
  • Un footer discret sous les colonnes indique : §7+ N joueurs supplémentaires (N = total - 45)
  • Aucune pagination — la tab list n'est pas un menu, elle doit etre consultable en un coup d'oeil

Pseudo tres long

  • Tronquer avec fontRenderer.trimStringToWidth() a la largeur disponible moins 3px, puis ajouter §8...
  • La largeur disponible est : largeur colonne - 8px (skin) - 4px (gap) - 22px (ping) - 4px (margin droite)

Joueur sans grade (grade par defaut)

  • Si rankPrefix est vide ou null : ne pas afficher de prefixe, afficher uniquement le pseudo en blanc (0xFFFFFFFF)
  • La priorite de tri est 0 (valeur par defaut de rankPriority)

Skin non encore charge

  • Si info.getLocationSkin() retourne une ResourceLocation mais que la texture n'est pas encore chargee en GPU, Minecraft affichera automatiquement une texture blanche/rose
  • Utiliser new ResourceLocation("textures/entity/steve.png") comme fallback explicite quand getLocationSkin() retourne la texture par defaut

Joueur dans un autre monde (dimension)

  • getPlayerInfoMap() liste tous les joueurs du serveur, pas seulement ceux de la dimension du joueur local
  • Pas de filtrage par dimension — afficher tous les joueurs connectes, quelle que soit leur dimension

TAB maintenu pendant l'ouverture d'un GUI

  • Le Mixin injecte dans renderPlayerlist qui n'est appele par vanilla que quand la condition TAB est vraie
  • Si un GUI EriAPI est ouvert par-dessus la tab list, le rendu de la tab list s'arrete naturellement (Minecraft n'appelle plus renderPlayerlist)

Connexion au serveur pendant l'affichage

  • Le TabListCache se met a jour toutes les 20 ticks, donc un nouveau joueur apparait dans la liste en moins d'une seconde
  • L'invalidation du cache est egalement declenchee par FMLNetworkEvent a la connexion d'un joueur
