Telechargement & Mise a jour du Jeu

Systeme de gestion des fichiers du jeu : telechargement initial, mises a jour delta, verification d'integrite SHA-256, et protection anti-tamper. Gere par le launcher, transparent pour le joueur.

01 Contexte & Objectifs

Probleme

Le serveur EriniumFaction utilise une vingtaine de mods custom (EriniumFaction, EriAPI, etc.) et des fichiers de configuration specifiques. Les joueurs doivent avoir exactement les bons fichiers, aux bonnes versions, pour se connecter. Distribuer et maintenir ces fichiers manuellement est impossible avec 500-1000 joueurs. Le launcher s'appuie sur la librairie minecraft-java-core pour gerer le telechargement de Minecraft vanilla (versions, librairies, assets, natives) et utilise CleanRoom (fork de Forge 1.12.2 compatible Java 21+) comme modloader.

Objectifs

  • Installation zero-effort — le joueur installe le launcher, clique "JOUER", et tout se telecharge automatiquement
  • Mises a jour delta — quand un mod est mis a jour, seuls les fichiers changes sont re-telecharges (pas tout le pack)
  • Verification d'integrite — avant chaque lancement, verifier que les fichiers critiques n'ont pas ete modifies (anti-triche)
  • Resilience — reprendre les telechargements interrompus, gerer les erreurs reseau, permettre le jeu offline avec les fichiers caches
  • Performance — telechargements paralleles, progression fluide, pas de freeze de l'UI

Contraintes

  • Taille totale estimee — ~500 Mo pour l'ensemble des fichiers (CleanRoom + mods + configs + resource packs), plus ~300 Mo pour Minecraft vanilla (gere par minecraft-java-core)
  • Bande passante — le CDN doit supporter 1000 telechargements simultanes en periode de pic (lancement/MAJ)
  • Minecraft vanilla automatique — le launcher telecharge Minecraft 1.12.2 automatiquement via minecraft-java-core (versions, librairies, assets, natives depuis les serveurs Mojang). Le joueur n'a PAS besoin d'avoir Minecraft deja installe

02 Architecture du Manifeste

Format du manifeste

Le manifeste est un fichier JSON heberge sur Vercel (meme domaine que le site web). Il est genere automatiquement par un script Node.js qui scanne un dossier source et calcule les hash SHA-256 de chaque fichier. Les fichiers de jeu sont eux aussi servis en statique par Vercel.

Hebergement sur Vercel

  • Manifeste — https://eriniumfaction.vercel.app/launcher/manifest.json
  • Fichiers — https://eriniumfaction.vercel.app/launcher/files/{path}
  • Avantages — meme domaine que le site, CDN global Vercel, zero cout supplementaire, deploiement automatique

Structure dans le projet web

EriniumFactionWeb/ ├── launcher-files/ # Dossier source (PAS dans public/) │ ├── mods/ │ │ ├── eriniumfaction-1.0.0.jar │ │ ├── eriapi-1.2.0-1.12.2.jar │ │ └── ... │ ├── config/ │ │ └── eriniumfaction_*.cfg │ └── resourcepacks/ │ └── EriniumRP.zip ├── public/launcher/ # Genere par le script │ ├── manifest.json # Manifeste auto-genere │ └── files/ # Copie des fichiers avec hash │ ├── mods/ │ ├── config/ │ └── resourcepacks/ └── scripts/ └── generate-manifest.js # Script de generation

Script de generation automatique

Le script scripts/generate-manifest.js :

  • Scanne recursivement le dossier launcher-files/
  • Calcule le hash SHA-256 de chaque fichier
  • Genere l'URL de telechargement : https://{SITE_URL}/launcher/files/{path}
  • Detecte la categorie automatiquement depuis le dossier parent (mods/ → "mod", config/ → "config", etc.)
  • Copie les fichiers dans public/launcher/files/
  • Ecrit le manifeste dans public/launcher/manifest.json
  • Calcule totalSize et fileCount automatiquement

Commande : node scripts/generate-manifest.js — a executer avant chaque deploiement quand les fichiers changent.

Peut etre integre dans le script npm run build pour automatiser.

04 Installation CleanRoom & Minecraft vanilla

minecraft-java-core : moteur de telechargement vanilla

Le launcher utilise la librairie NPM minecraft-java-core (npm install minecraft-java-core) pour gerer entierement le telechargement de Minecraft 1.12.2 vanilla. Cette librairie prend en charge :

  • Versions Minecraft — telechargement depuis le manifest officiel Mojang (launchermeta.mojang.com)
  • Librairies — resolution et telechargement de toutes les librairies requises par MC 1.12.2
  • Assets — telechargement de l'index d'assets et de tous les fichiers d'assets (sons, textures, langues)
  • Natives — telechargement et extraction des librairies natives (LWJGL, etc.) pour la plateforme courante
  • Runtime Java — detection ou telechargement du runtime Java si necessaire
  • Lancement — construction du classpath et lancement du jeu avec les bons arguments
i
Avantage — le joueur n'a PAS besoin d'avoir Minecraft installe au prealable. minecraft-java-core gere tout le telechargement vanilla automatiquement. Le launcher se suffit a lui-meme.

CleanRoom : remplacement de Forge

CleanRoom est un fork de Forge 1.12.2 developpe par CleanroomMC, compatible avec Java 21+ tout en conservant une compatibilite totale avec l'API Forge et les mods existants. C'est un drop-in replacement : meme API, memes mods, mais avec le support des versions modernes de Java.

  • GitHub — https://github.com/CleanroomMC/Cleanroom/releases
  • Version actuelle — cleanroom-0.5.7-alpha
  • Installateur — cleanroom-0.5.7-alpha-installer.jar
  • Universel — cleanroom-0.5.7-alpha-universal.jar

Flow d'installation (methode principale)

L'installateur CleanRoom fonctionne exactement comme l'installateur Forge : il extract les librairies, le profil version et configure le classpath automatiquement. C'est la methode par defaut.

  • 1. Minecraft vanilla — minecraft-java-core verifie si MC 1.12.2 est deja en cache. Si non, telechargement complet (versions, libraries, assets, natives) depuis les serveurs Mojang
  • 2. CleanRoom installer — telecharger cleanroom-0.5.7-alpha-installer.jar depuis notre CDN ou les GitHub Releases (releases)
  • 3. Execution de l'installateur — java -jar cleanroom-0.5.7-alpha-installer.jar --installClient {gameDir}. L'installateur :
    • Extract le jar universel CleanRoom
    • Telecharge et installe toutes les librairies necessaires
    • Cree le profil version JSON (comme Forge le fait)
    • Configure le classpath et les tweakers
  • 4. Verification — verifier exit code 0, presence des fichiers CleanRoom dans le dossier
  • 5. Fichiers custom — telecharger nos mods, configs, resource packs depuis notre manifeste/CDN

Detection version et mise a jour

  • Si CleanRoom est deja installe — verifier la version dans le profil JSON local vs la version du manifeste distant
  • Si version differente → re-telecharger l'installateur et re-executer
  • Si meme version → passer directement aux fichiers custom

Hash check : algorithme

const crypto = require('crypto'); const fs = require('fs'); async function hashFile(filePath) { return new Promise((resolve, reject) => { const hash = crypto.createHash('sha256'); const stream = fs.createReadStream(filePath); stream.on('data', chunk => hash.update(chunk)); stream.on('end', () => resolve(hash.digest('hex'))); stream.on('error', reject); }); } async function verifyFile(filePath, expectedHash) { const actualHash = await hashFile(filePath); return actualHash === expectedHash; }

Optimisation : hash partiel

  • Fichiers volumineux (> 50 Mo) — pour la verification rapide (pas le telechargement), hasher uniquement les 1 premiers Mo + 1 derniers Mo + la taille totale. Si ca match → considerer valide. Si ca mismatch → hash complet pour confirmer
  • Cache de hash — stocker les hash calcules en memoire pendant la session. Ne pas re-hasher un fichier deja verifie dans la meme session

06 Telechargement Parallele

Architecture du downloader

Le telechargement utilise un pool de connexions HTTP paralleles pour maximiser la bande passante. Le nombre de telechargements simultanes est configurable.

ParametreValeurDescription
Connexions paralleles4Nombre de fichiers telecharges simultanement (configurable 1-8)
Timeout par fichier60 secondesTimeout HTTP pour chaque fichier individuel
Retries par fichier3Nombre de tentatives avant de declarer un echec
Delai entre retries2 secondesBackoff exponentiel : 2s, 4s, 8s
Taille chunk64 KoTaille des chunks pour le streaming (progression granulaire)

Progression

Le renderer recoit des mises a jour de progression via IPC toutes les 200ms :

// Donnees envoyees via IPC 'download:progress' { "currentFile": "mods/eriniumfaction-1.0.0.jar", // Fichier en cours "currentFileProgress": 0.65, // 0.0 → 1.0 "currentFileSize": 10485760, // Taille totale "currentFileDownloaded": 6815744, // Octets telecharges "totalFiles": 17, // Nombre total de fichiers "completedFiles": 8, // Fichiers termines "totalBytes": 524288000, // Octets totaux "downloadedBytes": 278921216, // Octets telecharges "speed": 5242880, // Vitesse en octets/s "eta": 47 // Temps restant estime (secondes) }

Affichage dans l'UI

  • Barre de progression globale — remplace le bouton JOUER. Affiche le pourcentage global et la vitesse
  • Texte descriptif — "Telechargement... 8/17 fichiers (53%) — 5.0 Mo/s — ~47s restantes"
  • Annulation — bouton "Annuler" pour arreter le telechargement. Les fichiers partiels sont supprimes

Resume de telechargement

  • Fichiers complets — si un telechargement est interrompu, les fichiers deja entierement telecharges et verifies (hash OK) ne sont pas re-telecharges
  • Fichiers partiels — les fichiers partiellement telecharges sont supprimes et re-telecharges depuis le debut. Pas de HTTP Range (complexite disproportionnee pour des fichiers de quelques Mo)
  • Fichiers temporaires — pendant le telechargement, les fichiers sont ecrits avec l'extension .tmp. Renommes en nom final uniquement apres verification du hash. Evite les fichiers corrompus en cas de crash

Gestion de l'espace disque

  • Verification espace libre — avant le telechargement, verifier qu'il y a assez d'espace disque (taille totale + 20% de marge). Si insuffisant → message d'erreur clair avec la taille requise
  • Nettoyage ancien — quand un fichier est supprime du manifeste (mod retire), le launcher le supprime automatiquement du disque lors de la prochaine mise a jour
  • Bouton "Reparer" — dans les parametres, un bouton "Verifier et reparer les fichiers" qui force un re-hash de TOUS les fichiers et re-telecharge ceux qui ne correspondent pas

08 Anti-Tamper Fichiers

Whitelist des mods

Le dossier mods/ est sous controle strict. Seuls les fichiers listes dans le manifeste sont autorises :

  • Avant chaque lancement — scanner le dossier mods/. Lister tous les fichiers .jar presents
  • Comparer avec le manifeste — chaque fichier doit correspondre a une entree du manifeste (meme nom, meme hash)
  • Fichier non autorise — si un .jar est present dans mods/ mais PAS dans le manifeste → le deplacer dans un dossier mods_disabled/ et avertir le joueur
  • Fichier modifie — si un .jar a un hash different du manifeste → re-telecharger la version correcte
!
Anti-cheat client — cette verification empeche l'ajout de mods de triche (xray, reach hack, etc.) dans le dossier mods/. Un joueur qui ajoute un mod non autorise verra son mod deplace et ne pourra pas jouer avec. Combine avec la verification cote serveur (token + HWID), cela rend la triche par mods beaucoup plus difficile.

Verification en arriere-plan

  • Au lancement du launcher — hash check rapide des fichiers critiques (mods). Si un fichier est modifie, le bouton JOUER affiche "Reparation necessaire" au lieu de "JOUER"
  • Pendant le jeu — pas de verification (eviter les freeze du jeu et les I/O pendant le gameplay)
  • A la fermeture du jeu — optionnel : re-verifier les fichiers pour detecter si un mod a ete modifie pendant la session (attaque par swap de fichier pendant le runtime)

Fichiers config : politique mixte

  • Configs serveur-obligatoires — certaines configs (ex: connexion serveur, parametres gameplay) sont marquees "critical": true dans le manifeste. Elles sont ecrasees a chaque mise a jour
  • Configs joueur — les configs video, keybinds, options graphiques ne sont PAS dans le manifeste. Le joueur peut les modifier librement
  • Premier lancement — toutes les configs du manifeste sont copiees comme base. Le joueur peut ensuite personnaliser les configs non-critiques

09 CDN & Hebergement

Hebergement : Vercel (meme que le site)

Les fichiers de jeu et le manifeste sont heberges directement sur Vercel, dans le dossier public/launcher/ du projet web. Vercel sert ces fichiers en statique via son CDN global — meme domaine, zero cout supplementaire, deploiement automatique a chaque push.

i
Pourquoi Vercel ? — Pas besoin d'un CDN separe. Vercel a un CDN global integre (edge network), sert les fichiers statiques sans limite de bande passante sur le free tier (100 Go/mois), et le deploiement est automatique. Pour ~500 Mo de fichiers de jeu, c'est largement suffisant. Si un jour le trafic explose, on migrera les fichiers vers Cloudflare R2.

Script de generation scripts/generate-manifest.js

Un script Node.js qui automatise tout :

// Execution node scripts/generate-manifest.js // Ce que fait le script : // 1. Scanne recursivement launcher-files/ (dossier source) // 2. Pour chaque fichier : // - Calcule le hash SHA-256 // - Detecte la categorie depuis le dossier parent (mods/ → "mod", config/ → "config") // - Genere l'URL : https://{SITE_URL}/launcher/files/{path} // - Determine si le fichier est critique (mods = oui, configs = non) // 3. Copie tous les fichiers dans public/launcher/files/ // 4. Ecrit public/launcher/manifest.json avec toutes les metadonnees // 5. Affiche un resume : X fichiers, Y Mo total, Z nouveaux/modifies

Le script peut etre integre dans package.json :

"scripts": { "manifest": "node scripts/generate-manifest.js", "prebuild": "node scripts/generate-manifest.js" }

URLs de telechargement

// Manifeste https://eriniumfaction.vercel.app/launcher/manifest.json // Fichiers https://eriniumfaction.vercel.app/launcher/files/mods/eriniumfaction-1.0.0.jar https://eriniumfaction.vercel.app/launcher/files/config/eriniumfaction.cfg https://eriniumfaction.vercel.app/launcher/files/resourcepacks/EriniumRP.zip // Configuration custom domain sur Cloudflare R2 // Bucket R2 : erinium-launcher // Custom domain : cdn.erinium.fr

Processus de mise a jour (cote developpeur)

  • 1. Build le mod — generer le jar avec ./gradlew build
  • 2. Calculer le hash — sha256sum build/libs/eriniumfaction-1.0.0.jar
  • 3. Uploader sur R2 — via CLI wrangler r2 object put erinium-launcher/mods/eriniumfaction-1.0.0.jar --file=build/libs/eriniumfaction-1.0.0.jar
  • 4. Mettre a jour le manifeste — modifier le hash et la version dans manifest.json, uploader
  • 5. Verification — tester le telechargement depuis le launcher sur un poste de test
i
Automatisation future — a terme, un script CI/CD (GitHub Actions) pourrait automatiser les etapes 2-4 : build → hash → upload R2 → update manifeste → commit. Pour l'instant, processus manuel.

10 Gestion d'Erreurs

Erreurs reseau

ErreurComportementMessage joueur
Manifeste distant inaccessibleUtiliser le manifeste local si present. Avertir le joueur"Impossible de verifier les mises a jour. Vous jouez avec la derniere version connue."
Telechargement echoue (1 fichier)Retry 3 fois avec backoff exponentiel. Si echec final → erreur"Echec du telechargement de [fichier]. Verifiez votre connexion."
Telechargement echoue (CDN down)Tous les telechargements echouent. Annuler la queue"Le serveur de telechargement est indisponible. Reessayez plus tard."
Connexion perdue pendant DLMettre en pause. Reprendre automatiquement quand la connexion revient (poll toutes les 5s)"Connexion perdue. Reprise automatique..."
Hash mismatch apres DLRe-telecharger (max 3 fois). Si persiste → signaler une possible corruption CDN"Fichier corrompu. Nouveau telechargement en cours..."

Erreurs disque

ErreurComportementMessage joueur
Espace disque insuffisantCalculer avant de commencer. Ne pas demarrer le DL"Espace disque insuffisant. Il faut X Mo libres, vous avez Y Mo."
Permission refuseeTenter de creer le repertoire. Si echec → demander au joueur"Impossible d'ecrire dans [dossier]. Verifiez les permissions ou changez le repertoire dans les parametres."
Fichier verrouilleAttendre 2s, retenter. Si persiste → avertir"Le fichier [nom] est utilise par un autre programme. Fermez Minecraft et reessayez."

Mode degradation gracieuse

  • Offline total — si aucune connexion internet et que les fichiers locaux sont presents et valides → permettre le lancement (le joueur pourra jouer en solo ou sur LAN, meme si le serveur est injoignable). Afficher un bandeau "Mode hors ligne"
  • Manifeste corrompu — si le manifeste local est illisible → le supprimer et forcer un re-telechargement complet
  • Fichier manquant non-critique — si un fichier non-requis (resource pack optionnel) echoue → continuer le lancement sans ce fichier, avertir le joueur

11 Plan de Tests

Tests fonctionnels

  • Premier lancement — telecharge MC vanilla via minecraft-java-core, installe CleanRoom, telecharge tous les fichiers du manifeste, cree le repertoire
  • Lancement normal — manifeste identique, verification rapide des fichiers critiques, lancement
  • Mise a jour 1 fichier — un mod change de version. Seul ce fichier est re-telecharge
  • Mise a jour massive — 10+ fichiers changes. Telechargement parallele, progression correcte
  • Fichier supprime du manifeste — un mod est retire. Le fichier local est supprime
  • Fichier modifie par le joueur — un mod est modifie manuellement. Detecte et re-telecharge
  • Mod non autorise ajoute — un .jar inconnu dans mods/. Deplace dans mods_disabled/

Tests de resilience

  • Coupure reseau pendant DL — pause automatique, reprise quand la connexion revient
  • Crash du launcher pendant DL — les fichiers .tmp sont ignores au redemarrage, re-DL correct
  • CDN lent — timeout fonctionne, retry fonctionne, l'UI ne freeze pas
  • Espace disque plein — erreur detectee AVANT le telechargement
  • Manifeste distant corrompu — JSON invalide → message d'erreur, utiliser le local si possible

Tests de performance

  • Hash check 50 fichiers — moins de 5 secondes sur un SSD, moins de 15 secondes sur un HDD
  • Telechargement 500 Mo — utilisation correcte de la bande passante (pas de goulot cote launcher)
  • Memoire — le downloader ne charge pas les fichiers en RAM (streaming direct vers disque)
  • UI responsive — la progression se met a jour fluidement, pas de saccade, pas de freeze

Tests multi-plateforme

  • Windows — chemins %APPDATA%, detection .minecraft, permissions dossier
  • Linux — chemins ~/.eriniumfaction, detection ~/.minecraft, permissions chmod
  • macOS — chemins ~/Library, detection Minecraft.app, permissions
