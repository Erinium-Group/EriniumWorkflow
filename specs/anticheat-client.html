Anti-Cheat Client

Un mod client dedie a la detection des cheats invisibles au serveur : injections JVM, processus malveillants, modifications memoire, comportements suspects et ressources alterees. Inspire de VAC (Valve Anti-Cheat) et Vanguard (Riot Games), adapte a l'ecosysteme Minecraft Forge 1.12.2.

Securite Mod Separe DLL Native (Windows) 4 Couches de Detection Chiffrement AES-256-GCM Ban Wave System

01 Contexte & Objectifs

Positionnement dans l'ecosysteme

EriniumAntiCheat est un mod Forge 1.12.2 separe d'EriniumFaction, avec son propre Mod ID (eriniumanticheat). Il vient completer les systemes de securite existants :

Systeme existantCouvertureStatut
Anti-Cheat ServeurCheckMovement, CheckCombat, CheckWorld, CheckPacket, CheckMining9/9
ModIntegrityCheckerVerification des mods charges (hash, manifest, mods interdits)DONE
AuthentificationJWT + HWID binding + anti-VPN + token rotationDONE
Anti-Cheat ClientDetection cote client : injections, processus, memoire, comportementA FAIRE

Inspiration

  • VAC (Valve Anti-Cheat) : Scanning passif des processus et DLLs, ban waves differes, pas de mode kernel. VAC collecte silencieusement et ban par vagues pour eviter de reveler les signatures detectees.
  • Vanguard (Riot Games) : Verification d'integrite en temps reel, challenge-response, heartbeat continu. Vanguard prouve que l'anti-cheat tourne en permanence, pas seulement au lancement.
  • Adaptation Minecraft : Pas de driver kernel (trop invasif pour un mod MC). On reste en userland avec JNI pour les appels natifs Windows. La couche Java pure fonctionne sur tous les OS.

Objectifs de design

  • Detection en profondeur : Detecter ce que le serveur ne peut pas voir — agents Java attaches, processus cheat en arriere-plan, DLLs injectees, bytecode modifie au runtime, resource packs x-ray.
  • Faux positifs minimaux : Whitelist exhaustive des logiciels legitimes (NVIDIA, Discord, OBS, etc.). Les processus inconnus sont remontes sans action punitive immediate.
  • Resilience au contournement : Heartbeat continu avec challenge-response. Desactiver l'anti-cheat apres le check initial n'est pas possible — le serveur detecte l'absence de heartbeat.
  • Performance : Les scans sont espaces et asynchrones. Impact < 2 FPS en moyenne. Le scanning natif (DLL) est optimise pour ne pas impacter le rendu.
  • Transparence : Les joueurs savent que l'anti-cheat tourne. Pas de collecte de donnees personnelles au-dela des processus et DLLs du process Java.

Ce que le systeme ne fait pas

  • Pas de kernel driver : Aucun composant ring-0. Tout tourne en userland (ring-3).
  • Pas de keylogger : L'anti-cheat n'enregistre pas les frappes clavier. L'analyse d'input ne concerne que les clics souris et les mouvements camera dans Minecraft.
  • Pas de screenshot : Aucune capture d'ecran n'est envoyee au serveur.
  • Pas de ban automatique : Les violations SUSPICIOUS sont collectees pour des ban waves manuelles ou programmees. Seules les violations CRITICAL declenchent un kick immediat.
⚠
Dependances : Le mod declare required-after:eriapi et required-after:eriniumfaction. EriAPI fournit le framework (events, network, scheduler), EriniumFaction fournit le BanManager et le systeme d'authentification (JWT, HWID).

03 Couche 1 — JVM Integrity

Couche entierement en Java pur, fonctionne sur tous les OS. Elle verifie l'integrite de la JVM elle-meme : agents attaches, classes injectees, mixins non autorises, bytecode modifie.

AgentDetector

Detecte les agents Java attaches a la JVM, utilises par les cheats avances pour modifier le bytecode au runtime.

Methode de detectionDetailNiveau
Arguments JVMScan de -javaagent:, -agentlib:, -agentpath: dans ManagementFactory.getRuntimeMXBean().getInputArguments()CRITICAL
Instrumentation APITentative d'acces a java.lang.instrument.Instrumentation via reflection pour verifier si un agent est attache au runtime (post-launch attach)CRITICAL
Attach APIVerification de la presence de com.sun.tools.attach.VirtualMachine dans le classpath (indicateur d'outil d'attach)CRITICAL
public class AgentDetector { public ScanResult scan() { List<String> violations = new ArrayList<>(); // 1. Scan des arguments JVM List<String> args = ManagementFactory.getRuntimeMXBean().getInputArguments(); for (String arg : args) { if (arg.startsWith("-javaagent:") || arg.startsWith("-agentlib:") || arg.startsWith("-agentpath:")) { violations.add("AGENT_ARG: " + arg); } } // 2. Verification Instrumentation API (attach post-launch) try { Class<?> instrClass = Class.forName("sun.instrument.InstrumentationImpl"); // Si la classe existe et est chargee, un agent est attache violations.add("INSTRUMENTATION_LOADED"); } catch (ClassNotFoundException e) { // Attendu — pas d'agent } // 3. Verification Attach API dans le classpath try { Class.forName("com.sun.tools.attach.VirtualMachine"); violations.add("ATTACH_API_AVAILABLE"); } catch (ClassNotFoundException e) { // Attendu } return new ScanResult("AgentDetector", violations); } }

ClassLoaderAuditor

Enumere toutes les classes chargees dans la JVM et detecte les ClassLoaders non standard injectes par des clients cheats.

  • Enumeration des classes : Utilise Instrumentation.getAllLoadedClasses() si disponible, sinon parcourt les ClassLoaders connus via reflection.
  • Manifest de classes autorisees : Un fichier manifest (signe, distribue par le launcher) contient la liste des packages/prefixes autorises. Tout package inconnu est flagge.
  • ClassLoaders non standard : Detecte les ClassLoaders qui ne sont pas AppClassLoader, ExtClassLoader, LaunchClassLoader (Forge), ou MixinClassLoader (Mixin). Un ClassLoader inconnu indique une injection.
  • Packages suspects : Blacklist de packages connus de cheats : net.wurstclient, me.earth.earthhack, org.impact, net.ccbluex.liquidbounce, me.aristois, meteordevelopment, com.lambda, dev.tigr.ares, etc.

MixinAuditor

Verifie que seuls les mixins autorises sont appliques sur les classes Minecraft. Les clients cheats utilisent souvent Mixin pour modifier le comportement du jeu.

  • Manifest de mixins autorises : Le serveur envoie au login la liste des fichiers mixin JSON autorises (mixins.eriniumfaction.json, mixins.eriniumanticheat.json, mixins.eriapi.json). Tout fichier mixin non liste est suspect.
  • Enumeration des mixins appliques : Interroge le MixinEnvironment pour lister les transformers et configurations actives.
  • Classe cible monitoring : Verifie que les classes critiques (EntityPlayerSP, NetHandlerPlayClient, RenderGlobal, etc.) n'ont pas de mixins non autorises appliques.

BytecodeVerifier

Calcule et verifie les checksums SHA-256 des classes critiques de Minecraft. Si le bytecode d'une classe change au runtime (via agent, retransform, ou classloader custom), le checksum ne correspondra plus.

Classe surveilleeRaison
EntityPlayerSPMouvement du joueur, input, sprint, fly
NetHandlerPlayClientEnvoi/reception de packets, timer hack
PlayerControllerMPInteractions, attaques, reach hack
RenderGlobalRendu du monde, wallhack, x-ray, ESP
EntityRendererCamera, FOV, zoom hack, freecam
Minecraft (main class)Timer (vitesse de jeu), displayGuiScreen
GuiIngameHUD, crosshair overlay, ESP overlay
WorldX-ray via getBlockState override
MovementInputFromOptionsAuto-walk, strafe, scaffold
KeyBindingManipulation des touches, auto-click

Fonctionnement : Au premier chargement (postInit), les checksums SHA-256 sont calcules pour chaque classe cible via ClassLoader.getResourceAsStream(). Ces checksums de reference sont stockes en memoire. Un scan periodique (toutes les 60 secondes) recalcule les checksums et compare avec les references. Si un checksum differe, la classe a ete modifiee au runtime.

⚠
Attention Mixin : Les mixins legitimes (EriniumFaction, EriAPI) modifient aussi le bytecode. Le BytecodeVerifier doit capturer les checksums apres l'application des mixins autorises (a postInit), pas avant. Ainsi, les modifications legales sont dans la reference.

04 Couche 2 — Process & OS Scanning

Couche utilisant la DLL native via JNI. Active uniquement sur Windows x64. Scanne les processus, DLLs, fenetres et hooks au niveau du systeme d'exploitation.

Auto-detection OS

public class OSDetector { public enum OS { WINDOWS, LINUX, MAC, UNKNOWN } private static final OS CURRENT; static { String osName = System.getProperty("os.name").toLowerCase(); if (osName.contains("win")) CURRENT = OS.WINDOWS; else if (osName.contains("nux") || osName.contains("nix")) CURRENT = OS.LINUX; else if (osName.contains("mac")) CURRENT = OS.MAC; else CURRENT = OS.UNKNOWN; } public static OS getOS() { return CURRENT; } public static boolean isNativeAvailable() { return CURRENT == OS.WINDOWS; } }

NativeBridge (JNI)

Interface Java declarant les methodes natives implementees dans la DLL.

public class NativeBridge { private static boolean loaded = false; public static boolean load(String dllPath) { try { System.load(dllPath); loaded = true; } catch (UnsatisfiedLinkError e) { loaded = false; } return loaded; } // Process scanning public static native String[] enumerateProcesses(); public static native String[] enumerateLoadedDlls(); public static native String[] enumerateWindowTitles(); // Debugger detection public static native boolean isDebuggerPresent(); public static native boolean isRemoteDebuggerPresent(); public static native boolean ntQueryDebugPort(); // Hook detection public static native boolean checkIATIntegrity(); // Self-verification public static native byte[] getMemoryHash(); // SHA-256 de la DLL en memoire }

ProcessScanner

Enumere tous les processus en cours via CreateToolhelp32Snapshot (Win32 API) et compare contre la blacklist et la whitelist.

  • Enumeration : La DLL appelle CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0) puis itere avec Process32First / Process32Next pour obtenir la liste des noms de processus.
  • Comparaison : Chaque processus est compare contre la blacklist de cheats connus (section 9), puis contre la whitelist. Les processus non classes sont marques comme "inconnus" et remontes au serveur.
  • Matching : La comparaison est case-insensitive et utilise des patterns glob pour les variantes (wurst*.exe, *liquidbounce*, etc.).

DebuggerDetector

Detecte les debuggers attaches au processus Java. Un debugger permet de modifier le comportement du jeu en temps reel (breakpoints, modification de variables).

API Win32DetectionContournement connu
IsDebuggerPresent()Debugger userland attache (OllyDbg, x64dbg)Patchable en memoire — combine avec les autres
CheckRemoteDebuggerPresent()Debugger distant ou kernel debuggerHooking possible — c'est pourquoi HookDetector existe
NtQueryInformationProcess(DebugPort)Verifie si un port debug est attribue au processus (le plus fiable)Necessite un driver kernel pour contourner

Les trois methodes sont combinees : si au moins une retourne true, la violation est CRITICAL. L'utilisation de trois methodes rend le contournement plus difficile car il faut les patcher toutes.

DllInjectionDetector

Enumere les DLLs chargees dans le processus Java et detecte les injections.

  • Enumeration : La DLL utilise EnumProcessModules pour lister toutes les DLLs chargees dans le processus javaw.exe.
  • Whitelist : Les DLLs attendues sont whitelistees : DLLs JVM (jvm.dll, java.dll, zip.dll...), DLLs systeme Windows (kernel32.dll, ntdll.dll, user32.dll...), DLLs NVIDIA (nvoglv64.dll, nvinjectfilter64.dll...), DLLs AMD, DLLs Intel, DLLs Forge/Minecraft.
  • Detection : Toute DLL non whitelistee est suspecte. Le chemin complet de la DLL et son hash sont remontes au serveur.

WindowScanner

Enumere les fenetres visibles via EnumWindows (Win32 API) et compare les titres avec une blacklist de cheats.

  • Titres blacklistes : "Wurst", "Impact Client", "LiquidBounce", "Aristois", "Future Client", "Meteor Client", "Ares", "Lambda", "Sigma", "Rise Client", "Novoline", "Exhibition", "Remix", etc.
  • Patterns suspects : Titres contenant "inject", "cheat", "hack", "esp", "aimbot", "auto clicker" (case-insensitive).
  • Niveau : Un titre blackliste exact est CRITICAL. Un pattern suspect est SUSPICIOUS.

HookDetector

Verifie l'integrite des fonctions Win32 API utilisees par l'anti-cheat. Les cheats avances hookent les API systeme pour cacher leurs processus ou DLLs.

  • IAT Hooking : Verifie que l'Import Address Table (IAT) du processus n'a pas ete modifiee. Compare les adresses des fonctions importees avec les adresses reelles dans les DLLs systeme.
  • Inline Hooking : Verifie les premiers octets des fonctions critiques (NtQueryInformationProcess, EnumProcessModules, IsDebuggerPresent) pour detecter les JMP/CALL patches.
  • Niveau : Toute alteration des fonctions API est CRITICAL.
⚠
Antivirus : Certains antivirus (Avast, Norton, Kaspersky) injectent des DLLs dans les processus pour la surveillance. Ces DLLs sont dans la whitelist pour eviter les faux positifs. Les DLLs d'antivirus connues : aswhooka.dll (Avast), nvinject.dll (Norton), klhk.dll (Kaspersky), etc.

05 Couche 3 — Behavior Analysis

Couche en Java pur, fonctionne sur tous les OS. Analyse les patterns de comportement du joueur cote client pour detecter les auto-clickers, les aim assists et les timer hacks.

InputAnalyzer

Analyse les patterns de clic et de visee du joueur. Un humain a une variance naturelle dans ses actions, tandis qu'un outil automatise est statistiquement regulier.

Fenetre d'analyse : Les 200 derniers clics sont conserves en buffer circulaire. L'analyse est executee toutes les 100 ticks (5 secondes) si le joueur est en combat (a attaque une entite dans les 5 dernieres secondes).

ⓘ
Jitter clicking : Le serveur utilise un mod PvP 1.8 (pas de cooldown d'attaque). Le jitter clicking est une technique LEGITIME qui peut atteindre 12-16 CPS avec une variance moderee. Les seuils sont calibres pour ne pas punir le jitter clicking. Seuls les CPS anormalement hauts (> 20) ou la regularite statistiquement impossible declenchent une alerte.

MovementAnalyzer

Complement client au CheckMovement du serveur. Le client envoie des donnees supplementaires que le serveur ne peut pas connaitre :

  • Position camera : Le pitch/yaw du joueur, pour verifier la coherence entre la direction regardee et le mouvement.
  • Velocite appliquee : La velocite calculee par le client (motionX, motionY, motionZ), que le serveur peut comparer avec ses propres calculs.
  • Etat des touches : Quelles touches de mouvement sont enfoncees (forward, back, left, right, jump, sneak), pour detecter les inconsistances (ex: mouvement vers l'avant sans touche forward = scaffold ou auto-walk).
  • Collision state : Si le client detecte une collision avec un bloc, pour verifier la coherence avec le noclip (le joueur traverse les blocs).

Frequence : Les donnees sont envoyees toutes les 10 ticks (0.5 secondes) via le channel chiffre. Le serveur correle les donnees client avec ses propres checks.

PacketTimingAnalyzer

Detecte le timer hack — la modification de la vitesse de jeu (Minecraft.timer) qui accelere ou ralentit le tick rate du client.

  • Mesure : Le client mesure le temps reel (System.nanoTime) entre chaque envoi de packet de mouvement (CPacketPlayer). En fonctionnement normal, l'intervalle est de ~50ms (20 ticks/seconde).
  • Detection : Si l'intervalle moyen sur 20 packets est < 40ms (timer > 1.25x) ou > 65ms (timer < 0.77x), c'est suspect. Le ratio reel/attendu est envoye au serveur.
  • Tolerance : Une tolerance de +/- 10ms est appliquee pour compenser les variations de framerate et les microlags.
  • Frequence : L'analyse est executee toutes les 2 secondes (40 ticks).

06 Couche 4 — Resource Integrity

Couche en Java pur, fonctionne sur tous les OS. Verifie que les ressources du jeu (textures, shaders, configs) n'ont pas ete modifiees pour obtenir un avantage (x-ray, fullbright, wallhack).

ResourcePackValidator

Detecte les resource packs x-ray qui rendent les blocs transparents pour voir les minerais a travers le terrain.

Textures surveillees

TextureRaison de la surveillance
stone.pngRendu transparent = voir les minerais dans la pierre
dirt.pngRendu transparent = voir les donjons souterrains
netherrack.pngRendu transparent = voir le quartz/ancient debris au Nether
cobblestone.pngRendu transparent = voir a travers les murs construits
sand.png / gravel.pngRendu transparent = voir les blocs de dessous
water_still.png / water_flow.pngRendu transparent = voir sous l'eau
lava_still.png / lava_flow.pngRendu transparent = voir sous la lave

Methode de detection :

  • Hash SHA-256 : Le serveur envoie les hash des textures vanilla attendues. Le client hash les textures reellement chargees et compare. Si le hash differe, le resource pack a modifie la texture.
  • Analyse de transparence : Pour chaque texture surveillee, le client analyse le canal alpha moyen des pixels. Si la transparence moyenne depasse 50% sur une texture censee etre opaque, c'est un x-ray pack.
  • Fullbright detection : Verifie si les textures ont ete modifiees pour emettre de la lumiere (fichier lightmap modifie ou shaders qui ignorent l'eclairage).

ShaderValidator

Detecte les shaders modifies pour wallhack ou ESP (Entity Shader Pack).

  • Shader files : Enumere les fichiers .vsh / .fsh charges. Verifie que seuls les shaders vanilla ou autorises (OptiFine, nos shaders custom) sont presents.
  • Wallhack patterns : Analyse le code GLSL pour detecter les patterns suspects : discard conditionnel sur le depth buffer (rendre les blocs proches invisibles), modification du gl_FragColor.a a 0 pour les blocs non-minerai.
  • ESP patterns : Detection de rendu d'entites a travers les murs (depth test desactive pour les entites, outline shader sur les joueurs).

ConfigValidator

Verifie que les fichiers de configuration critiques du mod n'ont pas ete modifies manuellement pour obtenir un avantage.

  • options.txt : Detecte les valeurs suspectes — gamma > 1.0 (fullbright), renderDistance anormalement eleve ou faible selon le contexte.
  • optionsof.txt (OptiFine) : Detecte les options x-ray potentielles — fast render + specific chunk loading patterns.
  • Configs du mod : Verifie l'integrite des fichiers config/EriniumFaction/ et config/EriniumAntiCheat/ contre les valeurs attendues signees par le serveur.
ⓘ
Gamma / brightness : La gamma 1.0 dans options.txt est le maximum vanilla. Les valeurs superieures (souvent 15.0 ou 1000.0) donnent un fullbright. Cependant, certains joueurs utilisent des ecrans sombres et augmentent la gamma. Le seuil SUSPICIOUS est fixe a gamma > 5.0, et CRITICAL a gamma > 100.0.

07 Communication Client-Serveur

Channel Forge

Le mod enregistre un channel Forge custom ef_anticheat pour toute communication entre le client et le serveur. Ce channel est distinct du channel principal d'EriniumFaction (ef_main).

Chiffrement AES-256-GCM

Toute communication sur le channel est chiffree avec AES-256-GCM (Galois/Counter Mode) pour empecher le sniffing et le spoofing des messages anti-cheat.

Etablissement de la cle de session : Client Serveur | | | 1. Login reussi (JWT + HWID valides) | |--------------------------------------------->| | | | 2. Serveur genere DH keypair ephemere | | Envoie DH public + nonce (signe JWT) | |<---------------------------------------------| | | | 3. Client genere DH keypair ephemere | | Calcule shared secret (DH) | | Derive AES-256 key (HKDF-SHA256) | | Envoie DH public | |--------------------------------------------->| | | | 4. Serveur calcule le meme shared secret | | Derive la meme AES-256 key | | | | 5. Communication chiffree AES-256-GCM | |<============================================>|

Details techniques :

  • Diffie-Hellman : Groupes DH 2048 bits ephemeres. Une nouvelle paire de cles est generee a chaque connexion. Forward secrecy : compromettre une session ne compromet pas les autres.
  • HKDF : Le shared secret DH est passe par HKDF-SHA256 avec le JWT token comme salt et "ef_anticheat_session" comme info pour deriver la cle AES-256.
  • GCM : Chaque message utilise un nonce unique (compteur 96 bits). Le tag d'authentification GCM (128 bits) garantit l'integrite — un message modifie en transit est rejete.
  • Rotation : La cle de session est re-derivee toutes les 30 minutes (via un nouveau DH exchange) pour limiter l'impact d'une compromission.

Challenge-Response

Le serveur envoie periodiquement des challenges au client pour prouver que l'anti-cheat tourne reellement et n'a pas ete desactive ou patche.

Type de challengeDescriptionReponse attendue
MEMORY_HASHDemande le hash SHA-256 de la DLL en memoireHash correspondant au hash connu du serveur
SCANNER_STATEDemande l'etat des scanners (actifs, derniere execution, nb violations)Structure JSON signee avec timestamp recent
SPECIFIC_SCANDemande un scan specifique (ex: "scan processus maintenant")Resultat du scan demande avec timestamp

RANDOM_COMPUTEEnvoie un seed et demande un calcul deterministe utilisant du code interne de l'ACResultat du calcul — prouve que le code AC est en memoire et executable

Timeout : Le client a 10 secondes pour repondre a un challenge. Passe ce delai, la violation est comptee comme un heartbeat manque.

Heartbeat

Mecanisme de verification continue. Le serveur envoie des heartbeats a intervalles aleatoires pour empecher le client de predire quand le prochain check aura lieu.

  • Intervalle : Aleatoire entre 30 et 120 secondes (configurable). L'aleatoire empeche un cheat de reactiver l'AC juste avant le heartbeat.
  • Contenu : Chaque heartbeat contient un micro-challenge (question aleatoire sur l'etat interne : hash d'un scanner, timestamp du dernier scan, etc.).
  • Timeout : Le client a N secondes (configurable, par defaut 15) pour repondre. Un timeout est compte comme un heartbeat manque.
  • Tolerance : 3 heartbeats manques consecutifs declenchent un kick immediat. Un seul heartbeat manque est logue mais tolere (lag reseau).

Rapport de scan

Le client envoie regulierement (toutes les 5 minutes) un rapport de scan agrege contenant :

  • Liste des processus : Noms des processus en cours (sans donnees personnelles — juste le nom executable).
  • DLLs chargees : Liste des DLLs dans le processus Java (noms et hash).
  • Classes suspectes : Packages/classes detectes qui ne sont pas dans le manifest.
  • Violations detectees : Toutes les violations depuis le dernier rapport, avec timestamp et details.
  • Metriques : CPS moyen, ecart-type des clics, checksum des classes critiques, etat des scanners.

12 Securite & Anti-Tamper

Verification de la DLL

La DLL native est elle-meme protegee contre la modification.

  • Hash au chargement : Avant System.load(), le mod calcule le hash SHA-256 du fichier eriniumanticheat.dll et le compare avec le hash attendu (inclus dans le manifest du launcher). Si le hash differe, la DLL n'est pas chargee et une violation CRITICAL est envoyee.
  • Hash en memoire : La DLL expose une methode getMemoryHash() qui retourne le SHA-256 de son propre code en memoire. Le serveur peut demander ce hash via challenge-response pour verifier que la DLL n'a pas ete patchee en memoire apres chargement.
  • Timestamp : La DLL verifie sa propre date de compilation. Si la date est trop ancienne par rapport a la version attendue, elle refuse de fonctionner (protege contre l'utilisation d'anciennes versions vulnérables).

Chiffrement des communications

  • Man-in-the-middle : Le DH exchange est lie au JWT du joueur (le JWT sert de salt). Un attaquant ne peut pas intercepter et modifier les messages sans connaitre le JWT.
  • Replay attacks : Chaque message chiffre contient un nonce (compteur) et un timestamp. Les messages dupliques ou rejoues sont rejetes.
  • Spoofing : Le tag GCM garantit l'authenticite. Un message forge (faux rapport de scan) sera rejete car le tag ne correspondra pas.

Heartbeat anti-desactivation

  • Intervalles aleatoires : Le joueur/cheat ne peut pas predire quand le prochain heartbeat arrivera. Desactiver l'AC "entre les heartbeats" est impossible car l'intervalle est inconnu.
  • Micro-challenges : Chaque heartbeat contient une question specifique qui necessite que le code de l'anti-cheat soit en memoire et fonctionnel pour repondre. Un emulateur qui repond "tout va bien" sans executer les vrais scans echouera sur les micro-challenges.
  • Cumul : Les heartbeats manques sont cumules. Meme un seul heartbeat manque est logue et peut contribuer au score de violation SUSPICIOUS.

Obfuscation (production)

En production, les classes Java d'EriniumAntiCheat sont obfusquees avec ProGuard (ou equivalent). L'objectif n'est pas d'empecher le reverse engineering (impossible a 100%), mais de le ralentir et d'augmenter le cout.

  • Classes renommees : Toutes les classes internes (scanners, handlers, etc.) sont renommees en noms courts (a, b, c...). Seules les classes d'interface publique (@Mod, events Forge) gardent leurs noms.
  • Strings chiffrees : Les chaines de caracteres sensibles (noms de processus blacklistes, patterns de detection) sont chiffrees au compile-time et dechiffrees au runtime.
  • Flow obfuscation : Les methodes critiques (hash, challenge-response) ont un control flow opaque — le flux d'execution est rendu non trivial a suivre dans un decompileur.

Challenge-Response : preuve d'execution

Le challenge RANDOM_COMPUTE est la protection la plus forte contre l'emulation. Le serveur envoie un seed aleatoire, et le client doit executer un calcul deterministe qui utilise du code interne de l'anti-cheat. Le resultat prouve que le code est en memoire et s'execute correctement.

// Exemple simplifie de challenge RANDOM_COMPUTE // Le vrai calcul est plus complexe et change entre versions // Serveur envoie : seed = 0x4A3F7C82 // Client calcule : long result = seed; result ^= AgentDetector.getLastScanTimestamp(); // Necessite que AgentDetector tourne result = Long.rotateLeft(result, 13); result ^= BytecodeVerifier.getCriticalHash(); // Necessite les checksums en memoire result = result * 0x5DEECE66DL + 0xBL; result ^= NativeBridge.getMemoryHash()[0]; // Necessite la DLL chargee // Client repond : result // Serveur verifie : meme calcul cote serveur avec les valeurs attendues
ⓘ
Mise a jour du challenge : Le calcul RANDOM_COMPUTE change entre les versions du mod. Chaque mise a jour du mod invalide les emulateurs existants car le calcul est different. C'est une course aux armements deliberee — chaque mise a jour force les tricheurs a refaire le reverse engineering.

Limitations conceptuelles

  • Course aux armements : L'anti-cheat client est fondamentalement une course aux armements. Un attaquant avec suffisamment de temps et de competences peut toujours contourner un anti-cheat userland. L'objectif n'est pas l'invulnerabilite — c'est de rendre le triche suffisamment couteux pour decourager la majorite.
  • Client-side trust : Toute donnee venant du client est potentiellement forgeable. C'est pourquoi le systeme est concu en defense en profondeur — 4 couches independantes, heartbeat continu, challenge-response. Un attaquant doit contourner TOUTES les couches simultanement.
  • Vie privee : Le scan de processus est le point le plus sensible. La whitelist exhaustive et le fait de ne remonter que les noms (pas les chemins complets, pas les arguments de ligne de commande) est un compromis delibere entre securite et vie privee.
  • Performance : Les scans doivent etre espaces pour ne pas impacter le gameplay. Un tricher qui active son cheat pendant les 60 secondes entre deux scans natifs peut techniquement l'utiliser brievement. C'est mitige par la couche 3 (behavior analysis) qui tourne en continu.

Evolutions futures

  • Port Linux : DLL native portee en .so pour la couche 2 sur Linux.
  • Machine Learning : Entrainer un modele sur les patterns d'input (clics, visee, mouvement) pour detecter les cheats inconnus par anomalie statistique plutot que par signature.
  • Screenshot on demand : Option opt-in pour envoyer un screenshot au serveur sur demande admin (avec consentement explicite du joueur).
  • Anti-cheat community : Partager les signatures de cheats entre serveurs utilisant EriniumAntiCheat (blacklist communautaire).
ⓘ
Philosophie : L'anti-cheat parfait n'existe pas. L'objectif est de rendre le cout du triche superieur au benefice. Avec 4 couches de detection, un heartbeat continu, des ban waves, et des mises a jour regulieres, la majorite des tricheurs seront detectes ou decourages. Les quelques-uns qui passent seront geres par le staff via les rapports et le dashboard admin.
