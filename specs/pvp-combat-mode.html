PvP — Mecanique Serveur

Mode PvP — Combat Tag

Systeme de "combat mode" qui verrouille le joueur dans le combat pendant 15 secondes apres avoir recu ou inflige des degats PvP. Bloque les teleportations de fuite et penalise la deconnexion en cours de combat (combat log).

Forge 1.12.2 PvP 1.8 (no cooldown) Server-side Anti-fuite 500-1000 joueurs EriAPI + OverlayMod

01Contexte & Objectifs

Problemes identifies

Sur un serveur faction PvP 1.8, deux comportements brisent l'equilibre du combat :

  • Fuite par teleportation — Un joueur sur le point de mourir utilise /home, /spawn, /tpa, /back, /f home ou /minage pour sortir instantanement du combat. Sa mort est evitee artificiellement.
  • Combat log — Un joueur ferme brutalement le jeu (Alt+F4, coupure reseau, crash) pour conserver ses items et eviter la mort. Sur un serveur sans protection, le personnage disparait simplement.

Ces comportements sont particulierement impactants en PvP 1.8 (jitter click, pas de cooldown) ou les combats sont courts et intenses — 15 secondes de lock est la norme du secteur.

Solution

Un "Combat Tag" : des que deux joueurs echangent des coups, les deux entrent en mode combat pour 15 secondes (reconfigurable). Ce delai se remet a zero a chaque nouveau coup. Pendant ce temps :

  • Toute commande de teleportation est bloquee avec un message d'erreur indicatif
  • Si le joueur se deconnecte, il est tue cote serveur et ses items sont deposes comme une mort normale (sans perk de retention ECO_MONOPOLY)
i
PvP 1.8 : L'attribut ATTACK_SPEED est inutile sur ce serveur. Aucune mecanique du combat tag ne se base sur l'attack speed. La duree de 15 secondes est calibree pour le rythme rapide du PvP 1.8 sans cooldown.

02Activation du Mode PvP

Conditions d'activation

Le mode combat s'active via l'evenement LivingHurtEvent (apres reduction des degats, donc apres armure). Les conditions exactes :

  • Source de degats — DamageSource.getTrueSource() instanceof EntityPlayerMP est vrai
  • Anti self-damage — La source et la cible sont des entites differentes (pas de self-inflict)
  • Les deux joueurs — L'attaquant ET la victime entrent tous les deux en mode combat
  • Reset du timer — Chaque nouveau coup PvP (dans n'importe quel sens) remet le compteur a 15 secondes pour les deux

Conditions de non-activation

  • Safe zones — Si le chunk de la victime ou de l'attaquant est une safe zone (faction claim safe, spawn, etc.), le mode combat ne s'active pas
  • Degats de mobs, chute, environnement — Uniquement les degats d'un autre joueur declenchent le combat tag
  • Sorts AOE sur allies — Les degats de type erinium_spell appliques a un allie de faction ne declenchent pas le mode combat. En revanche, un sort AOE qui touche un ennemi le declenche normalement
  • Duree zero — Si combatDurationSeconds est a 0 ou enabled est false dans la config, le systeme est desactive
!
Note implementation : Utiliser LivingHurtEvent (post-armor) plutot que LivingAttackEvent (pre-armor) pour eviter d'activer le combat tag sur des coups bloques completement par l'armure ou annules par d'autres mods.

03Restrictions en Mode Combat

Vue d'ensemble

Quand un joueur est en mode combat, les commandes de teleportation et toute action qui declencherait un warmup de teleportation sont bloquees. Le message affiche est en rouge : Vous etes en combat ! Attendez Xs.

BLOQUEES en mode combat
  • /spawn
  • /home, /sethome
  • /warp
  • /tpa, /tpahere, /tpaccept
  • /back
  • /rtp
  • /f home (teleport vers la base de faction)
  • /minage (teleport vers le monde minier)
  • Tout warmup WarmupManager
AUTORISEES en mode combat
  • /f info, /f list, /f chat (lecture seule)
  • /msg, /r (messages prives)
  • /money, /eris (lecture seule economie)
  • /ac, /servperf (commandes admin)
  • /tp admin (bypasse le combat tag — les admins ont toujours le contrôle)
✓
Implementation recommandee : Centraliser le check dans le WarmupManager. Avant de demarrer n'importe quel warmup, verifier CombatManager.isInCombat(player.getUniqueID()). Les commandes qui n'utilisent pas de warmup (comme /f home instantane) doivent faire leur propre check en debut de handler.

04Combat Log — Deconnexion en Combat

Definition

Un "combat log" se produit quand un joueur se deconnecte pendant qu'il est en mode combat, quelle qu'en soit la raison : bouton Quitter, Alt+F4, coupure reseau, crash client.

Comportement serveur

A la reception de PlayerEvent.PlayerLoggedOutEvent, si le joueur est en mode combat :

  • Tuer le joueur cote serveur — Appeler un handler de mort special qui bypasse les perks de retention (ECO_MONOPOLY)
  • Drop 100% des items — Le perk ECO_MONOPOLY (qui permet de garder 50% des items a la mort) est ignore. C'est une penalite intentionnelle pour la fuite.
  • Credit de kill — Le dernier attaquant (UUID stocke dans CombatData) recoit le credit de kill (compteur kills, XP, etc.)
  • Broadcast — Message diffuse a tous les joueurs en ligne : [Combat] NomDuJoueur a fui le combat et a ete tue.
  • Log admin — L'evenement est logue pour review admin (timestamp, joueur, dernier attaquant, coordonnees)
!
Securite : L'anti-cheat ne doit pas penaliser le joueur mort cote serveur pour des violations de mouvement apres sa mort. Exempter les combat-log kills des checks velocity/position de l'anti-cheat.

Cas particulier : mort normale pendant le combat

Si le joueur meurt normalement (PvP, chute, etc.) pendant le mode combat, le combat mode est efface et le flux de mort normal s'applique (ECO_MONOPOLY actif). Seule la deconnexion en combat bypass ECO_MONOPOLY.

Scenario ECO_MONOPOLY Kill credit Broadcast
Mort normale en combat (PvP) Actif Attaquant Non
Mort normale hors combat Actif N/A Non
Deconnexion en combat (combat log) Bypasse Dernier attaquant Oui
Deconnexion hors combat N/A N/A Non

05Overlay — Indicateur Combat

Description visuelle

Un petit overlay affiche l'etat du combat mode en temps reel. Il n'apparait que quand le joueur est en mode combat et disparait quand le timer expire.

  • Icone — Epees croisees (⚔) ou icone epee — symbole universellement reconnu pour le combat
  • Texte countdown — "15s", "14s", ..., "1s" — mis a jour chaque seconde
  • Couleur — Fond rouge semi-transparent (0xCC CC2200), texte blanc, bordure rouge vif
  • Position par defaut — Top center (configurable via l'editeur d'overlay)
  • Layer — POST_ALL (au-dessus de tout le reste du HUD)

Implementation

L'overlay est un OverlayMod EriAPI (extends OverlayMod, register via OverlayModManager). Il ne fait jamais de RenderGameOverlayEvent direct ni d'OpenGL brut.

Les donnees de combat (inCombat, secondsLeft) sont recues depuis le serveur via les champs @Sync de CombatPlayerData. Le client ne calcule pas lui-meme le countdown — il affiche ce que le serveur lui envoie chaque seconde via sync.

!
Regle absolue : Ne jamais utiliser RenderGameOverlayEvent avec OpenGL direct pour cet overlay. Toujours extends OverlayMod et OverlayModManager.register() conformement au systeme EriAPI.

06Architecture Technique

Package

Toutes les classes du systeme combat tag sont dans : fr.eriniumgroup.eriniumfaction.combat

CombatManager (singleton serveur)

public class CombatManager { private static CombatManager INSTANCE; // UUID → CombatData (lastHitTick, lastAttackerUUID, inCombat) private final Map<UUID, CombatData> combatMap = new HashMap<>(); public void enterCombat(EntityPlayerMP player, EntityPlayerMP attacker); public boolean isInCombat(UUID playerId); public int getRemainingSeconds(UUID playerId); public void clearCombat(UUID playerId); // appele a la mort normale public void onServerTick(); // toutes les 20 ticks : expire les timers + sync clients }

Performance : le onServerTick() ne s'execute que toutes les 20 ticks (1 seconde). Il itere uniquement les entrees actives dans combatMap, pas tous les joueurs du serveur. Les entrees expirees sont supprimees de la map.

CombatData (POJO interne)

public class CombatData { public long lastHitTick; // tick du dernier coup PvP public UUID lastAttackerUUID; // dernier attaquant (pour kill credit) }

CombatEventHandler

@Mod.EventBusSubscriber(modid = EriniumFaction.MODID) public class CombatEventHandler { @SubscribeEvent public static void onLivingHurt(LivingHurtEvent e) { // Si source = EntityPlayerMP ET cible = EntityPlayerMP ET source != cible // ET pas en safe zone → CombatManager.enterCombat(victim, attacker) } @SubscribeEvent public static void onPlayerLogout(PlayerEvent.PlayerLoggedOutEvent e) { // Si isInCombat(uuid) → executer le combat log kill } @SubscribeEvent public static void onPlayerDeath(LivingDeathEvent e) { // Si joueur meurt normalement → clearCombat(uuid) } }

CombatPlayerData (@EriData)

@EriData(modId = "eriniumfaction", scope = DataScope.PLAYER, persistOnDeath = false) public class CombatPlayerData { @Sync public boolean inCombat = false; @Sync public int combatSecondsLeft = 0; // Ces deux champs sont automatiquement synces au client via EriAPI // Le CombatOverlay lit ces valeurs cote client }

CombatOverlay (client-side)

public class CombatOverlay extends OverlayMod { // Lit CombatPlayerData.inCombat et combatSecondsLeft // Si inCombat = false → ne rend rien // Si inCombat = true → affiche icone + "Xs" en rouge // Layer : POST_ALL // Enregistrement : OverlayModManager.register(new CombatOverlay()) }

CombatConfig (@EriConfig)

@EriConfig(modid = EriniumFaction.MODID, filename = "eriniumfaction_combat") public class CombatConfig { public static boolean enabled = true; public static int combatDurationSeconds = 15; public static boolean killOnCombatLog = true; public static boolean bypassMonopolyOnCombatLog = true; public static boolean broadcastCombatLog = true; }

08Configuration

Fichier

Genere automatiquement par @EriConfig dans run/config/eriniumfaction_combat.cfg.

Cle Valeur par defaut Description
enabled true Active ou desactive completement le systeme combat tag
combatDurationSeconds 15 Duree du mode combat en secondes. Reinitialisee a chaque coup PvP.
killOnCombatLog true Tuer le joueur qui se deconnecte en combat. Si false → uniquement un avertissement dans les logs.
bypassMonopolyOnCombatLog true Ignorer le perk ECO_MONOPOLY lors d'un combat log (100% des items droppes).
broadcastCombatLog true Diffuser un message public quand un joueur fait un combat log.

09Messages / Lang

Cles de traduction

(15s) eriniumfaction.combat.expired=Vous n'etes plus en combat. # Commande bloquee eriniumfaction.combat.blocked=Vous etes en combat ! Attendez %ds. # Combat log eriniumfaction.combat.log=&c[Combat] %s a fui le combat et a ete tue. eriniumfaction.combat.log_broadcast=%s a quitte en plein combat et a ete tue !

Notes de style

  • Les messages d'entree en combat sont envoyes en rouge via EriChat
  • Le message "combat expired" est envoye en vert — le joueur est soulagé
  • Les messages de combat log utilisent le prefixe [Combat] en rouge pour etre immediatement identifies dans le chat
  • Le %d dans blocked est remplace par getRemainingSeconds()

Tests de performance (500-1000 joueurs)

  • 1000 joueurs en mode combat simultane — verifier que onServerTick ne depasse pas 0.1 ms/tick
  • 100 combats PvP simultanes — aucune fuite memoire dans combatMap
  • Expiration massive de timers au meme tick — pas de spike TPS
  • Sync CombatPlayerData a 1000 joueurs — debit packets acceptable
