Reconnaissance de Base

Outils technologiques pour detecter les bases ennemies non-claim : la Station de Recherche par tiers (Valterite / Aetherite / Erinium), le Bloc de Vision Souterraine, et plusieurs gadgets de reconnaissance complementaires.

Planifie Tier 1 Valterite Tier 2 Aetherite Tier 3 Erinium Scan async batche Alerte PvP live

01 Contexte & Objectifs

Probleme actuel

Sur un serveur PvP Faction, trouver les bases ennemies non-claim est une activite centrale du jeu. Actuellement, les joueurs doivent explorer manuellement des centaines de chunks — c'est long, frustrant, et favorise uniquement les joueurs qui passent le plus de temps en jeu plutot que ceux qui jouent intelligemment.

Le systeme de Reconnaissance introduit des outils technologiques a progression par tier pour detecter les bases cachees, avec un equilibrage fin entre precision, cout d'utilisation et risque PvP genere.

Objectifs de design

  • Activite de reconnaissance active : donner aux joueurs un moyen tactique et risque de chercher des bases non-claim, pas juste une solution automatique.
  • Tension PvP inherente : le scan alerte les joueurs a proximite en temps reel. Utiliser un scanner, c'est s'exposer. Le scanner doit etre protege.
  • Progression par tier : plus le tier est eleve, plus la precision est haute — mais le cout, le risque et l'engagement augmentent en proportion.
  • Faux rapports comme mecanique de jeu : les tiers 1 et 2 generent des coordonnees fictives melangees aux vraies. Le joueur ne peut pas distinguer le vrai du faux sans explorer physiquement. Cela preserve l'exploration comme element de jeu.
  • Performance 500-1000 joueurs : le scan est entierement asynchrone et batche. Il ne bloque jamais le thread principal. Maximum 3 scans simultanes globalement.
⚠
Contrainte serveur : Avec 500-1000 joueurs, plusieurs scans peuvent etre lances en meme temps. Le ScanManager limite les scans actifs a 3 globalement. Les scans supplementaires sont mis en queue et demarrent automatiquement quand une place se libere. Aucun scan ne bloque le tick serveur.

Ce que le systeme ne fait pas

  • Pas de detection des chunks claim : le scan ignore totalement les territoires de faction. Seules les bases non-claimees sont detectees.
  • Pas de coordonnee exacte pour les tiers bas : tier 1 et 2 donnent des coordonnees approximatives avec offset aleatoire. L'exploration physique reste necessaire.
  • Pas de chargement force des chunks : les chunks non charges sont scanes via lecture de fichiers region (.mca) sur disque — aucun chargement en memoire, aucune entite instanciee. Le mod ne force jamais le chargement permanent de chunks.
  • Pas de X-ray client : le Cave Block est un affichage decoratif local, pas un bypass du rendu vanilla.

02 Unclaim Finder Station

Principe general

Un bloc placable qui scanne les chunks non-claim dans son rayon pour detecter des concentrations de TileEntities (coffres, fours, dispensers, hoppers, etc.). Apres un temps de scan configurable par tier, il genere un rapport papier (item custom) contenant les coordonnees approximatives des bases detectees, melangees a des faux rapports selon le tier.

Regles fondamentales du bloc

  • Durabilite limitee : le bloc possede un nombre d'utilisations fixes selon le tier. Chaque scan reussi consomme 1 point de durabilite.
  • Durabilite persistante dans l'item : quand on casse le bloc, la durabilite restante est ecrite dans le NBT de l'item drop. Quand on le repose, la durabilite est restauree depuis le NBT. Un bloc use a 3 utilisations restantes garde ces 3 utilisations apres etre deplace.
  • Detection non-claim uniquement : avant d'analyser un chunk, le scanner verifie s'il est claim par une faction. Si oui, il est skip entierement. Les territories de faction sont invisibles au scanner.
  • Clustering avant rapport : le scanner ne donne pas une coordonnee par TileEntity. Il regroupe les TEs proches en clusters (voir Section 05) et ne rapporte qu'un point par cluster. Un entrepot de 30 coffres = 1 coordonnee dans le rapport.
  • Coordonnees imprecises par tier : un offset aleatoire est applique sur X et Z de chaque cluster avant d'etre ecrit dans le rapport. L'amplitude de l'offset depend du tier.
  • Alerte immediate au lancement : DES LE LANCEMENT du scan (pas a la fin), tous les joueurs dans le rayon de detection recoivent un message chat : "Une onde de recherche a traverse votre position." Cela cree de la tension : les joueurs proches savent qu'un scanner est actif.
  • Tier 3 — position revelee : pour le scanner Erinium, l'alerte envoyee aux joueurs inclut les coordonnees exactes du bloc scanner. Le joueur qui scanne doit donc defendre son bloc pendant 5 minutes. Si le bloc est detruit avant la fin du scan, le rapport est perdu et la durabilite est quand meme consommee.
⚠
Regles d'alerte strictes : L'alerte est envoyee au DEBUT du scan, pas a la fin. Pour le Tier 3 (Erinium), le message inclut les coordonnees exactes du scanner. Ces regles sont non-negociables — elles constituent l'equilibre PvP du systeme.

TileEntities considerees comme "indicateurs de base"

Le scanner ne detecte que ces types de TileEntities — les blocs utilitaires qui indiquent clairement une occupation humaine :

  • TileEntityChest — coffre simple et double
  • TileEntityFurnace — four standard
  • TileEntityDispenser — distributeur
  • TileEntityDropper — droppeur
  • TileEntityHopper — entonnoir
  • TileEntityBrewingStand — etabli de brassage
  • TileEntityShulkerBox — boite de shulker
  • TileEntityEnchantmentTable — table d'enchantement

Les blocs non-fonctionnels (portes, torches, escaliers) sont ignores. Un joueur qui pose uniquement des blocs decoratifs ne sera pas detecte.

04 Le Rapport (Unclaim Finder Station Paper)

Item custom : unclaim_report

  • Non-stackable (maxStackSize = 1) — chaque rapport est unique
  • Droppable et echangeable — le rapport peut etre vole, echange, donne. Il a une valeur marchande entre joueurs.
  • Stockage NBT complet : toutes les donnees sont dans le NBT de l'item, rien en memoire serveur apres generation

Structure NBT du rapport

// Structure NBT de l'item unclaim_report { "timestamp": 1234567890L, // System.currentTimeMillis() "scanner_x": 1200, // position du bloc scanner "scanner_y": 64, "scanner_z": -800, "tier": 1, // 1=Valterite, 2=Aetherite, 3=Erinium "scanner_player": "UUID...", // UUID du joueur qui a lance le scan "scanner_name": "PlayerName", "clusters": [ { "x": 942, "y": 58, "z": -1104, // coordonnees avec offset applique "density": 2, // 0=Faible, 1=Moyenne, 2=Forte "te_count": 18, // nombre de TEs detectees dans ce cluster "fake": 0 // 0=vrai, 1=faux — flag JAMAIS visible par le joueur }, // ... autres clusters (vrais + faux melanges aleatoirement) ] }

GUI du rapport (clic droit avec le papier en main)

  • EriGuiScreen — pas de Container, affichage pur client-side
  • Titre : "Rapport de Reconnaissance — [date formattee]"
  • Scanner par : nom du joueur + tier utilise
  • Pour chaque cluster : coordonnees X/Y/Z (avec offset), densite estimee (Faible / Moyenne / Forte), distance depuis la position du scanner
  • Aucune indication fake/vrai — tous les clusters sont affiches identiquement
  • Coordonnees cliquables : clic sur un cluster copie /tp X Y Z dans le chat (utile pour les admins). Pour les joueurs normaux, c'est juste une reference visuelle.
  • Style Liquid Glass — panneau translucide, backdrop-filter, coins arrondis, animations EriAPI
✓
Valeur du rapport : Le rapport est un item physique dans le monde — il peut etre vole dans les coffres d'une faction, vendu entre joueurs, ou decouvert sur un joueur tue. Un rapport Erinium (precision +/- 50 blocs, 0 faux) est une information de haute valeur tactique.

05 Algorithme de Scan

Vue d'ensemble — 5 etapes

Le scan est entierement asynchrone. Il ne bloque jamais le thread principal du serveur. La logique est distribuee sur plusieurs ticks via EriScheduler.repeat().

⚠
Serveur 500-1000 joueurs : plusieurs scans simultanes sont possibles. Les regles de performance ci-dessous sont non-negociables : pas de boucle bloquante, pas de chargement force de chunks, max 3 scans actifs globaux, batching obligatoire. Les lectures de fichiers region sont asynchrones (thread separe), les resultats sont mis en cache par region pour 5 minutes, et le nombre de lectures disque est limite a 4 par tick (configurable).

Etape 1 — Collecte (async/batch)

  • Le scan ne traite PAS tous les chunks d'un coup. Il est decoup en iterations : 4 chunks par tick (configurable, defaut 4).
  • Les chunks sont divises en anneaux concentriques depuis le scanner. Chaque iteration traite le prochain lot de chunks non-traites.
  • Pour chaque chunk dans le rayon :
    1. Verifier si le chunk est claim par une faction → si oui, skip
    2. Verifier si le chunk est charge en memoire :
      • Si charge : lire les TileEntities directement depuis la memoire via chunk.getTileEntityMap().values() (lecture instantanee, aucun I/O)
      • Si non charge : lire les donnees NBT du chunk depuis le fichier region (.mca) sur le disque, sans charger le chunk en memoire. Le fichier region est a world/region/r.X.Z.mca (chaque region contient 32×32 chunks). Lire le tag NBT Level.TileEntities avec RegionFileCache ou un RandomAccessFile direct. Extraire les positions et types des TEs. Cette operation est faite de facon asynchrone via EriScheduler.async() pour ne pas bloquer le thread serveur. Les donnees de region parsees sont mises en cache 5 minutes pour eviter de relire le meme fichier plusieurs fois pendant un scan.
    3. Filtrer : garder uniquement les types "indicateurs de base" (voir Section 02)
    4. Stocker les positions dans une liste temporaire locale au ScanTask
ⓘ
Pourquoi la lecture des fichiers region est sans impact sur les performances : Lire le NBT d'un chunk depuis un fichier .mca n'est pas un chargement de chunk — aucune entite n'est instanciee, aucun tick n'est enregistre, aucune memoire de chunk n'est allouee. C'est une lecture de donnees binaires brutes sur le disque, effectuee dans un thread separee via EriScheduler.async(). Le thread principal du serveur n'est jamais bloque. Le cache par region (5 min) garantit que le meme fichier n'est pas relu plusieurs fois pendant la duree d'un scan.

Etape 2 — Clustering

Regrouper les TileEntities collectees en clusters significatifs :

  • Algorithme greedy simple : pour chaque TE non-assignee, creer un nouveau cluster et y ajouter toutes les TEs a moins de 30 blocs.
  • Un cluster est valide uniquement si il contient au moins 3 TileEntities. Les fours isoles dans les grottes de survie sont ignores.
  • Pour chaque cluster valide : calculer la coordonnee centrale (moyenne X, Y, Z de toutes les TEs du cluster) + compter le nombre total de TEs (= densite).
  • Seuils de densite : 3-8 TEs = Faible, 9-20 TEs = Moyenne, 21+ TEs = Forte.

Etape 3 — Application de l'imprecision

Pour chaque cluster, appliquer un offset aleatoire a la coordonnee centrale avant de l'ecrire dans le rapport :

// precision = amplitude de l'offset (500, 200, ou 50 selon le tier) int offsetX = random.nextInt(precision * 2) - precision; // [-precision, +precision] int offsetY = random.nextInt(precision / 2) - precision / 4; // Y moins decale int offsetZ = random.nextInt(precision * 2) - precision; // Y est borne pour rester dans les limites du monde int reportedX = cluster.centerX + offsetX; int reportedY = Math.max(1, Math.min(255, cluster.centerY + offsetY)); int reportedZ = cluster.centerZ + offsetZ;

Etape 4 — Ajout des faux rapports

  • Generer N faux clusters (3 pour Tier 1, 2 pour Tier 2, 0 pour Tier 3).
  • Position aleatoire a l'interieur du rayon de scan (pas en dehors).
  • Densite aleatoire parmi les 3 niveaux (Faible / Moyenne / Forte).
  • Le flag fake = 1 est ecrit dans le NBT mais n'est jamais affiche au joueur dans le GUI.
  • Melange aleatoire : les vrais et faux clusters sont melanges dans la liste finale. L'ordre est aleatoire pour que le premier cluster ne soit pas systematiquement le vrai.

Etape 5 — Generation et livraison du rapport

  • Creer l'item unclaim_report avec le NBT complet.
  • Dropper l'item dans le monde au-dessus du bloc scanner (Y+1).
  • Consommer 1 point de durabilite du TileEntity.
  • Si durabilite = 0 apres consommation : briser le bloc, pas de drop (ou drop d'un item vide pour recup les materiaux — a definir).

ScanManager — limitation des scans simultanes

// ScanManager.java — Singleton public class ScanManager { private static final int MAX_CONCURRENT_SCANS = 3; private final List<ScanTask> activeScans = new ArrayList<>(); private final Queue<ScanTask> pendingQueue = new LinkedList<>(); public boolean tryStartScan(ScanTask task) { if (activeScans.size() < MAX_CONCURRENT_SCANS) { activeScans.add(task); task.start(); return true; } pendingQueue.add(task); // Notifier le joueur qu'il est en queue return false; } public void onScanComplete(ScanTask task) { activeScans.remove(task); if (!pendingQueue.isEmpty()) { ScanTask next = pendingQueue.poll(); activeScans.add(next); next.start(); } } public void cancelScansFor(UUID playerUUID) { // Appele quand un joueur quitte le serveur activeScans.removeIf(t -> t.getOwner().equals(playerUUID) && t.cancel()); pendingQueue.removeIf(t -> t.getOwner().equals(playerUUID)); } }

ScanTask — batching par tick

// ScanTask.java — une tache de scan public class ScanTask { private static final int CHUNKS_PER_TICK = 4; private final List<ChunkPos> chunksToScan; // pre-calcule au moment du start private int currentIndex = 0; private final List<BlockPos> collectedTEs = new ArrayList<>(); private ScheduledTask scheduledTask; public void start() { scheduledTask = EriScheduler.repeat(1, () -> { if (cancelled) { scheduledTask.cancel(); return; } int end = Math.min(currentIndex + CHUNKS_PER_TICK, chunksToScan.size()); for (int i = currentIndex; i < end; i++) { processChunk(chunksToScan.get(i)); } currentIndex = end; if (currentIndex >= chunksToScan.size()) { finalize(); scheduledTask.cancel(); } }); } private void processChunk(ChunkPos pos) { // 1. Skip si claim if (FactionClaimManager.isClaimed(pos)) return; if (world.isChunkLoaded(pos.x, pos.z, false)) { // 2a. Chunk charge en memoire — lecture directe, aucun I/O Chunk chunk = world.getChunk(pos.x, pos.z); for (TileEntity te : chunk.getTileEntityMap().values()) { if (isBaseIndicator(te)) collectedTEs.add(te.getPos()); } } else { // 2b. Chunk non charge — lire le NBT depuis le fichier region (.mca) sans charger le chunk // Calcul du fichier region : r.(chunkX>>5).(chunkZ>>5).mca // Lecture asynchrone via EriScheduler.async() pour ne pas bloquer le thread serveur // Cache par region (5 min) : RegionDataCache.get(regionX, regionZ) RegionDataCache.getAsync(world, pos, regionTEs -> { for (BlockPos tePos : regionTEs) { if (isBaseIndicatorByType(tePos, regionTEs)) collectedTEs.add(tePos); } }); // La lecture I/O est non-bloquante : le resultat est integre au collectedTEs quand disponible } } }

06 Cave Block (Bloc de Vision Souterraine)

Principe

Un bloc transparent special qui permet aux joueurs de visualiser les espaces vides souterrains sous leur position. Quand un joueur regarde a travers un Cave Block, il peut detecter des grottes, tunnels ou espaces amenages sous le sol.

ⓘ
Ce n'est pas du X-ray : le Cave Block ne bypasse pas le rendu vanilla. Il utilise une detection serveur (scan des 32 blocs sous lui) et emet des particules indicatrices cote client. Le joueur voit les particules, pas les blocs souterrains directement.

Comportement technique

  • Rendu : transparent comme du verre (BlockRenderLayer.TRANSLUCENT), texture distincte avec une teinte bleutee
  • Detection souterraine : le TileEntity scanne periodiquement les 32 blocs en dessous de lui (Y-1 a Y-32) pour detecter des espaces vides (blocs AIR ou semi-transparents)
  • Frequence de scan : une fois toutes les 5 secondes (100 ticks) pour ne pas peser sur les TPS avec de nombreux blocs places
  • Emission de particules : quand un espace vide est detecte, emettre des particules SPELL_WITCH (ou particule custom) depuis la face inferieure du bloc, avec une intensite proportionnelle au volume d'air detecte. Plus il y a de vide sous le bloc, plus les particules sont denses.
  • Pas de TileEntity persistant si aucun espace : si le sol est completement solide, le bloc passe en mode "inactif" et ne tick plus jusqu'a ce qu'un bloc en dessous soit retire

Craft

Craft du Cave Block : G G G G = Verre (Glass) G A G A = Cristal Aetherite G G G Resultat : 1x Cave Block

Limitations intentionnelles

  • Le Cave Block ne revele pas les coordonnees exactes des espaces — juste leur presence et approximativement leur volume
  • Il doit etre physiquement place au sol — pas utilisable en l'air
  • Les espaces vides artificiels (bases creusees) et naturels (grottes) sont traites identiquement — le joueur doit explorer pour determiner la nature du vide

07 Idees Additionnelles (Propositions)

Ces trois features sont des propositions — elles ne font pas partie du scope initial mais s'integrent naturellement dans le systeme de reconnaissance.

7.1 — Mine Proximity Detector (Detecteur de Proximite)

Un item portable (tenu en main) qui vibre/emet un son quand le joueur s'approche d'une zone avec beaucoup de TileEntities non-claim. Portee : 50 blocs. Ne donne PAS de coordonnees — juste une indication de proximite (son plus intense = plus proche). Consomme de la durabilite passif au fil du temps (1 point par minute d'utilisation). Tier unique : Aetherite. Durabilite : 60 minutes. Utile pour "pister" une base apres avoir recu un rapport imprecis.

7.2 — Scanner Portatif (Handheld Scanner)

Un item tenu en main secondaire. Affiche un overlay HUD via le systeme OverlayMod d'EriAPI avec une boussole qui pointe vers le cluster de TEs non-claim le plus proche dans un rayon de 100 blocs. Consomme 1 point de durabilite par minute active. Ne donne PAS de coordonnees exactes — seulement une direction. Tier Erinium. Durabilite : 3 heures d'utilisation. Ideal en phase de chasse apres avoir exploite un rapport d'Aetherite Scanner.

7.3 — Leurre de Base (Decoy Station)

Principe

Un bloc qui genere de fausses TileEntities fantomes dans un rayon de 250 blocs. Ces TEs fantomes apparaissent dans les scans de l'Unclaim Finder Station mais n'existent pas physiquement. Le joueur place le leurre pour tromper les scanners ennemis et les envoyer au mauvais endroit.

Fonctionnement technique — chunks decharges

Le leurre ecrit ses fausses TEs directement dans les fichiers region (.mca) au moment du placement. Ainsi, meme si le chunk est decharge, les fausses TEs sont detectees par le scan (qui lit les region files pour les chunks non-charges). Quand le leurre est casse ou expire, les fausses TEs sont nettoyees des fichiers region.

Restrictions anti-spam

  • Maximum 1 leurre actif par faction — impossible d'en placer un deuxieme tant que le premier existe.
  • Interdit dans les chunks claim — le leurre ne peut PAS etre place dans un chunk claim par quelque faction que ce soit (ni la sienne, ni allie, ni ennemi). Il doit etre en territoire sauvage/neutre uniquement.
  • Distance minimum de 200 blocs entre le leurre et le chunk claim le plus proche de la faction du joueur. Empeche de mettre le leurre juste a cote de sa propre base pour creer de la confusion.

Difficulte d'obtention

Craft tres cher :

E N E E = Fragment d'Erinium N S N N = Nether Star E N E S = Oeil d'Ender (Ender Eye) Resultat : 1x Decoy Station (usage unique)
  • Usage unique — l'item est consomme au placement (durabilite 1).
  • Expiration 24h en temps reel — pas en ticks de jeu, en temps reel (System.currentTimeMillis() stocke en NBT du TileEntity). Apres 24h le bloc se detruit automatiquement et les fausses TEs sont nettoyees.

Possibilite d'echec

  • 15% de chance de dysfonctionnement au placement — le leurre ne genere rien, l'item est quand meme consomme. Le joueur recoit un message : "Le leurre a dysfonctionné... Les composants sont perdus." Pas de remboursement.
  • Detection par les scanners — a chaque scan ennemi qui traverse la zone du leurre, le scanner a une chance de detecter que le signal est faux :
    • Tier 1 (Valterite) : 0% de detection — le leurre passe toujours pour vrai.
    • Tier 2 (Aetherite) : 30% de chance de detection — si detecte, le rapport marque ce cluster comme "Signal instable" (indice subtil, pas une certitude).
    • Tier 3 (Erinium) : 60% de chance de detection — meme marquage "Signal instable".
  • Le marquage "Signal instable" n'est PAS une confirmation que c'est un leurre — le joueur doit quand meme aller verifier. Mais c'est un indice.

Donnees du leurre

  • Nombre de fausses TEs generees : 8-15 (aleatoire) repartis dans le rayon de 250 blocs.
  • Types de fausses TEs : melanges (coffres, fours, dispensers) pour simuler une vraie base.
  • Les fausses TEs sont regroupees en 2-4 clusters pour simuler 2-4 fausses bases.

09 Architecture Technique

Structure du package

fr.eriniumgroup.eriniumfaction.recon/ ├── station/ │ ├── BlockFinderStation.java // Bloc 3 tiers (meta/property TIER_1/2/3), extends Block │ ├── TileFinderStation.java // TileEntity : durabilite, timer, lancement scan │ ├── ScanTask.java // Logique de scan batchee (N chunks/tick via EriScheduler) │ ├── ScanCluster.java // Data class : centerX/Y/Z, teCount, density, fake │ └── ScanManager.java // Singleton : gestion scans actifs (max 3), queue ├── report/ │ ├── ReportItem.java // Item custom unclaim_report, maxStack=1 │ ├── ReportData.java // Serialization/deserialization NBT du rapport │ └── GuiReport.java // EriGuiScreen pour afficher le contenu du rapport ├── cave/ │ ├── BlockCave.java // Bloc transparent, BlockRenderLayer.TRANSLUCENT │ ├── TileCave.java // TileEntity : scan 32 blocs sous, update 100 ticks │ └── CaveParticleHandler.java // Emission particules client-side ├── ReconItems.java // Registration items (unclaim_report, items optionnels) ├── ReconBlocks.java // Registration blocs (finder_station x3, cave_block) └── ReconRecipes.java // Recettes de craft (EriRecipe.shaped)

Points techniques critiques

  • ScanTask utilise EriScheduler.repeat(1, ...) : un callback par tick, traitant 4 chunks maximum par appel. Jamais de boucle bloquante.
  • Durabilite NBT bi-directionnelle : onBlockPlacedBy() lit le NBT de l'ItemStack et initialise la durabilite du TileEntity. onBlockHarvested() ecrit la durabilite restante dans le drop ItemStack. La durabilite ne se perd jamais lors d'un deplacement.
  • Faux rapports generes cote serveur uniquement : le client recoit l'item avec le NBT complet. Le flag fake est dans le NBT mais le GuiReport ne l'affiche jamais. Il est invisible pour le joueur, mais visible pour le code.
  • Chunks non-charges = lecture region async : si un chunk n'est pas en memoire, le ScanTask lit son NBT depuis le fichier region .mca via RegionDataCache.getAsync() sans jamais charger le chunk. Le cache region evite les relecures redondantes (TTL 5 minutes). Maximum 4 lectures region par tick (configurable). Le thread serveur n'est jamais bloque — la lecture I/O est deleguee a un thread separee via EriScheduler.async().
  • Annulation au deconnect : PlayerEvent.PlayerLoggedOutEvent appelle ScanManager.cancelScansFor(player.getUniqueID()). Aucun scan orphelin ne reste actif.
  • Annulation si bloc detruit : le TileEntity s'invalide (invalidate()) quand le bloc est casse. Le ScanTask verifie tile.isInvalid() a chaque iteration et s'annule si vrai.
  • BlockFinderStation — 3 tiers en un bloc : utiliser une PropertyEnum TIER (VALTERITE, AETHERITE, ERINIUM) stockee dans le blockstate. Evite de creer 3 classes de blocs separees. La logique de tier est delegue au TileEntity via TileFinderStation.getTier().
  • TileCave — early return agressif : si aucun espace vide detecte lors du dernier scan, le TileEntity passe en mode inactif et ne retick plus jusqu'a ce qu'un BlockUpdate adjacent le reveille.
