Système de Faction

Document de référence exhaustif pour l'implémentation du système de faction PvP, basé sur EriAPI. Toutes les données, APIs, commandes, GUIs et flux réseau sont décrits ici.

Minecraft 1.12.2 EriAPI Framework Forge 14.23.5.2864 Java 8 SpongePowered Mixin

01 Vue d'ensemble

Le système de faction d'EriniumFaction implémente le modèle classique PvP que tous les joueurs de serveurs Minecraft connaissent : création de groupes, revendication de territoire (claims), gestion des relations diplomatiques, et mécanique de power pour équilibrer l'expansion.

Objectif

Offrir une expérience faction familière et éprouvée, enrichie par des GUIs modernes via EriAPI tout en maintenant les commandes classiques /f pour les joueurs qui les préfèrent.

Piliers du système

  • Factions — groupes de joueurs avec hiérarchie (Leader, Officer, Member, Recruit)
  • Territory / Claims — revendication de chunks avec contrainte de power
  • Power — ressource par joueur qui se régénère et se perd à la mort, limite les claims
  • Diplomatie — relations ALLY / NEUTRAL / ENEMY entre factions
  • Home — point de téléportation sur territoire propre
  • Banque — trésorerie commune à la faction
  • Chat multi-canal — global, faction-only, et alliances
  • GUIs — tous les écrans sont construits avec EriAPI (EriGuiScreen, 1920×1080 design coords)
◆
Règle absolue : tout ce qui est couvert par EriAPI (GUI, Data, Events, Keys, Chat, Scheduler, Network) DOIT utiliser EriAPI. Aucun boilerplate Forge brut là où l'API existe.

04 FactionManager — API centrale

Singleton : FactionManager.getInstance() — classe : fr.eriniumgroup.eriniumfaction.faction.FactionManager

Toute la logique métier passe par ce manager. Les handlers d'events, commandes et GUIs l'appellent exclusivement.

Cycle de vie

public void init(World world)
Charge FactionWorldData depuis DataManager. Appelé dans FMLServerStartingEvent. Définit la référence de monde pour les saves.
public void save()
Persiste FactionWorldData via DataManager. Appelé à l'arrêt du serveur et toutes les 1200 ticks (auto-save).
World getWorld()
Retourne la référence au monde serveur courant. Utilisé pour accéder à FactionWorldData.

Création / Suppression

FactionInfo createFaction(EntityPlayerMP player, String name, String tag)
Crée une nouvelle faction. Retourne le FactionInfo créé ou null si validation échoue.
Validations : joueur sans faction, nom unique, tag unique, longueur et pattern respectés, solde suffisant (factionCreationCost). Effets : crée FactionInfo avec UUID, met à jour FactionPlayerData (factionId, factionRole=LEADER), envoie message EriChat, sauvegarde.
boolean disbandFaction(String factionId)
Dissout une faction. Retourne false si la faction n'existe pas.
Effets : supprime tous les claims de cette faction, reset FactionPlayerData de chaque membre (factionId="", factionRole=""), supprime toutes les relations impliquant cette faction, retire les invitations pendantes, sauvegarde.

Membres

boolean invitePlayer(String factionId, EntityPlayerMP inviter, EntityPlayerMP target)
Envoie une invitation. Validations : inviter est OFFICER+, target n'est pas déjà dans une faction, target non déjà invité, faction non pleine. Crée l'entrée dans invitations avec timestamp d'expiration. Envoie message cliquable EriChat au target.
boolean acceptInvitation(EntityPlayerMP player, String factionId)
Accepte une invitation. Vérifie que l'invitation existe et n'est pas expirée, que la faction n'est pas pleine. Met à jour FactionPlayerData, ajoute le joueur à FactionInfo avec le rôle defaultRole.
boolean declineInvitation(EntityPlayerMP player, String factionId)
Refuse une invitation. Supprime l'entrée dans invitations, notifie la faction via EriChat.
boolean joinFaction(EntityPlayerMP player, String factionId)
Rejoindre une faction ouverte (open=true). Mêmes validations qu'acceptInvitation. Impossible si la faction est fermée.
boolean leaveFaction(EntityPlayerMP player)
Le joueur quitte sa faction. Si le joueur est le seul membre : disbandFaction automatique. Si leader avec d'autres membres : refusé (doit d'abord transférer le leadership).
boolean kickPlayer(String factionId, EntityPlayerMP kicker, String targetUUID)
Exclut un membre. Validation : kicker a une priority strictement supérieure à la priority du target. Un OFFICER ne peut pas kick un autre OFFICER.
boolean promotePlayer(String factionId, String targetUUID)
Monte le rôle d'un joueur d'un cran : RECRUIT→MEMBER→OFFICER. Un OFFICER ne peut pas promouvoir au rang LEADER (utiliser transferLeadership).
boolean demotePlayer(String factionId, String targetUUID)
Descend le rôle d'un joueur d'un cran : OFFICER→MEMBER→RECRUIT. Impossible sur le LEADER.
boolean transferLeadership(String factionId, String newLeaderUUID)
Transfère le leadership. L'ancien leader devient OFFICER. Met à jour leaderUUID et leaderName dans FactionInfo, ainsi que les memberRoles des deux joueurs.

Claims

boolean claimChunk(EntityPlayerMP player)
Claim le chunk où se trouve le joueur.
Validations : joueur est OFFICER+, power total suffisant (getMaxClaims > getClaimCount), chunk non déjà claimé par une autre faction (sauf si overclaim activé et power insuffisant chez la cible), distance minimale entre factions respectée, non en zone protégée (spawn/warzone), maxClaimsPerFaction non atteint, claimOnlyConnected respecté si activé. Débite claimCost si configuré.
boolean unclaimChunk(EntityPlayerMP player)
Retire la revendication du chunk actuel. Le chunk doit appartenir à la faction du joueur. Rembourse unclaimRefundMultiplier * claimCost.
boolean unclaimAll(String factionId)
Retire tous les claims d'une faction d'un coup. Réservé au LEADER ou à un appel interne (disband).
boolean autoClaimToggle(EntityPlayerMP player)
Active/désactive le mode auto-claim : chaque chunk visité par le joueur est automatiquement claimé. État stocké temporairement côté serveur (non persisté).
String getFactionAt(int chunkX, int chunkZ, int dimension)
Retourne le factionId qui possède ce chunk, ou null si wilderness.
int getClaimCount(String factionId)  |  int getMaxClaims(String factionId)
getClaimCount : compte les claims depuis FactionWorldData. getMaxClaims : Math.min(maxClaimsPerFaction, getTotalPower(factionId) * claimsPerPower).
boolean isOverclaimed(String factionId)
True si getClaimCount > getMaxClaims (faction vulnérable à l'overclaim).

Power

float getPlayerPower(EntityPlayerMP player)
Calcule le power actuel avec régénération lazy : power += min((now - lastPowerUpdateTime) / 60000 * powerRegenPerMinute, maxPower - power). Met à jour lastPowerUpdateTime et persiste. Retourne la valeur clamped dans [0, maxPower].
float getTotalPower(String factionId)
Somme du power de tous les membres (calculé via getPlayerPower sur les joueurs en ligne, valeur stockée pour les hors-ligne).
void onPlayerDeath(EntityPlayerMP player)
Soustrait powerLossOnDeath au power du joueur (min 0). Met à jour totalDeaths et lastDeathTime. Envoie notification au joueur.
void updatePlayerPower(EntityPlayerMP player)
Force le recalcul et la sauvegarde du power. Utilisé au login pour intégrer la régénération hors-ligne (si powerRegenOffline=true).

Relations diplomatiques

boolean setRelation(String factionId, String otherFactionId, RelationType type)
Définit une relation entre deux factions.
Validations : les deux factions existent, pas soi-même, maxAlliances respecté pour ALLY. L'alliance est un double-opt-in : elle n'est effective que quand les deux factions ont déclaré ALLY à l'autre. La clé dans relations est construite avec les deux IDs triés alphabétiquement.
RelationType getRelation(String f1, String f2)  |  List<FactionInfo> getAllies(String factionId)  |  List<FactionInfo> getEnemies(String factionId)
Lecture des relations. getRelation retourne NEUTRAL si aucune entrée dans la map.

Home

boolean setHome(String factionId, EntityPlayerMP player)
Définit le home de la faction aux coordonnées actuelles du joueur. Validation : le chunk doit être claimé par la faction.
boolean teleportHome(EntityPlayerMP player)
Démarre le compte à rebours de TP (teleportDelay secondes) via EriScheduler. Si teleportCancelOnMove ou teleportCancelOnDamage, écoute les events correspondants et annule via task.cancel().

Lookups & utilitaires

FactionInfo getFaction(String factionId)
FactionInfo getFactionByName(String name)
FactionInfo getFactionByTag(String tag)
FactionInfo getPlayerFaction(EntityPlayerMP player)
FactionInfo getPlayerFaction(String playerUUID)
List<FactionInfo> getAllFactions()
FactionRole getPlayerRole(String playerUUID)
boolean hasFaction(EntityPlayerMP player)
boolean isLeader(EntityPlayerMP player)
boolean isOfficerOrAbove(EntityPlayerMP player)
Méthodes de lecture et vérification. Retournent null si non trouvé. La sérialisation/désérialisation de FactionInfo depuis le JSON est transparente.

Invitations

boolean hasInvitation(String playerUUID, String factionId)
List<FactionInfo> getPendingInvitations(String playerUUID)
void cleanExpiredInvitations()
cleanExpiredInvitations est appelée dans le tick auto-save (toutes les 1200 ticks). Supprime les entrées dont le timestamp est dépassé.

Chat

void setChatMode(EntityPlayerMP player, ChatMode mode)
ChatMode getChatMode(EntityPlayerMP player)
Modifie/lit le canal de chat actif. Persisté dans FactionPlayerData.chatMode.

07 Events Forge à écouter

Tous les listeners sont enregistrés via EriEvents ou @Mod.EventBusSubscriber sur les handlers dédiés.

Event Forge Handler Action
FMLServerStartingEvent FactionLifecycleHandler Appelle FactionManager.init(world), enregistre FactionCommand, démarre l'EriScheduler de nettoyage toutes les 1200 ticks.
FMLServerStoppingEvent FactionLifecycleHandler Appelle FactionManager.save() pour persister toutes les données.
PlayerLoggedInEvent FactionLifecycleHandler Appelle DataManager pour charger FactionPlayerData, updatePlayerPower(), met à jour memberNames dans FactionInfo avec le nom courant du joueur.
PlayerLoggedOutEvent FactionLifecycleHandler Sauvegarde les données du joueur. Annule le mode auto-claim s'il était actif.
LivingDeathEvent FactionPowerHandler Si la victime est un EntityPlayerMP : appelle onPlayerDeath(). Si l'auteur est aussi un EntityPlayerMP : incrémente totalKills.
ServerTickEvent (END) FactionLifecycleHandler Toutes les 1200 ticks : cleanExpiredInvitations() et FactionManager.save() (auto-save). Géré via EriScheduler.repeat().
BlockBreakEvent FactionProtectionHandler Vérifie getFactionAt(chunk). Si le chunk appartient à une faction différente, cancel l'event (protection territoire).
BlockPlaceEvent FactionProtectionHandler Même logique que BlockBreakEvent. Seuls les membres de la faction propriétaire peuvent placer.
PlayerInteractEvent.RightClickBlock FactionProtectionHandler Protège les interactions (coffres, portes, leviers, boutons) sur les chunks claimés. Cancel si joueur non autorisé.
LivingAttackEvent / AttackEntityEvent FactionPvPHandler Gère le friendly fire : si friendlyFireFaction=false, cancel les dégâts entre membres de la même faction. Si friendlyFireAllies=false, cancel entre alliances. Si pvpDeniedInOwnTerritory=true, cancel le PvP sur territoire propre.
ServerChatEvent FactionChatHandler Intercepte les messages. Si chatMode=FACTION, redirige le message aux membres en ligne via EriChat. Si chatMode=ALLY, le distribue aux membres de la faction ET des alliances. Préfixe le message avec le tag de faction si showFactionInChat=true.
PlayerChangedDimensionEvent FactionTerritoryHandler Réinitialise l'affichage de territoire après changement de dimension.
PlayerTickEvent (ServerTickEvent) ou PlayerMoveEvent FactionTerritoryHandler Vérifie périodiquement si le joueur a changé de chunk. Si oui : récupère la nouvelle faction, affiche un title/subtitle avec le nom de la faction (si territoryTitleEnabled). Gère aussi l'auto-claim si activé.

09 Flux de données réseau

Toute la communication GUI ↔ Serveur passe par GuiNetworkHandler d'EriAPI. Identifiant canal global : "faction_gui".

Enregistrement (CommonProxy.preInit)

// Côté serveur — traite les actions envoyées par les GUIs clients GuiNetworkHandler.registerActionHandler("faction_gui", (player, action, data) -> { switch (action) { case "create": FactionManager.getInstance().createFaction(player, data.getString("name"), data.getString("tag")); break; case "invite": // data = UUID cible break; case "claim": FactionManager.getInstance().claimChunk(player); break; case "unclaim": FactionManager.getInstance().unclaimChunk(player); break; case "promote": case "demote": case "kick": case "setRelation": case "allianceAccept": case "allianceDeny": // ... etc. } });

Enregistrement (ClientProxy.init)

// Côté client — associe un identifiant GUI à sa classe GuiNetworkHandler.registerGui("faction_gui", data -> new GuiFactionMain(data)); // Enregistrement de la touche d'ouverture EriKeys.create("eriniumfaction.open_menu") .key(Keyboard.KEY_F) .category("Erinium Faction") .context(KeyContext.IN_GAME) .onPress(() -> GuiNetworkHandler.requestOpenGui("faction_gui")) .register();

Actions Client → Serveur

CLIENT → SRV
sendAction("faction_gui", "create", nbt{name, tag}) — Créer une faction depuis le GUI
CLIENT → SRV
sendAction("faction_gui", "invite", nbt{targetName}) — Inviter un joueur
CLIENT → SRV
sendAction("faction_gui", "claim", nbt{chunkX, chunkZ, dim}) — Claim un chunk depuis la carte
CLIENT → SRV
sendAction("faction_gui", "promote", nbt{targetUUID}) — Promouvoir un membre
CLIENT → SRV
sendAction("faction_gui", "demote", nbt{targetUUID}) — Rétrograder un membre
CLIENT → SRV
sendAction("faction_gui", "kick", nbt{targetUUID}) — Kick un membre
CLIENT → SRV
sendAction("faction_gui", "setRelation", nbt{factionId, relationType}) — Changer une relation
CLIENT → SRV
sendAction("faction_gui", "allianceAccept", nbt{factionId}) — Accepter une demande d'alliance
CLIENT → SRV
sendAction("faction_gui", "join", nbt{factionId}) — Rejoindre une faction ouverte

Données Serveur → Client

SRV → CLIENT
GuiNetworkHandler.openGui(player, "faction_gui", nbt{hasFaction, factionJson}) — Ouvre le GUI avec données initiales
SRV → CLIENT
sendData(player, "faction_gui", "faction_info", nbt{factionJson}) — Rafraîchit les données affichées
SRV → CLIENT
sendData(player, "faction_gui", "map_data", nbt{chunksJson}) — Envoie les données de la carte (chunks autour du joueur)
SRV → CLIENT
sendData(player, "faction_gui", "action_result", nbt{success, message}) — Résultat d'une action (pour Notification)

Implémentation côté GUI

public class GuiFactionMain extends EriGuiScreen implements IGuiDataReceiver { @Override protected void buildGui() { // Construction avec les données initiales reçues à l'ouverture } @Override public void onGuiDataReceived(String key, NBTTagCompound data) { if ("faction_info".equals(key)) { refreshDisplay(data); // Mise à jour des labels, stats } else if ("action_result".equals(key)) { if (data.getBoolean("success")) { NotificationManager.success(data.getString("message")); } else { NotificationManager.error(data.getString("message")); } } } }
✓
Principe de base : le serveur est toujours la source de vérité. Les GUIs sont en lecture seule sauf pour envoyer des actions. Chaque action déclenche une validation serveur et un retour action_result. Les données affichées ne sont jamais modifiées localement avant confirmation serveur.
