Faction Banner System

Editeur pixel art in-game permettant a une faction niveau 15 de dessiner une banniere 16x24 affichee sur les capes des joueurs et sur les blocs banniere blancs en territoire revendique.

Milestone Niv.15 Mixin LayerCape Mixin TileEntityBannerRenderer 16x24 Pixels DynamicTexture Cache

01 Contexte & Objectifs

Pourquoi ce systeme ?

Le systeme de niveaux de faction (Faction Enhanced) recompense la progression collective par des deblocages progressifs. Atteindre le niveau 15 est un accomplissement significatif qui merite une recompense visuelle forte et persistante : une identite graphique propre a la faction, visible par tous les joueurs du serveur.

La banniere de faction remplit ce role. C'est un embleme cree pixel par pixel par les membres de la faction, affiche fierement sur leurs capes et sur les blocs du monde qu'ils controlent. Elle transforme le territoire en territoire identifie.

Milestone Niveau 15

  • Prerequis strict : la faction doit avoir atteint le niveau Enhanced 15 pour deverrouiller l'acces a l'editeur.
  • Avant le niveau 15, toutes les commandes /f banner retournent un message d'erreur indiquant le niveau requis.
  • Une fois deverrouille, la banniere persiste indefiniment meme si la faction repasse sous le niveau 15 (la banniere deja creee reste valide, seule la creation/modification est bloquee).

Objectifs de design

  • Identite visuelle — chaque faction a une signature graphique unique reconnaissable depuis n'importe quelle cape ou bloc.
  • Engagement collaboratif — l'editeur est accessible aux Leader et Officier, renforçant la gouvernance de faction.
  • Integration monde — les blocs banniere blancs en territoire deviennent des marqueurs territoriaux visuels sans modifier de donnees vanilla.
  • Opt-in joueur — le joueur choisit d'afficher ou non la banniere sur sa cape, preservant la liberte de personnalisation.
  • Performance zero-impact — texture generee une seule fois et mise en cache ; aucune generation par frame.

Outils disponibles

  • Pinceau pixel — clic gauche pose la couleur selectionnee sur le pixel survole. Clic gauche maintenu = peinture continue. Clic droit = gomme temporaire.
  • Remplissage (bucket) — flood-fill 4-connexe sur le pixel cible. Ne traverse pas les bordures de couleur differente.
  • Tout effacer — remet tous les pixels en transparent (G). Demande une confirmation via une popup EriAPI.
  • Grille ON/OFF — affiche/cache les separateurs de pixels pour faciliter le travail fin.

Restrictions d'acces

  • Seuls les Leader et Officer peuvent ouvrir et utiliser l'editeur.
  • Tentative d'un membre simple → message d'erreur EriChat : "Seuls les Officiers et le Leader peuvent modifier la banniere."
  • Faction sous niveau 15 → message : "La banniere se deverrouille au niveau 15 (niveau actuel : X)."

Flux de sauvegarde

  • Clic "Sauvegarder" → encode le tableau 16x24 en chaine 384 chars.
  • Envoie via GuiNetworkHandler.sendAction("banner_save", nbt) avec le champ bannerData.
  • Le serveur valide et persiste. Renvoie un acquittement "banner_saved" au client.
  • Reception → notification NotificationManager succes verte : "Banniere sauvegardee !"
  • En cas d'erreur serveur → notification rouge.

04 Format de Donnees

Chaine 384 caracteres

La banniere est stockee comme une chaine hexadecimale compacte de 384 caracteres (16 colonnes × 24 lignes = 384 pixels, 1 char par pixel).

Lecture : pixel a l'index i = y * 16 + x (y = ligne 0-23, x = colonne 0-15).

Stockage

  • FactionInfo.factionBanner (String, 384 chars) — deja declare comme placeholder dans FactionInfo, a remplir via DataManager.
  • FactionPlayerData.showBannerCape (boolean, @Sync) — nouveau champ, indique si le joueur accepte d'afficher la banniere sur sa cape. Valeur par defaut : true a la creation de la faction (ou quand la banniere est creee pour la premiere fois).

05 Rendu Cape

Mixin cible

La classe a intercepter est net.minecraft.client.renderer.entity.layers.LayerCape. La methode visee est doRenderLayer. On injecte a HEAD avec cancellable = true.

@Mixin(LayerCape.class) public class MixinLayerCape { @Inject(method = "doRenderLayer", at = @At("HEAD"), cancellable = true) private void onDoRenderLayer(AbstractClientPlayer player, float limbSwing, float limbSwingAmount, float partialTicks, float ageInTicks, float netHeadYaw, float headPitch, float scale, CallbackInfo ci) { // Delegue a BannerCapeCallback — AUCUN import mod ici if (BannerCapeCallback.shouldOverrideCape(player.getUniqueID())) { ResourceLocation tex = BannerCapeCallback.getCapeTexture(player.getUniqueID()); if (tex != null) { // Rendu de la cape avec la texture de faction // (copie du rendu vanilla en substituant la texture) BannerCapeCallback.renderCapeWithTexture(player, tex, limbSwing, limbSwingAmount, partialTicks, ageInTicks, netHeadYaw, headPitch, scale); ci.cancel(); // empeche le rendu vanilla de la cape } } } }

BannerCapeCallback

  • shouldOverrideCape(UUID) — retourne true si le joueur a une faction avec banniere ET que showBannerCape = true. Lit depuis une Map<UUID, Boolean> mise a jour par les packets @Sync.
  • getCapeTexture(UUID) — retourne la ResourceLocation de la texture DynamicTexture generee par BannerTextureCache.
  • renderCapeWithTexture(...) — recopie le rendu cape vanilla (bind texture, rendu du ModelCape) en substituant la texture.

Format texture cape (64x32)

Le format standard de cape Minecraft est 64x32 pixels. La banniere 16x24 est centree dans ce format.

Texture cape 64x32 : ┌────────────────────────────────────────────────────────────────┐ y=0 │ │ │ ┌────────────────┐ │ y=4 │ │ BANNIERE 16x24│ centree : x=24, y=4 │ │ │ │ │ │ │ │ │ │ └────────────────┘ │ y=28 │ │ └────────────────────────────────────────────────────────────────┘ y=32
  • Pixels de la banniere places a partir de x=24, y=4 dans la texture 64x32.
  • Le reste de la texture est transparent (0x00000000).
  • Les pixels "G" (transparent) de la banniere laissent voir la cape skin du joueur en dessous si elle existe, sinon restent transparents.

Toggle joueur

  • Commande /f banner cape — inverse FactionPlayerData.showBannerCape.
  • Le champ est @Sync → envoye automatiquement au client du joueur.
  • BannerCapeCallback ecoute les mises a jour @Sync et met a jour sa Map interne.
  • Si le joueur n'a pas de cape skin, la banniere s'affiche seule sur la cape invisible (comportement acceptable).

06 Rendu Bloc Banniere

Mixin cible

La classe a intercepter est net.minecraft.client.renderer.tileentity.TileEntityBannerRenderer, methode render. Injection a HEAD avec cancellable = true.

@Mixin(TileEntityBannerRenderer.class) public class MixinTileEntityBannerRenderer { @Inject(method = "render", at = @At(value = "INVOKE", target = "Lnet/minecraft/client/renderer/GlStateManager;color(FFFF)V", ordinal = 0), cancellable = true) private void onRender(TileEntityBanner te, double x, double y, double z, float partialTicks, int destroyStage, float alpha, CallbackInfo ci) { // Delegue a BannerBlockCallback — aucun import mod long pos = te.getPos().toLong(); String tex = BannerBlockCallback.getFactionTexturePath(pos); if (tex != null) { BannerBlockCallback.renderFactionBanner(te, x, y, z, partialTicks, tex); ci.cancel(); } } }
⚠
Alternative plus robuste : si l'injection INVOKE s'avere fragile, utiliser @At("HEAD") avec cancellable = true, en verifiant la position dans BannerBlockCallback avant de canceller. Preferer HEAD si possible per les conventions du projet.

BannerBlockCallback

  • getFactionTexturePath(long blockPos) — cherche si la position appartient a un territoire de faction ayant une banniere. Retourne le path de ResourceLocation ou null.
  • La Map <Long, String> (position → factionId) est mise a jour par le systeme de territoire a chaque claim/unclaim, via un packet client dedie.
  • renderFactionBanner(...) — rendu complet du modele de banniere avec la texture de faction substituee. Copie du rendu vanilla sans les patterns de teinture.

Detection banniere blanche sans pattern

  • La substitution ne s'applique que si le TileEntityBanner n'a aucun pattern (te.getPatterns() == null || te.getPatterns().isEmpty()).
  • La couleur de base doit etre blanche (ItemDye.DyeColor.WHITE).
  • Si le joueur a pose une banniere avec des motifs vanilla, la substitution ne s'applique pas — le rendu vanilla prend le relai.

Portee du rendu

  • Client-side uniquement — les donnees du bloc restent 100% vanilla.
  • Le rendu de faction s'affiche pour tous les joueurs dont le client a recu le packet de mapping territoire.
  • Texture bloc : format 64x64, la banniere 16x24 est repetiee ou centree selon le look desire (a definir lors de l'implementation).

Toutes les sous-commandes banner sont integrees dans le FactionCommand existant sous /f banner [sous-commande]. L'auto-completion est obligatoire pour toutes les sous-commandes.

Tableau des commandes

CommandePermissionDescriptionConditions

Comportement detaille

  • /f banner — appelle Minecraft.getMinecraft().displayGuiScreen(new GuiBannerEditor(factionData)) cote client. Si execut cote serveur, envoie packet openGui. Les donnees de banniere actuelles sont chargees dans l'editeur pour permettre la modification.
  • /f banner give — cree un ItemStack de banniere blanche et ajoute un tag NBT FactionBanner: true + FactionId: "xxx". Donne l'item au joueur. Ce tag est purement decoratif ; le bloc pose se comporte normalement, mais BannerBlockCallback detecte le NBT si besoin.
  • /f banner cape — lit FactionPlayerData.showBannerCape, l'inverse, sauvegarde, envoie confirmation EriChat : "Banniere sur cape : activee / desactivee."

Messages d'erreur

  • Joueur sans faction → "Vous n'etes pas dans une faction."
  • Faction niveau insuffisant → "La banniere de faction se deverrouille au niveau 15. (Niveau actuel : X)"
  • Rang insuffisant pour l'editeur → "Seuls les Officiers et le Leader peuvent modifier la banniere de faction."
  • Aucune banniere definie pour /f banner give → "Votre faction n'a pas encore de banniere. Utilisez /f banner pour en creer une."

08 Performance & Cache

BannerTextureCache — cycle de vie

La classe BannerTextureCache (client-side singleton) gere la generation et l'invalidation des DynamicTexture Minecraft.

public class BannerTextureCache { private static final Map<String, ResourceLocation> capeCache = new HashMap<>(); private static final Map<String, ResourceLocation> blockCache = new HashMap<>(); private static final Map<String, DynamicTexture> capeTextures = new HashMap<>(); private static final Map<String, DynamicTexture> blockTextures = new HashMap<>(); public static void invalidate(String factionId) { // Supprimer les DynamicTextures existantes via TextureManager deleteTexture(capeCache.remove(factionId)); deleteTexture(blockCache.remove(factionId)); capeTextures.remove(factionId); blockTextures.remove(factionId); } public static ResourceLocation getCapeTexture(String factionId, String bannerData) { if (!capeCache.containsKey(factionId)) { generateCapeTexture(factionId, bannerData); // genere UNE fois } return capeCache.get(factionId); } // idem pour getBlockTexture(...) }

Regles de performance

  • Generation unique — la texture est generee une seule fois par faction, lors du premier rendu apres reception des donnees. Jamais en doRenderLayer directement.
  • Invalidation sur update — a chaque fois que le serveur diffuse un "banner_update", le client appelle BannerTextureCache.invalidate(factionId). La prochaine frame regenere la texture.
  • Cleanup a la deconnexion — lors de l'evenement FMLNetworkEvent.ClientDisconnectionFromServerEvent, appeler BannerTextureCache.clearAll() pour liberer toutes les DynamicTextures enregistrees dans le TextureManager.
  • Pas de generation async — les DynamicTextures Minecraft doivent etre creees sur le thread principal OpenGL. Pas de threading pour cette operation.
  • Limitation — maximum 1 texture cape (64x32) + 1 texture bloc (64x64) par faction active. En pratique: 50 factions max = 100 DynamicTextures, negligeable en VRAM.

Generation de texture (algorithme)

// Generer la texture cape 64x32 depuis la chaine 384 chars int[] pixels = new int[64 * 32]; // tous transparents for (int py = 0; py < 24; py++) { for (int px = 0; px < 16; px++) { char c = bannerData.charAt(py * 16 + px); int argb = BannerColorMap.toARGB(c); // 0x00000000 si 'G' pixels[(4 + py) * 64 + (24 + px)] = argb; // offset x=24, y=4 } } DynamicTexture dt = new DynamicTexture(64, 32); System.arraycopy(pixels, 0, dt.getTextureData(), 0, pixels.length); dt.updateDynamicTexture(); ResourceLocation rl = Minecraft.getMinecraft().getTextureManager() .getDynamicTextureLocation("faction_cape_" + factionId, dt);

09 Integrations

FactionEnhancedManager — gate niveau 15

  • FactionEnhancedManager.getLevel(factionId) est appele dans BannerCommand avant toute action.
  • Si le niveau est inferieur a 15, la commande est rejetee avec un message explicatif.
  • Le level check est serveur-side uniquement ; le GUI peut etre ouvert cote client mais le Save sera refuse par le serveur.

FactionInfo — champ banniere

  • Le champ factionBanner existe deja dans FactionInfo comme placeholder. L'implantation le remplira avec la chaine 384 chars.
  • Le champ est @Sync pour que les clients membres recoivent automatiquement la banniere lors de la connexion et lors des mises a jour.

FactionPlayerData — cape toggle

  • Nouveau champ : @Sync public boolean showBannerCape = true; — synced au client du joueur, persistOnDeath = true preserve la preference.
  • Ce champ est lu par BannerCapeCallback via la Map UUID → boolean.

Systeme de territoire — bloc banniere

  • Chaque fois qu'un chunk est claim ou unclaim, le systeme de territoire envoie un packet de mise a jour au client listant les positions de blocs banniere dans le chunk.
  • BannerBlockCallback maintient une Map<Long, String> (blockPos.toLong() → factionId) mise a jour par ces packets.
  • Alternativement : verifier a la volee depuis le TileEntity si sa position est dans un chunk claim (depuis une map statique de claims synchronisee). Choix d'implementation laisse au developpeur.

Anti-exploit

  • La validation de permission (Leader/Officer + niveau 15) se fait exclusivement cote serveur dans BannerNetworkHandler.
  • Un joueur ne peut pas modifier la banniere d'une autre faction, meme avec un client modifie.
  • Le rendu bloc ne s'applique qu'aux blocs en territoire revendique par la faction ; hors territoire, le rendu est vanilla.
  • La texture n'est jamais generee si la chaine banniere est vide ou invalide (protection contre injection de donnees corrompues).

10 Plan de Tests

Tests editeur GUI

  • Ouvrir l'editeur en tant que Leader d'une faction niveau 15 → succes.
  • Ouvrir l'editeur en tant que membre simple → message d'erreur.
  • Ouvrir l'editeur avec faction niveau 14 → message niveau insuffisant.
  • Dessiner pixel par pixel avec chaque couleur de la palette → verifier mapping ARGB.
  • Outil remplissage sur zone monochrome → flood-fill correct.
  • Outil remplissage sur zone multi-couleur → n'affecte que la couleur cible.
  • Tout effacer → confirmation demandee, canvas redevient transparent.
  • Toggle grille → affichage/masquage correct.
  • Sauvegarde → chaine 384 chars correctement encodee et recue par le serveur.
  • Chargement d'une banniere existante → pixels correctement decodes et affiches.
  • Apercu en temps reel → le panneau de droite reflate les modifications immediates.

Tests format de donnees

  • Chaine avec caracteres invalides → rejetee par la validation regex serveur.
  • Chaine de longueur incorrecte → rejetee.
  • Chaine pleine de 'G' → banner consideree comme vide / non affichee.
  • Round-trip : encode → decode → reencode → chaine identique.

Tests rendu cape

  • Joueur avec banniere de faction + showBannerCape = true → cape de faction affichee.
  • Joueur avec banniere de faction + showBannerCape = false → cape vanilla (ou absente).
  • /f banner cape → toggle correct, persistance apres relog.
  • Joueur sans cape skin → banniere seule visible sans artefact.
  • Joueur avec cape skin → banniere superposee a la cape (ou substitution selon implementation).
  • Modification de la banniere → invalidation du cache → nouvelle texture visible sans relog.
  • Deconnexion → clearAll() appele, pas de DynamicTexture residuelle.
  • Rendu a differentes resolutions (720p, 1080p, 4K) → pas de distorsion.

Tests rendu bloc banniere

  • Poser un bloc banniere blanche dans un territoire de faction avec banniere → rendu de faction.
  • Poser un bloc banniere blanche hors territoire → rendu vanilla (blanc uni).
  • Poser un bloc banniere blanche avec motifs vanilla dans territoire → rendu vanilla (motifs preserves).
  • Banniere de couleur non blanche dans territoire → rendu vanilla.
  • Modification de la banniere → blocs existants refletes immediatement.
  • Unclaim du territoire → blocs redeviennent vanilla.

Tests performance et cache

  • 50 factions avec bannieres differentes → 100 DynamicTextures max, FPS stable.
  • Appel doRenderLayer 60x/sec → aucune generation de texture par frame (cache hit).
  • Invalidation forcee → regeneration en 1 frame, pas de texture manquante (frame noire).
  • Stress test : modification rapide de banniere 10x en 10 secondes → pas de fuite memoire GPU.

Tests cas limites

  • Faction dissoute apres creation de banniere → textures nettoyees, capes redeviennent normales.
  • Joueur quittant une faction avec banniere → cape redevient normale au prochain login.
  • Serveur sans EriAPI → mod ne charge pas (dependance declaree).
  • Donnees banniere corrompues en NBT → validation serveur rejette, valeur precedente preservee.
