Systeme de Trade

Echange securise entre deux joueurs avec un vrai Container Forge anti-duplication, validation mutuelle avec reset anti-scam, et support multi-actifs (items, XP, Dollars, Eris).

Planifie Anti-Duplication Anti-Scam Multi-Actifs Forge Container Validation Mutuelle Mail Fallback

01 Contexte & Objectifs

Vision du systeme

Le trade est l'echange direct et securise entre deux joueurs. Contrairement au HDV (asynchrone), le trade est synchrone : les deux joueurs doivent etre en ligne, proches l'un de l'autre, et valider la transaction simultanement. Le systeme est concu avec une priorite absolue : zero duplication, zero scam.

Les joueurs peuvent echanger :

  • Items : n'importe quel ItemStack, via un vrai Container Forge (pas de GUI fictif).
  • Experience (XP) : points d'XP bruts (pas des niveaux).
  • Dollars ($) : monnaie principale du serveur.
  • Eris : monnaie premium du serveur.

Objectifs de design

  • Zero duplication : les items dans le trade sont dans un vrai inventaire Forge (Container). Ils ne sont JAMAIS dans deux endroits en meme temps. Pas de copie, pas de reference, pas de "ghost item".
  • Anti-scam : si l'un des deux joueurs modifie quoi que ce soit apres que l'autre ait valide, les DEUX validations sont reinitilisees. Le joueur scamme est force de relire le contenu avant de revalider.
  • Validations serveur : toute la logique de trade est cote serveur. Le client n'est qu'un affichage. Aucune action client ne peut creer un etat incoherent.
  • Timeout : un trade inactif expire apres un delai configurable (defaut: 2 minutes). Les items sont rendus.
  • Proximite : les deux joueurs doivent etre a moins de X blocs l'un de l'autre (configurable, defaut: 10 blocs).

Ce que le systeme ne fait pas

  • Pas de trade asynchrone : les deux joueurs doivent etre en ligne et proches. Pour l'asynchrone, il y a le HDV.
  • Pas de trade force : le joueur cible doit accepter la demande de trade.
  • Pas de trade avec soi-meme : un joueur ne peut pas se trader.
  • Pas de trade pendant le combat : si l'un des joueurs est en combat tag, le trade est refuse.

04 Forge Container — Anti-Duplication

ContainerTrade

Le ContainerTrade est un vrai Container Forge qui gere les inventaires des deux joueurs pendant le trade. C'est la piece maitresse de la securite anti-duplication.

public class ContainerTrade extends Container { // Slots 0-3 : Items offerts par joueur A (cote gauche du GUI) // Slots 4-7 : Items offerts par joueur B (cote droit du GUI) // Slots 8-43 : Inventaire du joueur actuel (36 slots standard) private final IInventory tradeInvA; // InventoryBasic(4 slots) pour A private final IInventory tradeInvB; // InventoryBasic(4 slots) pour B // Le joueur A peut interagir avec les slots 0-3 (ses offres) // Le joueur B peut interagir avec les slots 4-7 (ses offres) // Aucun joueur ne peut toucher les slots de l'autre @Override public ItemStack transferStackInSlot(EntityPlayer player, int index) { // Shift-click : transfert depuis inventaire vers ses propres slots trade // Et inversement } @Override public boolean canInteractWith(EntityPlayer player) { // Verifie que le joueur fait partie du trade // Verifie la proximite } @Override public void onContainerClosed(EntityPlayer player) { // Rendre les items de CE joueur a son inventaire // Si inventaire plein, drop au sol ou mail } }

Pourquoi un vrai Container

  • Atomicite Forge : quand un joueur place un item dans un slot du Container, l'item est RETIRE de son inventaire et PLACE dans le Container. Il n'existe qu'a un seul endroit a tout moment.
  • Slot locking : les slots de l'autre joueur sont en lecture seule. Le Container refuse les interactions sur les slots qui ne lui appartiennent pas.
  • onContainerClosed : si le GUI est ferme (deconnexion, Escape), Forge appelle automatiquement onContainerClosed qui rend les items. Pas de perte possible.
  • Synchro native : Forge synchronise automatiquement le contenu du Container entre client et serveur. Pas besoin de packets custom pour les items.
⚠
Deux Container instances : Chaque joueur ouvre le meme Container logique via IGuiHandler. Joueur A voit le Container avec ses slots editables (0-3) et les slots de B en lecture seule (4-7). Joueur B voit l'inverse. Le mapping des slots est gere dans ContainerTrade en fonction du joueur qui ouvre.

Gestion des 4 slots par joueur

  • 4 slots : volontairement limite a 4 slots (pas 27) pour simplifier le GUI et reduire les risques. Si un joueur veut echanger plus, il fait plusieurs trades.
  • Slot restrictions : aucune restriction sur le type d'item. Tout item Minecraft valide peut etre trade.
  • Stack size : respecte les stack sizes natifs des items (64 pour la plupart, 16 pour les oeufs, 1 pour les outils).

05 Interface Graphique

GUI du trade

+====================================================================+ | ECHANGE [Annuler] | |====================================================================| | | | VOS OFFRES OFFRES DE JoueurB | | --------------------------- --------------------------- | | [Dia x64] [Epee ] [ ] [Or x32 ] [Arc ] [ ] | | [ ] [ ] [ ] [ ] [ ] [ ] | | --------------------------- --------------------------- | | | | XP : [___500___] XP : 200 | | $ : [__1,000__] $ : 0 | | Eris : [____50___] Eris : 25 | | | | ------------------------------------------------------------- | | | | Votre validation : [ ] Non valide | | Validation de JoueurB : [X] Valide | | | | [ VALIDER L'ECHANGE ] | | | | ========================================================= | | [VOTRE INVENTAIRE — 36 slots] | | ========================================================= | +====================================================================+

Elements du GUI

  • Titre : "ECHANGE" avec le nom de l'autre joueur.
  • Cote gauche (vert) : vos offres. 4 slots de Container interactifs + champs XP/$/Eris (NumericField). Le joueur place ses items via drag&drop standard Forge.
  • Cote droit (rouge) : offres de l'autre joueur. 4 slots en lecture seule (le Container refuse les interactions). Affichage des offres monetaires non editables.
  • Champs monetaires : NumericField pour XP, Dollars, Eris. Le joueur saisit le montant qu'il offre. Max = ce qu'il possede.
  • Statut de validation : deux indicateurs montrant si chaque joueur a valide. Checkbox non cliquable, mise a jour en temps reel via packet.
  • Bouton Valider : quand le joueur clique, son indicateur passe a "Valide". Si les deux sont valides, un countdown de 3 secondes apparait avant l'execution.
  • Bouton Annuler : annule le trade immediatement. Les items sont rendus.
  • Inventaire : les 36 slots d'inventaire du joueur en bas (standard Forge Container).

Reset anti-scam

C'est la mecanique de securite cle du trade :

  • Toute modification = reset : si un joueur modifie quoi que ce soit (ajoute/retire un item, change un montant XP/$/Eris), les DEUX validations sont reinitalisees (confirmedA = false, confirmedB = false).
  • Notification visuelle : un flash rouge et un message "L'offre a ete modifiee — veuillez revalider" apparait pour les deux joueurs.
  • Countdown annule : si un countdown de 3 secondes etait en cours, il est annule.
  • Raison : empeche le scam classique ou un joueur retire un item a la derniere seconde apres que l'autre ait valide.
⚠
Synchronisation des offres monetaires : Les offres XP/$/Eris ne sont PAS dans le Container Forge (qui ne gere que les ItemStack). Elles sont synchronisees via des packets custom (trade_update_offer). Chaque modification d'un champ monetaire envoie un packet au serveur, qui met a jour la TradeSession et notifie l'autre joueur.

06 Machine a Etats

Flux complet du trade

1. /trade JoueurB | v [PENDING] ----JoueurB refuse ou timeout 30s----> [CANCELLED] | JoueurB accepte (/trade accept ou clic chat) | v [NEGOTIATING] ----Annulation ou timeout 2min----> [CANCELLED] | | Les deux joueurs cliquent Valider Items rendus | v [BOTH_CONFIRMED] ----Modification = retour a----> [NEGOTIATING] | Countdown 3 secondes termine | v [COMPLETING] | Execution atomique : - Items A -> inventaire B - Items B -> inventaire A - XP/$/Eris transferes | v [COMPLETED] --- Message de confirmation aux deux joueurs

Details des transitions

  • PENDING → NEGOTIATING : la cible accepte. Le serveur ouvre le ContainerTrade pour les deux joueurs via IGuiHandler.
  • NEGOTIATING → BOTH_CONFIRMED : les deux confirmedA et confirmedB sont true. Un countdown de 3 secondes commence (affiche dans le GUI).
  • BOTH_CONFIRMED → NEGOTIATING : n'importe quelle modification (Container change ou offre monetaire) reset les confirmations.
  • BOTH_CONFIRMED → COMPLETING : le countdown de 3 secondes se termine sans modification. L'execution commence.
  • COMPLETING → COMPLETED : l'execution atomique est terminee avec succes.
  • Tout etat → CANCELLED : annulation manuelle, timeout, deconnexion, distance trop grande, mort d'un joueur.

Countdown de 3 secondes

Le countdown sert a donner un dernier moment de reflexion aux deux joueurs. Pendant le countdown :

  • Un timer visuel est affiche (3... 2... 1...).
  • Toute modification annule le countdown et reset les validations.
  • Les joueurs peuvent encore annuler.
  • Implementation : EriScheduler.delay(60, ...) (3 secondes = 60 ticks) avec verification de l'etat avant execution.

10 Cas Limites & Securite

Anti-duplication

  • Container Forge : la base de toute la securite. Les items sont dans le Container, pas dans deux inventaires a la fois.
  • Pas de snapshot : le systeme ne prend JAMAIS de "copie" ou de "snapshot" des items. Il deplace les ItemStack reels.
  • Execution atomique : l'echange final (COMPLETING) deplace les items dans le meme tick. Si une erreur survient, les items restent dans le Container et sont rendus via onContainerClosed.
  • Lock de session : chaque joueur ne peut avoir qu'un seul trade actif. TradeManager refuse toute nouvelle demande si le joueur est deja en trade.

Anti-scam

  • Reset de validation : toute modification (item ou montant) reset les deux validations. Le scam "je retire un item apres que tu aies valide" est impossible.
  • Countdown : 3 secondes d'attente apres double validation. Donne le temps de voir la modification eventuelle.
  • Lecture seule : les offres de l'autre joueur sont toujours visibles en lecture seule. Pas de zone cachee.
  • Detection Container : le Container Forge appelle detectAndSendChanges() a chaque tick. Toute modification des slots est detectee cote serveur et declenche le reset.

Deconnexion

  • Pendant NEGOTIATING : le Container se ferme automatiquement (onContainerClosed). Les items sont rendus a l'inventaire du joueur. Si l'inventaire est plein, les items sont envoyes par mail (type TRADE).
  • Pendant BOTH_CONFIRMED : le countdown est annule. Meme comportement que NEGOTIATING.
  • Pendant COMPLETING : ce state dure un seul tick. Si par miracle le serveur crash pendant ce tick, les items sont dans le Container et seront perdus. C'est le seul cas de perte possible et il est extremement improbable.
  • Items de l'autre joueur : quand un joueur se deconnecte, SES items sont rendus a SON inventaire. Les items de l'AUTRE joueur sont rendus a l'AUTRE joueur.

Distance

  • Verification continue : a chaque tick pendant le trade, la distance entre les deux joueurs est verifiee. Si elle depasse maxDistance, le trade est annule.
  • Dimension : les deux joueurs doivent etre dans la meme dimension.
  • Monde different : un changement de dimension annule le trade.

Validation des offres monetaires

  • Verification au confirm : au moment ou les deux joueurs valident (pas au moment de la saisie), le serveur verifie que chaque joueur possede bien les XP/$/Eris offerts.
  • Verification a l'execution : re-verification au moment de l'execution (3 sec plus tard). Si un joueur a depense ses $ entre-temps (via une autre commande), le trade est annule.
  • Debit atomique : les montants sont debites PUIS credites dans le meme tick. Si le debit echoue, pas de credit.

Divers

  • Trade avec soi-meme : refuse avec un message d'erreur.
  • Combat tag : si allowInCombat = false, refuse avec un message "Vous etes en combat".
  • Demandes multiples : un joueur ne peut avoir qu'une demande sortante a la fois. Les anciennes expirent.
  • Toggle off : si la cible a desactive les demandes de trade, le demandeur recoit "Ce joueur n'accepte pas les trades".
  • Items blacklisted : si un joueur tente de placer un item blackliste dans le Container, le serveur le refuse et le rend.
⚠
Edge case XP : L'XP Minecraft est stockee en points d'experience (pas en niveaux). Un joueur peut offrir plus de points qu'il n'en a en niveaux apparents (car la courbe XP est non-lineaire). Le systeme doit toujours travailler en POINTS bruts, pas en niveaux. Verification : player.experienceTotal >= offeredXP.

11 Integration Mail

Fallback en cas d'inventaire plein

Si les items du trade ne peuvent pas etre rendus a l'inventaire d'un joueur (deconnexion + inventaire plein), ils sont envoyes par mail :

  • Type : MailSenderType.TRADE
  • Message : "Trade annule — vos items ont ete renvoyes"
  • Items : les ItemStack qui etaient dans les slots de trade du joueur
  • Argent : aucun montant n'est inclus dans le mail (les offres monetaires ne sont pas debitees si le trade est annule)

Cas d'utilisation

  • Deconnexion : le joueur se deconnecte avec des items dans le trade. onContainerClosed tente de les rendre. Si echec, mail.
  • Crash serveur : les items dans le Container sont perdus (pas de persistence du Container). Le systeme de mail ne peut pas couvrir ce cas. C'est un risque accepte (tres rare).
ⓘ
Priorite : inventaire > drop > mail. Le systeme tente d'abord de placer les items dans l'inventaire. Si impossible (plein + hors ligne), il envoie par mail. Le drop au sol n'est utilise que si le joueur est en ligne et que son inventaire est plein (cas ou onContainerClosed echoue et que le joueur est encore la).
🔧
Implementation — livraison hors ligne : utiliser MailManager.getInstance().sendMail(UUID, ...) pour envoyer les items et l'argent aux joueurs hors ligne. Le MailManager fonctionne par UUID et ne necessite pas que le joueur soit connecte.

12 Plan de Tests

Tests fonctionnels

  • Trade basique : A offre 32 diamants, B offre $5,000. Les deux valident. Verifier : A recoit $5,000, B recoit 32 diamants, inventaires corrects.
  • Trade multi-actifs : A offre 16 diamants + $500 + 200 XP. B offre 1 epee + 50 Eris. Verifier tous les transferts.
  • Annulation : A place des items, B place des items, A clique Annuler. Verifier que tous les items sont rendus.
  • Demande refusee : A demande un trade, B refuse. Verifier pas de GUI ouvert, message de refus.
  • Timeout demande : A demande un trade, attendre 30 secondes. Verifier l'expiration.
  • Timeout inactivite : ouvrir un trade, ne rien faire pendant 2 minutes. Verifier l'annulation.
  • Toggle off : B fait /trade toggle, A essaie /trade B. Verifier le refus.

Tests anti-scam

  • Reset au retrait : A valide. B valide. A retire un diamant. Verifier : les deux validations sont reset.
  • Reset a l'ajout : les deux valident. B ajoute un item. Verifier le reset.
  • Reset montant : les deux valident. A change son offre de $500 a $400. Verifier le reset.
  • Countdown annule : countdown en cours (3... 2...). A retire un item. Verifier : countdown annule, reset.
  • Scam impossible : A et B valident. Pendant le countdown, A retire un item. B est notifie et doit revalider. Le trade ne s'execute pas.

Tests anti-duplication

  • Item dans Container : A place un diamant dans le trade. Verifier que le diamant n'est plus dans son inventaire.
  • Double trade : A est en trade avec B. A essaie /trade C. Verifier le refus.
  • Packet forge — slot interdit : forger un packet pour interagir avec les slots de l'autre joueur. Verifier le refus.
  • Fermeture GUI : A place des items, ferme le GUI (Escape). Verifier que les items sont rendus.
  • Deconnexion : A place des items, se deconnecte. Verifier que les items sont dans son inventaire a la reconnexion (ou en mail si plein).
  • Execution atomique : trade reussi. Verifier que AUCUN item n'a ete duplique (compter le total avant/apres).

Tests monetaires

  • Solde insuffisant au confirm : A offre $10,000 mais n'a que $5,000. Les deux valident. Verifier l'erreur "Solde insuffisant".
  • Solde change entre confirm et execution : A offre $10,000 et a $10,000. Les deux valident. Pendant le countdown, A depense $5,000 via /shop. Verifier l'annulation du trade.
  • XP insuffisant : A offre 5000 XP mais n'en a que 3000. Verifier le refus.
  • Montant negatif : forger un packet avec dollars = -1000. Verifier le refus.
  • Montant > max : offrir $2,000,000 (max = 1,000,000). Verifier le clamp.

Tests de distance

  • Hors de portee : A est a 15 blocs de B (max = 10). /trade B. Verifier le refus.
  • S'eloigner : trade en cours, A marche au-dela de 10 blocs. Verifier l'annulation apres 1-2 secondes de delai.
  • Dimension differente : A teleporte dans le Nether pendant le trade. Verifier l'annulation.

Tests de performance

  • 100 trades simultanes : 200 joueurs en 100 trades paralleles. Verifier l'impact TPS (< 2ms/tick).
  • Verification de distance : 100 trades actifs, verification de distance chaque tick. Mesurer le cout.
