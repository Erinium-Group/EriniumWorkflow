Cahier des charges — Interface web du serveur EriniumFaction (Next.js / React / TailwindCSS)

1. Vue d'ensemble

Le site web EriniumFaction est la vitrine publique du serveur et le portail d'acces pour les joueurs. Il sert trois objectifs :

  • Acquisition — attirer de nouveaux joueurs via le SEO, le design attractif et les CTA clairs
  • Retention — donner aux joueurs connectes un tableau de bord riche (stats, achats, votes, securite)
  • Monetisation — boutique premium et boutique de votes integrees avec paiement multi-methode
Scope : Ce cahier des charges couvre UNIQUEMENT le frontend (Next.js). Le backend (API Node.js, base de donnees, paiements) fera l'objet d'un spec separe. Les endpoints API listes ici sont des contrats d'interface que le backend devra implementer.

3.2. Authentification cote frontend — Discord OAuth + TOTP optionnel

Principe : L'authentification repose exclusivement sur Discord OAuth2. Il n'y a pas de systeme email/mot de passe. Le joueur se connecte avec son compte Discord. Une double authentification optionnelle (TOTP) peut etre activee depuis le dashboard pour securiser le compte.

Flow de connexion :

  1. Le joueur clique sur "Se connecter avec Discord" (bouton dans le header ou sur une page protegee)
  2. Redirect vers https://discord.com/oauth2/authorize avec les scopes identify et email
  3. Le joueur autorise l'application sur Discord et est redirige vers /auth/callback?code=xxx
  4. Le frontend envoie le code au backend : POST /api/auth/discord/callback
  5. Le Worker Cloudflare echange le code contre un token Discord, recupere les infos utilisateur (id, username, email, avatar)
  6. Le Worker cree ou met a jour l'utilisateur dans D1, genere un JWT session et un refresh token, les stocke dans des cookies httpOnly
  7. Si l'utilisateur a active le TOTP : le backend retourne { requireTotp: true }. Le frontend redirige vers /auth/totp ou le joueur saisit son code TOTP a 6 chiffres
  8. Si pas de TOTP : connexion terminee, redirect vers le dashboard ou la page d'origine

Mecanismes de session :

  • JWT — stocke dans un cookie httpOnly, secure, sameSite=strict. Jamais dans localStorage (XSS vulnerable). Expire en 15 minutes
  • Refresh token — cookie separe, rotation a chaque usage. Duree de vie : 7 jours
  • Middleware Next.js — middleware.ts a la racine. Intercepte les routes protegees (/dashboard/*, /shop/checkout). Redirige vers la page de connexion Discord si pas de session valide
  • Refresh automatique — le client API intercepte les 401, tente un refresh via POST /api/auth/refresh. Si le refresh echoue, redirect vers Discord OAuth
  • CSRF — token CSRF dans les headers pour les mutations (POST/PUT/DELETE). Genere par le backend, stocke dans un cookie non-httpOnly lisible par JS

TOTP (double authentification optionnelle) :

  • Activable depuis /dashboard/security (voir section 5.4.2)
  • Utilise un algorithme TOTP standard (RFC 6238), compatible Google Authenticator, Authy, etc.
  • Si active : apres le retour Discord OAuth, une etape supplementaire demande le code TOTP avant d'emettre la session JWT complete
  • Le secret TOTP est stocke chiffre dans D1 (cle de chiffrement dans les variables d'environnement Cloudflare)

5.2.1. Bouton de connexion Discord (dans le Header et pages protegees)

Il n'y a pas de page /login dediee. Quand un utilisateur non connecte tente d'acceder a une page protegee (ou clique sur le bouton du header), il est redirige directement vers Discord OAuth.

Bouton "Se connecter avec Discord" :

  • Fond #5865F2 (Discord blurple), coins arrondis 8px, icone SVG Discord a gauche du texte
  • Hover : brightness +10%, transition 150ms
  • Press : scale 0.98
  • Au clic : appel GET /api/auth/discord/authorize qui retourne l'URL de redirection Discord. Le frontend redirige le navigateur vers cette URL
  • Le query param redirect est sauvegarde en sessionStorage avant la redirection vers Discord pour pouvoir y revenir apres connexion (ex: /shop/checkout)

Flow complet :

  1. Le joueur clique "Se connecter avec Discord"
  2. Redirect vers https://discord.com/oauth2/authorize?client_id=XXX&redirect_uri=SITE/auth/callback&response_type=code&scope=identify+email
  3. Le joueur autorise l'application sur Discord
  4. Discord redirige vers /auth/callback?code=xxx
  5. La page callback envoie le code au backend : POST /api/auth/discord/callback
  6. Le backend echange le code contre un token Discord, recupere les infos utilisateur, cree/met a jour le compte dans D1, genere JWT + refresh token en cookies
  7. Si TOTP active : le backend retourne { requireTotp: true, tempToken: "xxx" }. Le frontend redirige vers /auth/totp
  8. Si pas de TOTP : le backend retourne { success: true } + Set-Cookie. Le frontend redirige vers le dashboard ou la page d'origine sauvegardee dans sessionStorage

5.2.2. Callback Discord OAuth (/auth/callback)

Page technique, pas visible longtemps par l'utilisateur. Elle :

  1. Recupere le parametre code dans l'URL
  2. Affiche un loader : logo + "Connexion en cours..." + spinner
  3. Envoie POST /api/auth/discord/callback avec le code
  4. Si succes sans TOTP : redirect vers dashboard ou page d'origine
  5. Si TOTP requis : redirect vers /auth/totp avec le tempToken en state
  6. Si erreur (code invalide, expire, banni) : affiche un message d'erreur avec un bouton "Reessayer" qui relance le flow Discord OAuth

API requise (callback)

7. SEO et performance

7.1. SEO technique

  • Sitemap XML — genere dynamiquement par Next.js (app/sitemap.ts). Inclut : homepage, boutique, vote, classement, news, articles individuels. Exclut : dashboard, auth, legal
  • robots.txt — autorise tout sauf /dashboard/, /api/, /auth/
  • Canonical URLs — sur chaque page pour eviter le contenu duplique (FR vs EN)
  • Structured Data — Schema.org JSON-LD sur les pages pertinentes (WebSite, Organization, Article, ItemList, Product)
  • Hreflang — <link rel="alternate" hreflang="fr" href="..." /> et hreflang="en" sur toutes les pages
  • Meta tags — title unique par page (max 60 chars), description unique (max 155 chars), Open Graph (og:title, og:description, og:image), Twitter Card

7.2. Performance

  • Target Core Web Vitals :
    MetriqueObjectif
    LCP (Largest Contentful Paint)< 2.5s
    FID (First Input Delay)< 100ms
    CLS (Cumulative Layout Shift)< 0.1
  • Images — format WebP, next/image pour l'optimisation automatique (redimensionnement, lazy loading, srcset). Textures MC : PNG avec image-rendering: pixelated
  • Fonts — next/font pour Inter (self-hosted, no CLS). Subset latin uniquement
  • Bundle — tree shaking, dynamic imports pour les pages lourdes (graphiques, carrousel). Objectif : < 200KB JS initial (gzip)
  • Cache — assets statiques : Cache-Control: public, max-age=31536000, immutable. Pages ISR : s-maxage=60, stale-while-revalidate
  • Preloading — preload des fonts critiques, preconnect vers les APIs externes (mc-heads.net, Stripe)

7.3. Accessibilite (WCAG 2.1 AA)

  • Navigation clavier — tous les elements interactifs sont focusables avec Tab, actionnables avec Enter/Space. Focus visible (ring gradient)
  • Contraste — ratio minimum 4.5:1 pour le texte normal, 3:1 pour le texte large. Verifier avec le fond sombre
  • Screen readers — semantic HTML (nav, main, article, section, aside). ARIA labels sur les icones sans texte, les boutons d'action, les modales
  • Formulaires — labels explicites (pas de placeholder-only), messages d'erreur associes via aria-describedby, aria-invalid sur les champs en erreur
  • Animations — respecter prefers-reduced-motion : desactiver les particules, reduire les transitions a des fades simples
  • Images — alt text descriptif sur toutes les images informatives. Images decoratives : alt=""
  • Modales — focus trap, fermeture avec Escape, retour du focus a l'element declencheur

8. Securite frontend

  • XSS — React echappe par defaut. Ne JAMAIS utiliser dangerouslySetInnerHTML sauf pour le contenu d'articles (sanitize avec DOMPurify avant)
  • CSRF — token CSRF dans les headers de toutes les requetes mutatives
  • Auth cookies — httpOnly, secure, sameSite=strict. Pas de JWT dans localStorage
  • Content Security Policy — headers CSP stricts : default-src 'self', autoriser uniquement les sources connues (Stripe JS, mc-heads.net, Cloudflare Turnstile)
  • Rate limiting — cote backend, mais le frontend doit aussi desactiver les boutons pendant les requetes pour eviter le spam
  • Input validation — double validation : client-side (Zod, UX immediate) + server-side (backend, securite reelle). Ne JAMAIS faire confiance au client seul
  • Sensitive data — ne JAMAIS afficher en clair : HWID complet (masquer partiellement), IP complete (masquer le dernier octet), token JWT, email Discord complet (masquer partiellement), secret TOTP
  • Dependencies — npm audit dans le CI, dependabot active pour les mises a jour de securite

9. Monitoring et analytics

  • Error tracking — Sentry (free tier) pour capturer les erreurs JS client et les erreurs API. Source maps uploadees en CI
  • Analytics — Plausible Analytics (self-hosted ou cloud, RGPD-friendly, pas de cookies). Metrics : pages vues, visiteurs uniques, sources de trafic, pays. PAS de Google Analytics (RGPD problematique)
  • Uptime monitoring — UptimeRobot ou BetterStack pour surveiller la disponibilite du site et de l'API
  • Performance — Vercel Analytics (si heberge sur Vercel) ou Web Vitals reporting custom
