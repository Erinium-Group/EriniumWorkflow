Framework d'Animation de Blocks 3D

Pipeline complet pour animer des blocks 3D dans Minecraft Forge 1.12.2 : import de modeles Blockbench, edition d'animations dans un editeur Python dedie, et rendu temps reel via TESR. Concu comme module EriAPI reutilisable par tout mod.

EriAPI TESR Blockbench Python Editor Forge 1.12.2

01 Contexte & Objectifs

Probleme actuel

Les blocks Minecraft definis par JSON sont completement statiques. Il n'existe aucun mecanisme natif pour animer la geometrie d'un block : pas de rotation de groupes, pas de translation de pieces, pas de keyframes. Les seuls "mouvements" possibles passent par des hacks (swap de blockstate, animation de texture UV) qui sont limites, saccades et couteux en performance.

Pour des blocks comme une lootbox qui s'ouvre, une tourelle qui pivote, un coffre-fort avec un mecanisme, une machine avec des engrenages, ou un portail avec des anneaux rotatifs, il faut un vrai systeme d'animation 3D avec des keyframes, des easings, et un rendu fluide via TileEntitySpecialRenderer.

Solution proposee

Un framework complet en 3 composants integre a EriAPI :

  • Runtime Java (EriAPI) : parseur de modeles Blockbench, moteur d'animation par keyframes, rendu TESR optimise avec interpolation partialTicks.
  • Format .erianim : format JSON declaratif pour definir des animations (keyframes, easings, events, endBehavior) sans toucher au code Java.
  • Editeur Python : application desktop PyQt6 + OpenGL pour creer visuellement les animations, avec timeline, viewport 3D, preview en temps reel, et export .erianim.

Cible

Le framework est un module EriAPI, pas une feature EriniumFaction. Il doit etre generique et reutilisable par tout mod qui depend d'EriAPI. EriniumFaction sera le premier consommateur (lootbox, machines de faction, etc.).

⚠
Contrainte serveur 500-1000 joueurs : des dizaines voire centaines de blocks animes peuvent etre charges simultanement dans les chunks visibles. Le runtime TESR doit etre optimise pour le rendu batche en IDLE, l'absence de tick quand aucune animation ne joue, et le cache agressif des modeles parses. Cote serveur, une animation = un etat + un timestamp, jamais de sync par tick.

Objectifs de design

  • Zero code pour animer : un moddeur exporte un .json depuis Blockbench, cree un .erianim dans l'editeur Python, et enregistre le block avec 3 lignes de builder Java. Pas de classe custom a ecrire.
  • Fidelite Blockbench : le rendu in-game doit etre visuellement identique au preview Blockbench — memes pivots, memes rotations, memes UV.
  • Animations declaratives : tout est defini dans le .erianim (keyframes, easings, events, comportement de fin). Le code Java ne fait que declencher playAnimation("open").
  • Editeur visuel : les artistes/designers creent les animations dans un outil graphique, pas en ecrivant du JSON a la main.
  • Performance : rendu batchable en IDLE, pas de tick inutile, cache des modeles, distance de rendu configurable.
  • Extensibilite : events d'animation (son, particule, callback Java) pour synchroniser la logique de jeu avec les keyframes.

Ce que le framework ne fait pas

  • Pas d'animation d'entites : ce framework est pour les blocks (TileEntity + TESR). Les entites utilisent le systeme d'animation vanilla de Minecraft.
  • Pas de physique : les animations sont pre-calculees (keyframes). Pas de simulation physique en temps reel.
  • Pas d'animation procedurale : chaque mouvement est defini par des keyframes explicites, pas par des algorithmes proceduraux.
  • Pas de skinning/bones : les modeles sont des boxes rigides groupees. Pas de deformation de mesh ni de squelette.

ⓘ
Separation des responsabilites : le runtime Java ne sait rien de l'editeur Python. L'editeur ne sait rien du runtime. Le format .erianim est le contrat entre les deux. Un artiste peut creer des animations sans connaitre Java, et un developpeur peut les utiliser sans connaitre l'editeur.

03 Format de modele — Lecteur JSON Blockbench

Objectif

Parser les fichiers .json exportes depuis Blockbench (format "Java Block/Item") et construire une hierarchie Java exploitable par le TESR. Le parseur doit supporter les groupes imbriques, les pivots, les rotations initiales, les UV par face, et les multi-textures.

ℹ
Modele de reference : un exemple complet de modele 3D avec multi-textures (11 textures, 75 elements, groupes imbriques) est disponible dans C:\Users\killi\Pictures\Mcreator-Resources\3d models\lootbox\. Contient le lootbox.json (export Blockbench), le .bbmodel source, et toutes les textures PNG. Utiliser ce modele comme reference pour comprendre les mappings textures, la hierarchie des groupes, et le format des UV.

Structure Blockbench JSON parsee

Le fichier Blockbench contient 3 sections principales :

Champ JSONTypeDescription
texturesObjectMap des textures. Deux formats : par index "0": "modid:blocks/name" ou par nom "side_cable.png": "side_cable". Le parseur doit supporter les deux.
elementsArrayListe des boxes (cubes). Chaque element a from, to, rotation, faces
groupsArrayHierarchie de groupes. Chaque groupe a un name, origin (pivot), children (indices d'elements ou sous-groupes)

Structure Java resultante

public class AnimatedBlockModel { private final String modelId; // "modid:block/name" private final List<ModelGroup> rootGroups; // Groupes racine private final Map<String, ModelGroup> groupsByName; // Acces rapide par nom private final Map<Integer, ResourceLocation> textures; // Index -> texture private final List<ModelElement> allElements; // Tous les elements a plat } public class ModelGroup { private final String name; // "lid", "hinge", "base" private final float[] origin; // Pivot point [x, y, z] private final List<ModelElement> elements; // Boxes de ce groupe private final List<ModelGroup> children; // Sous-groupes private boolean visible; // Visibilite courante } public class ModelElement { private final float[] from; // [x, y, z] min private final float[] to; // [x, y, z] max private final float rotationAngle; // Rotation statique private final String rotationAxis; // "x", "y", ou "z" private final float[] rotationOrigin; // Pivot de rotation statique private final Map<EnumFacing, FaceData> faces; // UV par face } public class FaceData { private final float[] uv; // [u1, v1, u2, v2] private final int textureIndex; // Index dans la map de textures private final int uvRotation; // 0, 90, 180, 270 private final int tintIndex; // -1 si pas de tint }

Conversion des coordonnees

Blockbench utilise un systeme ou 1 block = 16 pixels, avec l'origine (0,0,0) en bas-coin-nord-ouest du block. Le parseur convertit les coordonnees en unites OpenGL :

  • Position : pixel / 16.0f — une box de [0,0,0] a [16,16,16] devient [0,0,0] a [1,1,1] en OpenGL.
  • Pivot : meme conversion origin / 16.0f. Le pivot [8,8,8] dans Blockbench = [0.5, 0.5, 0.5] en OpenGL (centre du block).
  • UV : les UV Blockbench sont en pixels de texture. Convertis en uv / textureSize (generalement 16, mais peut etre 32, 64, etc. selon la texture).
  • Rotation : les angles Blockbench sont en degres, identiques en OpenGL. Les axes X/Y/Z suivent la convention Minecraft (Y = haut).
✓
Multi-textures : un modele Blockbench peut utiliser plusieurs textures (une par face ou par groupe). Le runtime charge toutes les textures referencees et bind la bonne avant chaque face. Pour optimiser, les faces utilisant la meme texture sont regroupees dans le meme draw call.

04 Runtime TESR — Rendu anime

AnimatedBlockTESR

Le renderer est un TileEntitySpecialRenderer<AnimatedBlockTileEntity> qui gere le rendu de TOUS les blocks animes enregistres via EriAPI. Un seul TESR est enregistre pour le type AnimatedBlockTileEntity — il dispatch le rendu selon le modele associe au block.

public class AnimatedBlockTESR extends TileEntitySpecialRenderer<AnimatedBlockTileEntity> { @Override public void render(AnimatedBlockTileEntity te, double x, double y, double z, float partialTicks, int destroyStage, float alpha) { // 1. Recuperer le modele depuis le cache AnimatedBlockModel model = AnimModelCache.get(te.getModelId()); // 2. Recuperer l'etat d'animation interpole AnimationController ctrl = te.getAnimationController(); AnimationState state = ctrl.getInterpolatedState(partialTicks); // 3. Push matrix globale GlStateManager.pushMatrix(); GlStateManager.translate(x + 0.5, y, z + 0.5); // 4. Rendre chaque groupe racine recursivement for (ModelGroup group : model.getRootGroups()) { renderGroup(group, state); } GlStateManager.popMatrix(); } private void renderGroup(ModelGroup group, AnimationState state) { if (!state.isGroupVisible(group.getName())) return; GlStateManager.pushMatrix(); // Translate au pivot float[] pivot = group.getOrigin(); GlStateManager.translate(pivot[0], pivot[1], pivot[2]); // Appliquer rotation/translation/scale animees float[] rot = state.getRotation(group.getName()); float[] trans = state.getTranslation(group.getName()); float[] scale = state.getScale(group.getName()); GlStateManager.translate(trans[0], trans[1], trans[2]); GlStateManager.rotate(rot[0], 1, 0, 0); GlStateManager.rotate(rot[1], 0, 1, 0); GlStateManager.rotate(rot[2], 0, 0, 1); GlStateManager.scale(scale[0], scale[1], scale[2]); // Revenir du pivot GlStateManager.translate(-pivot[0], -pivot[1], -pivot[2]); // Rendre les elements du groupe for (ModelElement elem : group.getElements()) { renderElement(elem); } // Rendre les sous-groupes recursivement for (ModelGroup child : group.getChildren()) { renderGroup(child, state); } GlStateManager.popMatrix(); } }

Etats d'animation (AnimState)

EtatDescriptionTick ?Rendu
IDLEAucune animation en cours. Pose par defaut.NONFastTESR batche ou TESR statique
PLAYINGAnimation en cours de lecture.OUITESR anime avec partialTicks
PAUSEDAnimation en pause a un tick specifique.NONTESR statique a la frame de pause

Synchronisation client/serveur

Le serveur ne calcule JAMAIS les frames d'animation. Il envoie uniquement :

  • Un packet de declenchement : {animation: "open", startTick: 12345, endBehavior: "freeze"}
  • Un packet de reset : {action: "reset", instant: false}
  • Un packet de pause/resume : {action: "pause"} ou {action: "resume", resumeTick: 12350}

Le client recoit le packet et son AnimationController calcule la frame courante a chaque rendu via : float progress = (currentWorldTick - animStartTick + partialTicks) / duration. Aucune donnee d'animation ne transite chaque tick.

ⓘ
ITickable conditionnel : le TileEntity implemente ITickable mais le update() ne fait rien si animationState est IDLE, FROZEN, PAUSED ou WAITING_SERVER. Le tick n'est actif que pendant PLAYING et ROLLING_BACK, pour declencher les events d'animation (sons, particules, callbacks) au bon moment. Le rendu graphique est gere par le TESR via partialTicks, pas par le tick.

Render distance

Chaque type de block anime a une distance de rendu max configurable via le builder (renderDistance(64)). Le TESR surcharge getMaxRenderDistanceSquared() sur le TileEntity pour retourner distance * distance. Au-dela de cette distance, le TESR ne dessine rien — le block apparait comme un cube invisible ou un placeholder.

05 Format .erianim — Specification complete

Vue d'ensemble

Le format .erianim est un fichier JSON qui definit une ou plusieurs animations pour un modele Blockbench donne. Chaque animation contient des pistes (tracks) de keyframes par groupe du modele, des parametres de comportement de fin, de blend, et des events synchronises.

Structure JSON complete

{ "format_version": 1, "model": "eriniumfaction:block/lootbox", "animations": { "open": { "duration": 20, "endBehavior": "freeze", "rollbackDuration": 10, "blendIn": 3, "blendOut": 3, "loopCount": 0, "loopPause": 0, "tracks": { "lid": { "rotation": [ { "tick": 0, "value": [0, 0, 0], "easing": "linear" }, { "tick": 10, "value": [-90, 0, 0], "easing": "ease_out" }, { "tick": 15, "value": [-85, 0, 0], "easing": "ease_in_out" }, { "tick": 20, "value": [-90, 0, 0], "easing": "ease_out" } ], "translation": [ { "tick": 0, "value": [0, 0, 0], "easing": "linear" } ], "scale": [ { "tick": 0, "value": [1, 1, 1], "easing": "linear" } ], "visible": [ { "tick": 0, "value": true } ] }, "sparkle_group": { "visible": [ { "tick": 0, "value": false }, { "tick": 12, "value": true } ], "scale": [ { "tick": 12, "value": [0, 0, 0], "easing": "elastic" }, { "tick": 20, "value": [1, 1, 1], "easing": "elastic" } ] } }, "events": [ { "tick": 5, "type": "sound", "value": "eriniumfaction:lootbox_creak" }, { "tick": 12, "type": "particle", "value": "sparkle", "count": 8, "spread": 0.5 }, { "tick": 15, "type": "sound", "value": "eriniumfaction:lootbox_open" }, { "tick": 18, "type": "cut" }, { "tick": 20, "type": "callback", "value": "onOpenComplete" } ] }, "close": { "duration": 15, "endBehavior": "rollback_instant", "tracks": { /* ... */ }, "events": [ /* ... */ ] }, "idle_breathe": { "duration": 40, "endBehavior": "loop", "tracks": { /* ... */ }, "events": [] } } }

Pistes de track (par groupe)

Chaque groupe du modele peut avoir jusqu'a 6 types de pistes :

PisteType valeurDescription
rotationfloat[3] (degres)Rotation autour du pivot du groupe [X, Y, Z]. Ajoutee a la rotation statique du modele.
translationfloat[3] (pixels Blockbench)Translation relative [X, Y, Z]. Convertie en unites OpenGL (/ 16.0f).
scalefloat[3]Echelle [X, Y, Z]. Valeur par defaut [1, 1, 1]. Scale 0 = invisible.
visiblebooleanVisibilite du groupe. Pas d'interpolation — changement instantane.
spinObject (speed + axis)Rotation continue autour d'un axe. Les keyframes definissent la vitesse (degres/seconde) et l'axe. L'easing s'applique sur la transition de vitesse entre keyframes — permet acceleration/deceleration naturelle. La rotation s'accumule dans le temps (pas de valeur cible).
textureObject (target + value)Changement de texture dynamique a un tick donne. target = nom de la texture dans le modele (ex: "side_core"), value = nom de la texture de remplacement. Pas d'interpolation — changement instantane. Peut aussi etre change depuis un callback Java via te.setTexture().

Piste spin — rotation continue

La piste spin est un type special qui fait tourner un groupe en continu a une vitesse donnee, au lieu d'interpoler entre des angles cibles. Ideal pour des engrenages, hologrammes, effets de charge, etc.

"hexadecagon": { "spin": [ { "tick": 0, "axis": "y", "speed": 0, "easing": "ease_in" }, { "tick": 20, "axis": "y", "speed": 360, "easing": "linear" }, { "tick": 60, "axis": "y", "speed": 360, "easing": "ease_out" }, { "tick": 80, "axis": "y", "speed": 0 } ] }

Resultat : le groupe accelere de 0 a 360°/s sur 1 seconde, tourne a vitesse constante pendant 2 secondes, puis decelere et s'arrete en 1 seconde.

ChampTypeRequisDescription
tickintOUITick de la keyframe de vitesse.
axisStringOUIAxe de rotation : "x", "y", ou "z".
speedfloatOUIVitesse de rotation en degres par seconde. 360 = un tour complet par seconde. Negatif = sens inverse.
easingStringNONEasing de la transition de vitesse depuis la keyframe precedente. Defaut : "linear".
ℹ
spin vs rotation : les deux sont mutuellement exclusifs sur un meme groupe. rotation interpole entre des angles cibles, spin accumule la rotation dans le temps a une vitesse variable. Si un groupe a les deux, spin est prioritaire.

Piste texture — changement de texture dynamique

La piste texture permet de changer la texture d'un groupe a un tick precis dans l'animation. Le changement est instantane (pas d'interpolation). C'est ideal pour des effets visuels lies a l'etat du block (rarete, activation, degats, etc.).

"front": { "texture": [ { "tick": 0, "target": "side_core", "value": "side_core" }, { "tick": 15, "target": "side_core", "value": "side_core_red" }, { "tick": 30, "target": "side_core", "value": "side_core" } ] }
ChampTypeRequisDescription
tickintOUITick du changement de texture.
targetStringOUINom de la texture dans le modele a remplacer (ex: "side_core").
valueStringOUINom de la texture de remplacement. Doit exister dans les resources du mod.

Textures dynamiques via callback Java

Les textures peuvent aussi etre changees depuis un callback Java, ce qui permet une logique serveur dynamique :

// Enregistrer un callback qui change les textures selon la rarete te.setAnimationCallback("onRarityReveal", (tile) -> { int rarity = calculateLootRarity(tile.getPos()); if (rarity >= 3) { tile.setTexture("rarity_top", "eriniumfaction:blocks/rarity_legendary"); tile.setTexture("side_core", "eriniumfaction:blocks/side_core_gold"); } else if (rarity >= 2) { tile.setTexture("rarity_top", "eriniumfaction:blocks/rarity_epic"); tile.setTexture("side_core", "eriniumfaction:blocks/side_core_purple"); } else { tile.setTexture("rarity_top", "eriniumfaction:blocks/rarity_common"); } }); // Dans le .erianim : // { "tick": 10, "type": "callback", "value": "onRarityReveal" } // → tick 10 : le serveur calcule la rarete et change les textures // → le changement sync au client via le TileEntity
✓
API TileEntity : te.setTexture(target, resourceLocation) set un override. te.clearTexture(target) revient a la texture par defaut du modele. te.clearAllTextures() reset tout. Les overrides sont stockes dans le NBT du TileEntity et synchronises au client automatiquement.

08 Animations blend & transitions

Principe

Quand un block passe d'une animation a une autre (ex: de "idle_breathe" a "open"), il ne doit pas y avoir de saut visuel brutal. Le systeme de blend interpole entre la pose actuelle du block et la premiere keyframe de la nouvelle animation sur une duree configurable.

blendIn

Defini sur chaque animation dans le .erianim. Quand playAnimation("open") est appele :

  • Tick 0 du blend : la pose du block est celle de la frame courante de l'animation precedente (ou la pose par defaut si IDLE).
  • Tick blendIn du blend : la pose est celle de la premiere keyframe (tick 0) de la nouvelle animation.
  • Interpolation : lineaire entre les deux poses, composant par composant (rotation, translation, scale) pour chaque groupe.
  • Pendant le blend : l'animation precedente est arretee. Le playhead de la nouvelle animation est a 0 et ne demarre qu'apres le blend.

blendOut

Defini sur chaque animation. Active quand l'animation se termine et qu'une autre prend le relais :

  • Si une nouvelle animation demarre pendant le blendOut : le blendOut de l'ancienne et le blendIn de la nouvelle se combinent. Le temps de transition = max(blendOut, blendIn) de la nouvelle.
  • Si aucune animation ne suit (passage a IDLE) : le blendOut interpole vers la pose par defaut (tick 0 du modele, pas d'animation).
  • blendOut = 0 : pas de transition en sortie, saut direct.

Cas particuliers

  • blendIn = 0 : pas de transition en entree. L'animation demarre immediatement, potentiel saut visuel.
  • Interruption pendant un blend : si une troisieme animation est declenchee pendant un blend A->B, le blend repart de la pose courante interpolee vers la premiere keyframe de C.
  • Loop avec blend : en mode loop, le blendOut de la fin du cycle se combine avec le blendIn du debut du cycle suivant. Si les deux sont 0, pas de transition (boucle seche).
ⓘ
Calcul de la pose de blend : pour chaque groupe, chaque composant (rotX, rotY, rotZ, transX, transY, transZ, scaleX, scaleY, scaleZ), la valeur intermediaire est : lerp(poseA[component], poseB[component], blendProgress) ou blendProgress = elapsedBlendTicks / blendDuration. La visibilite n'est PAS blendee — elle snap au premier tick de la nouvelle animation.

09 Gestion des endBehavior

Cycle de vie complet

Quand une animation atteint son dernier tick, le AnimationController evalue le endBehavior et transition vers l'etat approprie :

playAnimation("open") | v [blendIn ?] ──yes──> BLENDING_IN (blendIn ticks) ──> PLAYING | no | v v PLAYING ────────────────────────────────> Dernier tick atteint | +──────────────+────────────+────────────+──────────+ | | | | | v v v v v FROZEN ROLLING_BACK IDLE WAITING PLAYING (freeze) (rollback) (rollback _SERVER (loop) | _instant) | | v v v IDLE resetAnimation() restart (rollback done) | v IDLE ou ROLLING_BACK

freeze

  • Le TileEntity passe en etat FROZEN.
  • Le tick du TileEntity s'arrete (pas de update()).
  • Le TESR rend la derniere frame de l'animation (statique).
  • Pour sortir : playAnimation("autre"), resetAnimation(), ou stopAnimation().

rollback

  • Le TileEntity passe en etat ROLLING_BACK.
  • L'animation se joue en inverse, de la derniere frame vers la frame 0.
  • Duree du rollback = rollbackDuration ticks (defaut: moitie de duration).
  • Les easings sont inverses : un ease_out forward devient un ease_in en rollback.
  • Les events ne se declenchent PAS pendant le rollback.
  • A la fin du rollback, passage en IDLE.

rollback_instant

  • Snap immediat a la frame 0 (pose par defaut).
  • Passage direct en IDLE.
  • Peut causer un saut visuel — a utiliser pour les animations courtes ou mecaniques.

wait_server

  • Le TileEntity passe en etat WAITING_SERVER.
  • Le tick s'arrete, le TESR rend la derniere frame (comme freeze).
  • Le serveur doit appeler te.resetAnimation() pour declencher un rollback, ou te.resetAnimation(true) pour un retour instantane.
  • Cas d'usage : une lootbox s'ouvre, reste ouverte en attendant que le serveur genere le loot, puis se referme quand le loot est distribue.

loop

  • A la derniere frame, le playhead revient a la frame 0 et l'animation recommence.
  • Si blendOut et blendIn sont > 0, le rebouclage est fluide.
  • Les events se redeclenchent a chaque cycle.
  • Boucle infinie jusqu'a stopAnimation() ou playAnimation("autre").

loop_count

  • Comme loop mais s'arrete apres loopCount repetitions.
  • Apres la derniere repetition, passe en FROZEN a la derniere frame.
  • Si loopCount = 3, l'animation joue 3 fois au total (pas 3 fois apres la premiere).

loop_pause

  • Comme loop mais attend loopPause ticks a la frame 0 avant de recommencer.
  • Pendant la pause, le TileEntity est en IDLE (pas de tick, rendu statique frame 0).
  • Ideal pour les animations idle intermittentes (ex: une machine qui fait un cycle toutes les 3 secondes).
⚠
Persistence NBT : l'etat d'animation (currentAnimation, animStartTick, animationState) est sauvegarde en NBT sur le TileEntity. Quand un chunk est decharge puis recharge, l'animation reprend la ou elle en etait. Si le block etait en etat PLAYING et que plus de duration ticks se sont ecoules, le endBehavior est applique immediatement au chargement.

10 Performance & Optimisations

Objectif de performance

Le framework doit supporter 100 blocks animes simultanement visibles dans le champ de vision d'un joueur, dont 20 en animation active et 80 en IDLE, sans impact mesurable sur le framerate (delta < 1ms par frame a 60 FPS). Cote serveur, 500 blocks animes dans les chunks charges ne doivent pas impacter les TPS.

Tests unitaires — Parsing

  • Parse .erianim avec defauts : fichier sans les champs optionnels. Verifier que les valeurs par defaut sont appliquees (easing: linear, endBehavior: freeze, etc.).

Tests unitaires — Interpolation

  • Interpolation lineaire : 2 keyframes (tick 0: [0,0,0], tick 20: [90,0,0]). Verifier que tick 10 = [45,0,0].
  • Interpolation ease_out : verifier que la courbe est decelerative (valeur a mi-chemin > 50% de la progression lineaire).
  • Interpolation bounce : verifier que la valeur depasse la cible puis revient (overshoot characteristic).
  • Interpolation elastic : verifier l'oscillation amortie.
  • Interpolation avec 3+ keyframes : verifier que l'easing s'applique par segment (entre chaque paire de keyframes consecutives).
  • Interpolation partialTicks : verifier que la valeur entre deux ticks est correcte (ex: tick 5 + partialTicks 0.5 = progression equivalente a tick 5.5).
  • Interpolation multi-composant : rotation + translation + scale sur le meme groupe au meme tick. Verifier l'independance.

Tests unitaires — endBehavior

  • freeze : apres duration ticks, l'etat passe a FROZEN. La pose est celle de la derniere keyframe.
  • rollback : apres duration ticks, l'etat passe a ROLLING_BACK. Apres rollbackDuration ticks, etat = IDLE, pose = frame 0.
  • rollback_instant : apres duration ticks, etat = IDLE immediatement, pose = frame 0.
  • wait_server : apres duration ticks, etat = WAITING_SERVER. Appel resetAnimation() -> ROLLING_BACK ou IDLE.
  • loop : apres duration ticks, le playhead revient a 0 et PLAYING continue.
  • loop_count : apres N repetitions, etat = FROZEN.
  • loop_pause : apres chaque cycle, pause de loopPause ticks a frame 0, puis recommence.

Tests unitaires — Blend
