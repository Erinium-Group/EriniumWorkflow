Systeme d'Electricite RF

Generateurs et batteries bases sur le standard Redstone Flux (RF) de l'ecosysteme Forge. Cinq generateurs allant du charbon au noyau Erinium, quatre niveaux de batteries, et une fondation pour les features industrielles futures.

Planifie 5 generateurs 4 batteries Forge Energy compatible 1 200 RF/t max 100M RF stockage max

01 Contexte & Objectifs

Vision du systeme

EriniumFaction introduit un systeme d'electricite base sur le standard Redstone Flux (RF) de l'ecosysteme Forge. RF est le standard d'energie le plus repandu dans les mods Forge 1.12.2 — utilise par Thermal Expansion, EnderIO, Mekanism, Applied Energistics 2, etc. Ce systeme offre une couche de progression industrielle distincte de la progression PvP classique, et ouvre la porte a des features a forte valeur ajoutee.

Objectifs de design

  • Progression industrielle : ajouter un axe de jeu ou les joueurs investissent dans la production et le stockage d'energie pour debloquer des fonctionnalites avancees.
  • Economie d'energie : les generateurs et batteries cree une economie emergente — les factions qui maitrisent le RF obtiennent des avantages defensifs et logistiques.
  • Compatibilite native : nos blocs implementent les interfaces Forge Energy standard, compatibles avec tout mod utilisant RF ou FE sur le serveur.
  • Fondation pour le futur : chaque feature avancee (bouclier, teleporteur, tourelle) consommera du RF produit par ce systeme.
  • Performance serveur : les TileEntities ne tick que si elles ont du travail effectif — critere absolu sur un serveur 500-1000 joueurs.
ⓘ
Perimetre du mod : Les cables de transport d'energie seront geres par un mod externe dote de vrais modeles 3D (ex. : Immersive Engineering, Thermal Dynamics). Notre mod fournit uniquement les generateurs (production) et les batteries (stockage), qui exposent la capability CapabilityEnergy.ENERGY sur toutes leurs faces.

Ce que le systeme ne fait pas

  • Pas de cables propres : le transport d'energie inter-blocs est delegu a un mod externe. Nos blocs poussent/recoivent via les faces adjacentes uniquement.
  • Pas de multiblocs complexes : chaque generateur est un bloc unique, pas une structure multibloc.
  • Pas de GUI EriGuiScreen : les GUIs de machines utilisent le systeme Container + GuiContainer de Forge (compatible Forge 1.12.2, identique aux fours et tables de craft vanilla).

02 Standard RF / Forge Energy

Historique et compatibilite

En Minecraft 1.12.2, Forge a integre le systeme Forge Energy (interfaces IEnergyStorage, IEnergyProvider, IEnergyReceiver) qui est nativement compatible avec le standard RF de CoFH. Le ratio est 1:1 — 1 RF = 1 FE. Tout mod qui utilise l'une ou l'autre API peut interagir avec nos blocs sans configuration supplementaire.

Unites de mesure

  • RF (Redstone Flux) : unite d'energie de base. Valeur stockee en int ou long selon la capacite.
  • RF/t (RF par tick) : debit de production ou de consommation. 20 ticks = 1 seconde. Un generateur a 40 RF/t produit 800 RF/seconde.
  • MRF (mega RF) : usage courant pour les grandes capacites. 1 MRF = 1 000 000 RF.

Interfaces Forge implementees

InterfaceImplemente parRole
IEnergyStorageGenerateurs + BatteriesStockage interne, receiveEnergy, extractEnergy, getEnergyStored, getMaxEnergyStored
IEnergyProviderGenerateurs + BatteriesPousse l'energie vers les blocs adjacents chaque tick
IEnergyReceiverBatteries uniquementAccepte l'energie entrante sur les faces configurees
ITickableGenerateurs + BatteriesLogique de production / distribution chaque tick serveur
⚠
Capability Forge : L'energie est exposee via CapabilityEnergy.ENERGY dans la methode getCapability(Capability, EnumFacing) de la TileEntity. La face null (interne) retourne toujours la capability. Les faces specifiques peuvent etre configurees (voir Section 05 — GUI Batteries).

Modele de distribution

Chaque tick, les generateurs et batteries qui ont de l'energie disponible iterent les 6 faces adjacentes. Pour chaque face, si le bloc voisin expose CapabilityEnergy.ENERGY et peut recevoir de l'energie (canReceive() == true), on appelle receiveEnergy() avec la valeur minimale entre le debit max du generateur et la capacite libre du recepteur. L'energie est distribuee equitablement entre toutes les faces qui en acceptent.

3.1 — Generateur a Charbon (Coal Generator)

  • Tier : Basique — craft en fer + redstone
  • Production : 40 RF/t (800 RF/seconde)
  • Capacite interne : 50 000 RF
  • Combustibles acceptes : tous les combustibles du four vanilla (charbon, bois, planches, bois de charbon, etc.) — utiliser TileEntityFurnace.isItemFuel(itemStack)
  • Rendement charbon : 1600 ticks de combustion = 64 000 RF par charbon (1600 × 40 RF/t)
  • Rendement bois : 300 ticks = 12 000 RF par buche
  • Comportement : si la capacite interne est pleine, la combustion s'arrete (ne gaspille pas le combustible)

3.2 — Generateur Valterite (Valterite Dynamo)

  • Tier : Valterite — investissement mid-game
  • Production : 120 RF/t (2 400 RF/seconde)
  • Capacite interne : 200 000 RF
  • Combustibles acceptes : charbon (tous les combustibles four vanilla) + Valterite Raw (combustible exclusif)
  • Rendement charbon : 1600 ticks = 192 000 RF par charbon (rendement ameliore grace au tier)
  • Rendement Valterite Raw : 3200 ticks = 384 000 RF par Valterite Raw
  • Avantage cumulatif : meme avec du charbon ordinaire, le dynamo Valterite produit 3× plus de RF qu'un Coal Generator

3.3 — Reacteur Aetherite (Aetherite Reactor)

  • Tier : Aetherite — investissement high-game
  • Production : 400 RF/t (8 000 RF/seconde)
  • Capacite interne : 1 000 000 RF (1 MRF)
  • Combustible exclusif : Aetherite Crystal uniquement (pas de charbon)
  • Rendement Aetherite Crystal : 6400 ticks (320 secondes) = 2 560 000 RF par cristal
  • Note design : le reacteur necessite un approvisionnement continu en cristaux — encourage la construction d'une chaine de production

3.4 — Noyau d'Erinium (Erinium Core)

  • Tier : Erinium — endgame absolu
  • Production : 1 200 RF/t (24 000 RF/seconde)
  • Capacite interne : 5 000 000 RF (5 MRF)
  • Combustible exclusif : Erinium Fragment uniquement
  • Rendement Erinium Fragment : 12 800 ticks (640 secondes) = 15 360 000 RF par fragment
  • Usage typique : alimentation des boucliers de faction, tourelles, et extracteurs de minerais en continu

3.5 — Panneau Solaire (Solar Panel)

  • Tier : Basique (diamant + verre) — haut cout initial, production passive
  • Production jour clair : 20 RF/t — condition : world.isDaytime() && !world.isRaining()
  • Production jour nuageux / pluie : 5 RF/t — condition : world.isDaytime() && world.isRaining()
  • Production nuit : 0 RF/t — condition : !world.isDaytime()
  • Capacite interne : 10 000 RF (tampon de nuit)
  • Contrainte : le bloc immediatement au-dessus doit etre Blocks.AIR ou un bloc transparent — verifier world.canSeeSky(pos.up())
  • Pas de slot combustible : TileEntity ITickable mais sans inventaire interne

Quatre niveaux de batteries correspondent aux quatre tiers de materiaux du mod. Les batteries acceptent l'energie de toutes les faces configurees et la redistribuent vers les blocs adjacents. Elles sont le coeur de toute installation RF serieuse.

4.1 — Batterie Basique (Basic Cell)

  • Capacite : 100 000 RF
  • Debit entree max : 200 RF/t — evite la surcharge accidentelle
  • Debit sortie max : 200 RF/t
  • Materiau : Fer + Redstone + Bloc de Lapis (voir recette Section 06)
  • Usage typique : tampon local pour un Coal Generator, alimentation d'une seule machine

4.2 — Batterie Valterite (Valterite Cell)

  • Capacite : 1 000 000 RF (1 MRF) — 10× la Basic Cell
  • Debit entree/sortie max : 1 000 RF/t
  • Usage typique : stockage pour un Valterite Dynamo, alimentation de plusieurs machines

4.3 — Batterie Aetherite (Aetherite Cell)

  • Capacite : 10 000 000 RF (10 MRF)
  • Debit entree/sortie max : 5 000 RF/t
  • Usage typique : reservoir central d'une faction, alimentation du bouclier de faction

4.4 — Batterie Erinium (Erinium Cell)

  • Capacite : 100 000 000 RF (100 MRF) — stockage maximal
  • Debit entree/sortie max : 20 000 RF/t
  • Usage typique : hub energetique d'une grande faction endgame, alimentation simultanée de tourelles, bouclier, teleporteurs
ⓘ
Persistance NBT : Le niveau d'energie stocke est sauvegarde dans le NBT de la TileEntity a chaque modification. En cas de crash serveur, l'energie est conservee. Le niveau de charge est aussi lu depuis le NBT au chargement du chunk.

05 GUI des Machines

⚠
Systeme GUI : Les GUIs des machines utilisent le systeme Forge Container + GuiContainer (comme les fours, tables de craft, et notre table d'enchantement custom dans FactionChestGuiHandler). PAS EriGuiScreen — les machines doivent etre accessibles depuis le serveur sans client-side.

GUI des Generateurs

Chaque generateur partage le meme layout de GUI, adapte par une couleur de theme (gris pour Coal, cyan pour Valterite, violet pour Aetherite, or pour Erinium, jaune pour Solar).

  • Slot combustible : un slot en bas a gauche — accepte uniquement les combustibles valides pour ce generateur
  • Animation de flamme : flamme animee (4 frames) au-dessus du slot combustible, active/inactive selon l'etat de combustion — utiliser le champ burnTime synchronise via IIntArray
  • Barre d'energie RF : barre verticale a droite, hauteur proportionnelle au niveau d'energie. Couleur : rouge (0-25%) → orange (25-50%) → jaune (50-75%) → vert (75-100%)
  • Affichage numerique energie : label au-dessus de la barre — format 12 500 / 50 000 RF
  • Affichage debit : label en bas — format 40 RF/t quand actif, 0 RF/t quand inactif
  • Temps restant combustible : label sous la flamme — format ~80s base sur le burnTime restant

GUI Panneau Solaire

  • Pas de slot combustible (pas d'inventaire)
  • Barre d'energie RF verticale
  • Affichage numerique energie
  • Indicateur d'etat : Production : 20 RF/t (jour clair) / 5 RF/t (pluie) / Inactif (nuit)
  • Icone soleil/lune indiquant la condition actuelle

GUI des Batteries

  • Grande barre d'energie RF : barre verticale centrale, largeur proportionnelle a la capacite affichee
  • Affichage numerique energie : format 250 000 / 1 000 000 RF — format lisible avec espaces comme separateurs de milliers
  • Debit entree en temps reel : label Entree : +500 RF/t mis a jour toutes les 20 ticks
  • Debit sortie en temps reel : label Sortie : -200 RF/t
  • Configuration face par face : 6 boutons representant les 6 faces (NORTH, SOUTH, EAST, WEST, UP, DOWN). Chaque face peut etre configuree en :
    • Entree (BLUE) : la face accepte l'energie entrante
    • Sortie (ORANGE) : la face fournit de l'energie sortante
    • Desactivee (GREY) : aucune interaction sur cette face
  • Persistance face config : la configuration des faces est stockee en NBT (faceModes[6] int array, valeurs 0/1/2)

Synchronisation Container → GUI

Les donnees dynamiques (energie stockee, burnTime, debit) sont synchronisees via IIntArray (Forge 1.12.2 : addTrackingInt dans le Container). Le client recoit les mises a jour automatiquement. Pour les grandes valeurs (capacite 100M RF), utiliser deux champs int (high/low 32 bits) ou caster en long cote serveur avant de splitter.

TileGeneratorBase — logique commune

public abstract class TileGeneratorBase extends TileEntity implements ITickable, IEnergyStorage { // Energie stockee (int suffit pour les generateurs <= 5M RF) protected int energyStored; protected int maxEnergy; // defini par sous-classe protected int rfPerTick; // production par tick protected int burnTime; // ticks restants pour le combustible actuel protected int totalBurnTime; // duree totale du combustible actuel (pour la flamme) // Inventaire combustible (1 slot pour les generateurs a combustible) protected ItemStackHandler fuelHandler = new ItemStackHandler(1); @Override public void update() { // OPTIMISATION : ne tick que si necessaire if (world.isRemote) return; boolean wasActive = burnTime > 0; if (burnTime > 0) { burnTime--; if (energyStored + rfPerTick <= maxEnergy) { energyStored += rfPerTick; markDirty(); } } // Essayer de charger le prochain combustible si energie non pleine if (burnTime == 0 && energyStored < maxEnergy) { tryConsumeFuel(); } // Pousser l'energie vers les blocs adjacents if (energyStored > 0) { pushEnergyToNeighbors(); } // Mise a jour du blockstate on/off si changement d'etat boolean isActive = burnTime > 0; if (wasActive != isActive) { updateBlockState(isActive); } } protected void pushEnergyToNeighbors() { for (EnumFacing facing : EnumFacing.values()) { TileEntity neighbor = world.getTileEntity(pos.offset(facing)); if (neighbor == null) continue; IEnergyStorage storage = neighbor.getCapability( CapabilityEnergy.ENERGY, facing.getOpposite()); if (storage != null && storage.canReceive()) { int toSend = Math.min(energyStored, rfPerTick); int received = storage.receiveEnergy(toSend, false); energyStored -= received; if (received > 0) markDirty(); } } } // writeToNBT / readFromNBT : sauvegarder energyStored, burnTime, fuelHandler }

TileEnergyCell — logique batterie

public class TileEnergyCell extends TileEntity implements ITickable, IEnergyStorage { // Pour Erinium Cell : capacite 100M → utiliser long private long energyStored; private final long maxEnergy; private final int maxTransfer; // Configuration des faces : 0=OFF, 1=ENTREE, 2=SORTIE private final int[] faceModes = new int[6]; // index = EnumFacing.getIndex() @Override public void update() { if (world.isRemote) return; // Pousser l'energie sur les faces configurees en SORTIE if (energyStored > 0) { pushEnergyToOutputFaces(); } } // La capability est exposee par face selon la configuration @Override public <T> T getCapability(Capability<T> capability, EnumFacing facing) { if (capability == CapabilityEnergy.ENERGY) { if (facing == null) return this; // acces interne toujours int mode = faceModes[facing.getIndex()]; if (mode != 0) return this; // ENTREE ou SORTIE = capability exposee } return super.getCapability(capability, facing); } @Override public boolean canReceive() { // Retourne true si au moins une face est en mode ENTREE for (int mode : faceModes) if (mode == 1) return true; return false; } }

Optimisation TPS — regles obligatoires

⚠
Serveur 500-1000 joueurs : Des dizaines de TileEntities RF potentiellement actives en meme temps. Chaque regle ci-dessous est non-negociable.
  • Early return si inactif : if (burnTime == 0 && energyStored == 0) return; — un bloc vide et froid ne fait rien
  • Early return client-side : toujours la premiere instruction de update()
  • markDirty() conditionnel : appeler uniquement si l'energie a reellement change
  • Distribution limitee : pousser au maximum rfPerTick RF par tick, meme si la batterie contient bien plus
  • Pas d'allocation dans update() : ne pas creer de new ArrayList, new ItemStack, etc. dans la boucle de tick
  • Solar Panel : verifier world.isDaytime() en premier — si nuit et energie = 0, retour immediat
  • Batch markDirty : pour le solar panel qui produit 5-20 RF/t, grouper les saves toutes les 10 ticks minimum

09 Textures & Assets

Generateurs — 5 blocs

Chaque generateur necessite un set de textures complet. La face avant (FACING) est distincte des autres faces. L'etat on/off est gere via un BlockStateProperty POWERED de type boolean.

  • coal_generator : palette grise/rouge. Face avant : grille avec flamme (etat on = lueur). Textures : coal_generator_front_on, coal_generator_front_off, coal_generator_side, coal_generator_top
  • valterite_dynamo : palette cyan/acier. Face avant avec engrenages et lueur cyan en etat on
  • aetherite_reactor : palette violet/noir. Face avant avec cristaux et rayonnement violet en etat on
  • erinium_core : palette or/noir. Face avant avec noyau pulsant et halos dores en etat on
  • solar_panel : palette bleue/argent. Une seule texture (pas d'etat on/off). Face superieure avec motif de cellules photovoltaiques

Batteries — 4 blocs

Les batteries n'ont pas d'orientation (pas de FACING). Elles ont 4 etats visuels selon le niveau de charge, geres via une IntegerProperty CHARGE_LEVEL (0, 1, 2, 3).

  • CHARGE_LEVEL 0 : 0–25% — indicateurs eteints, bloc fonce
  • CHARGE_LEVEL 1 : 25–50% — un indicateur allume, rouge/orange
  • CHARGE_LEVEL 2 : 50–75% — deux indicateurs allumes, orange/jaune
  • CHARGE_LEVEL 3 : 75–100% — tous indicateurs allumes, vert clair
  • Chaque batterie a 4 textures correspondant aux 4 niveaux + 1 texture de base pour les cotes

Blockstates JSON — structure obligatoire

// Exemple : blockstates/coal_generator.json { "variants": { "facing=north,powered=false": { "model": "eriniumfaction:coal_generator_off" }, "facing=north,powered=true": { "model": "eriniumfaction:coal_generator_on" }, "facing=south,powered=false": { "model": "eriniumfaction:coal_generator_off", "y": 180 }, "facing=south,powered=true": { "model": "eriniumfaction:coal_generator_on", "y": 180 }, "facing=east,powered=false": { "model": "eriniumfaction:coal_generator_off", "y": 90 }, "facing=east,powered=true": { "model": "eriniumfaction:coal_generator_on", "y": 90 }, "facing=west,powered=false": { "model": "eriniumfaction:coal_generator_off", "y": 270 }, "facing=west,powered=true": { "model": "eriniumfaction:coal_generator_on", "y": 270 } } }

Dimensions et format des textures

  • Format : PNG 32×32 pixels (compatible 1.12.2, bon compromis visuel/performance)
  • Generation : SVG pixel art → ImageMagick → PNG selon workflow docs/ASSETS_WORKFLOW.md
  • Nommage : textures/blocks/coal_generator_front_on.png, coal_generator_front_off.png, coal_generator_side.png, coal_generator_top.png
  • Item : chaque bloc a un modele item pointant vers la texture de face off (models/item/coal_generator.json)
