Securite — Systeme de Prevention

Anti Use Bug

Framework serveur de prevention, detection et mitigation des exploits de duplication, crashs de chunks et attaques par saturation de donnees NBT en environnement Minecraft 1.12.2 multi-mods.

Priorite CRITIQUE Forge 1.12.2 Mixin + EriAPI 500-1000 joueurs Zero Trust Client

01Contexte & Objectifs

Qu'est-ce qu'un "Use Bug" ?

Un "use bug" designe toute technique exploitant des failles dans la logique de jeu pour obtenir un avantage illegitime, en particulier :

  • Duplication d'items — obtenir plus d'objets qu'il n'en a ete investis via des exploits de containers, de craft, de paquets ou de mort.
  • Crash de chunk — saturer un chunk avec des donnees NBT massives jusqu'a ce que le jeu ne puisse plus sauvegarder ou charger ce chunk, provoquant une perte de donnees ou une dupe involontaire.
  • NBT data bombs — creer des stacks d'items avec des metadonnees excessivement grandes (livres ecrits, cartes, enseignes) pour provoquer des lags, des OOM ou des crashs serveur.
  • Exploits de conteneurs — utiliser des conditions de course (race conditions) dans les interactions de coffres, fours, hoppers pour dupliquer des items.
  • Exploits de mode creatif — envoyer des paquets de creation d'item illicites depuis un client modifie.

Contexte du serveur

EriniumFaction cible 500 a 1000 joueurs simultanees avec 20+ mods actifs sur un monde faction PvP et une dimension de minage. Dans cet environnement :

  • L'economie du serveur est entierement basee sur la rareté des ressources et les echanges entre factions. Une seule technique de dupe peut ruiner l'equilibre en quelques heures.
  • La charge du serveur est elevee : les exploits de saturation NBT peuvent declencher des cascades de lag sur des machines sous pression.
  • Les joueurs de factions PvP sont souvent des joueurs experimentes qui connaissent et partagent activement les exploits connus.
  • Il est impossible de moderer manuellement 1000 joueurs en permanence.

Philosophie : Zero Trust Client

!
Le client ne doit jamais etre fait confiance. Toute donnee, paquet, ou action provenant du client doit etre validee independamment cote serveur. Le serveur maintient sa propre representation de l'etat du monde et refuse tout ce qui devie de cet etat attendu.

Objectifs

  • Prevention proactive — Bloquer les exploits avant qu'ils ne prennent effet, pas apres.
  • Detection en temps reel — Identifier les comportements suspects des qu'ils se produisent.
  • Absence de faux positifs — Ne pas penaliser le jeu normal. Les limites doivent etre suffisamment hautes pour ne jamais impacter un joueur legitime.
  • Impact performance minimal — Le systeme ne doit pas ajouter plus de 1ms par tick sur le thread principal.
  • Configurabilite totale — Tous les seuils sont ajustables sans recompilation.
  • Audit complet — Chaque detection est logguee avec assez de contexte pour investigation.

03Chunk NBT Bomb

!
Exploit CRITIQUE. Des joueurs remplissent des chunks de coffres contenant des livres ecrits avec un nombre maximal de pages, des bannières complexes, ou des cartes imbriquees. Quand le chunk tente d'etre sauvegarde, la serialisation NBT echoue ou depasse la limite de taille du fichier de region, provoquant une perte de chunk — et potentiellement une duplication si le serveur recharge une ancienne version du chunk.

Mecanisme de l'exploit

  • Un livre ecrit peut avoir jusqu'a 50 pages et 256 caracteres par page = ~12,800 chars de texte brut.
  • Chaque page est stockee en NBT avec overhead JSON : une page peut representer jusqu'a ~800 bytes en NBT.
  • Un coffre double contient 54 slots. 54 livres pleins = ~43,200 chars. Mais chaque slot peut etre stacke a 64 en mode creatif : 54 * 64 = 3,456 livres dans un coffre.
  • Le format de chunk NBT de Minecraft 1.12.2 n'a pas de limite stricte de taille cote serveur Forge. Un chunk avec 100 coffres pleins de livres peut atteindre plusieurs megaoctets de donnees NBT.
  • Le format de region (.mca) est limite a 1 MB par chunk. Un depassement produit un ChunkSaveException et corrompt le chunk.

Solution : ChunkNBTMonitor

Le ChunkNBTMonitor maintient un cache de la taille NBT estimee pour chaque chunk charge. Cette taille est mise a jour incrementalement a chaque modification de TileEntity, pas a chaque tick.

// Estimation taille NBT d'un TileEntity public class ChunkNBTMonitor { // Cache: ChunkPos -> estimated NBT bytes private final Map<ChunkPos, Integer> chunkSizeCache = new ConcurrentHashMap<>(); // Limites configurables public static final int MAX_CHUNK_NBT_BYTES = 524288; // 512 KB par chunk public static final int MAX_TE_NBT_BYTES = 65536; // 64 KB par TileEntity public static final int MAX_TILE_ENTITIES_PER_CHUNK = 256; // max TE par chunk @SubscribeEvent public void onChunkWatch(ChunkWatchEvent.Watch event) { // Calculer la taille NBT initiale quand le chunk se charge EriScheduler.delay(1, () -> scanChunk(event.getChunk())); } public void onTileEntityModified(TileEntity te) { // Mettre a jour le cache incrementalement ChunkPos pos = new ChunkPos(te.getPos()); int newSize = estimateTENBTSize(te); chunkSizeCache.merge(pos, newSize, (Integer old, Integer add) -> old + add); checkChunkLimit(pos, te.getWorld()); } }

Mesures de protection

  • Limite NBT par chunk — Si la taille NBT estimee depasse MAX_CHUNK_NBT_BYTES (512 KB par defaut), les nouveaux items NBT lourds sont refuses et l'admin est notifie.
  • Limite NBT par TileEntity — Avant d'accepter la serialisation d'un TileEntity, sa taille NBT est verifiee. Si > 64 KB, le TE est vidange puis loggue.
  • Limite de TileEntities par chunk — Un chunk ne peut pas contenir plus de 256 TileEntities. Au-dela, les nouveaux placements de blocks TE sont refuses.
  • Nettoyage automatique — Si un chunk depasse la limite, les TileEntities les plus lourds sont identifies et leurs contenus sont supprimes (items droppes au sol ou detruits selon config).
  • Hook de sauvegarde — Mixin sur AnvilChunkLoader.writeChunkToNBT pour intercepter la serialisation et refuser si la taille depasse la limite avant ecriture.
// Mixin sur AnvilChunkLoader @Mixin(AnvilChunkLoader.class) public abstract class MixinAnvilChunkLoader { @Inject(method = "writeChunkToNBT", at = @At("HEAD"), cancellable = true) private void onWriteChunk(Chunk chunk, World world, NBTTagCompound compound, CallbackInfo ci) { if (ChunkNBTMonitor.getInstance().isChunkOverLimit(chunk)) { ExploitLogger.log("CHUNK_NBT_OVERFLOW", chunk.getPos(), null); ChunkNBTMonitor.getInstance().cleanChunk(chunk); // Ne pas annuler - le chunk sauvegarde apres nettoyage } } }

04Container Duplication

Mecanisme de la race condition

Dans Minecraft 1.12.2, lorsqu'un joueur interagit avec un conteneur (coffre, four, table de craft), le serveur traite les paquets CPacketClickWindow et met a jour l'inventaire. Sous forte charge serveur (lag) :

  • Le client peut envoyer plusieurs paquets rapides avant de recevoir la confirmation serveur.
  • Un double-clic ou shift-clic rapide peut etre traite deux fois si la validation de l'etat precedent est manquee.
  • La desynchronisation client-serveur cree une fenetre ou l'item semble exister dans deux endroits simultanement.

ContainerInteractionValidator

Le ContainerInteractionValidator maintient un historique reconstruisant l'etat serveur-autoritatif de chaque interaction de conteneur par joueur.

public class ContainerInteractionValidator { // Dernier tick d'interaction par joueur par container private final Map<UUID, ContainerState> playerStates = new HashMap<>(); // Cooldown minimum entre interactions (ticks) public static final int MIN_INTERACTION_GAP_TICKS = 1; public boolean validateClick(EntityPlayerMP player, Container container, int slotId, int button, ClickType type, ItemStack expected) { ContainerState state = playerStates.getOrDefault(player.getUniqueID(), new ContainerState()); // 1. Verifier que le tick d'interaction precedent est passe if (state.lastTick == player.world.getTotalWorldTime()) { // Meme tick = paquet double -> refuser flagExploit(player, "CONTAINER_DOUBLE_INTERACT"); return false; } // 2. Verifier que l'item dans le slot correspond a notre etat attendu ItemStack serverSlot = container.getSlot(slotId).getStack(); if (!ItemStack.areItemStacksEqual(serverSlot, expected)) { flagExploit(player, "CONTAINER_DESYNC"); // Forcer resync inventaire player.sendContainerToPlayer(container); return false; } state.lastTick = player.world.getTotalWorldTime(); return true; } }

Verification de la somme des items

Apres chaque transaction de conteneur completee, le systeme compare le nombre total d'items dans l'inventaire du joueur + le conteneur avant et apres la transaction. Si la somme augmente d'un item qui n'etait pas attendu, la transaction est annulee et l'inventaire est restaure.

// Snapshot avant transaction InventorySnapshot before = InventorySnapshot.take(player, container); // [transaction vanilla] // Verification apres transaction InventorySnapshot after = InventorySnapshot.take(player, container); if (!before.isConsistentWith(after)) { ExploitLogger.log("ITEM_COUNT_MISMATCH", player); before.restore(player, container); // Rollback }

05Packet Manipulation

Vecteurs d'attaque

  • CPacketCreativeInventoryAction — Les clients en mode creatif peuvent envoyer ce paquet pour donner n'importe quel item. Des clients modifies l'envoient en mode survie.
  • CPacketClickWindow avec itemStack modifie — Un client modifie peut envoyer un paquet avec un ItemStack dont le NBT a ete gonfle ou les metadonnees alterees.
  • CPacketWindowConfirmation modifie — Confirmer des transactions que le serveur n'a jamais initiees.

PacketValidator — Mixin sur NetHandlerPlayServer

@Mixin(NetHandlerPlayServer.class) public abstract class MixinNetHandlerPlayServer { @Shadow public EntityPlayerMP player; // Intercepter les clics d'inventaire @Inject(method = "processClickWindow", at = @At("HEAD"), cancellable = true) private void validateClickWindow(CPacketClickWindow pkt, CallbackInfo ci) { if (!PacketValidator.getInstance().validateClickWindow(player, pkt)) { ci.cancel(); } } // Intercepter les actions creatives (ne doit JAMAIS arriver en survie) @Inject(method = "processCreativeInventoryAction", at = @At("HEAD"), cancellable = true) private void validateCreativeAction(CPacketCreativeInventoryAction pkt, CallbackInfo ci) { if (player.interactionManager.getGameType() != GameType.CREATIVE) { ExploitLogger.log("CREATIVE_PACKET_IN_SURVIVAL", player); PacketValidator.getInstance().flagViolation(player, 10); // Violation severe ci.cancel(); } } }

Validation des ItemStack dans les paquets

  • Taille NBT par item — Tout ItemStack recu dans un paquet est inspecte. Si sa taille NBT depasse 8 KB, le paquet est refuse.
  • Validite de l'item — L'item doit exister dans le registre Forge. Les items avec des metadatas impossibles sont refuses.
  • Coherence du count — Un ItemStack dans un slot de conteneur ne peut pas depasser maxStackSize. Les stacks > max (manipules par client) sont refuses.
  • Hash verification — Pour chaque slot d'inventaire, le systeme maintient un hash de l'etat attendu. Si le paquet presente un item qui ne correspond pas au hash, la transaction est rejetee et l'inventaire est resynchronise.

06Drop / Pickup Exploits

Vecteurs d'attaque

Ces exploits exploitent les fenetres temporelles durant des transitions d'etat ou le serveur peut perdre le suivi des items :

CRITIQUEDeath Dupe
Dropper un item au sol au moment exact de la mort. Sous lag, le serveur traite le drop et la mort dans le meme tick — l'item est a la fois dans le loot de mort et au sol.
Solution : bloquer les drops d'inventaire pendant les 2 ticks precedant et suivant la mort.
HAUTTP Dupe
Interagir avec un conteneur pendant un teleportation. L'item peut exister dans deux etats simultanes si la TP ne vide pas correctement les conteneurs ouverts.
Solution : fermer de force tous les conteneurs ouverts avant chaque teleportation.
HAUTDimension Change Dupe
Placer des items dans un conteneur en cours de transition de dimension. La desync entre dimensions permet a l'item d'exister dans les deux.
Solution : lock inventaire pendant la transition dimensionnelle (flag joueur).
MOYENOverflow Pickup
Ramasser des items alors que l'inventaire est plein sur un client modifie. Certains clients envoient des faux paquets de pickup pour forcer des items supplementaires.
Solution : valider capacite inventaire cote serveur avant chaque pickup.

Implementation

// Flag de lock inventaire par joueur @EriCapability(modId = "eriniumfaction", target = AttachTarget.PLAYER) public class SecurityCapData { public boolean inventoryLocked = false; public long lockUntilTick = 0; public String lockReason = ""; } // Mixin sur EntityPlayer.dropItem @Inject(method = "dropItem(Lnet/minecraft/item/ItemStack;Z)Lnet/minecraft/entity/item/EntityItem;", at = @At("HEAD"), cancellable = true) private void preventDropWhileLocked(ItemStack stack, boolean dropAround, CallbackInfoReturnable<EntityItem> cir) { if (EriCap.get((EntityPlayer)(Object)this, SecurityCapData.class).inventoryLocked) { cir.setReturnValue(null); // Annuler le drop } }

07Crafting Exploits

Shift-click Craft sous lag

Le craft par shift-clic repete tente de fabriquer autant d'exemplaires que possible en une seule interaction. Sous lag :

  • Le client peut envoyer plusieurs paquets CPacketClickWindow de shift-clic sur l'output de craft avant que le serveur ait retire les ingredients.
  • Resultat : l'output est collecte N fois mais les ingredients ne sont retires qu'une seule fois.

Solution : Post-Craft Ingredient Verification

@Mixin(ContainerWorkbench.class) public abstract class MixinContainerWorkbench { @Shadow public InventoryCrafting craftMatrix; @Shadow public IInventory craftResult; @Inject(method = "transferStackInSlot", at = @At("RETURN")) private void verifyCraftIngredients(EntityPlayer player, int slotIndex, CallbackInfoReturnable<ItemStack> cir) { if (slotIndex == 0) { // Slot 0 = output de craft ItemStack output = cir.getReturnValue(); if (!output.isEmpty()) { // Verifier que les ingredients sont bien consommes CraftingValidator.getInstance() .validateCraftTransaction(player, craftMatrix, output); } } } }

Cooldown de craft

Un cooldown de 1 tick est impose entre chaque operation de craft output pour un meme joueur. Ce cooldown est transparent pour le jeu normal (le craft est instantane dans le jeu vanilla), mais empeche les double-traitements de paquets sous lag.

08Book / Sign NBT Overflow

Attaques par saturation de texte

Minecraft 1.12.2 ne valide pas strictement la taille du contenu des livres ecrits ni des enseignes. Des clients malveillants peuvent envoyer :

  • Un livre avec 500+ pages (limite vanilla = 50 mais non enforced serveur).
  • Un livre avec des pages contenant des JSON Component imbriques exponentiellement.
  • Des enseignes avec des lignes contenant des JSON Component de grande profondeur.
  • Des livres avec des caracteres Unicode speciaux causant des problemes de rendering.
!
Impact : Un livre avec 500 pages de texte JSON imbriqué peut faire crasher le rendu du client et provoquer un lag du serveur lors de la serialisation. Des enseignes avec un JSON malformé peuvent faire crasher tous les clients dans le rayon de rendu.

Validation des paquets

@Inject(method = "processEditBook", at = @At("HEAD"), cancellable = true) private void validateEditBook(CPacketEditBook pkt, CallbackInfo ci) { NBTTagCompound bookTag = pkt.getBookNBT(); // 1. Limiter le nombre de pages NBTTagList pages = bookTag.getTagList("pages", 8); while (pages.tagCount() > UseBugConfig.MAX_BOOK_PAGES) { pages.removeTag(pages.tagCount() - 1); } // 2. Tronquer chaque page for (int i = 0; i < pages.tagCount(); i++) { String page = pages.getStringTagAt(i); if (page.length() > UseBugConfig.MAX_CHARS_PER_PAGE) { pages.set(i, new NBTTagString(page.substring(0, UseBugConfig.MAX_CHARS_PER_PAGE))); } } // 3. Verifier taille NBT totale int nbtSize = NBTSizeValidator.estimateSize(bookTag); if (nbtSize > UseBugConfig.MAX_BOOK_NBT_BYTES) { ExploitLogger.log("BOOK_NBT_OVERFLOW", player); ci.cancel(); } }

09Hopper / Dispenser / TileEntity Exploits

Machines de duplication par hopper

Des joueurs experimentes construisent des "machines" utilisant des combinations de hoppers, comparateurs, pistons et dispensers pour automatiser la duplication a faible taux mais de maniere continue. Les exploits classiques incluent :

  • Hopper-clock dupe — Un hopper connecte a un autre en boucle peut dupliquer un item toutes les N ticks si un comparateur mal configure cree un etat instable.
  • Piston-hopper item duplicate — Utiliser un piston pour pousser un bloc contenant des items pendant qu'un hopper tente de les extraire cree une race condition.
  • Dispenser-bucket dupe — Placer et ramasser des buckets (eau/lave) via dispenser dans des conditions specifiques.

Limites et moniteurs

  • Max hoppers par chunk — Limite a 64 hoppers par chunk (config). Au-dela, les nouveaux hoppers sont refuses au placement (message d'erreur au joueur).
  • Taux de transfert hopper — Si un hopper transfere plus de 64 items/seconde (impossible en vanilla normal), le hopper est desactive temporairement et l'admin est notifie.
  • Cap TileEntities par chunk — Limite totale de 256 TileEntities par chunk (hoppers + coffres + fours + tous les TE).
  • Monitor de TileEntities dupliques — Detection si le meme item UUID apparait dans deux TileEntities differents simultanement.
// Mixin sur TileEntityHopper.transferItems @Mixin(TileEntityHopper.class) public abstract class MixinTileEntityHopper { private int transfersThisSecond = 0; private long lastSecondTick = 0; @Inject(method = "transferItems", at = @At("RETURN")) private void monitorTransferRate(CallbackInfoReturnable<Boolean> cir) { if (!cir.getReturnValue()) return; TileEntityHopper self = (TileEntityHopper)(Object)this; long tick = self.getWorld().getTotalWorldTime(); if (tick - lastSecondTick >= 20) { transfersThisSecond = 0; lastSecondTick = tick; } transfersThisSecond++; if (transfersThisSecond > UseBugConfig.MAX_HOPPER_TRANSFERS_PER_SEC) { UseBugManager.getInstance().flagHopperAnomaly(self); } } }

10Architecture Technique

Package principal

Tout le code de securite est isole dans le package fr.eriniumgroup.eriniumfaction.security.usebug pour faciliter la maintenance et les audits de securite.

UseBugManager
Singleton central. Initialise et orchestre tous les sous-systemes. Point d'entree unique pour les flags d'exploit.
ChunkNBTMonitor
Surveille la taille NBT des chunks en temps reel. Cache incremental, scan initial au chargement.
ContainerInteractionValidator
Valide chaque clic de conteneur. Maintient l'etat attendu par joueur. Rollback si incoherence.
NBTSizeValidator
Estime la taille NBT de tout objet (ItemStack, TileEntity, Chunk). Utilitaire partage.
PacketValidator
Mixin sur NetHandlerPlayServer. Intercepte et valide tous les paquets inventaire entrants.
CraftingValidator
Verifie la coherence des transactions de craft. Comparaison ingredients avant/apres.
ExploitLogger
Log toutes les detections avec UUID joueur, coordonnees, timestamp, type d'exploit et details.
UseBugConfig
Toutes les limites configurables. Rechargeable en live via commande admin.

11Monitoring & Alertes Admin

Systeme d'alertes

Chaque detection d'exploit declenche une alerte multi-canal selon sa severite :

  • Console — Toujours loggue dans la console serveur avec level WARNING ou ERROR.
  • Fichier de log — Ecrit dans logs/exploits.log avec timestamp ISO, UUID joueur, coordonnees, type, details.
  • Chat admin — Via EriChat, envoi a tous les joueurs avec permission eriniumfaction.admin.security.
// Alerte admin via EriChat public void alertAdmins(String exploitType, EntityPlayerMP offender, String details) { for (EntityPlayerMP admin : FMLCommonHandler.instance().getMinecraftServerInstance() .getPlayerList().getPlayers()) { if (admin.canUseCommand(4, "eriniumfaction.admin.security")) { EriChat.create() .text("[SECURITY] ").color(TextFormatting.RED).bold() .text(offender.getName()).color(TextFormatting.YELLOW) .text(" — " + exploitType).color(TextFormatting.WHITE) .text(" | " + details).color(TextFormatting.GRAY) .to(admin); } } }

Rate limiting des violations

Si un joueur declenche plus de N violations en M secondes, il est automatiquement kicte avec un message de log pour investigation manuelle. Les seuils sont configurables :

  • Kick automatique — 5 violations distinctes en 30 secondes.
  • Ban temporaire 1h — 3 kicks consecutifs pour exploit dans la meme session.
  • Escalade admin — Tout kick pour exploit envoie une alerte persistante aux admins connectes.

Dashboard de statistiques

Les donnees d'exploit sont stockees via EriAPI DataManager dans un POJO monde :

@EriData(modId = "eriniumfaction", scope = DataScope.WORLD) public class SecurityWorldData { public Map<String, Integer> exploitCounts = new HashMap<>(); // type -> count public Map<String, Integer> playerViolations = new HashMap<>(); // UUID -> count public List<String> recentExploits = new ArrayList<>(); // derniers 100 exploits }

12Considerations de Performance

✓
Objectif : L'ensemble du systeme anti-usebug ne doit pas ajouter plus de 1ms par tick sur le thread principal avec 1000 joueurs actifs.

Strategies d'optimisation

  • Evaluation paresseuse (lazy) — Le scan NBT approfondi d'un chunk n'est effectue qu'a son chargement initial ou lors d'une modification de TileEntity, pas a chaque tick.
  • Cache incremental des tailles NBT — La taille NBT d'un chunk est mise a jour incrementalement (+/- la difference) quand un TE est modifie, pas recalculee en entier.
  • NBTSizeValidator rapide — L'estimation de taille est faite par comptage d'octets approximatif (pas par serialisation complete), ce qui est 50x plus rapide.
  • Validation paquet en O(1) — Le hashage d'inventaire utilise un entier 64-bit XOR des hashes d'items, pas une serialisation complete.
  • Monitoring hopper asynchrone — Les statistiques de taux de transfert hoppers sont calculees en arriere-plan toutes les 20 ticks, pas en temps reel.
  • Alertes admin en batch — Les alertes multiples pour le meme joueur dans le meme tick sont regroupees en un seul message.

14Plan de Tests

Scenarios de test par exploit

[TEST-01] Chunk NBT Bomb : Placer 100 coffres pleins de livres ecrits 100 pages dans un chunk. Verifier que le monitor detecte le depassement et clean automatiquement sans crash serveur.
[TEST-02] Container Race Condition : Simuler des paquets CPacketClickWindow en double dans le meme tick. Verifier que seul le premier est traite et le second est refuse.
[TEST-03] Creative Packet en Survie : Envoyer manuellement un CPacketCreativeInventoryAction depuis un joueur en mode survie. Verifier kick + log.
[TEST-04] Livre overflow : Envoyer un CPacketEditBook avec 500 pages. Verifier que le livre est tronque a 100 pages et que le paquet est accepte (pas refuse).
[TEST-05] ItemStack NBT gonfle : Envoyer un CPacketClickWindow avec un ItemStack ayant 100 KB de NBT. Verifier refus et resync inventaire.
[TEST-06] Death Dupe : Tester la mort avec un item dans la main droite au meme tick. Verifier que l'item n'est pas duplique.
[TEST-07] TP pendant conteneur ouvert : Teleporter un joueur ayant un coffre ouvert. Verifier fermeture du conteneur et absence de dupe.
[TEST-08] Hopper rate limit : Creer une machine hopper transferant 100 items/sec. Verifier flag et desactivation temporaire du hopper.
[TEST-09] Stress test 1000 coffres : Placer 1000 coffres dans un chunk (via commande admin test). Verifier que la limite est enforced a 256 TE et que les placements supplementaires sont refuses.
[TEST-10] Faux positifs : Jouer normalement pendant 1 heure (craft intensif, coffres pleins, editeur de livres). Verifier zero violation detectee.
[TEST-11] Performance : Charger 100 chunks simultanement avec des TE. Verifier que le tick time n'augmente pas de plus de 1ms.
[TEST-12] Map flood : Creer 20 cartes en 1 minute. Verifier rate limit a 5/minute et refus des suivantes.

Criteres de validation finale

  • Aucun des 12 exploits ci-dessus ne peut etre execute avec succes.
  • Aucun faux positif sur 1 heure de jeu normal par 3 joueurs de test.
  • Impact performance < 1ms/tick mesure sur serveur de test avec 100 joueurs.
  • Tous les exploits detectes sont loggues dans logs/exploits.log.
  • Les alertes admin sont recues en moins de 2 ticks apres detection.

Note : Les tests d'exploit reels ne doivent etre effectues que sur un serveur de test isole avec une copie de la world de production. Ne jamais tester sur la production directement.
