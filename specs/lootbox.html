Systeme de Lootbox

Des blocs lootbox configurables places dans le monde par les joueurs, contenant un lootpool defini par config. A l'ouverture, un seul tier de rarete est tire au sort, puis N items de ce tier sont generes et propulses en l'air avec une animation spectaculaire.

Planifie Config-Driven 6 Tiers de Rarete Animation 3D Premium — /give only

01 Contexte & Objectifs

Vision du systeme

Les lootbox sont des blocs physiques que les joueurs placent dans le monde et ouvrent par clic droit. Chaque type de lootbox est defini par un dossier de configuration contenant le lootpool (liste d'items possibles avec quantites et rarete) et la texture du bloc. A l'ouverture, un seul tier de rarete est tire au sort selon les poids configures, puis N items de ce tier sont selectionnes aleatoirement et propulses en l'air depuis le bloc avec une animation spectaculaire.

Les lootbox ne sont pas craftables. Ce sont des items premium distribues uniquement par les administrateurs via /give. Elles servent de recompense pour des events, des achats boutique, ou des objectifs de progression.

Objectifs de design

  • Spectacle : L'ouverture d'une lootbox doit etre un moment memorable — animation, particules, sons, items qui volent. Les joueurs a proximite doivent voir l'ouverture.
  • Equite : Les probabilites par tier sont fixes et transparentes. Pas de manipulation des odds.
  • Extensibilite : Ajouter un nouveau type de lootbox = creer un dossier avec les fichiers config. Pas de code Java.
  • Securite : Impossible de dupliquer, recuperer ou manipuler une lootbox en cours d'ouverture.

Ce que le systeme ne fait pas

  • Pas de craft : Aucune recette ne produit de lootbox.
  • Pas d'achat en jeu : Les lootbox ne s'achetent pas dans un GUI avec une monnaie. Elles sont distribuees par les admins.
  • Pas de preview : Le joueur ne sait pas ce qu'il va obtenir avant l'ouverture (pas de "roulette" — c'est un tirage instantane avec animation de revelation).
  • Pas d'ouverture depuis l'inventaire : La lootbox DOIT etre placee en tant que bloc puis ouverte par clic droit. Pas d'ouverture directe depuis la main.

02 Architecture Config-Driven

Arborescence

Au demarrage du serveur, le systeme scanne config/Lootboxes/. Chaque sous-dossier est un type de lootbox.

config/Lootboxes/ ├── StarterBox/ <-- Type "StarterBox" │ ├── lootbox.info <-- Metadonnees du type │ ├── lootpool.json <-- Pool d'items avec rarete │ └── textures/ <-- TOUTES les textures du modele 3D │ ├── corner_cube.png │ ├── corner_joint.png │ ├── interior_cube.png │ ├── logo_animated.png <-- Peut etre personnalise par type │ ├── rarity_top.png │ ├── rocks.png │ ├── side_cable.png │ ├── side_core.png │ ├── side_middle_plate.png │ ├── side_plate.png │ ├── side_plate_core.png │ ├── top_pedestral.png │ ├── rarity_beam.png <-- Texture par defaut du beam │ ├── rarity_beam_common.png <-- Textures variables par rarete │ ├── rarity_beam_uncommon.png (appliquees dynamiquement pendant │ ├── rarity_beam_rare.png l'animation via le callback │ ├── rarity_beam_epic.png "calculrarity") │ ├── rarity_beam_legendary.png │ └── rarity_beam_mythic.png ├── PvPBox/ │ ├── lootbox.info │ ├── lootpool.json │ └── textures/ ├── LegendaryBox/ │ ├── lootbox.info │ ├── lootpool.json │ └── textures/ └── EventHalloween/ ├── lootbox.info ├── lootpool.json └── textures/
ⓘ
Modele 3D et animation hardcodes : Le modele 3D Blockbench (lootbox.json) et le fichier d'animation (lootbox.erianim.json) sont hardcodes dans le mod (dans assets/eriniumfaction/models/block/ et assets/eriniumfaction/animations/block/). Ils sont identiques pour TOUTES les gammes de lootbox. Seules les textures changent d'un type a l'autre — elles sont fournies par le serveur dans le dossier textures/ de la config de chaque type. Les textures rarity_beam_XX.png ne sont PAS referencees dans le modele 3D JSON — ce sont des textures variables appliquees dynamiquement pendant l'event d'ouverture via le callback calculrarity de l'animation.

Cycle de chargement

  • 1. Scan : Au FMLServerStartingEvent, scanner config/Lootboxes/.
  • 2. Validation : Chaque dossier doit contenir lootbox.info, lootpool.json et un sous-dossier textures/ avec les textures du modele 3D. Les dossiers invalides sont ignores avec un WARNING.
  • 3. Registration des blocs : Le modele 3D Blockbench et l'animation d'ouverture sont hardcodes dans le mod (models/block/lootbox.json + animations/block/lootbox.erianim.json). Ils sont identiques pour tous les types de lootbox. Seules les textures (fournies dans le dossier config de chaque type) changent.
  • 4. Indexation : Le LootboxRegistry mappe les IDs vers les metadonnees et lootpools parses.
  • 5. Sync client : Au login, envoyer la liste des types de lootbox disponibles et leurs textures aux clients.
⚠
Registration dynamique : Les blocs Forge doivent etre enregistres au RegistryEvent.Register<Block> (preInit), mais le scan config se fait au ServerStartingEvent (plus tard). Solution : enregistrer un bloc generique BlockLootbox qui utilise les metadonnees de tile entity pour determiner le type. Le registre de blocs ne contient qu'un seul ID : eriniumfaction:lootbox. Le type de lootbox est stocke dans le TileEntity associe au bloc place.

Bloc generique vs types multiples

Plutot que d'enregistrer N blocs (un par type de lootbox), on enregistre un seul bloc eriniumfaction:lootbox avec un TileEntity qui stocke le type. Avantages :

  • Registration statique : Le bloc est enregistre au preInit, avant que la config ne soit lue.
  • Pas de limite de types : Ajouter un type de lootbox n'ajoute pas un nouveau bloc au registre Forge.
  • Texture dynamique : Le renderer du bloc lit le type depuis le TileEntity et applique la texture correspondante.

L'item correspondant est aussi un seul ItemBlock. Le type est stocke dans les NBT de l'ItemStack (tag LootboxType). Quand le joueur place le bloc, le type est copie des NBT vers le TileEntity.

ⓘ
Mecanique de tirage : A l'ouverture, un seul tier est selectionne selon les poids weight_* (tirage pondere). Ensuite, qty_* items sont tires au hasard dans le pool de ce tier. Exemple : avec weight_rare=15.0 et qty_rare=3, si le tier Rare est selectionne, le joueur recoit 3 items tires dans le pool rare. Les tiers sans entries dans le lootpool sont automatiquement exclus du tirage de rarete (leur poids est ignore).

textures/ (dossier)

Dossier contenant toutes les textures du modele 3D Blockbench pour ce type de lootbox. Les 13 textures de base (corner_cube, corner_joint, interior_cube, logo_animated, rarity_top, rocks, side_cable, side_core, side_middle_plate, side_plate, side_plate_core, top_pedestral, rarity_beam) remplacent les textures par defaut du modele. Les textures rarity_beam_*.png sont des variantes appliquees dynamiquement pendant l'animation d'ouverture selon le tier de rarete selectionne. Format PNG, dimensions libres (recommande : 64x64 ou 128x128).

lootpool.json

Le fichier JSON definissant le pool d'items disponibles pour ce type de lootbox. Les entries sont groupees par tier de rarete.

// config/Lootboxes/StarterBox/lootpool.json { "entries": { "common": [ { "item": "minecraft:iron_sword", "min": 1, "max": 1, "duplicate": true, "nbt": null }, { "item": "minecraft:iron_ingot", "min": 5, "max": 15, "duplicate": true, "nbt": null } ], "uncommon": [ { "item": "minecraft:diamond", "min": 1, "max": 5, "duplicate": true, "nbt": null } ], "rare": [ { "item": "minecraft:diamond_sword", "min": 1, "max": 1, "duplicate": false, "nbt": "{ench:[{id:16,lvl:5}]}" } ], "epic": [ { "item": "eriniumfaction:erinium_fragment", "min": 1, "max": 3, "duplicate": true, "nbt": null } ], "legendary": [ { "item": "eriniumfaction:valterite_chestplate", "min": 1, "max": 1, "duplicate": false, "nbt": "{ench:[{id:0,lvl:6}]}" } ], "mythic": [ { "item": "eriniumfaction:erinium_sword", "min": 1, "max": 1, "duplicate": false, "nbt": "{ench:[{id:16,lvl:7},{id:19,lvl:3}]}" } ] } }
Champ Type Description
entries Object Map des pools d'items groupes par tier de rarete. Cles : common, uncommon, rare, epic, legendary, mythic. Chaque cle contient un Array d'entries.
entries.<tier>[].item String Registry name de l'item (ex: minecraft:diamond_sword). Peut inclure un metadata avec :meta (ex: minecraft:dye:4 pour le lapis).
entries.<tier>[].min int (>=1) Quantite minimum de l'item donne
entries.<tier>[].max int (>=min) Quantite maximum de l'item donne. La quantite reelle est aleatoire entre min et max (inclusif).
entries.<tier>[].duplicate boolean true (defaut) : cette entry peut etre tiree plusieurs fois dans la meme ouverture. false : une fois tiree, l'entry est retiree du pool pour cette ouverture (pas de doublon).
entries.<tier>[].nbt String / null Optionnel — tag NBT a appliquer a l'item (format SNBT Minecraft). Permet d'ajouter des enchantements, des noms custom, des attributs, etc.
ⓘ
Selection dans un tier : Quand un tier est selectionne, la lootbox tire qty_* items aleatoirement parmi les entries de ce tier. Le comportement de remise depend du champ duplicate de chaque entry : si duplicate=true (defaut), l'entry peut etre tiree plusieurs fois dans la meme ouverture. Si duplicate=false, l'entry est retiree du pool apres avoir ete tiree (pas de doublon). Si le pool se vide avant d'atteindre le nombre d'items demande, le tirage s'arrete.

Rendu du bloc (client)

Le bloc utilise un TileEntitySpecialRenderer (TESR) pour le rendu. Le modele 3D est un coffre stylise (universel pour tous les types). La texture est chargee dynamiquement depuis le LootboxRegistry en fonction du type stocke dans le TileEntity.

  • Etat normal : Le coffre est ferme, avec un subtil effet de particules lumineux autour (couleur basee sur le tier le plus rare ayant des entries dans le lootpool).
  • Etat ouverture : Animation du couvercle qui s'ouvre, lumiere intense sortant de l'interieur, particules de couleur du tier selectionne.
  • Etat ouvert : Le bloc disparait avec un fade-out + particules de dispersion.

06 Mecanique d'Ouverture

Sequence d'ouverture

L'ouverture se declenche quand un joueur fait un clic droit sur un bloc lootbox place. La sequence complete est :

  • 1. Clic droit (client) : Le client envoie un PacketLootboxOpen au serveur avec la position du bloc.
  • 2. Validation (serveur) : Le serveur verifie :
    • Le bloc a la position est bien un BlockLootbox avec un TileEntity valide.
    • Le TileEntity n'est pas deja en cours d'ouverture (opening == false).
    • Le joueur est a moins de 5 blocs du bloc (anti-cheat distance).
  • 3. Determination du loot (serveur) : Tirer un seul tier de rarete par tirage pondere (weight_*), puis tirer qty_* items aleatoirement dans le pool de ce tier. Generer les ItemStacks avec quantites aleatoires (entre min et max), NBT, et gestion du champ duplicate.
  • 4. Marquage ouverture (serveur) : Mettre opening = true sur le TileEntity. Rendre le bloc incassable (hardness = -1). Envoyer un PacketLootboxOpening a tous les joueurs dans un rayon de 64 blocs (position, type, tier selectionne, items obtenus).
  • 5. Animation (client, 3 secondes) : Tous les clients proches affichent l'animation d'ouverture (couvercle, lumiere, son).
  • 6. Ejection des items (serveur, apres 3 sec) : Apres l'animation, le serveur spawn les EntityItem au-dessus du bloc avec un vecteur de velocite vers le haut (propulsion). Chaque item est ejecte avec un leger decalage temporel (0.1 sec entre chaque) et un angle aleatoire.
  • 7. Suppression du bloc (serveur) : Le bloc est supprime du monde (world.setBlockToAir()).

Formule de velocite des items ejectes

// Velocite de base pour chaque item ejecte double baseY = 0.6; // Velocite verticale (vers le haut) double spreadXZ = 0.15; // Dispersion horizontale aleatoire // Pour chaque item : double vx = (random.nextDouble() - 0.5) * spreadXZ; double vy = baseY + random.nextDouble() * 0.2; // 0.6 a 0.8 double vz = (random.nextDouble() - 0.5) * spreadXZ; EntityItem entityItem = new EntityItem(world, pos.getX() + 0.5, pos.getY() + 1.0, pos.getZ() + 0.5, itemStack); entityItem.motionX = vx; entityItem.motionY = vy; entityItem.motionZ = vz; entityItem.setPickupDelay(40); // 2 secondes avant ramassage (temps de voir les items) world.spawnEntity(entityItem);
⚠
Anti-vol : Les items ejectes ont un pickupDelay de 40 ticks (2 secondes). Pendant ce delai, AUCUN joueur ne peut les ramasser. Apres le delai, les items sont ramassables par n'importe qui (premier arrive, premier servi). Il n'y a PAS de protection "owner only" — c'est un choix delibere pour encourager les joueurs a ouvrir leurs lootbox dans un endroit sur.

Recuperation du bloc avant ouverture

Un joueur qui a place une lootbox peut la recuperer (sans l'ouvrir) en faisant Shift + clic gauche sur le bloc. Le bloc est casse et l'item lootbox est rendu avec les memes NBT. Seul le joueur qui a pose le bloc (placedBy) peut le recuperer. Les autres joueurs ne peuvent ni casser ni recuperer le bloc.

07 Animation & Rendu — Framework EriAPI

Le rendu et l'animation de la lootbox utilisent le module Animation Blocks 3D d'EriAPI. Le bloc est rendu via un AnimatedBlockTESR avec un modele Blockbench 3D multi-textures. L'animation d'ouverture est definie dans un fichier .erianim.json et declenchee cote serveur.

Modele 3D

  • Fichier : assets/eriniumfaction/models/block/lootbox.json (export Blockbench, hardcode dans le mod)
  • 8 groupes racine : front, back, left, right, interiorcube, corner, Top, topplate
  • Sous-groupes : corner > joint > vertical/horizontal, etc.
  • 13 textures referencees dans le modele
  • Identique pour TOUS les types de lootbox — seules les textures changent (fournies dans le dossier config de chaque type)

Fichier d'animation

  • Fichier : assets/eriniumfaction/animations/block/lootbox.erianim.json (hardcode dans le mod)
  • Animation "Open" — duree 100 ticks (5 secondes)

Tracks utilises :

  • topplate : rotate [0, 360, 0] sur 40 ticks + translation [0, -10.426, 0] (le couvercle tourne et descend)
  • rocks : translation vers le bas avec easing elastic + spin Y a 1200 deg/s qui decelere a 0 (les roches descendent et tournent)
  • raritybeam : translation vers le haut avec easing elastic (la colonne de rarete monte)

Event a tick 0 : callback calculrarity — determine la rarete et applique la bonne texture sur le beam.

Callback calculrarity — Logique Java

Le callback est enregistre sur le TileEntity et execute au debut de l'animation. Il lit le tier de rarete selectionne depuis le LootResult stocke dans le TileEntity et applique la texture correspondante sur le beam :

// Enregistrement du callback sur le TileEntity te.setAnimationCallback("calculrarity", (tile) -> { String typeId = tile.getLootboxType(); LootboxType type = LootboxRegistry.getInstance().getType(typeId); Random rand = new Random(); // Generer le loot (tire une rarete + les items de ce tier) LootResult result = type.generateLoot(rand); tile.setLootResult(result); // Stocker pour la phase de drop // Appliquer la texture du beam selon la rarete selectionnee String beamTexture = "rarity_beam_" + result.getSelectedTier().name().toLowerCase(); tile.setTexture("rarity_beam", beamTexture); });

Gestion des tiers vides

Certaines lootbox peuvent ne pas avoir d'entries dans tous les tiers de rarete. Lors du tirage de rarete (rollRarity), les tiers sans entries dans le pool sont automatiquement exclus : leur poids est ignore et le total est recalcule uniquement sur les tiers qui ont des entries. Si aucun tier n'a d'entries (cas extreme), le tier par defaut COMMON est retourne en fallback.

Textures dynamiques

Le systeme de texture dynamique d'EriAPI (tile.setTexture()) est utilise a deux moments :

  • Au chargement du TileEntity : les textures du dossier config du type de lootbox sont appliquees sur le modele 3D — elles remplacent les textures par defaut du modele par celles fournies par le serveur dans config/Lootboxes/<type>/textures/.
  • Pendant l'animation : via le callback calculrarity, la texture rarity_beam est swappee vers la variante du tier selectionne (ex: rarity_beam_legendary.png si le tier Legendary a ete tire).

Registration du bloc

Le bloc est enregistre avec EriBlock en mode TESR-only (pas de rendu par le BlockModelRenderer — tout est fait par le TileEntitySpecialRenderer anime) :

// Un seul bloc pour tous les types de lootbox EriBlock.create(MODID, "lootbox") .material(Material.IRON) .hardness(-1) // Incassable sauf par le serveur .opaque(false) .fullCube(false) .tesrOnly("eriniumfaction:block/lootbox") .tileEntity(TileLootbox.class) .onRightClick(ctx -> LootboxManager.onOpen(ctx.world, ctx.pos, ctx.player)) .register();
ⓘ
Performance rendu : Le modele 3D est rendu par le AnimatedBlockTESR d'EriAPI qui gere le culling automatique et le batching des draw calls. L'animation ne tick que pendant les 5 secondes d'ouverture — le TileEntity n'a pas de update() permanent. Les textures config sont chargees une seule fois au login et cachees cote client.

08 Structures de Donnees

LootboxRegistry (serveur, singleton)

public class LootboxRegistry { private static LootboxRegistry INSTANCE; // typeId -> LootboxType private final Map<String, LootboxType> types = new LinkedHashMap<>(); public LootboxType getType(String id) { ... } public Collection<LootboxType> getAllTypes() { ... } public void reload() { ... } }

LootboxType

public class LootboxType { private final String id; // "starterbox" private final String nameFr; // "Caisse de Depart" private final String nameEn; // "Starter Box" private final String texturePath; // Chemin absolu vers texture.png private final Map<LootboxTier, Float> rarityWeights; // Poids par tier (weight_*) private final Map<LootboxTier, Integer> qtyPerTier; // Nb items par tier selectionne (qty_*) private final Map<LootboxTier, List<LootboxEntry>> entries; // Entries par tier /** * Tire une rarete au hasard selon les poids configures. * Les tiers sans entries dans le pool sont exclus. */ public LootboxTier rollRarity(Random rand) { // Calculer le total des poids (en excluant les tiers vides) float totalWeight = 0; for (Map.Entry<LootboxTier, Float> e : rarityWeights.entrySet()) { List<LootboxEntry> pool = entries.get(e.getKey()); if (pool != null && !pool.isEmpty()) { totalWeight += e.getValue(); } } // Tirage pondere float roll = rand.nextFloat() * totalWeight; float cumulative = 0; for (Map.Entry<LootboxTier, Float> e : rarityWeights.entrySet()) { List<LootboxEntry> pool = entries.get(e.getKey()); if (pool == null || pool.isEmpty()) continue; cumulative += e.getValue(); if (roll <= cumulative) return e.getKey(); } return LootboxTier.COMMON; // fallback } /** * Genere le loot : tire une rarete, puis tire N items de cette rarete. * Respecte le champ duplicate de chaque entry. */ public LootResult generateLoot(Random rand) { LootboxTier selectedTier = rollRarity(rand); int qty = qtyPerTier.getOrDefault(selectedTier, 1); List<LootboxEntry> pool = new ArrayList<>(entries.get(selectedTier)); List<ItemStack> items = new ArrayList<>(); for (int i = 0; i < qty && !pool.isEmpty(); i++) { int idx = rand.nextInt(pool.size()); LootboxEntry entry = pool.get(idx); items.add(entry.createItemStack(rand)); // Si duplicate=false, retirer l'entry du pool pour cette ouverture if (!entry.isDuplicate()) { pool.remove(idx); } } return new LootResult(selectedTier, items); } }

LootboxEntry

public class LootboxEntry { private final String itemRegistryName; // "minecraft:diamond_sword" private final int metadata; // 0 par defaut, ou valeur apres le 2e ':' private final int min; // Quantite min private final int max; // Quantite max private final boolean duplicate; // true = peut etre tire plusieurs fois par ouverture private final String nbtString; // NBT SNBT ou null public boolean isDuplicate() { return duplicate; } public ItemStack createItemStack(Random rand) { Item item = Item.getByNameOrId(itemRegistryName); if (item == null) return ItemStack.EMPTY; int qty = min + (max > min ? rand.nextInt(max - min + 1) : 0); ItemStack stack = new ItemStack(item, qty, metadata); if (nbtString != null) { NBTTagCompound nbt = JsonToNBT.getTagFromJson(nbtString); stack.setTagCompound(nbt); } return stack; } }

LootResult

11 Cas Limites & Securite

Duplication de lootbox

  • Clic droit pendant l'ouverture : Rejete — le flag opening est check. Aucun second tirage possible.
  • Casser le bloc pendant l'ouverture : Impossible — la hardness est mise a -1 (incassable).
  • Piston pendant l'ouverture : Le bloc est immovable (Material.IRON + override getPushReaction() retourne BLOCK).
  • Explosion pendant l'ouverture : Resistance a 1200 — aucune explosion vanilla ne peut le detruire.
  • Hopper : Le bloc n'a pas d'inventaire, les hoppers ne peuvent rien extraire.

Lootbox type inexistant

Si un joueur place un item lootbox dont le type n'existe pas dans le registre (lootbox donnee avant un reload qui a supprime le type) :

  • Le bloc est place mais le clic droit affiche un message d'erreur en chat : "Ce type de lootbox n'est plus disponible."
  • Le joueur peut recuperer le bloc (Shift+clic gauche) et le conserver — au cas ou le type serait re-ajoute plus tard.

Lootpool vide pour un tier

Si le lootpool n'a aucune entry pour un tier configure dans weight_*, ce tier est automatiquement exclu du tirage de rarete (son poids est ignore). Le total des poids est recalcule uniquement sur les tiers qui ont des entries. Un WARNING est logue au chargement : "Lootbox [type] a weight_rare=15.0 mais aucune entry 'rare' dans le lootpool — tier exclu du tirage."

Item du lootpool inexistant

Si une entry reference un item qui n'existe pas dans le registre Forge (Item.getByNameOrId() retourne null), cette entry est ignoree lors du tirage. Un WARNING est logue au chargement. Si toutes les entries d'un tier sont invalides, le tier est traite comme un pool vide.

NBT invalide

Si le champ nbt d'une entry contient un SNBT invalide (JsonToNBT.getTagFromJson() leve une exception), l'item est genere SANS NBT. Un WARNING est logue.

Deconnexion pendant l'ouverture

Si le joueur deconnecte pendant les 3 secondes d'animation :

  • Le serveur continue la sequence normalement — les items sont spawnes au-dessus du bloc.
  • Les items tombent au sol et peuvent etre ramasses par d'autres joueurs (le pickupDelay expire normalement).
  • Le bloc est supprime.
  • Le joueur ne recoit PAS les items dans son inventaire a la reconnexion — ils sont au sol.

Serveur restart pendant l'ouverture

Si le serveur redémarre pendant une ouverture :

  • Le TileEntity est sauvegarde avec opening=true.
  • Au redemarrage, le TileEntity detecte opening=true au chargement et finalise l'ouverture : genere le loot, spawn les items, supprime le bloc.

Performance

  • Scan async : Le scan du dossier config se fait en async pour ne pas bloquer le thread serveur.
  • Pas de tick constant : Le TileEntityLootbox n'a PAS de update() tick permanent. Il ne tick que pendant l'ouverture (3 secondes) et s'arrete apres.
  • Packet broadcast cible : Les packets d'animation ne sont envoyes qu'aux joueurs dans un rayon de 64 blocs.
